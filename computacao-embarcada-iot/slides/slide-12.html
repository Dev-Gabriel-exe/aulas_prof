TF vs TFLite vs TFLite Micro — três runtimes, três mundos

TFLite Micro (TFLM) não tem alocação dinâmica de memória, não usa STL C++ completo, não depende de sistema operacional e não faz chamadas ao sistema. Ele roda em qualquer microcontrolador que tenha um compilador C++ — incluindo o ESP32, que tem hardware relativamente generoso para um MCU.
Mapa de memória — o que cabe onde no ESP32

O ESP32 tem 4MB de flash (onde fica o firmware e o modelo) e 520KB de RAM (onde ficam as variáveis, a tensor arena e os buffers de inferência). O modelo quantizado fica na flash como um array de bytes em C. Durante a inferência, o runtime usa uma região de RAM chamada tensor arena para alocar tensores intermediários.

Tensor Arena: bloco de RAM alocado estaticamente que o TFLM usa como "rascunho" durante a inferência — tensores de entrada, saída e intermediários. Você define o tamanho. Muito pequeno → erro de alocação. Muito grande → menos RAM para o resto do firmware.
Wi-Fi e TFLite Micro não convivem facilmente. O driver Wi-Fi do ESP32 usa ~100KB de RAM. Se você ativar Wi-Fi + MQTT + TFLite ao mesmo tempo, pode ficar sem RAM para a tensor arena. Estratégia: fazer a inferência, depois publicar o resultado via MQTT — nunca os dois simultaneamente no mesmo loop.
O modelo no ESP32 — FlatBuffer e o arquivo .h

Um modelo TFLite é serializado no formato FlatBuffer — uma representação binária compacta e zero-copy (pode ser lida diretamente da flash sem deserialização). No Arduino/ESP32, o modelo é convertido para um array de bytes em C e salvo como um arquivo .h incluído no sketch.

Python — converter .tflite para array C (.h)
# Converte um arquivo .tflite para um array de bytes C
# Execute no PC após treinar e exportar o modelo

def tflite_para_c(arquivo_tflite, nome_variavel="modelo_clima"):
    with open(arquivo_tflite, "rb") as f:
        dados = f.read()

    linhas = [f"// Gerado automaticamente — não editar"]
    linhas.append(f"#pragma once")
    linhas.append(f"#include <stdint.h>")
    linhas.append("")
    linhas.append(f"const unsigned int {nome_variavel}_len = {len(dados)};")
    linhas.append(f"alignas(8) const uint8_t {nome_variavel}[] = {{")

    hex_vals = [f"0x{b:02x}" for b in dados]
    for i in range(0, len(hex_vals), 12):
        linha = "  " + ", ".join(hex_vals[i:i+12]) + ","
        linhas.append(linha)

    linhas.append("};")

    saida = arquivo_tflite.replace(".tflite", ".h")
    with open(saida, "w") as f:
        f.write("\n".join(linhas))
    print(f"Gerado: {saida} ({len(dados)} bytes)")

tflite_para_c("modelo_clima.tflite")
alignas(8) é obrigatório no array do modelo. O FlatBuffer exige alinhamento de 8 bytes para leitura correta. Sem isso, o ESP32 gera erros de alinhamento (fault) na inicialização do modelo, difíceis de diagnosticar.

Python — treinar e quantizar modelo de clima (Keras)
import numpy as np
import tensorflow as tf

# Dataset sintético — [temperatura, umidade] → classe
# 0=Frio (<18°C)  1=Confortável (18–28°C, <70%)  2=Quente/Úmido
X = np.array([
  [12.0, 55.0], [15.5, 60.0], [10.0, 45.0],   # Frio
  [22.0, 50.0], [25.0, 55.0], [20.0, 48.0],   # Confortável
  [32.0, 80.0], [35.0, 85.0], [28.0, 75.0],   # Quente/Úmido
], dtype=np.float32)
y = np.array([0,0,0, 1,1,1, 2,2,2])

# Normalização — importante! Aplicar os mesmos valores no ESP32
X_mean = X.mean(axis=0); X_std = X.std(axis=0)
X_norm = (X - X_mean) / X_std
print(f"Média: {X_mean}  Desvio: {X_std}")  # guardar esses valores!

# Modelo pequeno — 2 entradas, 3 saídas
model = tf.keras.Sequential([
    tf.keras.layers.Dense(8, activation='relu', input_shape=(2,)),
    tf.keras.layers.Dense(3, activation='softmax')
])
model.compile(optimizer='adam', loss='sparse_categorical_crossentropy', metrics=['accuracy'])
model.fit(X_norm, y, epochs=200, verbose=0)
print(f"Acurácia: {model.evaluate(X_norm, y, verbose=0)[1]:.2%}")

# Converter para TFLite com quantização int8
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]

def gen_dataset():
    for x in X_norm: yield [x.reshape(1, 2).astype(np.float32)]

converter.representative_dataset = gen_dataset
converter.target_spec.supported_ops = [tf.lite.OpsSet.TFLITE_BUILTINS_INT8]
converter.inference_input_type  = tf.int8
converter.inference_output_type = tf.int8

tflite_model = converter.convert()
with open("modelo_clima.tflite", "wb") as f: f.write(tflite_model)
print(f"Modelo salvo: {len(tflite_model)} bytes")
Inferência no ESP32 — as 6 etapas do runtime TFLM

O ciclo de inferência do TFLite Micro no ESP32 segue sempre a mesma sequência de 6 etapas. Entender cada uma delas é o que permite depurar erros e otimizar o sistema.

Ciclo de inferência — setup() e loop()
1
Carregar o modelo (setup)
GetModel(modelo_clima) — lê o FlatBuffer do array em flash. Operação única no setup(). O modelo fica em flash, não ocupa RAM.
2
Criar o Interpreter (setup)
MicroInterpreter(model, resolver, tensor_arena, arena_size) — instancia o runtime com os ops registrados e a tensor arena. Operação única.
3
AllocateTensors (setup)
interpreter.AllocateTensors() — aloca todos os tensores dentro da arena. Se falhar, a arena é pequena demais. Retorna kTfLiteOk se ok.
4
Preencher o tensor de entrada (loop)
interpreter.input(0)->data.int8[i] = valor_quantizado — copia os dados do sensor para o buffer de input. Atenção à quantização: converter float para int8 usando scale e zero_point do tensor.
5
Executar a inferência (loop)
interpreter.Invoke() — executa todas as operações da rede neural. No modelo pequeno desta aula, leva menos de 1ms no ESP32 a 240MHz.
6
Ler o tensor de saída (loop)
interpreter.output(0)->data.int8[i] — lê as probabilidades de cada classe (em int8). Converter de volta para float: (val - zero_point) * scale. A classe predita é o índice de maior valor.

73 — apenas multiplicar por 100
B
0 — o valor já está normalizado, basta truncar para zero
C

Arduino C — TFLite Micro + DHT22 classificador de clima
#include <DHT.h>
#include "TensorFlowLite_ESP32.h"
#include "tensorflow/lite/micro/all_ops_resolver.h"
#include "tensorflow/lite/micro/micro_interpreter.h"
#include "tensorflow/lite/schema/schema_generated.h"
#include "modelo_clima.h"   // array do modelo

#define PINO_DHT    4
#define ARENA_SIZE  4096   // ajustar se AllocateTensors() falhar

// Parâmetros de normalização — mesmos do treino Python
const float MEAN_T = 23.28f, STD_T = 8.76f;
const float MEAN_U = 64.78f, STD_U = 14.23f;
const char* CLASSES[] = {"Frio", "Confortavel", "Quente/Umido"};

DHT dht(PINO_DHT, DHT22);

// Objetos TFLM — declarados globalmente (sem heap)
const tflite::Model*         model;
tflite::MicroInterpreter*    interpreter;
TfLiteTensor*               input;
TfLiteTensor*               output;
uint8_t tensor_arena[ARENA_SIZE];

int8_t quantizar(float val, float scale, int zero_point) {
  int q = (int)roundf(val / scale) + zero_point;
  return (int8_t)constrain(q, -128, 127);
}

void setup() {
  Serial.begin(115200);
  dht.begin();

  // 1. Carregar modelo
  model = tflite::GetModel(modelo_clima);
  if (model->version() != TFLITE_SCHEMA_VERSION) {
    Serial.println("Versão do modelo incompatível!"); while(true);
  }

  // 2. Resolver de ops — AllOpsResolver inclui tudo (mais seguro para lab)
  static tflite::AllOpsResolver resolver;

  // 3. Criar interpreter
  static tflite::MicroInterpreter static_interp(
    model, resolver, tensor_arena, ARENA_SIZE);
  interpreter = &static_interp;

  // 4. Alocar tensores
  if (interpreter->AllocateTensors() != kTfLiteOk) {
    Serial.println("AllocateTensors() falhou! Aumente ARENA_SIZE.");
    while(true);
  }

  input  = interpreter->input(0);
  output = interpreter->output(0);

  Serial.println("Modelo carregado!");
  Serial.printf("Input shape: [%d, %d]  tipo: %d\n",
    input->dims->data[0], input->dims->data[1], input->type);
  Serial.printf("Arena usada: %u / %d bytes\n",
    interpreter->arena_used_bytes(), ARENA_SIZE);
  delay(2000);
}

void loop() {
  float t = dht.readTemperature();
  float u = dht.readHumidity();
  if (isnan(t) || isnan(u)) { delay(5000); return; }

  // Normalizar com os parâmetros do treino
  float t_norm = (t - MEAN_T) / STD_T;
  float u_norm = (u - MEAN_U) / STD_U;

  // Preencher tensor de entrada (int8 quantizado)
  float in_scale = input->params.scale;
  int   in_zp    = input->params.zero_point;
  input->data.int8[0] = quantizar(t_norm, in_scale, in_zp);
  input->data.int8[1] = quantizar(u_norm, in_scale, in_zp);

  // Inferência
  unsigned long t0 = micros();
  interpreter->Invoke();
  unsigned long dt = micros() - t0;

  // Ler saída — converter int8 → float
  float out_scale = output->params.scale;
  int   out_zp    = output->params.zero_point;
  float probs[3];
  int   melhor = 0;
  for (int i = 0; i < 3; i++) {
    probs[i] = (output->data.int8[i] - out_zp) * out_scale;
    if (probs[i] > probs[melhor]) melhor = i;
  }

  Serial.printf("\nT=%.1f°C U=%.1f%%  → [%s]  (%luµs)\n",
    t, u, CLASSES[melhor], dt);
  Serial.printf("  Frio=%.2f  Conf=%.2f  Qte=%.2f\n",
    probs[0], probs[1], probs[2]);

  delay(5000);
}

// No setup(): pinMode(LED_BUILTIN, OUTPUT);

// Função de exibição — chamar após Invoke()
void imprimirResultado(float* probs, int melhor) {
  Serial.println("");
  for (int i = 0; i < 3; i++) {
    int barras = (int)(probs[i] * 10);
    Serial.printf("%-14s [", CLASSES[i]);
    for (int j = 0; j < 10; j++)
      Serial.print(j < barras ? '=' : ' ');
    Serial.printf("] %3d%% %s\n",
      (int)(probs[i] * 100), (i == melhor) ? "◀" : "");
  }
  // LED acende se classe 2 (Quente/Úmido)
  digitalWrite(LED_BUILTIN, melhor == 2 ? HIGH : LOW);
}
Exercício 03 — Medir latência por tipo de operação

Meça separadamente com micros() o tempo de: (a) leitura do DHT22, (b) normalização + quantização do input, (c) Invoke(), (d) desquantização e decisão do output. Imprima cada tempo e calcule o percentual de tempo que cada etapa representa no ciclo total. Qual etapa domina? Isso foi surpresa para você? Justifique.

Análise de Performance
Resultado esperado e análise
  • Leitura DHT22: ~250.000µs (250ms) — dominante total. O DHT22 tem protocolo de 1 fio lento por design.
  • Normalização + quantização: ~5–10µs — negligível.
  • Invoke() — inferência: ~800–1200µs — muito rápido para uma rede com 2 camadas densas pequenas.
  • Desquantização + decisão: ~3–5µs — negligível.
  • Conclusão: o gargalo não é o modelo de ML — é o sensor. Em sensores mais rápidos (acelerômetro SPI a 1kHz), a inferência pode ser 10–100× mais lenta que a leitura, justificando otimizações como MicroMutableOpResolver em vez de AllOpsResolver.
