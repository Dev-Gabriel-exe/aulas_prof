ThingSpeak e Adafruit IO — plataformas prontas para IoT
Quando usar plataforma pronta vs construir do zero

Nas semanas anteriores você construiu tudo do zero: MQTT local, servidor Flask, AWS IoT Core com Rules Engine. Isso dá controle total mas exige tempo de implementação. Para protótipos, projetos acadêmicos e MVPs, plataformas IoT prontas entregam visualização, armazenamento e alertas em minutos.

A escolha entre construir ou usar plataforma pronta não é binária — depende do estágio do projeto, do volume de dados e dos requisitos de privacidade. Conhecer as opções disponíveis é tão importante quanto saber implementar do zero.

ThingSpeak — publicando dados via HTTP com o ESP32

O ThingSpeak organiza dados em Channels (canais), cada um com até 8 fields (campos). Para enviar dados, faça uma requisição HTTP GET ou POST para a API com sua Write API Key. O ThingSpeak cria automaticamente gráficos para cada field.

Arduino C — ESP32 + DHT22 → ThingSpeak via HTTP
#include <WiFi.h>
#include <HTTPClient.h>
#include <DHT.h>

const char* SSID        = "NomeDaRede";
const char* SENHA       = "SenhaDaRede";
const char* API_KEY     = "SUA_WRITE_API_KEY";
const char* TS_URL      = "http://api.thingspeak.com/update";
#define PINO_DHT 4
#define INTERVALO 20000UL   // ThingSpeak free: mínimo 15s

DHT dht(PINO_DHT, DHT22);
unsigned long ultimoEnvio = 0;

void setup() {
  Serial.begin(115200);
  dht.begin();
  WiFi.begin(SSID, SENHA);
  Serial.print("Wi-Fi");
  while(WiFi.status()!=WL_CONNECTED){delay(400);Serial.print(".");}
  Serial.println(" OK");
  delay(2000);
}

void loop() {
  if (millis() - ultimoEnvio < INTERVALO) return;
  ultimoEnvio = millis();

  float t=dht.readTemperature(), u=dht.readHumidity();
  if(isnan(t)||isnan(u)) return;

  HTTPClient http;
  // GET com parâmetros na URL — cada fieldN é um campo do canal
  String url = String(TS_URL)
    + "?api_key=" + API_KEY
    + "&field1="  + String(t, 1)
    + "&field2="  + String(u, 1);

  http.begin(url);
  int code = http.GET();
  http.end();

  Serial.printf("ThingSpeak: T=%.1f U=%.1f → HTTP %d\n", t, u, code);
  // Resposta 200 com o entry_id = dado salvo com sucesso
}
Adafruit IO — MQTT nativo com feeds e triggers

O Adafruit IO suporta MQTT diretamente — você usa o mesmo PubSubClient da Semana 9, apenas apontando para o broker da Adafruit com autenticação por usuário e chave. Os feeds são equivalentes a tópicos MQTT, e o dashboard é configurado visualmente no site.

Arduino C — ESP32 → Adafruit IO via MQTT
#include <WiFi.h>
#include <PubSubClient.h>
#include <DHT.h>

const char* AIO_SERVER   = "io.adafruit.com";
const int  AIO_PORT     = 1883;              // ou 8883 para TLS
const char* AIO_USERNAME = "seu_usuario";
const char* AIO_KEY      = "sua_aio_key";

// Feeds no formato: usuario/feeds/nome-do-feed
const char* FEED_TEMP = "seu_usuario/feeds/temperatura";
const char* FEED_UMID = "seu_usuario/feeds/umidade";

WiFiClient   wc;
PubSubClient mqtt(wc);
DHT          dht(4, DHT22);

void conectar() {
  mqtt.setServer(AIO_SERVER, AIO_PORT);
  while(!mqtt.connect("esp32-aio", AIO_USERNAME, AIO_KEY)){
    delay(1000);
  }
  Serial.println("Adafruit IO conectado!");
}

void loop() {
  if(!mqtt.connected()) conectar();
  mqtt.loop();

  static unsigned long t0 = 0;
  if(millis()-t0 < 10000) return; t0=millis();

  float t=dht.readTemperature(), u=dht.readHumidity();
  if(isnan(t)||isnan(u)) return;

  // Publicar cada variável no seu feed
  mqtt.publish(FEED_TEMP, String(t,1).c_str());
  mqtt.publish(FEED_UMID, String(u,1).c_str());
  Serial.printf("AIO: T=%.1f U=%.1f\n", t, u);
}

Node-RED — programação visual por fluxo
O que é Node-RED — fluxos, nodes e wires

O Node-RED é uma ferramenta de programação visual baseada em fluxos (flows), criada pela IBM e atualmente mantida pela OpenJS Foundation. Em vez de escrever código imperativo, você arrasta blocos chamados nodes e os conecta com wires — o dado flui de node em node sendo transformado a cada etapa.

Cada node tem uma função específica: receber mensagens MQTT, fazer parse de JSON, calcular algo, enviar para uma API, mostrar em um gauge. A composição desses nodes forma um flow — equivalente a um script completo, mas visual e editável em tempo de execução sem reiniciar o servidor.

Anatomia de um flow Node-RED — MQTT → Dashboard
inputmqtt in
▶

▶

▶

msg.payload é o objeto que passa pelos wires — cada node pode ler, modificar e repassar. Tudo em JavaScript simples nos function nodes.
Nodes fundamentais — os blocos de construção
Input
mqtt in
Assina um tópico MQTT e injeta cada mensagem como msg.payload
Output
mqtt out
Publica msg.payload em um tópico MQTT ao receber uma mensagem
Parser
JSON
Converte string JSON → objeto JS e vice-versa automaticamente
Logic
function
JavaScript livre — transforma, filtra ou gera mensagens. return msg para repassar
Debug
debug
Exibe msg.payload na barra lateral — essencial para depurar flows
Input
inject
Injeta mensagem manualmente ou em intervalos — ótimo para testes sem sensor
Network
http request
Faz GET/POST para qualquer URL — integra com REST APIs externas
Router
switch
Roteia mensagens para saídas diferentes com base em condições
JavaScript — function node: extrair temp e classificar
// msg.payload chega como objeto JS após o node JSON:
// { temp: 27.3, umid: 64.8, dispositivo: "esp32-lab" }

var t = msg.payload.temp;
var u = msg.payload.umid;

// Classificar conforto
var conforto;
if      (t < 18)           conforto = "🥶 Frio";
else if (t > 28 || u > 75) conforto = "🥵 Quente/Úmido";
else                        conforto = "😊 Confortável";

// Enviar 3 mensagens independentes para diferentes widgets
var msgTemp = { payload: t };
var msgUmid = { payload: u };
var msgText = { payload: conforto };

return [ msgTemp, msgUmid, msgText ];
// Saída 1 → gauge temperatura
// Saída 2 → gauge umidade
// Saída 3 → text widget

🙋 Em um flow Node-RED, um function node recebe msg.payload como string '{"temp":27.3}' mas você tenta acessar msg.payload.temp e obtém undefined. Qual é a solução mais simples?
A

LAB — Node-RED: MQTT → Dashboard + controle do ESP32
Objetivo: criar um flow Node-RED que (1) assina o tópico MQTT do ESP32, (2) faz parse do JSON, (3) exibe temperatura num gauge e gráfico histórico, e (4) tem um botão que publica um comando de volta ao ESP32 via MQTT.
Passo a passo
1
Instalar Node-RED e o dashboard
No terminal: npm install -g --unsafe-perm node-red. Inicie com node-red. Acesse http://localhost:1880. Instale o dashboard: ☰ → Manage Palette → Install → buscar node-red-dashboard → Install.
2
Confirmar que o ESP32 está publicando
Antes de criar o flow, confirme com mosquitto_sub -h localhost -t "lab/#" -v que as mensagens do ESP32 estão chegando no broker local. O Node-RED se conecta ao mesmo broker.
3
Importar o flow via JSON
No editor Node-RED: ☰ → Import → cole o JSON do flow abaixo → Import. Configure o broker MQTT (localhost:1883) clicando duas vezes no node mqtt in. Clique Deploy.
4
Ver o dashboard funcionando
Acesse http://localhost:1880/ui. Você verá o gauge de temperatura atualizando a cada leitura do ESP32. Clique no botão "LED ON/OFF" e confirme no Serial Monitor do ESP32 que o comando chegou.

Exercício 01 — Alerta visual no dashboard

Adicione ao flow um node ui_text que exibe o indicador de conforto térmico ("Frio", "Confortável", "Quente/Úmido") com cor diferente para cada estado. No function node, mude msg.style para aplicar cor: vermelho para quente, azul para frio, verde para confortável. Use a sintaxe de HTML inline que o ui_text suporta: msg.payload = "<span style='color:red'>Quente</span>".

Node-RED · Dashboard
JavaScript — function node com alerta colorido
var t = msg.payload.temp || 0;
var u = msg.payload.umid || 0;
var texto, cor;

if      (t < 18)            { texto="🥶 Frio";          cor="#60a5fa"; }
else if (t > 28 || u > 75)  { texto="🥵 Quente/Úmido";  cor="#f87171"; }
else                          { texto="😊 Confortável";    cor="#2dd4bf"; }

var msgStatus = {
    payload: "<span style='color:" + cor + ";font-weight:700;font-size:1.1em'>"
             + texto + "</span>"
};
return [ {payload: t}, {payload: u}, msgStatus ];
Exercício 02 — Publicar no ThingSpeak via Node-RED

Adicione ao flow existente um node http request que publica os dados do DHT22 no ThingSpeak sempre que uma nova leitura chega. Monte a URL de atualização dinamicamente no function node com sua API key e os campos field1 (temperatura) e field2 (umidade). O Node-RED centraliza todos os destinos dos dados: MQTT broker local, dashboard E ThingSpeak — a partir de um único input MQTT.

Integração · Múltiplos destinos
JavaScript — function node para ThingSpeak via http request
var t = msg.payload.temp;
var u = msg.payload.umid;
var apiKey = "SUA_WRITE_API_KEY";

msg.method  = "GET";
msg.url     = "http://api.thingspeak.com/update?api_key="
              + apiKey + "&field1=" + t + "&field2=" + u;
return msg;
// Conectar saída deste function → node http request → node debug
Respeite o rate limit: adicione um node delay entre o function e o http request configurado para "Rate Limit 1 msg per 20 seconds" — evita que o ThingSpeak rejeite requisições acima do limite do free tier.
Exercício 03 — Notificação por e-mail com nodemailer

Instale o node node-red-node-email (Palette Manager) e configure-o para enviar um e-mail de alerta quando a temperatura ultrapassar 32°C. O flow deve: receber a mensagem MQTT → fazer parse → verificar threshold em um switch node → se acima do limite, enviar e-mail com subject "Alerta Temperatura" e body contendo o valor atual. Use debounce (node delay ou variável de contexto) para não enviar mais que 1 e-mail por hora.

Integração · Alertas
Estrutura do flow
  • mqtt in → json → switch: configurar o switch para "msg.payload.temp > 32" na saída 1 e "otherwise" na saída 2.
  • Debounce via contexto: no function node, verificar context.get('ultimoAlerta') — se foi há menos de 3600 segundos, retornar null (sem mensagem).
  • node email: configurar servidor SMTP (Gmail com app password, ou Ethereal para testes). Definir msg.topic como subject e msg.payload como body.
  • Para Gmail: ativar "App Passwords" em Conta Google → Segurança (não use a senha principal).
Exercício 04 — Desafio: dashboard multi-dispositivo

Expanda o flow para suportar dois ESP32 distintos publicando em tópicos diferentes: lab/esp32-01/dados e lab/esp32-02/dados. O dashboard deve mostrar os dados de ambos com labels distintos, e um terceiro widget exibe a diferença de temperatura entre os dois (calculada em um function node com estado de contexto). Teste publicando manualmente com mosquitto_pub para simular o segundo dispositivo caso não tenha dois ESP32 disponíveis.

Desafio · Multi-dispositivo
JavaScript — function node com contexto de dois dispositivos
// Contexto de fluxo compartilhado entre mensagens de dispositivos distintos
var id  = msg.payload.dispositivo || msg.topic;
var tmp = msg.payload.temp;

flow.set('temp_' + id, tmp);

var t1 = flow.get('temp_esp32-01') || 0;
var t2 = flow.get('temp_esp32-02') || 0;
var diff = Math.abs(t1 - t2);

return [
  { payload: t1, topic: 'esp32-01' },   // → chart 1
  { payload: t2, topic: 'esp32-02' },   // → chart 2 (mesmo chart, linhas diferentes)
  { payload: diff }                        // → gauge diferença
];
