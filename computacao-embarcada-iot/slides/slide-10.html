Bluetooth BLE — comunicação de curto alcance e baixíssimo consumo
BT Classic vs BLE — não são a mesma coisa

Muita gente confunde Bluetooth Classic (BT) com Bluetooth Low Energy (BLE). São protocolos distintos — mesma frequência (2.4GHz), mas arquiteturas completamente diferentes. O BT Classic foi projetado para streaming de áudio e transferência de arquivos. O BLE (Bluetooth 4.0+) foi criado para sensores e dispositivos que precisam funcionar meses ou anos com uma pilha.

Bluetooth Classic

  • Streaming de áudio (fones, caixas de som)
  • Transferência de arquivos em alta velocidade
  • Consumo alto — não adequado para baterias pequenas
  • Conexão ponto a ponto (pareamento)

Bluetooth Low Energy (BLE)

  • Sensores de saúde, wearables, beacons
  • Consumo ~10× menor — meses/anos com CR2032
  • Advertising: transmite sem conexão (beacons)
  • Conexão com smartphone sem app nativo (Web BLE)
O ESP32 suporta ambos — BT Classic e BLE — no mesmo chip. Para projetos IoT com sensores e bateria, use sempre BLE. Para áudio sem fio, use BT Classic. Nunca tente usar os dois simultaneamente no ESP32 sem alocar memória extra; há limitações de coexistência.
GATT — como os dados são organizados no BLE

O BLE usa o protocolo GATT (Generic ATTribute Profile) para organizar dados. A hierarquia é: Perfil → Serviço → Característica → Descritor. Cada elemento tem um UUID (Identifier) — padrão de 16 bits para perfis oficiais Bluetooth SIG, ou 128 bits para custom.

Estrutura GATT — ESP32 como servidor de sensor ambiental
Serviço: Sensor Ambiental · UUID: 0x181A (Environmental Sensing)
0x2A6E Temperature READ NOTIFY
0x2A6F Humidity READ NOTIFY
Serviço: Device Info · UUID: 0x180A
0x2A24 Model Number READ
0xFFFF Custom: Comando (UUID 128-bit) WRITE
READ cliente lê sob demanda · NOTIFY servidor envia automaticamente quando muda · WRITE cliente escreve no servidor
BLE Scanner e BLE Server no ESP32

O ESP32 pode atuar como Peripheral (server — anuncia serviços e aceita conexões) ou como Central (client — faz scan e conecta a outros dispositivos). No LAB de hoje usamos o modo Scanner para listar dispositivos BLE próximos com seus RSSI e nomes.

Arduino C — BLE Scanner (lista dispositivos próximos)
#include <BLEDevice.h>
#include <BLEScan.h>
#include <BLEAdvertisedDevice.h>

#define DURACAO_SCAN  5    // segundos por varredura

BLEScan* pScan;

// Chamada para cada dispositivo encontrado durante o scan
class Callbacks : public BLEAdvertisedDeviceCallbacks {
  void onResult(BLEAdvertisedDevice dev) {
    Serial.printf("  [%4d dBm]  %-30s  %s\n",
      dev.getRSSI(),
      dev.haveName() ? dev.getName().c_str() : "(sem nome)",
      dev.getAddress().toString().c_str());
  }
};

void setup() {
  Serial.begin(115200);
  BLEDevice::init("");

  pScan = BLEDevice::getScan();
  pScan->setAdvertisedDeviceCallbacks(new Callbacks());
  pScan->setActiveScan(true);   // solicita SCAN_RSP → mais info
  pScan->setInterval(100);
  pScan->setWindow(99);
}

void loop() {
  Serial.printf("\n=== Scan BLE (%ds) ===\n", DURACAO_SCAN);
  BLEScanResults resultados = pScan->start(DURACAO_SCAN, false);
  Serial.printf("%d dispositivo(s) encontrado(s)\n", resultados.getCount());
  pScan->clearResults();
  delay(5000);
}

LoRa, LoRaWAN e Zigbee — longa distância e redes mesh
LoRa — Long Range: como funciona a magia do alcance

O LoRa (Long Range) é uma tecnologia de modulação de rádio desenvolvida pela Semtech. Usa Chirp Spread Spectrum (CSS) — uma técnica que espalha o sinal em uma banda larga — o que permite receber sinais muito abaixo do nível de ruído. Resultado: alcance de 2 a 15km em área urbana, e até 50km em campo aberto, com consumo de energia baixíssimo.

A desvantagem é a taxa de dados: de 250bps a 50kbps, dependendo da configuração. LoRa é ideal para sensores que enviam poucos bytes a cada minuto ou hora — temperatura, nível de água, GPS, leituras de medidores — nunca para streaming de vídeo ou áudio.

Duty cycle: na maioria dos países, redes LoRa em banda ISM têm limite de 1% de duty cycle — o dispositivo só pode transmitir 1% do tempo. Isso é suficiente para 1 mensagem de 50 bytes por minuto no SF padrão. Projete o sistema respeitando esse limite ou use LoRaWAN com gerenciamento automático de ADR.
LoRaWAN — a rede por cima do rádio LoRa

LoRa é apenas a camada física (o rádio). LoRaWAN é o protocolo de rede que roda sobre ele — define como os dispositivos se autenticam, como os gateways encaminham pacotes, como o Network Server gerencia a rede e como o Application Server entrega os dados.

Classe A — transmite quando quiser; janela de downlink logo após TX. Menor consumo. Classe B — janelas de recepção periódicas sincronizadas com o gateway. Classe C — sempre ouvindo; maior consumo, latência mínima de downlink.
The Things Network (TTN) é a maior rede LoRaWAN pública e comunitária — gratuita para uso não-comercial. Basta um gateway LoRa conectado à internet para cobrir um bairro inteiro. No Brasil a cobertura ainda é limitada a capitais, mas empresas como V2COM e Everynet operam redes privadas.
Zigbee — o mesh que se organiza sozinho

O Zigbee (IEEE 802.15.4) opera em 2.4GHz com alcance de 10 a 100m por nó, mas sua grande sacada é a topologia mesh: cada dispositivo intermediário retransmite mensagens para outros, expandindo a cobertura sem infraestrutura adicional. É o protocolo de automação residencial mais usado — presente em lâmpadas Philips Hue, sensores IKEA Tradfri, dispositivos Tuya e Amazon Echo.

Coordinator: forma e gerencia a rede (PAN). Único obrigatório.
Router: precisa de energia constante (tomada) — retransmite pacotes de outros nós.
End Device: fala apenas com o seu "pai" (router ou coordinator) — pode hibernar para economizar bateria.
Zigbee vs Z-Wave vs Thread: os três são protocolos mesh para automação residencial em baixa potência. Z-Wave opera em 868/908MHz (menos interferência com Wi-Fi). Thread é o protocolo base do Matter — o novo padrão unificado apoiado por Apple, Google, Amazon e Samsung. O Matter pode rodar sobre Zigbee, Thread ou Wi-Fi.

Exercício 01 — Filtro por RSSI no Scanner

Modifique o código BLE Scanner para exibir apenas dispositivos com RSSI acima de -70 dBm (sinal razoável ou melhor). Para cada dispositivo filtrado, classifique o sinal textualmente: "Excelente" (acima de -50), "Bom" (-50 a -70). Ordene a saída do mais forte ao mais fraco usando um array de structs antes de imprimir.

LAB
Arduino C — Scanner com filtro e ordenação
#include <BLEDevice.h>
#include <BLEScan.h>
#include <algorithm>

struct Dispositivo {
  String nome;
  String addr;
  int    rssi;
};

BLEScan* pScan;
std::vector<Dispositivo> encontrados;

class CB : public BLEAdvertisedDeviceCallbacks {
  void onResult(BLEAdvertisedDevice d) {
    if (d.getRSSI() >= -70) {
      encontrados.push_back({
        d.haveName() ? d.getName().c_str() : "(sem nome)",
        d.getAddress().toString().c_str(),
        d.getRSSI()
      });
    }
  }
};

void setup() {
  Serial.begin(115200);
  BLEDevice::init("");
  pScan = BLEDevice::getScan();
  pScan->setAdvertisedDeviceCallbacks(new CB());
  pScan->setActiveScan(true);
}

void loop() {
  encontrados.clear();
  pScan->start(5, false);
  pScan->clearResults();

  // Ordenar por RSSI decrescente
  std::sort(encontrados.begin(), encontrados.end(),
    [](const Dispositivo& a, const Dispositivo& b){ return a.rssi > b.rssi; });

  Serial.printf("\n%d dispositivo(s) com sinal ≥ -70 dBm:\n", encontrados.size());
  for (auto& d : encontrados) {
    String qual = (d.rssi > -50) ? "Excelente" : "Bom";
    Serial.printf("  [%4d dBm] %-10s %-28s %s\n",
      d.rssi, qual.c_str(), d.nome.c_str(), d.addr.c_str());
  }
  delay(5000);
}
Exercício 02 — BLE Advertising Beacon

Configure o ESP32 como um beacon BLE que anuncia (advertises) o nome "ESP32-SALA" e um dado custom no payload: a leitura atual de temperatura do DHT22 codificada em 2 bytes (multiplique por 100 para preservar 2 casas decimais — ex: 27.30°C → 2730 = 0x0AAA). Sem estabelecer conexão — apenas advertising contínuo. Verifique no nRF Connect que o beacon aparece com o nome correto e o Manufacturer Data visível.

LAB
Arduino C — BLE Beacon com temperatura no payload
#include <BLEDevice.h>
#include <BLEAdvertising.h>
#include <DHT.h>

DHT dht(4, DHT22);
BLEAdvertising* pAdv;

void atualizarBeacon(float temp) {
  uint16_t t100 = (uint16_t)(temp * 100);

  BLEAdvertisementData advData;
  advData.setName("ESP32-SALA");

  // Manufacturer Data: Company ID (0xFFFF custom) + 2 bytes de temp
  String mfr = "\xFF\xFF";      // company ID (little-endian)
  mfr += (char)(t100 & 0xFF);   // byte baixo
  mfr += (char)(t100 >> 8);     // byte alto
  advData.setManufacturerData(mfr);

  pAdv->stop();
  pAdv->setAdvertisementData(advData);
  pAdv->start();
}

void setup() {
  Serial.begin(115200);
  dht.begin();
  BLEDevice::init("ESP32-SALA");
  pAdv = BLEDevice::getAdvertising();
  delay(2000);
  atualizarBeacon(dht.readTemperature());
}

void loop() {
  float t = dht.readTemperature();
  if (!isnan(t)) { atualizarBeacon(t); Serial.printf("Beacon: %.1f°C\n", t); }
  delay(30000);   // atualiza a cada 30s
}
Exercício 03 — Tabela de decisão pessoal

Crie uma tabela de decisão de protocolos IoT com 5 critérios que você considera mais importantes para projetos reais (ex: autonomia de bateria, alcance, custo do módulo, complexidade de implementação, disponibilidade de infraestrutura). Para cada critério, pontue Wi-Fi+MQTT, BLE, LoRaWAN e Zigbee de 1 a 5. Análise
Referência — não há resposta única
  • A maioria dos projetos de lab e prototipagem começa com Wi-Fi + MQTT pelo ecossistema maduro, facilidade de debug e infraestrutura já presente (roteador).
  • A troca para BLE é motivada por restrição de energia: sensor a bateria que precisa durar semanas ou meses.
  • A troca para LoRaWAN é motivada por alcance: quando o sensor está longe de qualquer rede Wi-Fi.
  • A troca para Zigbee é motivada por densidade de nós em espaço limitado com infraestrutura própria (automação residencial ou industrial).
Exercício 04 — Desafio: gateway BLE → MQTT

Implemente um gateway BLE-MQTT: um ESP32 (gateway) faz scan BLE continuamente e, para cada dispositivo encontrado com RSSI > -75 dBm, publica no broker MQTT (via Wi-Fi) no tópico gateway/ble/ENDERECO_MAC um JSON com {"rssi": -62, "nome": "ESP32-SALA", "ts": 1234567} onde ts é o valor de millis(). Isso simula um gateway IoT real que coleta dados BLE e os publica na infraestrutura de nuvem via MQTT.

Desafio · Integração
Arduino C — gateway BLE → MQTT
#include <BLEDevice.h>
#include <BLEScan.h>
#include <WiFi.h>
#include <PubSubClient.h>
#include <ArduinoJson.h>

const char* SSID = "Rede"; const char* SENHA = "senha";
const char* BROKER = "192.168.1.100";

WiFiClient wifi; PubSubClient mqtt(wifi); BLEScan* scan;

class CB : public BLEAdvertisedDeviceCallbacks {
  void onResult(BLEAdvertisedDevice d) {
    if (d.getRSSI() < -75) return;
    if (!mqtt.connected()) return;

    String addr = d.getAddress().toString().c_str();
    addr.replace(":", "");   // remove : para usar no tópico

    StaticJsonDocument<128> doc;
    doc["rssi"] = d.getRSSI();
    doc["nome"] = d.haveName() ? d.getName().c_str() : "?";
    doc["ts"]   = (unsigned long)millis();

    char buf[128], topico[64];
    serializeJson(doc, buf);
    snprintf(topico, sizeof(topico), "gateway/ble/%s", addr.c_str());
    mqtt.publish(topico, buf);
  }
};

void setup() {
  Serial.begin(115200);
  WiFi.begin(SSID, SENHA);
  while(WiFi.status() != WL_CONNECTED) delay(300);
  mqtt.setServer(BROKER, 1883);
  while(!mqtt.connect("ble-gateway")) delay(1000);
  BLEDevice::init("");
  scan = BLEDevice::getScan();
  scan->setAdvertisedDeviceCallbacks(new CB(), true);
  scan->setActiveScan(true);
}

void loop() {
  if(!mqtt.connected()) mqtt.connect("ble-gateway");
  mqtt.loop();
  scan->start(3, false);
  scan->clearResults();
}
