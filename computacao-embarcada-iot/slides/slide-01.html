Onde você está pisando

Você está no terceiro período de um curso de Inteligência Artificial. Até agora viu programação, banco de dados, fundamentos. Este semestre é diferente — vamos sair do software puro e entrar no mundo físico.

Computação Embarcada e IoT é a disciplina que conecta código com hardware. Ao final do semestre você vai ter programado um microcontrolador real, lido sensores físicos, transmitido dados via Wi-Fi e MQTT, e colocado um modelo de IA para rodar dentro de um chip com memória de poucos kilobytes.

Por que isso importa para IA? A maioria dos dados que alimentam modelos de ML vêm de sensores no mundo real — câmeras, acelerômetros, termômetros, microfones. Entender como esses dispositivos funcionam é entender de onde vêm os dados antes do pipeline.

O que é Computação Embarcada — e por que é diferente
Definição objetiva

Um sistema embarcado é um sistema computacional projetado para executar uma função específica, embutido dentro de um dispositivo maior. Ele tem recursos limitados — memória em KB ou MB, processador em MHz — e geralmente opera em tempo real, muitas vezes sem sistema operacional completo.

A palavra-chave é dedicado. O microcontrolador do airbag do seu carro faz uma coisa: detectar impacto e disparar o airbag. Não tem browser. Não tem planilha. Tem um loop que roda para sempre verificando um sensor de aceleração.

Definição da IEEE: "Um sistema embarcado é qualquer dispositivo que inclua um computador programável, mas que não seja, por si só, um computador de propósito geral."

O loop que nunca termina

Todo sistema embarcado tem uma estrutura fundamental: um setup (executa uma vez, inicializa tudo) e um loop (executa para sempre, verifica sensores, executa lógica, controla atuadores).

Arduino C — Estrutura básica
// setup(): executa UMA VEZ ao ligar ou resetar
void setup() {
  Serial.begin(9600);        // inicia comunicação serial
  pinMode(13, OUTPUT);     // configura pino 13 como saída (LED)
  // inicializa sensores, displays, módulos Wi-Fi...
}

// loop(): executa PARA SEMPRE (dezenas de vezes por segundo)
void loop() {
  digitalWrite(13, HIGH);   // liga LED
  delay(1000);               // aguarda 1 segundo
  digitalWrite(13, LOW);    // desliga LED
  delay(1000);               // aguarda 1 segundo
  // → esse é o "Hello World" do mundo embarcado: o Blink
}

Esse código é o Blink — o Hello World do Arduino. Na Semana 2 você vai escrever e rodar isso no hardware real.

IoT e Edge AI — quando o dispositivo pensa por conta própria
O que é IoT — além do buzzword

IoT (Internet of Things — Internet das Coisas) é a conexão de objetos físicos à internet para coletar, transmitir e agir sobre dados. Mas a definição técnica vai além: é a integração de sensores, processamento, comunicação e atuação em objetos do mundo real.

O mercado IoT tem previsão de 29 bilhões de dispositivos conectados até 2030. Geladeiras que pedem reposição de estoque. Lavouras que ativam irrigação automaticamente. Fábricas que detectam falha de equipamento antes de acontecer. Tudo isso começa num microcontrolador.

Arquitetura de um sistema IoT
Sensor
Coleta dados do mundo físico
MCU / Borda
Lê, filtra, processa localmente
Gateway
Protocolo → MQTT / HTTP
Nuvem
Armazena, analisa, modelos ML
Dashboard
Visualização e alertas

Cada etapa pode ser mapeada para ferramentas que você vai usar no semestre: DHT22 (sensor), ESP32 (MCU), MQTT (protocolo), AWS IoT Core (nuvem), Node-RED (dashboard).

Escolha do protocolo: não existe protocolo "melhor". Existe o protocolo certo para o caso de uso. Alcance, consumo de energia, latência e custo determinam a escolha. Nas semanas 10 e 11 vamos explorar cada um em profundidade.
Edge AI — quando o chip pensa

Tradicionalmente, um dispositivo IoT coleta dados e manda para a nuvem processar. O modelo de ML fica na nuvem, a inferência acontece lá, a resposta volta para o dispositivo. Isso funciona — mas tem problemas:

  • Latência: ida e volta para a nuvem leva 100ms–2s. Um carro autônomo não pode esperar 2 segundos para decidir frear.
  • Conectividade: e se não tiver internet? Um sistema industrial não pode parar por queda de rede.
  • Privacidade: imagem de câmera de segurança subindo para nuvem de terceiros — problema legal e ético.
  • Custo: enviar gigabytes de dados de sensores para a nuvem todo dia tem custo alto de banda e armazenamento.

A solução é Edge AI: colocar o modelo dentro do dispositivo. A inferência acontece localmente — sem internet, sem latência, sem exposição de dados.

TinyML: subárea do Edge AI focada em rodar modelos de ML em microcontroladores com <1 MB de RAM. Você vai fazer isso na Semana 12 — carregar um modelo treinado no Google Colab e rodar dentro de um ESP32 com 520 KB de RAM.

O que você precisa ter gravado
Sistema Embarcado

Computador dedicado a uma função específica. Recursos limitados. Opera em tempo real. Sem SO completo na maioria dos casos.

IoT

Objetos físicos conectados. Arquitetura: Sensor → MCU → Gateway → Nuvem → Dashboard. Cada protocolo tem seu caso de uso.

Edge AI

IA rodando no dispositivo, não na nuvem. Vantagens: latência zero, offline, privacidade. TinyML: modelos em <1 MB de RAM.
