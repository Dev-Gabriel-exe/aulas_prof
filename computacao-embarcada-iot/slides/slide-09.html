MQTT — o protocolo que HTTP não consegue ser em IoT
HTTP vs MQTT — por que mudar?

Na semana passada, o ESP32 mandava dados via HTTP POST. Funcionou bem para o lab, mas em produção essa abordagem tem problemas sérios: o ESP32 precisa conhecer o endereço do servidor, o servidor precisa estar sempre disponível, e cada envio abre uma nova conexão TCP — custosa em energia e tempo.

O MQTT (Message Queuing Telemetry Transport) foi criado em 1999 pela IBM para redes instáveis e dispositivos com poucos recursos — pense em sensores em plataformas de petróleo conectados via satélite. Ele resolve exatamente os pontos fracos do HTTP em IoT.

HTTP em IoT — limitações

  • Modelo request-response — cliente sempre inicia, servidor sempre espera
  • Dispositivo precisa saber o endereço do servidor — acoplamento direto
  • Para receber dados, o dispositivo precisa fazer polling (perguntar repetidamente)
  • Headers HTTP pesados — overhead alto para mensagens de poucos bytes
  • Sem suporte nativo a reconexão ou garantia de entrega

MQTT em IoT — vantagens

  • Modelo publish/subscribe — produtor e consumidor nunca se conhecem
  • Todos os dispositivos falam com o broker — único ponto de contato
  • Assinantes recebem dados em tempo real sem polling
  • Header mínimo de 2 bytes — ideal para payload pequeno
  • QoS garante entrega mesmo em redes instáveis; reconexão automática
O modelo Publish/Subscribe — três atores

No MQTT existem três papéis: o Publisher (publica dados em um tópico), o Subscriber (assina tópicos e recebe mensagens) e o Broker (intermediário central que recebe de publishers e distribui para subscribers). Publisher e Subscriber nunca se comunicam diretamente — nem precisam saber que o outro existe.

O broker não armazena os dados permanentemente (apenas com retain=true para a última mensagem). Ele é um roteador de mensagens em tempo real.
Tópicos MQTT — hierarquia e wildcards

Tópicos são strings que funcionam como endereços de roteamento. Usam / como separador de níveis — pense em caminhos de arquivo. Você projeta a hierarquia de tópicos antes de começar a programar: uma boa estrutura facilita filtros e escalabilidade.

Exemplo de hierarquia de tópicos — sistema de casa inteligente
casa
├─ casa/sala
│ ├─ casa/sala/temperatura → publica: 27.3
│ └─ casa/sala/umidade → publica: 64.8
├─ casa/quarto
│ └─ casa/quarto/temperatura
└─ casa/jardim
└─ casa/jardim/umidade_solo
casa/+/temperatura → wildcard + : um nível qualquer → recebe sala E quarto
casa/# → wildcard # : tudo abaixo → recebe todos os tópicos da casa
WildcardSignificadoExemploAssina
+Um nível qualquercasa/+/tempcasa/sala/temp, casa/quarto/temp — não casa/sala/sub/temp
#Zero ou mais níveis (sempre no final)casa/#tudo que começa com casa/
—Sem wildcard — tópico exatocasa/sala/tempapenas esse tópico específico
Tópicos que começam com $ são reservados pelo broker (ex: $SYS/broker/clients/connected). O wildcard # não os captura — você precisa assinar explicitamente.
Mosquitto — broker local em minutos

O Eclipse Mosquitto é o broker MQTT open-source mais usado, leve e fácil de instalar. Roda no mesmo computador do lab, escuta na porta padrão 1883 (sem TLS) e inclui duas ferramentas de linha de comando: mosquitto_pub e mosquitto_sub.

$ sudo apt install mosquitto mosquitto-clients
# Windows: baixar instalador em mosquitto.org
$ mosquitto -v # iniciar broker verboso (debug)
1700000000: mosquitto version 2.x.x starting
1700000000: Opening ipv4 listen socket on port 1883.
# Em outro terminal — assinar um tópico:
$ mosquitto_sub -h localhost -t "casa/#" -v
# Em outro terminal — publicar uma mensagem de teste:
$ mosquitto_pub -h localhost -t "casa/sala/temperatura" -m "27.3"
casa/sala/temperatura 27.3 ← aparece no sub acima
Por padrão, Mosquitto 2.x exige configuração para aceitar conexões externas. Crie um arquivo mosquitto.conf com: listener 1883 e allow_anonymous true. Inicie com mosquitto -c mosquitto.conf. Sem isso, o ESP32 não consegue se conectar a partir de outra máquina.
🙋 Em qual situação o modelo publish/subscribe do MQTT é claramente superior ao HTTP request-response?
A
Quando um servidor web precisa servir páginas HTML para muitos usuários simultâneos
B
Quando 50 sensores precisam enviar dados e 3 sistemas diferentes (dashboard, banco de dados, alarme) precisam receber todos esses dados em tempo real
C
Quando o cliente precisa receber uma resposta imediata e confirmada com conteúdo específico para aquela requisição
D
Quando os dados precisam ser transmitidos com criptografia TLS — o MQTT não suporta isso via HTTP

QoS, Retain, LWT e ESP32 com PubSubClient
QoS — Quality of Service: quem garante que a mensagem chegou?

Em redes Wi-Fi de laboratório, quase tudo funciona. Em produção — uma fábrica com interferência, um galpão com paredes de concreto, um sensor externo com sinal instável — mensagens se perdem. O MQTT define três níveis de garantia de entrega chamados QoS.

0
At most once
Dispara e esquece. Sem confirmação, sem retransmissão. Pode perder mensagens.
Leituras frequentes, perda aceitável
1
At least once
Entregue pelo menos uma vez. Broker confirma com PUBACK. Pode chegar duplicada.
Uso geral — dados de sensor
2
Exactly once
Entregue exatamente uma vez. Handshake de 4 passos. Mais lento e pesado.
Atuações críticas, billing, alarmes
Na prática: QoS 1 é o ponto de equilíbrio para 90% dos projetos IoT — garante entrega sem o overhead do QoS 2. Use QoS 0 para telemetria de alta frequência (temperatura a cada 1s) onde perder uma leitura não importa. Use QoS 2 apenas para comandos irreversíveis (abrir uma válvula, acionar um alarme de incêndio).
Retain e Last Will — dois recursos essenciais

Retained Message

Quando uma mensagem é publicada com retain=true, o broker armazena a última mensagem daquele tópico. Todo novo assinante recebe imediatamente esse valor ao se conectar — sem precisar esperar a próxima publicação.

Caso de uso típico: status de um dispositivo (online / offline). Um dashboard que abre já sabe o estado atual sem esperar.

Last Will and Testament (LWT)

Ao conectar, o cliente informa ao broker: "Se eu desconectar de forma inesperada (queda de energia, reset, perda de Wi-Fi), publique esta mensagem em nome de mim." O broker executa o LWT automaticamente quando detecta a queda.

Caso de uso típico: casa/esp32/status = "offline". Quando o ESP32 cai, o broker avisa todos os assinantes.
PubSubClient — MQTT no ESP32

A biblioteca PubSubClient by Nick O'Leary é a implementação MQTT mais usada no ecossistema Arduino/ESP32. Instale via Gerenciador de Bibliotecas (busque "PubSubClient"). Ela suporta MQTT 3.1.1, QoS 0 e 1, retain e LWT.

Arduino C — ESP32 MQTT Publisher com reconexão e LWT
#include <WiFi.h>
#include <PubSubClient.h>

const char* SSID       = "NomeDaRede";
const char* SENHA      = "SenhaDaRede";
const char* BROKER_IP  = "192.168.1.100";   // IP do PC com Mosquitto
const int  BROKER_PORT = 1883;
const char* CLIENT_ID   = "esp32-sensor-01";  // único por dispositivo

// Tópicos
const char* TOP_STATUS  = "lab/esp32/status";
const char* TOP_TEMP    = "lab/esp32/temperatura";
const char* TOP_UMID    = "lab/esp32/umidade";
const char* TOP_CMD     = "lab/esp32/comando";  // subscribe

WiFiClient   wifiClient;
PubSubClient mqtt(wifiClient);

// ── Callback: chamada ao receber mensagem em tópico assinado
void callbackMQTT(char* topico, byte* payload, unsigned int tamanho) {
  String msg = "";
  for (int i = 0; i < (int)tamanho; i++) msg += (char)payload[i];

  Serial.printf("[MQTT ←] %s : %s\n", topico, msg.c_str());

  if (msg == "ping") {
    mqtt.publish("lab/esp32/pong", "pong");
  }
}

void conectarWifi() {
  WiFi.mode(WIFI_STA);
  WiFi.begin(SSID, SENHA);
  Serial.print("Wi-Fi");
  while (WiFi.status() != WL_CONNECTED) { delay(500); Serial.print("."); }
  Serial.println(" OK · IP: " + WiFi.localIP().toString());
}

void conectarMQTT() {
  mqtt.setServer(BROKER_IP, BROKER_PORT);
  mqtt.setCallback(callbackMQTT);

  while (!mqtt.connected()) {
    Serial.print("MQTT...");

    // connect(clientId, user, pass, willTopic, willQos, willRetain, willMsg)
    if (mqtt.connect(CLIENT_ID, "", "", TOP_STATUS, 1, true, "offline")) {
      Serial.println(" conectado!");
      mqtt.publish(TOP_STATUS, "online", true);   // retain=true
      mqtt.subscribe(TOP_CMD, 1);               // QoS 1
    } else {
      Serial.printf(" falhou (rc=%d). Nova tentativa em 5s\n", mqtt.state());
      delay(5000);
    }
  }
}

void setup() {
  Serial.begin(115200);
  conectarWifi();
  conectarMQTT();
}

void loop() {
  if (!mqtt.connected()) conectarMQTT();
  mqtt.loop();   // processa mensagens recebidas e mantém conexão viva
}
mqtt.loop() é obrigatório no loop(). Sem ele, o cliente não processa mensagens recebidas, não responde ao keepalive do broker e é desconectado em poucos segundos. Se você usar delay() por muito tempo no loop(), o MQTT pode perder a conexão — use millis() para timing.

B
Last Will Testament — o broker re-envia a última mensagem ao detectar nova conexão
C

Arduino C — ESP32 + DHT22 → MQTT completo
#include <WiFi.h>
#include <PubSubClient.h>
#include <ArduinoJson.h>
#include <DHT.h>

const char* SSID      = "NomeDaRede";
const char* SENHA     = "SenhaDaRede";
const char* BROKER_IP = "192.168.1.100";
const char* CLIENT_ID = "esp32-dht-01";

#define PINO_DHT   4
#define TIPO_DHT   DHT22
#define INTERVALO  10000UL

const char* TOP_DADOS  = "lab/esp32/dados";
const char* TOP_STATUS = "lab/esp32/status";
const char* TOP_CMD    = "lab/esp32/comando";

DHT          dht(PINO_DHT, TIPO_DHT);
WiFiClient   wifiClient;
PubSubClient mqtt(wifiClient);
unsigned long ultimaPublicacao = 0;

void callbackMQTT(char* topico, byte* payload, unsigned int tam) {
  String msg = "";
  for (int i = 0; i < (int)tam; i++) msg += (char)payload[i];
  Serial.printf("[CMD] %s : %s\n", topico, msg.c_str());
  if (msg == "ping") mqtt.publish("lab/esp32/pong", "pong");
}

void conectarWifi() {
  WiFi.mode(WIFI_STA); WiFi.begin(SSID, SENHA);
  Serial.print("Wi-Fi");
  while (WiFi.status() != WL_CONNECTED) { delay(500); Serial.print("."); }
  Serial.println(" OK · " + WiFi.localIP().toString());
}

void conectarMQTT() {
  mqtt.setServer(BROKER_IP, 1883);
  mqtt.setCallback(callbackMQTT);
  while (!mqtt.connected()) {
    Serial.print("MQTT...");
    if (mqtt.connect(CLIENT_ID, "", "", TOP_STATUS, 1, true, "offline")) {
      Serial.println(" OK");
      mqtt.publish(TOP_STATUS, "online", true);
      mqtt.subscribe(TOP_CMD, 1);
    } else {
      Serial.printf(" erro rc=%d\n", mqtt.state()); delay(5000);
    }
  }
}

void publicarDados(float t, float u) {
  StaticJsonDocument<128> doc;
  doc["temp"]      = t;
  doc["umid"]      = u;
  doc["dispositivo"] = CLIENT_ID;

  char buf[128];
  serializeJson(doc, buf);

  // publish(topico, payload, retained)
  if (mqtt.publish(TOP_DADOS, buf, false)) {   // QoS 0 padrão
    Serial.printf("[PUB] T=%.1f U=%.1f\n", t, u);
  } else {
    Serial.println("[PUB] Falhou — buffer cheio?");
  }
}

void setup() {
  Serial.begin(115200);
  dht.begin();
  conectarWifi();
  conectarMQTT();
  delay(2000);
}

void loop() {
  if (WiFi.status() != WL_CONNECTED) conectarWifi();
  if (!mqtt.connected()) conectarMQTT();
  mqtt.loop();   // SEMPRE chamar — mantém conexão e processa callbacks

  if (millis() - ultimaPublicacao >= INTERVALO) {
    ultimaPublicacao = millis();
    float t = dht.readTemperature();
    float u = dht.readHumidity();
    if (!isnan(t) && !isnan(u)) publicarDados(t, u);
    else Serial.println("[ERRO] Leitura DHT inválida");
  }
}
Python — subscriber paho-mqtt com JSON
# subscriber_mqtt.py — pip install paho-mqtt
import paho.mqtt.client as mqtt
import json
from datetime import datetime

BROKER = "192.168.1.100"   # mesmo IP do Mosquitto
PORT   = 1883
TOPICO = "lab/#"            # assina tudo abaixo de lab/

def on_connect(client, userdata, flags, rc):
    if rc == 0:
        print(f"Conectado ao broker {BROKER}:{PORT}")
        client.subscribe(TOPICO, qos=1)
        print(f"Assinando: {TOPICO}\n")
    else:
        print(f"Falha na conexão — rc={rc}")

def on_message(client, userdata, msg):
    ts = datetime.now().strftime("%H:%M:%S")
    payload = msg.payload.decode("utf-8")

    # Tenta fazer parse como JSON
    try:
        dados = json.loads(payload)
        t = dados.get("temp", "?")
        u = dados.get("umid", "?")
        disp = dados.get("dispositivo", "?")
        print(f"[{ts}] {msg.topic}")
        print(f"       Temp={t}°C  Umid={u}%  de={disp}\n")
    except json.JSONDecodeError:
        # Mensagem não é JSON (ex: status "online"/"offline")
        print(f"[{ts}] {msg.topic} : {payload}\n")

def on_disconnect(client, userdata, rc):
    print(f"Desconectado (rc={rc}) — reconectando...")

client = mqtt.Client(client_id="python-sub-01")
client.on_connect    = on_connect
client.on_message    = on_message
client.on_disconnect = on_disconnect

client.connect(BROKER, PORT, keepalive=60)
print("Aguardando mensagens... (Ctrl+C para sair)")
client.loop_forever()   # bloqueia e processa eventos indefinidamente

Exercício 02 — Tópicos separados vs JSON único

O código do LAB publica temperatura e umidade juntos em um único tópico como JSON (lab/esp32/dados). Uma abordagem alternativa seria publicar em tópicos separados: lab/esp32/temperatura com o valor 27.3 (número puro) e lab/esp32/umidade com 64.8. Implemente essa segunda abordagem no ESP32 e no Python. Depois discuta: quais são as vantagens e desvantagens de cada abordagem em um projeto real?

LAB + Análise
Arduino C — tópicos separados
// Na função publicarDados(), substituir serializeJson por:
char bufT[16], bufU[16];
dtostrf(t, 4, 1, bufT);   // float para string: "27.3"
dtostrf(u, 4, 1, bufU);
mqtt.publish("lab/esp32/temperatura", bufT);
mqtt.publish("lab/esp32/umidade",     bufU);
Análise comparativa
  • Tópicos separados: mais simples para assinantes que só precisam de um valor; integração direta com ferramentas como Node-RED e InfluxDB (sem parser JSON); facil usar retain por variável.
  • JSON único: garante que temperatura e umidade chegam como um snapshot atômico do mesmo instante; reduz número de mensagens MQTT; mais fácil de versionar e adicionar campos.
  • Recomendação: use JSON para dados que pertencem ao mesmo instante de leitura. Use tópicos separados quando os consumidores são heterogêneos e cada um precisa só de um campo.

Python — subscriber com CSV + resumo pandas
import paho.mqtt.client as mqtt
import json, csv, os, time, threading
from datetime import datetime

BROKER  = "192.168.1.100"
ARQUIVO = "historico.csv"
CABECALHO = ["timestamp", "dispositivo", "temperatura", "umidade"]

# Cria arquivo com cabeçalho se não existir
if not os.path.exists(ARQUIVO):
    with open(ARQUIVO, "w", newline="") as f:
        csv.writer(f).writerow(CABECALHO)

def on_message(client, userdata, msg):
    try:
        d = json.loads(msg.payload.decode())
        ts = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
        row = [ts, d.get("dispositivo"), d.get("temp"), d.get("umid")]
        with open(ARQUIVO, "a", newline="") as f:
            csv.writer(f).writerow(row)
        print(f"[{ts}] T={row[2]}°C U={row[3]}%")
    except: pass

def resumo_periodico():
    while True:
        time.sleep(300)   # a cada 5 minutos
        try:
            import pandas as pd
            df = pd.read_csv(ARQUIVO)
