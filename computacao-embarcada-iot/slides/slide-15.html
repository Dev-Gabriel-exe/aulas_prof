AWS IoT Core — arquitetura, Things e Device Shadow
O que muda ao sair do Mosquitto local para a nuvem

O Mosquitto que usamos na Semana 9 é excelente para laboratório: sem autenticação, sem criptografia, IP local fixo. Em produção, esses três pontos são problemas sérios — qualquer dispositivo na mesma rede consegue publicar ou assinar seus tópicos, e os dados trafegam em texto puro.

O AWS IoT Core é um broker MQTT gerenciado que resolve esses problemas: suporta milhões de dispositivos simultâneos, exige autenticação mútua via certificados X.509, criptografa toda comunicação com TLS 1.2+, e conecta nativamente a todos os serviços AWS. A diferença no código do ESP32 é menor do que parece — a mudança principal é usar WiFiClientSecure em vez de WiFiClient.

Device Shadow — sincronizando estado com dispositivos offline

O Device Shadow é um dos recursos mais poderosos do AWS IoT Core — e resolve o problema mais difícil de IoT: como enviar um comando a um dispositivo que pode estar offline no momento do envio?

O Shadow é um documento JSON persistido na nuvem com duas seções: desired (o que a aplicação quer que o dispositivo faça) e reported (o que o dispositivo reportou como estado atual). O ESP32 assina o tópico de delta e aplica as mudanças ao reconectar — sem perder comandos enviados quando estava offline.

Documento Shadow — controle de LED remoto
Desired — o que a app quer
JSON
{
  "state": {
    "desired": {
      "led": "ON",
      "intervalo": 10
    }
  }
}
Reported — o que o ESP32 diz
JSON
{
  "state": {
    "reported": {
      "led": "OFF",
      "temp": 27.3,
      "umid": 64.8
    }
  }
}
Quando desired ≠ reported, o IoT Core gera automaticamente um documento delta e o publica no tópico $aws/things/NOME/shadow/update/delta. O ESP32 assina esse tópico, aplica o delta e publica o novo reported. Se estiver offline, recebe o delta ao reconectar.
Tópicos reservados do Shadow:
Publicar update: $aws/things/NOME/shadow/update
Receber delta: $aws/things/NOME/shadow/update/delta
Buscar estado atual: $aws/things/NOME/shadow/get

TLS mútuo, certificados X.509 e Rules Engine
TLS mútuo — por que é diferente do HTTPS comum

No HTTPS comum (banco, e-commerce), apenas o servidor apresenta um certificado — você verifica que está falando com o banco certo. Em IoT de produção, isso não é suficiente: o servidor também precisa verificar quem é o dispositivo que está se conectando. Isso é a autenticação mútua (mTLS — mutual TLS).

No AWS IoT Core, cada dispositivo tem seu próprio certificado X.509, assinado pela CA da AWS. O ESP32 apresenta esse certificado ao conectar, e a AWS verifica que ele pertence a um Thing registrado com a política correta. Sem certificado válido: conexão recusada, sem exceção.

Os 3 arquivos de certificado — o que cada um faz
🏛️
CA Root — AmazonRootCA1.pem
Certificado da Autoridade Certificadora da Amazon. O ESP32 usa para verificar que está falando com um servidor legítimo da AWS — evita ataques man-in-the-middle. Mesmo para todos os dispositivos. Baixar em: https://www.amazontrust.com/repository/AmazonRootCA1.pem
📜
Device Certificate — xxxxx-certificate.pem.crt
Certificado único do dispositivo, gerado no console AWS ou programaticamente. O ESP32 apresenta isso para provar sua identidade à AWS. Um certificado diferente por dispositivo. Associar ao Thing e à Policy no console.
🔑
Private Key — xxxxx-private.pem.key
Chave privada RSA do dispositivo — par criptográfico do certificado. Nunca sai do dispositivo. Se vazar, revogar o certificado imediatamente no console. Não existe cópia na AWS.
NUNCA commite certificados e chaves privadas no GitHub. Adicione os arquivos .pem ao .gitignore antes de criar o primeiro commit. Em projetos reais use AWS Secrets Manager ou variáveis de ambiente seguras. A chave privada comprometida dá acesso total ao dispositivo na nuvem.
Rules Engine — roteando mensagens para o ecossistema AWS

O Rules Engine do AWS IoT Core usa uma linguagem SQL-like para selecionar e transformar mensagens MQTT e enviá-las para outros serviços. Uma regra define: (1) de qual tópico selecionar, (2) quais campos do JSON pegar, (3) para qual serviço enviar.

🗄️
→ S3
SELECT * FROM
'sensor/dados'
Salva cada mensagem como objeto JSON no S3. Ideal para histórico bruto, data lake, análise posterior com Athena.
📊
→ DynamoDB
SELECT temp, umid,
timestamp() as ts
FROM 'sensor/dados'
Insere campos específicos em tabela DynamoDB. Leitura em tempo real por aplicações web/mobile.
⚡
→ Lambda
SELECT * FROM
'sensor/dados'
WHERE temp > 35
Aciona função Lambda apenas quando temperatura ultrapassa 35°C. Lambda pode enviar e-mail, SMS ou notificação push.
Timestream é o serviço AWS otimizado para séries temporais IoT — escalável, com retenção configurável e integração nativa com Grafana. Para projetos IoT em produção que precisam de dashboards de time-series, Timestream + Grafana é a combinação padrão da indústria.
Policy IoT — o que o dispositivo pode e não pode fazer

A política (Policy) é um documento JSON que define exatamente quais ações o certificado pode executar — publicar, assinar, conectar — e em quais tópicos. Princípio do menor privilégio: um sensor de temperatura não precisa de permissão para publicar em tópicos de outro dispositivo.

JSON — Policy IoT mínima para ESP32 sensor
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "iot:Connect",
      "Resource": "arn:aws:iot:*:*:client/esp32-sensor-*"
    },
    {
      "Effect": "Allow",
      "Action": "iot:Publish",
      "Resource": [
        "arn:aws:iot:*:*:topic/sensor/dados",
        "arn:aws:iot:*:*:topic/$aws/things/*/shadow/update"
      ]
    },
    {
      "Effect": "Allow",
      "Action": "iot:Subscribe",
      "Resource": "arn:aws:iot:*:*:topicfilter/$aws/things/*/shadow/update/delta"
    },
    {
      "Effect": "Allow",
      "Action": "iot:Receive",
      "Resource": "arn:aws:iot:*:*:topic/$aws/things/*/shadow/update/delta"
    }
  ]
}
🙋 Por que o AWS IoT Core usa autenticação mútua (mTLS) em vez de apenas usuário e senha como o Mosquitto local?
A
Porque a AWS não suporta MQTT com usuário e senha — é uma limitação técnica da plataforma
B
Porque senha pode ser interceptada ou vazada; o certificado X.509 + chave privada garante que só o dispositivo legítimo com a chave correta consegue autenticar — mesmo que o certificado seja copiado, sem a chave privada não serve
C
Para que a AWS possa cobrar por dispositivo — o certificado serve como sistema de billing por Thing
D
Porque em redes IoT o ESP32 não tem memória suficiente para armazenar uma senha longa

Arduino C — ESP32 + DHT22 → AWS IoT Core via MQTT/TLS + Device Shadow
#include <WiFi.h>
#include <WiFiClientSecure.h>
#include <PubSubClient.h>
#include <ArduinoJson.h>
#include <DHT.h>

// ── Configurações ──────────────────────────────────────────
const char* SSID       = "NomeDaRede";
const char* SENHA      = "SenhaDaRede";
const char* AWS_ENDPOINT= "xxxxx.iot.us-east-1.amazonaws.com";
const char* THING_NAME = "esp32-lab";
#define PINO_DHT  4
#define PINO_LED  2
#define INTERVALO 10000UL

// ── Certificados — substituir pelo conteúdo real ───────────
const char CA_CERT[] PROGMEM = R"EOF(
-----BEGIN CERTIFICATE-----
... cole o conteúdo do AmazonRootCA1.pem aqui ...
-----END CERTIFICATE-----
)EOF";

const char DEVICE_CERT[] PROGMEM = R"EOF(
-----BEGIN CERTIFICATE-----
... cole o conteúdo do xxxxx-certificate.pem.crt aqui ...
-----END CERTIFICATE-----
)EOF";

const char PRIVATE_KEY[] PROGMEM = R"EOF(
-----BEGIN RSA PRIVATE KEY-----
... cole o conteúdo do xxxxx-private.pem.key aqui ...
-----END RSA PRIVATE KEY-----
)EOF";

// ── Tópicos ────────────────────────────────────────────────
char TOP_DADOS[64], TOP_SHADOW_UPDATE[80], TOP_SHADOW_DELTA[80];

DHT           dht(PINO_DHT, DHT22);
WiFiClientSecure wifiSecure;
PubSubClient  mqtt(wifiSecure);
unsigned long ultimoEnvio = 0;
bool          ledLigado   = false;

// ── Callback — Shadow delta recebido ──────────────────────
void callbackMQTT(char* topico, byte* payload, unsigned int tam) {
  String msg = "";
  for(int i=0;i<(int)tam;i++) msg+=(char)payload[i];
  Serial.printf("[DELTA] %s\n", msg.c_str());

  StaticJsonDocument<256> doc;
  if (!deserializeJson(doc, msg)) {
    const char* ledVal = doc["state"]["led"] | "";
    if (String(ledVal) == "ON")  { ledLigado=true;  digitalWrite(PINO_LED, HIGH); }
    if (String(ledVal) == "OFF") { ledLigado=false; digitalWrite(PINO_LED, LOW);  }

    // Reportar novo estado ao Shadow
    StaticJsonDocument<128> rep;
    rep["state"]["reported"]["led"] = ledLigado ? "ON" : "OFF";
    char buf[128]; serializeJson(rep, buf);
    mqtt.publish(TOP_SHADOW_UPDATE, buf);
  }
}

void conectar() {
  WiFi.begin(SSID, SENHA);
  Serial.print("Wi-Fi");
  while(WiFi.status()!=WL_CONNECTED){delay(400);Serial.print(".");}
  Serial.println(" OK");

  wifiSecure.setCACert(CA_CERT);
  wifiSecure.setCertificate(DEVICE_CERT);
  wifiSecure.setPrivateKey(PRIVATE_KEY);

  mqtt.setServer(AWS_ENDPOINT, 8883);
  mqtt.setCallback(callbackMQTT);
  mqtt.setBufferSize(1024);   // Shadow JSONs podem ser grandes

  Serial.print("AWS IoT");
  while (!mqtt.connect(THING_NAME)) {
    Serial.print("."); delay(1000);
  }
  Serial.println(" Conectado!");
  mqtt.subscribe(TOP_SHADOW_DELTA);
}

void setup() {
  Serial.begin(115200);
  dht.begin();
  pinMode(PINO_LED, OUTPUT);
  snprintf(TOP_DADOS,          sizeof(TOP_DADOS),         "sensor/dados");
  snprintf(TOP_SHADOW_UPDATE,  sizeof(TOP_SHADOW_UPDATE),  "$aws/things/%s/shadow/update",       THING_NAME);
  snprintf(TOP_SHADOW_DELTA,   sizeof(TOP_SHADOW_DELTA),   "$aws/things/%s/shadow/update/delta",  THING_NAME);
  conectar();
  delay(2000);
}

void loop() {
  if(!mqtt.connected()) conectar();
  mqtt.loop();

  if(millis() - ultimoEnvio >= INTERVALO) {
    ultimoEnvio = millis();
    float t=dht.readTemperature(), u=dht.readHumidity();
    if(isnan(t)||isnan(u)) return;

    // Publicar dados do sensor
    StaticJsonDocument<128> dados;
    dados["temp"]=t; dados["umid"]=u; dados["dispositivo"]=THING_NAME;
    char buf[128]; serializeJson(dados, buf);
    mqtt.publish(TOP_DADOS, buf);

    // Reportar estado no Shadow
    StaticJsonDocument<128> shadow;
    shadow["state"]["reported"]["temp"]=t;
    shadow["state"]["reported"]["umid"]=u;
    shadow["state"]["reported"]["led"]=ledLigado?"ON":"OFF";
    serializeJson(shadow, buf);
    mqtt.publish(TOP_SHADOW_UPDATE, buf);

    Serial.printf("Publicado: T=%.1f U=%.1f LED=%s\n",t,u,ledLigado?"ON":"OFF");
  }
}
A primeira conexão TLS pode levar 5–15 segundos no ESP32 — a handshake mTLS exige troca de certificados e geração de chaves de sessão, que são operações pesadas para um MCU a 240MHz. Conexões subsequentes (reconexão) são mais rápidas. Aumente o timeout do PubSubClient se necessário: mqtt.setSocketTimeout(30).

Exercício 01 — Criar uma Rule para S3

No console AWS IoT Core, crie uma Rule que salva cada mensagem publicada em sensor/dados como um objeto JSON no S3. O nome do objeto deve incluir o timestamp: use a função SQL timestamp() na projeção. Verifique no S3 que os objetos estão chegando após 2 ciclos de publicação do ESP32. Qual seria o custo mensal estimado no free tier para 1 dispositivo publicando a cada 10 segundos por 30 dias?

Console AWS + Cálculo
Cálculo de custo
  • 1 mensagem a cada 10s × 60s × 60min × 24h × 30 dias = 259.200 mensagens/mês.
  • AWS IoT Core free tier: 250.000 mensagens/mês. Excesso: ~9.200 × $0,000001 = ~$0,009. Custo praticamente zero.
  • S3: cada objeto ~100 bytes. 259.200 × 100B ≈ 26MB. Free tier S3: 5GB. Custo S3 = $0.
  • SQL da Rule: SELECT *, timestamp() AS ts FROM 'sensor/dados'. Action: S3, bucket configurado, key: dados/${timestamp()}.json.

Exercício 03 — Rule com filtro e Lambda simulado

Crie uma segunda Rule no AWS IoT que filtra apenas mensagens com temperatura acima de 30°C: SELECT temp, umid FROM 'sensor/dados' WHERE temp > 30. Configure a ação para republicar em um novo tópico alertas/temperatura. No MQTT Test Client, assine alertas/# e confirme que apenas as leituras acima do threshold chegam nesse tópico. Aquece o DHT22 com a mão para disparar o alerta.

Console AWS · Rules Engine
Configuração da Rule no console
  • Rule query statement: SELECT temp, umid, timestamp() AS ts FROM 'sensor/dados' WHERE temp > 30
  • Action: Republish → tópico: alertas/temperatura · QoS 0
  • Criar uma IAM Role para que o IoT Core tenha permissão de publicar no tópico destino (o console cria automaticamente).
  • Resultado: para temperatura de 28°C, nenhuma mensagem em alertas/. Para 31°C, mensagem chega imediatamente com apenas os campos temp, umid e ts.
Exercício 04 — Desafio: visualizar no Grafana com Amazon Timestream

Crie uma Rule que envia dados do DHT22 para uma tabela no Amazon Timestream (free tier: 1GB armazenamento, 3 meses gratuitos). Instale o Grafana localmente (ou use Grafana Cloud gratuito), adicione o datasource Timestream e crie um painel com gráfico de linha de temperatura ao longo do tempo. Esse é o stack padrão de visualização IoT em produção da AWS. Documente com um print do painel funcionando.

Desafio · Stack completo
Roteiro do desafio
  • Timestream: AWS Console → Timestream → Create database (Standard) → Create table (ex: sensores_iot). Configurar retention: in-memory 1h, magnetic 7 dias.
  • Rule Timestream: SELECT temp, umid FROM 'sensor/dados' → Action: Timestream → database/table configurados → Dimensions: dispositivo=${dispositivo}.
  • Grafana datasource: instalar plugin Amazon Timestream → configurar com credenciais IAM → criar query: SELECT time, measure_value::double FROM sensores_iot.sensores_iot WHERE measure_name='temp' ORDER BY time DESC LIMIT 100.
  • IAM: criar usuário com policy AmazonTimestreamFullAccess (ou ReadOnlyAccess para Grafana).
