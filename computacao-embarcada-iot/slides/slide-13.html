07
I2C · SPI · UART
  • I2C: 2 fios (SDA+SCL), endereço 7-bit, pull-ups 4.7kΩ. Uno: A4=SDA, A5=SCL. ESP32: GPIO21/22.
  • SPI: 4 fios (MOSI, MISO, SCK, CS), 1 CS por slave, full-duplex, até 80MHz.
  • UART: 2 fios (TX→RX cruzado), assíncrono, mesmo baud rate obrigatório.
  • OLED SSD1306: endereço 0x3C, biblioteca Adafruit. display.display() é obrigatório para enviar o buffer.
Scanner I2COLED buffer
08
ESP32 · Wi-Fi · HTTP
  • ESP32: 2 cores 240MHz, 520KB RAM, GPIOs em 3.3V — não 5V tolerante. GPIO 6–11 reservados.
  • WiFi.begin() → loop WL_CONNECTED → WiFi.localIP().
  • HTTPClient: GET busca dados, POST envia. http.end() obrigatório sempre.
  • ArduinoJson: StaticJsonDocument + serializeJson para montar JSON; deserializeJson para parsear.
3.3Vhttp.end()
09
MQTT
  • Pub/Sub: Publisher e Subscriber não se conhecem — o Broker roteia. Mosquitto porta 1883.
  • Tópicos hierárquicos: casa/sala/temp. Wildcards: + (1 nível), # (todos abaixo).
  • QoS 0=dispara/esquece · QoS 1=pelo menos 1x · QoS 2=exatamente 1x.
  • Retain: broker guarda última mensagem. LWT: broker publica "offline" se cair. mqtt.loop() obrigatório.
mqtt.loop()retain/LWT

BLE · LoRa · Zigbee
  • BLE: GATT (Serviço→Característica), meses de bateria, 10–100m. Advertising = beacon sem conexão.
  • LoRa: CSS, 2–15km, 250bps–50kbps. SF↑=mais alcance/menos taxa. Duty cycle 1%.
  • LoRaWAN: LoRa + rede. End Node → Gateway → Network Server → App. Classes A/B/C.
  • Zigbee: mesh 2.4GHz. Coordinator + Router (sempre ligado) + End Device (bateria).

Edge AI
  • 3 camadas: Edge (<10ms) · Fog (1–100ms) · Cloud (100ms–2s).
  • 4 drivers para processar na borda: latência, privacidade, offline, banda.
  • Pipeline Edge AI: treinar na nuvem (float32) → converter → quantizar int8 → deploy no MCU.
  • Quantização int8: modelo 4× menor, inferência 4–10× mais rápida, perda <2% de acurácia.

TinyML
  • TFLite Micro: sem OS, sem heap dinâmico, RAM mínima ~20KB. Modelo fica na flash como array .h.
  • Tensor Arena: bloco de RAM estático para tensores. AllocateTensors() falha se pequeno demais.
  • Ciclo: GetModel → Interpreter → AllocateTensors (setup). Input → Invoke → Output (loop).
  • Quantização: int8 = round(float / scale) + zero_point. alignas(8) no array do modelo.

Q1Semana 7
Você conecta dois módulos OLED SSD1306 idênticos no mesmo barramento I2C sem alterar nenhuma configuração. O que acontece?
A
Ambos funcionam normalmente — o Arduino alterna entre os dois automaticamente
B
Conflito de barramento — ambos respondem em 0x3C simultaneamente corrompendo os dados; nenhum funciona corretamente
C
O segundo OLED ignora os comandos pois o I2C só suporta um dispositivo por barramento
D
O Arduino detecta automaticamente o conflito e usa SPI em vez de I2C
✅ Dois dispositivos com o mesmo endereço I2C causam conflito. Ambos respondem quando o master chama 0x3C, gerando colisão no barramento. Para usar dois OLEDs iguais, um deles precisa ter o endereço alterado para 0x3D via pino SA0.
Q2Semana 7
No protocolo UART, qual é a consequência de configurar baud rates diferentes nos dois dispositivos (ex: 9600 de um lado e 115200 do outro)?
A
A comunicação fica mais lenta, usando o menor dos dois baud rates
B
Um erro é gerado e a transmissão para automaticamente
C
Os dados recebidos ficam corrompidos — caracteres aleatórios, pois o receptor amostra os bits no tempo errado
D
Nada — UART é assíncrono e não depende de clock, portanto o baud rate não importa
✅ UART é assíncrono — sem clock compartilhado. A sincronização depende de ambos os lados usarem exatamente o mesmo baud rate. Com baud rates diferentes, o receptor amostra os bits nos momentos errados, resultando em dados completamente corrompidos — típico sintoma: lixo no Serial Monitor.
Q3Semana 8
Um aluno conecta a saída de 5V de um Arduino Uno diretamente ao GPIO4 de um ESP32 para enviar um sinal digital. O que provavelmente acontecerá?
A
Funciona normalmente — o ESP32 detecta 5V como HIGH sem problema
B
O GPIO do ESP32 pode ser danificado permanentemente — os pinos operam em 3.3V e não são tolerantes a 5V
C
O ESP32 converte automaticamente 5V para 3.3V internamente
D
O sinal é ignorado — o ESP32 só aceita sinais de outros ESP32
✅ Os GPIOs do ESP32 operam em 3.3V e NÃO são tolerantes a 5V. Aplicar 5V diretamente pode ultrapassar a tensão máxima do pino e causar dano permanente ao chip. Para comunicação entre Arduino (5V) e ESP32 (3.3V), use um divisor de tensão ou level shifter.
Q4Semana 8
O HTTPClient retorna código -1 ao tentar fazer um POST para http://192.168.1.50:5000/dados. O Wi-Fi está conectado. Qual NÃO é uma causa possível?
A
O servidor Flask não está em execução no endereço 192.168.1.50
B
O firewall do PC está bloqueando a porta 5000
C
O JSON enviado no body está malformado
D
O ESP32 e o PC estão em redes diferentes (ESP32 no hotspot do celular, PC no Wi-Fi da sala)
✅ JSON malformado não causa código -1. O código -1 (HTTP_CODE_ERROR) significa que a conexão TCP nem chegou a ser estabelecida — o ESP32 não encontrou o servidor. JSON malformado causaria HTTP 400 (Bad Request), pois o servidor recebe a requisição mas rejeita o conteúdo. As opções A, B e D impedem a conexão TCP.
Q5Semana 9
Um dashboard IoT conecta ao broker MQTT e assina o tópico sensor/+/temperatura. Quais mensagens ele receberá?
A
Apenas sensor/sala/temperatura — o + só substitui "sala"
B
sensor/sala/temperatura, sensor/quarto/temperatura, sensor/jardim/temperatura — qualquer tópico com exatamente 3 níveis em que o 3° é "temperatura"
C
Todos os tópicos que começam com "sensor/" — o + substitui tudo que vem depois
D
sensor/sala/temperatura e sensor/sala/sub/temperatura — o + aceita múltiplos níveis
✅ O wildcard + substitui exatamente um nível. sensor/+/temperatura recebe qualquer tópico com 3 níveis onde o primeiro é "sensor" e o terceiro é "temperatura". NÃO recebe sensor/sala/sub/temperatura (que tem 4 níveis). Para múltiplos níveis use # (mas sempre no final).
Q6Semana 9
Um sensor publica a cada 60 segundos com retain=true. Um novo cliente assina o tópico 30 segundos após a última publicação. O que ele recebe imediatamente?
A
Nada — precisa esperar a próxima publicação do sensor
B
A última mensagem publicada com retain=true — o broker entrega imediatamente ao novo assinante
C
O histórico completo de todas as mensagens publicadas até agora
D
Uma mensagem de erro informando que o tópico está vazio
✅ Retained message: o broker armazena a ÚLTIMA mensagem de um tópico com retain=true. Todo novo assinante recebe essa última mensagem imediatamente ao se inscrever, sem esperar a próxima publicação. Isso não é um histórico completo — apenas a última mensagem.
Q7Semana 10
Um engenheiro precisa instalar 200 sensores de temperatura em uma fazenda de 800 hectares. Cada sensor envia 1 leitura por hora, alimentado por bateria, sem infraestrutura de rede disponível. Qual protocolo é mais adequado?
A
Wi-Fi — instalar roteadores nos pontos mais altos da fazenda
B
BLE — os sensores fazem relay entre si cobrindo a fazenda toda
C
LoRaWAN — km de alcance com 1 gateway, 1 mensagem/hora cabe no duty cycle de 1%, bateria de anos com Classe A
D
Zigbee mesh — os sensores formam a rede entre si sem gateway
✅ LoRaWAN é a escolha correta. O alcance de km cobre a fazenda com 1 ou poucos gateways. 1 mensagem/hora × 200 sensores está dentro do duty cycle. Bateria de anos com deep sleep entre transmissões (Classe A). BLE alcança 100m por salto e Zigbee precisa de Routers com alimentação constante. Wi-Fi consome energia demais.
Q8Semana 10
Qual é a diferença fundamental entre LoRa e LoRaWAN?
A
LoRa é mais rápido, LoRaWAN é mais lento — a diferença é apenas de velocidade
B
LoRa é a camada física (modulação de rádio). LoRaWAN é o protocolo de rede que roda sobre LoRa — define autenticação, gateways, network server e classes de dispositivo
C
LoRaWAN é o nome europeu para LoRa — são a mesma tecnologia com nomes regionais diferentes
D
LoRa usa 915MHz, LoRaWAN usa 868MHz — a diferença é apenas de frequência
✅ LoRa é a tecnologia de rádio (camada física), LoRaWAN é a rede. Analogia: LoRa é como o cabo Ethernet (meio físico), LoRaWAN é o protocolo TCP/IP que roda sobre ele. Você pode usar LoRa sem LoRaWAN (ponto a ponto), mas LoRaWAN sempre requer LoRa como camada física.
Q9Semana 11
Um hospital quer usar visão computacional para pré-triagem de raio-X de pacientes. Os dados são imagens médicas sigilosas (LGPD). A rede do hospital cai frequentemente. Onde o processamento deve ocorrer?
A
Nuvem pública (AWS/Azure) — mais poder de processamento e modelos mais precisos
B
Borda/Fog local — dois drivers forçam a borda: privacidade (LGPD proíbe dados médicos em nuvem pública) e resiliência offline (rede instável)
C
Nuvem privada — desde que seja privada, LGPD não se aplica
D
Não é viável — visão computacional requer GPU e não pode rodar na borda
✅ Dois drivers forçam processamento na borda: privacidade (imagens de saúde não podem sair do perímetro hospitalar segundo LGPD/CFM) e resiliência offline (rede instável). Hardware como NVIDIA Jetson Orin permite inferência de visão computacional de alto desempenho na borda sem internet.
Q10Semana 11
Por que a quantização int8 é essencial para rodar modelos de ML em microcontroladores como o ESP32?
A
Porque int8 é mais preciso que float32 para dados de sensores
B
Porque a biblioteca TFLite Micro só aceita o tipo int8 — float32 não é suportado
C
Reduz o modelo ~4× (1 byte vs 4 bytes por peso), cabe na RAM limitada do MCU e acelera a inferência em hardware sem FPU — com perda de acurácia geralmente menor que 2%
D
Para reduzir o consumo de energia — int8 gasta menos energia por operação que float32
✅ Quantização int8 resolve os dois principais gargalos do MCU: memória e velocidade. Float32 usa 4 bytes por peso, int8 usa 1 byte — redução de 4×. No ESP32 sem FPU dedicada, operações inteiras são muito mais rápidas que ponto flutuante. A perda de acurácia é geralmente <2% para modelos de classificação.
Q11Semana 12
O código chama interpreter->AllocateTensors() e recebe kTfLiteError. Qual é a causa mais provável e como corrigir?
A
O modelo .tflite está corrompido — exportar novamente do Python
B
A versão do esquema TFLite é incompatível — atualizar a biblioteca
C
A tensor arena é pequena demais — aumentar o valor de ARENA_SIZE até que AllocateTensors() retorne kTfLiteOk
D
O ESP32 não tem RAM suficiente para TFLite Micro — usar Raspberry Pi
✅ AllocateTensors() falha quando a tensor arena é insuficiente. O TFLM precisa alocar todos os tensores (entrada, intermediários, saída) dentro da arena. Aumente ARENA_SIZE gradualmente (4096 → 8192 → 16384). Para calibrar: após uma inferência bem-sucedida, imprima interpreter->arena_used_bytes() e use esse valor + margem como ARENA_SIZE mínimo.
Q12Semana 12
Um valor float normalizado de -0.5 precisa ser quantizado para int8. O tensor de entrada tem scale=0.004 e zero_point=0. Qual é o valor int8 correto?
A
-50 — multiplicar por 100
B
0 — valores negativos normalizados mapeiam para 0 em int8
C
-125 — fórmula: round(-0.5 / 0.004) + 0 = round(-125) = -125, dentro do range int8 [-128, 127]
D
-128 — todos os valores negativos mapeiam para -128
✅ Fórmula: int8 = round(float_value / scale) + zero_point. round(-0.5 / 0.004) + 0 = round(-125.0) + 0 = -125. Com zero_point=0, o mapeamento é linear e centrado. -125 está dentro do range int8 válido [-128, 127]. A operação inversa (desquantização): float = (int8_val - zero_point) × scale = (-125 - 0) × 0.004 = -0.5 ✓
