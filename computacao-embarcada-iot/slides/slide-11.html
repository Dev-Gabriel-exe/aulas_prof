Edge Computing — a arquitetura em camadas e os 4 drivers
O problema que Edge Computing resolve

Por anos o modelo dominante foi simples: sensores coletam dados, enviam tudo para a nuvem, a nuvem processa e responde. Funciona bem quando a conexão é estável, a latência não importa e privacidade não é problema. Mas esse modelo quebra em um número crescente de cenários reais.

Imagine uma câmera de segurança industrial que precisa detectar um operador sem capacete e travar uma máquina em menos de 100ms. Nesse tempo, um pacote não vai à AWS, é processado e volta — a física não permite. Ou um veículo autônomo que não pode depender de conexão 4G para decidir se freia. Ou um hospital que não pode enviar imagens médicas de pacientes para servidores externos por questões de LGPD. Nesses casos, o processamento precisa acontecer na borda — no próprio dispositivo ou numa máquina próxima.

Os 4 drivers — por que mover processamento para a borda

A decisão entre processar na borda ou na nuvem é sempre um trade-off. Quatro fatores técnicos definem quando a borda vence:

Latência
Quando a decisão precisa acontecer em milissegundos. A física da rede não permite ir à nuvem e voltar nesse tempo.
Privacidade
Dados sensíveis — imagens médicas, biometria, dados pessoais — não podem sair do perímetro por regulação (LGPD, HIPAA) ou contrato.
Resiliência offline
O sistema precisa funcionar mesmo sem internet — minas, plataformas offshore, veículos em área rural, fábricas com rede própria.
Largura de banda
Câmeras 4K, sensores de vibração a 10kHz — enviar tudo para a nuvem custa caro e satura a rede. Processar na borda e enviar só eventos reduz 100× o tráfego.

Edge AI — inferência na borda, quantização e ferramentas
Edge AI — o que muda quando o modelo vai para a borda

Em Machine Learning clássico, o modelo roda em um servidor com GPUs, memória abundante e energia ilimitada. Em Edge AI, o modelo roda em um ESP32 com 520KB de RAM, sem sistema operacional completo, alimentado por bateria. Isso muda tudo — não em termos de teoria, mas de prática de engenharia.

O pipeline não muda de ponta a ponta: você ainda coleta dados, treina, avalia e itera. O que muda é a etapa de deployment: antes de carregar no dispositivo, o modelo precisa ser comprimido, convertido e otimizado para o hardware-alvo.

O modelo treinado na nuvem (float32, centenas de MB) precisa ser convertido e comprimido para caber na memória flash do dispositivo de borda (dezenas de KB).
Quantização — o que é e por que é essencial

Redes neurais treinadas em float32 (4 bytes por peso) são grandes demais para microcontroladores. A quantização converte os pesos para int8 (1 byte) ou int16, reduzindo o tamanho do modelo em 4× e acelerando a inferência em hardware sem FPU — sem perda significativa de acurácia na maioria dos casos.

Regra prática: quantização int8 reduz tamanho e latência em ~4× com perda de acurácia menor que 1–2% na maioria dos modelos de classificação e detecção de anomalias. Para ESP32 (sem unidade de ponto flutuante dedicada), int8 é 10× mais rápido que float32 em operações matriciais.

Exercício 01 — Gabaritos dos cenários

Após a discussão em grupo, anote a decisão que você considera correta para cada um dos 6 cenários (A–F) e os principais argumentos técnicos. Para os cenários onde você discordou do gabarito, escreva qual argumento mudaria sua resposta e qual requisito você não havia considerado inicialmente.

Conceito · Registro
Gabaritos comentados
  • A — Híbrido: lógica de tarifa e cálculo local no medidor (fog/borda), envio diário para servidor da concessionária (nuvem). Latência não importa; privacidade moderada; volume pequeno.
  • B — Borda pura: latência crítica (<500ms) + offline subterrâneo. Regra simples de threshold no ESP32 — não precisa de ML. Nunca depender de rede para salvar vida.

Exercício 02 — Calculando viabilidade de banda

Uma câmera de inspeção industrial gera frames de 1280×720 pixels em RGB (3 bytes/pixel) a 15fps. (a) Calcule o volume de dados brutos por hora em GB. (b) Supondo que a detecção de defeitos na borda reduz o envio para apenas os frames com defeito detectado, e que 2% das peças têm defeito, calcule o volume por hora após o filtro. (c) Qual é a redução percentual? Justifique por que essa redução torna a arquitetura Edge+Cloud viável onde Cloud-only não seria.

Cálculo + Análise
Resolução
  • (a) Bruto: 1280 × 720 × 3 bytes = 2.764.800 bytes/frame ≈ 2.76 MB/frame. A 15fps: 2.76 × 15 = 41.4 MB/s = 149 GB/hora. Para 24h = 3.574 GB/dia por câmera. Inviável.
  • (b) Com filtro edge (2% defeitos): 149 GB × 0,02 = 2,98 GB/hora = 71,6 GB/dia. Ainda alto, mas 50× menor.
  • (c) Redução: 98%. Mas o real ganho é maior — na prática não se envia o frame inteiro, apenas a região de interesse (bounding box) e metadados: pode chegar a 99,9% de redução. Uma linha de 1Gbps suporta centenas de câmeras com o modelo edge.
  • Na arquitetura Cloud-only com 1Gbps de uplink: 149 GB/hora = ~333 Mbps — saturaria uma linha de 1Gbps com apenas 3 câmeras. Com edge filtering, 1Gbps comporta centenas de câmeras.
Exercício 03 — Esboçando uma arquitetura híbrida

Projete a arquitetura de um sistema de monitoramento de qualidade do ar para uma cidade de médio porte com 50 sensores espalhados. Cada sensor mede CO₂, PM2.5, temperatura e umidade a cada 1 minuto. Os requisitos são: (a) alerta local imediato se CO₂ ultrapassar 1000ppm; (b) dashboard público online com dados em tempo real; (c) histórico de 2 anos para análise epidemiológica; (d) os dados são públicos — sem restrição de privacidade. Descreva quais tarefas ocorrem na borda (sensor), na fog (servidor municipal) e na nuvem.

Projeto de Arquitetura
Arquitetura sugerida
  • Borda (sensor ESP32): leitura dos sensores, threshold de CO₂ >1000ppm → LED/buzzer local imediato (latência <100ms), publicação via MQTT a cada 60s para o servidor municipal.
  • Fog (servidor municipal): broker MQTT local, agrega dados dos 50 sensores, persiste últimas 24h em banco local (resiliência se internet cair), calcula índice de qualidade do ar por zona, repassa para nuvem via HTTPS a cada 5 min.
  • Nuvem: banco de dados time-series (InfluxDB/TimescaleDB), API pública REST, dashboard Grafana, histórico de 2 anos, modelos de ML para previsão de qualidade do ar.
  • O driver de latência exige borda (alerta imediato). O volume pequeno (50 sensores × 4 variáveis) permite nuvem sem filtro. A resiliência justifica o fog como buffer intermediário.
Exercício 04 — Desafio: detectar anomalia no ESP32

Implemente no ESP32 um detector de anomalias simples usando z-score (sem modelo de ML): leia o DHT22 a cada 5 segundos e mantenha uma janela deslizante das últimas 10 leituras de temperatura. Calcule a média e o desvio padrão da janela. Se a leitura atual estiver a mais de 2 desvios padrões da média, classifique como "ANOMALIA" e publique via MQTT no tópico borda/anomalia com um JSON contendo o valor, a média e o z-score. Isso simula o padrão básico de Edge AI para detecção de anomalias em sensores industriais.

Desafio · Implementação
Arduino C — z-score anomaly detection no ESP32
#include <WiFi.h>
#include <PubSubClient.h>
#include <ArduinoJson.h>
#include <DHT.h>
#include <math.h>

#define JANELA     10
#define Z_THRESH   2.0f
#define PINO_DHT   4

DHT dht(PINO_DHT, DHT22);
WiFiClient wc; PubSubClient mqtt(wc);
float janela[JANELA]; int idx = 0; bool cheia = false;

float media() {
  int n = cheia ? JANELA : idx;
  float s = 0;
  for(int i=0;i<n;i++) s += janela[i];
  return n ? s/n : 0;
}

float desvio(float m) {
  int n = cheia ? JANELA : idx;
  if(n < 2) return 0;
  float var = 0;
  for(int i=0;i<n;i++) var += (janela[i]-m)*(janela[i]-m);
  return sqrt(var/n);
}

void setup() {
  Serial.begin(115200);
  dht.begin();
  // conectar Wi-Fi e MQTT conforme semanas anteriores...
  delay(2000);
}

void loop() {
  mqtt.loop();
  float t = dht.readTemperature();
  if(isnan(t)) { delay(5000); return; }

  // Insere na janela deslizante
  janela[idx] = t;
  idx = (idx + 1) % JANELA;
  if(idx == 0) cheia = true;
  if(!cheia) { delay(5000); return; }

  float m = media();
  float s = desvio(m);
  float z = (s > 0) ? abs(t - m) / s : 0;

  Serial.printf("T=%.1f M=%.1f S=%.2f Z=%.2f %s\n",
    t, m, s, z, z > Z_THRESH ? "*** ANOMALIA ***" : "normal");

  if(z > Z_THRESH && mqtt.connected()) {
    StaticJsonDocument<128> doc;
    doc["valor"] = t; doc["media"] = m; doc["zscore"] = z;
    char buf[128]; serializeJson(doc, buf);
    mqtt.publish("borda/anomalia", buf);
  }
  delay(5000);
}
