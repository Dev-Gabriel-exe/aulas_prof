Revisão Express — Semanas 1 a 5
5–8 minutos por semana. O objetivo não é re-ensinar — é garantir que os conceitos-chave estão frescos antes do simulado. Verbalize em voz alta, não apenas leia.
Semana 1
Computação Embarcada, IoT e Edge AI
  • Sistema embarcado = computador dedicado a uma função específica
  • Palavra-chave: dedicado — não executa qualquer software
  • MCU tem CPU + RAM + Flash + periféricos no mesmo chip
  • MPU tem apenas a CPU — depende de chips externos para RAM e armazenamento
  • Arquitetura IoT: Sensor → MCU → Gateway → Nuvem → Dashboard
  • Edge AI: inferência no dispositivo — sem internet, sem latência, sem exposição de dados
  • TinyML: modelos de ML rodando em <1 MB de RAM
Semana 2
Microcontroladores: Arduino, ESP32 e Pico
  • Arduino Uno: ATmega328P, 16MHz, 2KB RAM, 32KB Flash — sem Wi-Fi
  • ESP32: dual-core 240MHz, 520KB RAM, Wi-Fi + BLE — tensão lógica 3.3V
  • Raspberry Pi Pico: RP2040, 133MHz, 264KB RAM — C/C++ ou MicroPython
  • ATENÇÃO: ESP32 = 3.3V. Arduino = 5V. Misturar sem divisor de tensão = queima
  • Pinout: cada pino tem número, nome e função (pode ser digital, PWM, analógico, I2C)
  • Datasheet: Absolute Maximum Ratings = limites que nunca podem ser ultrapassados
  • setup() roda uma vez · loop() roda para sempre · LED_BUILTIN = pino 13 no Uno
Semana 3
Programação: Tipos, Digital, Analógico, PWM
  • unsigned long para millis() — int estouraria em 32 segundos
  • byte para valores 0–255 (PWM) · float para temperaturas com decimal
  • INPUT_PULLUP: lógica invertida — HIGH = solto, LOW = pressionado
  • Pino flutuante sem pull-up oscila aleatoriamente — use INPUT_PULLUP ou resistor externo
  • analogRead(A0) → 0 a 1023 (10-bit ADC, 0–5V) · resolução: ~4.9mV por passo
  • analogWrite(pino, 0–255): apenas pinos com ~ · 255 = 100% duty cycle
  • Debounce por software com millis(): aguarda 50ms de estabilidade antes de processar
  • delay() é bloqueante — use millis() para multitarefa sem travamento
Semana 4
Sensores: DHT22 e HC-SR04
  • DHT22: protocolo 1-Wire, precisão ±0.5°C, resolução 0.1°C — melhor que DHT11
  • Pull-up 10kΩ entre DATA e VCC — obrigatório, sem ele retorna NaN
  • Sempre verificar isnan() antes de usar a leitura do DHT22
  • Mínimo 2 segundos entre leituras — respeite o tempo de conversão
  • HC-SR04: TRIG = pulso 10µs · ECHO = tempo de retorno em µs
  • Distância (cm) = pulseIn(ECHO, HIGH) × 0.034 ÷ 2
  • Divide por 2 porque o som percorre ida E volta
  • Calibração: offset = referência − sensor · média de 10 amostras reduz ruído
Semana 5
Atuadores: Servo, Motor DC, Relé e o Ciclo Completo
  • Servo: biblioteca Servo.h · write(0–180) · PWM 50Hz interno · não use analogWrite()
  • Servo SG90: pico de corrente até 700mA — fonte externa para servos maiores
  • Motor DC: precisa de driver L298N (ponte H) — GPIO não fornece corrente suficiente
  • L298N: IN1/IN2 = direção · ENA com PWM = velocidade · IN1=IN2=HIGH = freio elétrico
  • Back-EMF: tensão reversa ao desligar motor — módulos L298N prontos têm diodos de proteção
  • Relé ativo-baixo: HIGH = desligado · LOW = ligado · inicialize com HIGH no setup()
  • Relé: COM + NO para ligar algo quando Arduino ativa · NC para o inverso
  • Histerese: zona morta em torno do limiar evita liga/desliga rápido (ex: ±0.5°C)
  • Ciclo completo: Sensor → Lógica (if/else) → Atuador → feedback contínuo
  • millis() para todas as tarefas simultâneas — ventilação + leitura + Serial sem travar

Q1
Conceito1,5 pts

Explique a diferença entre um microcontrolador e um microprocessador. Cite um exemplo de produto que usa cada um e justifique por que o componente escolhido é o mais adequado para aquele produto.

Microcontrolador (MCU): integra CPU, RAM, Flash e periféricos (ADC, PWM, UART) em um único chip. Opera com recursos limitados (KB de RAM, MHz de clock) e baixo consumo. Ideal para sistemas com função dedicada e custo/consumo restritos.

Microprocessador (MPU): contém apenas a CPU — depende de chips externos para RAM e armazenamento. Muito mais poderoso, mas caro e com consumo elevado. Requer sistema operacional para gerenciar recursos.

Exemplos válidos:
MCU — marca-passo cardíaco (ESP32, ARM Cortex-M): função única (monitorar e estimular coração), consumo em µA, bateria deve durar anos. Qualquer MCU simples resolve e é mais confiável que um sistema complexo.

MPU — smartphone (Qualcomm Snapdragon, Apple A-series): executa centenas de apps simultaneamente, toca vídeo 4K, roda modelos de IA — exige GHz e GB de RAM, justificando o MPU.
Q2
Análise de Código2,0 pts

O código abaixo tem três problemas que causarão comportamento incorreto. Identifique cada um, explique o impacto e escreva a correção.

Arduino C — com bugs
#define BTN  2
#define LED  9

int ultimoToggle = 0;
bool estadoLED = false;

void setup() {
  pinMode(BTN, INPUT);
  pinMode(LED, OUTPUT);
}

void loop() {
  if (digitalRead(BTN) == HIGH) {
    if (millis() - ultimoToggle > 50) {
      ultimoToggle = millis();
      estadoLED = !estadoLED;
      analogWrite(LED, estadoLED ? 255 : 0);
    }
  }
}
Problema 1 — int ultimoToggle = 0: int tem máximo de 32.767ms — em ~32 segundos, millis() ultrapassa esse valor e o cálculo millis() - ultimoToggle causa overflow, gerando resultado incorreto.
Correção: unsigned long ultimoToggle = 0;

Problema 2 — pinMode(BTN, INPUT) sem pull-up: sem resistor pull-up ou pull-down, o pino 2 fica flutuante quando o botão está solto — leituras aleatórias entre HIGH e LOW causarão toggles fantasmas do LED.
Correção: pinMode(BTN, INPUT_PULLUP); e ajustar a condição para == LOW (lógica invertida do INPUT_PULLUP).

Problema 3 — lógica de disparo contínua: o código dispara o toggle a cada 50ms enquanto o botão estiver pressionado — um pressionamento de 200ms geraria 4 toggles. Falta detectar apenas a borda de descida (transição de solto para pressionado).
Correção: usar variável de estado anterior e só acionar quando leitura != estadoAnterior && leitura == LOW.
Q3
Sensores1,5 pts

O HC-SR04 retornou duracao = 4118 µs. a) Calcule a distância em centímetros. b) Um colega argumenta que dividir por 2 na fórmula não é necessário porque "o sensor já considera isso". Corrija o argumento com uma explicação técnica precisa.

a) Cálculo:
Distância = 4118 × 0.034 ÷ 2 = 140.012 ÷ 2 ≈ 70 cm

b) Correção do argumento:
O HC-SR04 emite um pulso sonoro que viaja ATÉ o objeto e depois retorna ao sensor. O pulseIn(ECHO, HIGH) mede o tempo total dessa jornada completa — ida mais volta. Como queremos apenas a distância entre o sensor e o objeto (só a ida), é obrigatório dividir por 2. Se não dividir, o resultado seria 140 cm em vez de 70 cm — o dobro da distância real. O sensor NÃO faz essa divisão internamente: ele apenas gera um pulso HIGH cuja duração é proporcional ao tempo de voo completo.
Q4
Análise de Código — DHT222,0 pts

O código abaixo deveria ler temperatura com DHT22 e acender um LED quando a temperatura passar de 28°C, mas apresenta comportamento errático. Identifique os dois problemas e corrija.

Arduino C — com bugs
#include <DHT.h>
DHT dht(2, DHT22);

void setup() {
  Serial.begin(9600);
  dht.begin();
  pinMode(13, OUTPUT);
}

void loop() {
  float temp = dht.readTemperature();

  if (temp > 28.0) {
    digitalWrite(13, HIGH);
  } else {
    digitalWrite(13, LOW);
  }

  Serial.println(temp);
  delay(500);
}
Problema 1 — delay(500) viola o tempo mínimo do DHT22: o DHT22 precisa de mínimo 2 segundos entre leituras consecutivas. Com delay(500), o sensor é consultado a cada 500ms — ele não consegue completar a conversão interna e retorna NaN frequentemente. O LED pode acender ou apagar de forma aleatória baseado em NaN.
Correção: alterar para delay(2000) ou, melhor ainda, usar millis() com intervalo de 3000ms.

Problema 2 — ausência de verificação isnan(): quando a leitura falha (NaN), a comparação NaN > 28.0 retorna false no AVR — o LED apaga mesmo que a temperatura real seja alta. O Serial Monitor exibirá "nan" sem aviso de erro.
Correção: adicionar antes do if:
Correção
if (isnan(temp)) {
  Serial.println("ERRO: leitura inválida — verifique pull-up");
  delay(2000);
  return;
}
Q5
Análise1,5 pts

Um sistema de irrigação automática usa um relé para ligar uma bomba d'água quando a umidade do solo cai abaixo de 30%. Sem histerese, a bomba liga e desliga centenas de vezes por minuto quando a umidade está exatamente em 30%. a) Explique tecnicamente por que isso acontece. b) Descreva como implementar histerese para resolver o problema, com valores concretos.

a) Por que acontece (efeito de chatterring):
O sensor de umidade tem ruído intrínseco — mesmo com o solo em 30%, as leituras variam entre 29.8% e 30.2% entre amostras. Sem histerese, a condição umidade < 30 ora é verdadeira (29.8%) ora é falsa (30.2%), fazendo o relé comutar a cada ciclo do loop. Relés não foram projetados para esse uso — a bobina superaquece e os contatos mecânicos se desgastam em minutos.

b) Implementação de histerese com valores concretos:
Define dois limiares distintos: liga a bomba quando umidade cai abaixo de 28% e desliga apenas quando umidade sobe acima de 35%. A zona morta entre 28% e 35% garante que o relé não comuta enquanto o solo está nessa faixa.
Arduino C — histerese
#define LIGA_ABAIXO   28   // liga bomba quando umidade < 28%
#define DESLIGA_ACIMA 35   // desliga quando umidade > 35%

bool bombaLigada = false;

void controlarBomba(float umidade) {
  if (!bombaLigada && umidade < LIGA_ABAIXO) {
    digitalWrite(RELE, LOW);   // liga
    bombaLigada = true;
  }
  else if (bombaLigada && umidade > DESLIGA_ACIMA) {
    digitalWrite(RELE, HIGH);  // desliga
    bombaLigada = false;
  }
  // Na zona 28–35%: mantém estado atual
}
Q6
Código Completo2,0 pts

Escreva o código Arduino completo que: (1) lê a distância com HC-SR04 (TRIG=9, ECHO=10) a cada 300ms usando millis(); (2) controla a velocidade de um motor DC via L298N (ENA=6, IN1=4, IN2=5) proporcionalmente à distância — quanto mais perto o objeto, mais devagar o motor; (3) para o motor completamente quando o objeto está a menos de 10 cm; (4) imprime distância e velocidade no Serial Monitor.

Arduino C — gabarito Q6
#define TRIG 9
#define ECHO 10
#define ENA  6
#define IN1  4
#define IN2  5
#define DIST_PARAR   10    // cm — para o motor
#define DIST_MAX    100    // cm — velocidade máxima

unsigned long ultimaMedicao = 0;

long lerDistancia() {
  digitalWrite(TRIG, LOW);  delayMicroseconds(2);
  digitalWrite(TRIG, HIGH); delayMicroseconds(10);
  digitalWrite(TRIG, LOW);
  return pulseIn(ECHO, HIGH, 30000) * 0.034 / 2;
}

void setup() {
  Serial.begin(9600);
  pinMode(TRIG, OUTPUT); pinMode(ECHO, INPUT);
  pinMode(ENA, OUTPUT);
  pinMode(IN1, OUTPUT);  pinMode(IN2, OUTPUT);
  // Motor sempre para frente
  digitalWrite(IN1, HIGH); digitalWrite(IN2, LOW);
}

void loop() {
  if (millis() - ultimaMedicao >= 300) {
    ultimaMedicao = millis();
    long dist = lerDistancia();

    int velocidade;
    if (dist <= DIST_PARAR || dist == 0) {
      velocidade = 0;
    } else {
      // Mais longe = mais rápido · constrain limita 0–255
      velocidade = constrain(
        map(dist, DIST_PARAR, DIST_MAX, 0, 255),
        0, 255
      );
    }
    analogWrite(ENA, velocidade);

    Serial.print("Dist: "); Serial.print(dist);
    Serial.print("cm | Vel PWM: ");
    Serial.println(velocidade);
  }
}
Ponto crítico: constrain() garante que map() nunca retorne valor fora de 0–255, mesmo para distâncias extremas. millis() permite que outras tarefas sejam adicionadas sem travar o loop.
Q7
Prática LAB3,5 pts

Descreva em detalhes como montar e programar um sistema de ventilação automática usando DHT22 e relé. Inclua: (a) esquema de ligação com todos os pinos e componentes necessários; (b) a lógica de controle com três estados e histerese; (c) por que usar millis() em vez de delay(); (d) o que aconteceria se o resistor pull-up do DHT22 fosse esquecido.

a) Ligação:
DHT22: VCC→5V, DATA→pino 8 + resistor 10kΩ entre DATA e 5V, GND→GND.
Relé: VCC→5V, GND→GND, IN→pino 7. Ventilador entre COM e NO do relé.
LEDs indicadores (opcional): pinos 2 (verde), 3 (amarelo), 4 (vermelho) com resistores 220Ω.

b) Lógica com histerese:
Estado 0 (Frio, <25°C): relé HIGH (desligado), LED verde.
Estado 1 (Morno, 25–30°C): relé LOW (ligado), LED amarelo.
Estado 2 (Quente, >30°C): relé LOW (ligado, máxima), LED vermelho.
Histerese ±0.5°C: só muda de estado quando ultrapassa limiar + 0.5°C ou cai abaixo do limiar - 0.5°C.

c) Por que millis() e não delay():
delay(3000) trava o microcontrolador por 3 segundos — nenhum outro código executa nesse período. Com millis(), o loop continua rodando centenas de vezes por segundo, permitindo verificar botões, atualizar displays e responder a eventos simultaneamente. Sistemas IoT reais precisam de múltiplas tarefas concorrentes.

d) Se o pull-up for esquecido:
O pino DATA fica em estado indefinido (flutuante) entre leituras. O protocolo 1-Wire do DHT22 requer que o barramento seja puxado para HIGH pelo pull-up quando está em repouso. Sem isso, o sinal é corrompido e readTemperature() retorna NaN. Sem verificação isnan(), o sistema pode tomar decisões com base em NaN — o relé pode ligar ou desligar de forma imprevisível.
