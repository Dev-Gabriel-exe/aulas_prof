Os limites do FastAPI para serving de ML em escala

O FastAPI que construímos na Semana 9 é sólido e serve bem a maioria dos casos — até o momento em que a escala aumenta. Quando o volume de requisições cresce, começam a aparecer problemas que o FastAPI por si só não resolve.

  • Sem batching dinâmico: cada requisição chama model.predict() individualmente. Se chegarem 100 requisições simultâneas, o modelo é chamado 100 vezes — mesmo que processar um batch de 100 de uma vez fosse muito mais eficiente.
  • Sem otimização de GPU: o FastAPI não tem mecanismo nativo para aproveitar GPU de forma eficiente, gerenciar filas de inferência ou fazer pin de memória entre requisições.
  • Empacotamento manual: empacotar o modelo + código + dependências + Dockerfile requer esforço manual a cada novo modelo. Não há padronização de como salvar, versionar e servir.
  • Sem suporte nativo a múltiplos modelos: servir um pipeline com preprocessador + modelo principal + pós-processador exige código de orquestração manual.
A regra de ouro: FastAPI é a escolha certa para a maioria dos times começando, ou quando a lógica de negócio é complexa e você precisa de controle total. BentoML e Triton fazem sentido quando o throughput e a eficiência de inferência passam a ser gargalo.
O que é BentoML

BentoML é um framework de serving especializado para modelos de ML. O nome vem de "Bento" — a marmita japonesa que tem todos os componentes em compartimentos separados, organizados e portáteis. A analogia é exata: um Bento ML empacota modelo, runner, API e dependências em uma unidade portátil.

Os três conceitos centrais:

Runner

  • Abstração que isola o modelo do servidor HTTP
  • Gerencia o pool de workers de inferência
  • Suporta batching automático entre requisições
  • Escala independentemente da camada HTTP

Service

  • Define os endpoints da API (análogo ao FastAPI app)
  • Conecta requisições ao runner correto
  • Declarativo: tipos de entrada/saída definem o contrato
  • Gera documentação automática igual ao FastAPI

Bento

  • Artefato final: modelo + código + dependências + config
  • Imutável e versionado automaticamente
  • Pode ser convertido em imagem Docker com um comando
  • Pode ser publicado em BentoCloud ou registry próprio

bentofile.yaml

  • Define dependências Python do serviço
  • Aponta qual arquivo é o service principal
  • Configura labels, versão e metadados
  • Substitui o Dockerfile para o build do Bento

Salvando o modelo no BentoML Model Store

O BentoML tem seu próprio registro local de modelos — separado do MLflow. O primeiro passo é salvar o modelo nesse store, que vai rastrear versões automaticamente.

Python — salvar modelo no BentoML store
import bentoml
import pickle
from sklearn.ensemble import RandomForestClassifier

# Opção 1: salvar diretamente do objeto sklearn
with open("models/model.pkl", "rb") as f:
    model = pickle.load(f)

bento_model = bentoml.sklearn.save_model(
    name="classificador-churn",
    model=model,
    labels={"framework": "sklearn", "dataset": "v2"},
    metadata={"accuracy": 0.9450, "n_features": 10}
)

print(f"Modelo salvo: {bento_model.tag}")
# Output: classificador-churn:a1b2c3d4ef5678

# Listar modelos no store
for m in bentoml.models.list():
    print(m.tag, m.creation_time)
Criando o Service BentoML
Python — service.py
from __future__ import annotations
import numpy as np
import bentoml
from bentoml.io import NumpyNdarray, JSON
from pydantic import BaseModel
from typing import List

# Criar um runner a partir do modelo salvo no store
# "latest" pega a versão mais recente automaticamente
runner = bentoml.sklearn.get("classificador-churn:latest").to_runner()

# Definir o serviço com seus runners
svc = bentoml.Service("servico-churn", runners=[runner])

class InputData(BaseModel):
    features: List[float]

class OutputData(BaseModel):
    prediction: int
    probability: float

# Endpoint de predição individual
@svc.api(input=JSON(pydantic_model=InputData), output=JSON(pydantic_model=OutputData))
async def predict(data: InputData) -> OutputData:
    X = np.array(data.features).reshape(1, -1)
    # await runner.predict.async_run() — não bloqueia o event loop
    pred  = await runner.predict.async_run(X)
    proba = await runner.predict_proba.async_run(X)
    return OutputData(
        prediction=int(pred[0]),
        probability=round(float(proba[0][int(pred[0])]), 4)
    )

# Endpoint batch — aceita múltiplas instâncias
@svc.api(input=NumpyNdarray(shape=(-1, 10), dtype=np.float32), output=NumpyNdarray())
async def predict_batch(X: np.ndarray) -> np.ndarray:
    return await runner.predict.async_run(X)

bentoml containerize vs Dockerfile manual: o bentoml containerize gera uma imagem Docker otimizada automaticamente — com multi-stage build, usuário não-root e cache de layers já configurados. Para a maioria dos casos de sklearn, é mais simples e seguro que escrever o Dockerfile manualmente.
Batching dinâmico — o diferencial de performance

O batching dinâmico é a capacidade de acumular múltiplas requisições individuais e processá-las juntas em uma única chamada ao modelo. Isso é especialmente vantajoso quando a overhead de inicializar a computação é maior do que processar dados extras.

Python — configurar batching no runner
from bentoml._internal.runner.utils import Params

# Runner com batching dinâmico configurado
runner = bentoml.sklearn.get("classificador-churn:latest").to_runner(
    max_batch_size=32,          # máximo de itens por batch
    max_latency_ms=100,         # espera no máximo 100ms para completar o batch
)

# Com max_latency_ms=100:
# - Se chegarem 32 requisições em menos de 100ms → processa batch de 32
# - Se passarem 100ms sem completar 32 → processa o que tiver (ex: 7)
# Isso maximiza throughput sem sacrificar latência de forma excessiva

O trade-off do batching:

Sem batching (req individual)
~120 req/s
Batch size 8
~480 req/s
Batch size 32
~900 req/s
Batch size 128
~1100 req/s

* Valores ilustrativos com RandomForest em CPU. Ganhos reais dependem do modelo, hardware e distribuição de carga.

Custo do batching: latência individual aumenta — uma requisição que antes era processada em 5ms pode levar até max_latency_ms (100ms) esperando outras. Para aplicações com SLA de latência estrito (<10ms), o batching pode não ser adequado. Para sistemas que priorizam throughput (pipelines de scoring em lote, por exemplo), é o padrão certo.

Triton Inference Server — o que é e para que serve

O Triton é um servidor de inferência open-source da NVIDIA, otimizado para GPU mas funcional em CPU também. Ele foi projetado para cenários de alta escala onde se precisa servir múltiplos modelos simultaneamente com máxima eficiência de hardware.

  • Multi-framework: suporta modelos TensorFlow, PyTorch, ONNX, TensorRT, scikit-learn e outros com o mesmo servidor — sem precisar de um servidor dedicado por framework.
  • Concorrência de modelo: pode rodar múltiplas instâncias do mesmo modelo em paralelo, em diferentes GPUs ou na mesma GPU com particionamento de memória.
  • Batching dinâmico e sequencial: mais sofisticado que o BentoML — suporta preferred batch sizes e delay scheduler customizável.
  • Métricas nativas: expõe métricas Prometheus automaticamente — latência por percentil, throughput, utilização de GPU — sem configuração adicional.
Para esta disciplina: o Triton é apresentado conceitualmente. Para usá-lo com GPU real, é necessário hardware NVIDIA com drivers adequados. A demo prática desta semana usa BentoML, que funciona em qualquer CPU. O Triton aparece novamente na Semana 14 com Kubernetes.
Como o Triton organiza modelos — Model Repository

O Triton usa uma estrutura de diretórios chamada Model Repository. Cada modelo fica numa pasta com nome e versão, e um arquivo config.pbtxt descreve o contrato de entrada/saída.

Estrutura do Model Repository
model_repository/
│
├── classificador-churn/
│   ├── config.pbtxt        # configuração do modelo
│   └── 1/                  # versão 1
│       └── model.onnx      # modelo no formato ONNX
│
└── preprocessador/
    ├── config.pbtxt
    └── 1/
        └── model.py        # backend Python customizado
config.pbtxt — configuração do modelo
name: "classificador-churn"
backend: "onnxruntime"
max_batch_size: 64

input [
  {
    name: "features"
    data_type: TYPE_FP32
    dims: [10]
  }
]

output [
  {
    name: "prediction"
    data_type: TYPE_INT64
    dims: [1]
  }
]

dynamic_batching {
  preferred_batch_size: [8, 16, 32]
  max_queue_delay_microseconds: 50000  # 50ms
}

Passo 02 — Criar service.py e bentofile.yaml

Crie o service.py com um runner do modelo e um endpoint /predict que recebe JSON com features: List[float] e retorna predição. Crie o bentofile.yaml apontando para o service. Suba localmente com bentoml serve service:svc --reload e teste com curl.

Básico
Python — service.py
import numpy as np
import bentoml
from bentoml.io import JSON
from pydantic import BaseModel
from typing import List

runner = bentoml.sklearn.get("classificador-churn:latest").to_runner()
svc = bentoml.Service("servico-churn", runners=[runner])

class In(BaseModel):
    features: List[float]

@svc.api(input=JSON(pydantic_model=In), output=JSON())
async def predict(data: In):
    X = np.array(data.features).reshape(1, -1)
    pred = await runner.predict.async_run(X)
    return {"prediction": int(pred[0])}

Passo 04 — Comparativo de throughput

Com ambas as APIs rodando (FastAPI na porta 8000 e BentoML na porta 3000), use o hey (ferramenta de benchmark HTTP) para enviar 500 requisições com 10 workers concorrentes para cada endpoint e compare as métricas de throughput (req/s) e latência média.

Médio
Bash — instalar hey e rodar benchmark
# Instalar hey (Go tool)
go install github.com/rakyll/hey@latest
# ou: brew install hey  (macOS)

# Benchmark FastAPI (porta 8000)
hey -n 500 -c 10 \
  -m POST \
  -H "Content-Type: application/json" \
  -d '{"features":[0.5,-1.2,0.8,1.1,-0.3,0.6,1.5,-0.9,0.2,0.7]}' \
  http://localhost:8000/predict

# Benchmark BentoML (porta 3000)
hey -n 500 -c 10 \
  -m POST \
  -H "Content-Type: application/json" \
  -d '{"features":[0.5,-1.2,0.8,1.1,-0.3,0.6,1.5,-0.9,0.2,0.7]}' \
  http://localhost:3000/predict
Com modelo sklearn simples, a diferença de throughput entre FastAPI e BentoML sem batching tende a ser pequena (ambos em CPU). A diferença fica evidente ao ativar batching dinâmico no BentoML e replicar o teste com alta concorrência.
Passo 05 — Desafio: ativar batching e medir o impacto

Modifique o runner para adicionar max_batch_size=16 e max_latency_ms=50. Suba novamente e repita o benchmark com -c 20 (20 workers concorrentes). Compare o throughput com e sem batching. Em quais condições o batching ajuda mais — baixa ou alta concorrência? Por quê?

Desafio
Critério de conclusão: o Bento deve estar rodando no Docker (gerado via bentoml containerize, sem Dockerfile manual), respondendo corretamente em localhost:3000/predict, e você deve ter um número de throughput comparativo entre FastAPI e BentoML anotado.
