Questão 1 · Kubernetes — Semana 14

Um Deployment de modelo ML tem replicas: 3 e a seguinte estratégia de atualização: maxUnavailable: 0 e maxSurge: 1. O que acontece quando você executa kubectl set image deployment/modelo modelo=modelo:2.0?

A
Os 3 pods antigos são encerrados simultaneamente e 3 novos com a imagem 2.0 são iniciados
B
Um pod com imagem 2.0 é iniciado (total: 4 pods). Quando passa na readinessProbe, um pod antigo é removido. Processo se repete até todos serem 2.0, nunca havendo menos de 3 pods rodando
C
O Kubernetes aguarda o operador aprovar manualmente a atualização antes de iniciar qualquer pod novo
D
O update falha porque maxUnavailable: 0 não é compatível com maxSurge: 1

Resposta: B
Com maxUnavailable: 0 e maxSurge: 1, o Kubernetes garante zero downtime. Ele sobe um pod extra (total: 4) antes de remover qualquer pod antigo. O pod novo só recebe tráfego depois de passar na readinessProbe — que para modelos ML verifica se o modelo carregou com sucesso via /health. Só então um pod antigo é removido. O processo se repete até todos os pods estarem na versão 2.0, com no mínimo 3 pods ativos em todo momento.

Questão 2 · Celery — Semana 15

Um cliente envia 500.000 registros para pontuação via POST /jobs/batch. O endpoint retorna imediatamente o JSON {"task_id": "abc123", "status": "PENDING"}. Três minutos depois, o cliente consulta GET /jobs/abc123 e recebe {"status": "SUCCESS", "resultado": {...}}. Qual o nome deste padrão de arquitetura?

A
Streaming síncrono — o servidor processa em paralelo e o cliente aguarda em stream
B
Polling assíncrono — o trabalho é enfileirado, o cliente verifica periodicamente o status usando o task_id
C
Webhook reverso — o servidor notifica o cliente assim que o processamento termina
D
Long polling — a conexão HTTP fica aberta até o processamento terminar

Resposta: B
O padrão é polling assíncrono. O POST retorna imediatamente com um task_id (sem bloquear). O cliente repete chamadas GET periodicamente (polling) até receber SUCCESS. Este padrão é o mais simples de implementar com Celery — POST enfileira a task com .delay(), GET usa AsyncResult(task_id).status. O webhook (C) é o oposto: o servidor que notifica o cliente proativamente. O long polling (D) mantém a conexão aberta esperando, o que bloquearia o servidor e exigiria timeout muito maior.

Questão 3 · JWT — Semana 16

Uma API de ML escala horizontalmente para 5 réplicas no Kubernetes. Por que JWT é preferível a sessões armazenadas em banco de dados para autenticação nesse cenário?

A
JWT é mais seguro porque o token é criptografado e não pode ser interceptado por terceiros
B
JWT é stateless: cada réplica valida o token localmente com a chave secreta, sem precisar consultar um banco de sessões compartilhado — o que eliminaria gargalo e ponto único de falha
C
JWT permite que o usuário veja o conteúdo do token, facilitando o debug de problemas de autenticação
D
JWT expira automaticamente, enquanto sessões em banco precisam de limpeza manual periódica

Resposta: B
A vantagem central do JWT em ambientes horizontalmente escalados é a natureza stateless: a réplica que recebe a requisição não precisa de nenhuma comunicação com banco de dados ou outras réplicas para validar o token — ela verifica a assinatura HMAC com a chave secreta e decodifica o payload. Com sessões em banco, todas as 5 réplicas precisariam consultar o mesmo banco a cada requisição, criando gargalo de I/O e ponto único de falha. A opção A é parcialmente falsa (o payload do JWT é só codificado em base64, não criptografado). A D é verdadeira mas não é a razão principal para preferi-lo em ambiente distribuído.

Questão 4 · SHAP — Semana 17

Um modelo de crédito prevê que o cliente João não deve receber empréstimo (predição = 0, probabilidade = 0.82). Os valores SHAP calculados são: historico_credito: -0.31, dividas_abertas: +0.28, renda_mensal: -0.15. O valor base do modelo é 0.40. O que significam esses valores?

A
O histórico de crédito ruim e as dívidas abertas contribuem juntos para aumentar o risco, enquanto a renda reduz o risco
B
O bom histórico de crédito e a boa renda reduzem o score de risco, mas as dívidas abertas aumentam — resultando no score final de 0.82 (0.40 - 0.31 + 0.28 - 0.15 + outros = 0.82)
C
Os valores SHAP representam a importância global de cada feature para o modelo, não a explicação desta predição específica
D
O modelo base já prediz 0.40 de risco; os SHAP values somam e subtraem para chegar à predição final, sendo positivos os que aumentam e negativos os que reduzem o risco de inadimplência

Resposta: D
Esta questão testa a interpretação correta dos valores SHAP locais. O valor base (0.40) é a predição média do modelo sobre o conjunto de treino. Os SHAP values medem a contribuição de cada feature para afastar a predição deste cliente do valor base: dividas_abertas: +0.28 significa que as dívidas deste cliente aumentam o score de risco em 0.28 em relação à média. historico_credito: -0.31 significa que o histórico deste cliente reduz o risco em 0.31. A soma de todos os SHAP values + base_value = predição final (0.82). A opção B descreve a direção corretamente mas erra ao dizer que histórico de crédito "bom" reduz o risco (o enunciado não diz que o histórico é bom — um SHAP negativo significa que a feature deste cliente específico reduz o score em relação à média, não que o histórico é bom em termos absolutos).

Questão 5 · Rate Limiting — Semana 16

Uma empresa concorrente suspeita que está sofrendo model extraction: um agente externo faz 50.000 consultas à API de inferência em 24h, possivelmente para reconstruir o modelo proprietário. O rate limiting por IP já está configurado em 100 req/min. Por que ele não resolveu o problema e qual seria a contramedida mais eficaz?

A
O rate limiting por IP falhou porque o atacante usou múltiplos IPs (proxies/VPN). A solução mais eficaz é rate limiting por usuário autenticado (JWT), que limita a conta independente do IP de origem
B
O rate limiting falhou porque o atacante usou HTTP/2 multiplexing. A solução é migrar para gRPC
C
O rate limiting por IP é suficiente — o problema deve estar em outra camada de segurança
D
A única solução para model extraction é tornar a API completamente privada, sem acesso externo

Resposta: A
Rate limiting por IP é facilmente contornado com proxies residenciais, VPNs ou redes de IPs rotativos. 100 req/min por IP, distribuídos em 1.000 IPs diferentes = 100.000 req/min sem violar nenhum limite individual. Rate limiting por conta autenticada (JWT) remove essa possibilidade: independente de quantos IPs o atacante use, ele está vinculado à sua conta. Complementarmente, sistemas de detecção de anomalias (padrão de consulta suspeito — inputs muito similares ou cobrindo o espaço de features sistematicamente) podem identificar model extraction antes que o dano seja significativo.

Questão 6 · Kafka — Semana 18

Uma empresa publica eventos de transações no tópico Kafka transacoes. Três sistemas distintos precisam processar cada evento: (1) detecção de fraude ML, (2) auditoria/logging, (3) atualização de saldo em tempo real. Qual a abordagem correta no Kafka?

A
Criar três tópicos diferentes (transacoes-fraude, transacoes-auditoria, transacoes-saldo) e o producer publica em todos os três
B
Criar um único consumer que processa os três casos e publica os resultados em tópicos separados
C
Criar três consumer groups distintos (grupo-fraude, grupo-auditoria, grupo-saldo), cada um consumindo o mesmo tópico transacoes de forma independente
D
Usar RabbitMQ em vez de Kafka, pois o Kafka só suporta um consumer por tópico

Resposta: C
Esta é a principal vantagem do Kafka sobre filas tradicionais: o modelo de consumer groups. Cada grupo tem seu próprio offset por partição — ou seja, cada grupo lê o evento de forma completamente independente. O grupo de fraude pode estar no offset 1.000, o de auditoria no 999 e o de saldo no 1.001, sem interferência. O evento persiste no log e cada grupo o lê no seu próprio ritmo. A opção A seria ineficiente (triplicar os dados no broker). A B acoplaria três responsabilidades distintas em um único consumer, criando um ponto de falha único. A D é factualmente incorreta — o Kafka suporta múltiplos consumer groups por design.

Questão 7 · Integração — Semanas 14–18

Uma API de ML em produção no Kubernetes apresenta os seguintes sintomas: (1) autenticação JWT funcionando, (2) algumas requisições retornam 503, (3) o HPA não está escalando mesmo com CPU em 85%, (4) o Grafana mostra latência P95 crescendo. Qual é a causa raiz mais provável e a sequência correta de investigação?

A
O JWT expirou para todos os usuários simultaneamente, causando reautenticação em massa que sobrecarregou o servidor
B
O modelo não está carregando corretamente: pods falham na readinessProbe e ficam fora do Service, reduzindo a capacidade efetiva. O HPA não escala porque as réplicas não-ready não contam para o cálculo de CPU média. Investigação: kubectl describe pods → logs → verificar /health
C
O rate limiting está configurado muito restritivo e está retornando 503 em vez de 429
D
O Kafka está sobrecarregando a API com eventos demais, saturando as conexões disponíveis

Resposta: B
Esta questão integra Kubernetes + monitoramento. Os sintomas se encaixam perfeitamente: se o modelo falha ao carregar (por ex., arquivo .pkl corrompido ou dependência ausente), os pods falham na readinessProbe e o Kubernetes os remove do Service — resultando em 503 para as requisições que chegam sem pods disponíveis. O HPA calcula a utilização de CPU como média sobre os pods ready, não todos os pods. Com apenas 1-2 pods ready em vez de 3, o denominador do cálculo diminui, mas a CPU por pod ready pode já estar tão alta que o comportamento parece inconsistente. A investigação correta é sempre: kubectl get pods → checar quais estão Running vs não-ready → kubectl describe pod para eventos → kubectl logs → verificar manualmente o endpoint /health.

Rubrica de Avaliação e Orientações de Entrega
O que é o Projeto Final

O Projeto Final consiste em construir um pipeline MLOps end-to-end para um problema de sua escolha, integrando pelo menos 6 dos componentes estudados na disciplina. Você apresentará em 10 minutos demonstrando o pipeline funcionando.

Liberdade de tema: escolha qualquer problema de ML que você tenha interesse — detecção de fraude, churn de clientes, recomendação, classificação de texto, previsão de demanda. O que importa é a infraestrutura MLOps, não a acurácia do modelo.
Rubrica de Avaliação — 100 pontos

Reprodutibilidade 20pts

  • Dados versionados com DVC (10pts)
  • Dockerfile correto com slim e boas práticas (5pts)
  • requirements.txt com versões fixas (5pts)

Pipeline e Tracking 20pts

  • Experimentos logados no MLflow (10pts)
  • Modelo promovido no Registry com critério (5pts)
  • Pipeline reproduzível (MLflow ou DVC) (5pts)

Serving 20pts

  • API FastAPI com endpoints documentados (10pts)
  • Validação Pydantic e tratamento de erros (5pts)
  • Health check funcional (5pts)

Monitoramento 15pts

  • Métricas Prometheus expostas (8pts)
  • Dashboard Grafana com pelo menos 2 painéis (7pts)

Segurança ou Explicabilidade 15pts

  • JWT + rate limiting (15pts), ou
  • SHAP global + endpoint /explain (15pts)

Apresentação 10pts

  • Demo ao vivo funcionando (5pts)
  • Clareza na explicação das escolhas (3pts)
  • README completo no repositório (2pts)
