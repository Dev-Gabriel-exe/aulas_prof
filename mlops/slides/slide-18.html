Celery vs Kafka — a diferença fundamental
  • Celery: fila de tarefas — trabalho assíncrono, resultado esperado
  • Kafka: stream de eventos — fluxo contínuo, sem fim, pub/sub
  • Celery: um produtor, um worker consome cada mensagem
  • Kafka: um evento pode ser consumido por múltiplos grupos independentes

Apache Kafka — Event Streaming para ML em Tempo Real
O que é Kafka e por que vai além do Celery

O Apache Kafka é uma plataforma de streaming de eventos distribuída. Diferente do RabbitMQ (fila de mensagens), o Kafka é um log distribuído e imutável: os eventos são persistidos por um período configurável e qualquer número de consumidores pode ler o mesmo evento de forma independente.

RabbitMQ / Celery:

  • Mensagem é consumida por um worker e removida
  • Ideal para tarefas com começo e fim
  • Resultado é armazenado e buscado pelo cliente
  • Não suporta reprocessamento de histórico

Kafka:

  • Evento persiste no log por dias/semanas
  • Múltiplos consumer groups leem o mesmo evento
  • Ideal para fluxos contínuos de dados
  • Reprocessamento histórico a qualquer momento
Caso de uso paradigmático em ML: detecção de fraude em transações bancárias. Cada transação é um evento publicado no tópico transacoes. Três consumer groups diferentes consomem o mesmo evento: (1) consumer de detecção de fraude ML, (2) consumer de auditoria/logging, (3) consumer de notificação ao cliente. Cada um processa de forma independente, sem que um afete o outro.

Producer, Consumer ML e Atividade Prática
Producer — publicar eventos de transação
Python — producer.py
import json, time, random
from kafka import KafkaProducer
from datetime import datetime

producer = KafkaProducer(
    bootstrap_servers=["localhost:9092"],
    value_serializer=lambda v: json.dumps(v).encode("utf-8"),
    key_serializer=lambda k: k.encode("utf-8")
)

def gerar_transacao():
    return {
        "transaction_id": f"TXN-{random.randint(10000,99999)}",
        "cliente_id"    : f"CLT-{random.randint(1000,9999)}",
        "valor"         : round(random.uniform(10, 5000), 2),
        "hora"          : datetime.now().hour,
        "pais_destino"  : random.choice(["BR", "US", "NG", "RU"]),
        "tentativas"    : random.randint(1, 5),
        "timestamp"     : datetime.now().isoformat()
    }

print("Publicando transações no tópico 'transacoes'...")
try:
    while True:
        txn = gerar_transacao()
        producer.send(
            topic="transacoes",
            key=txn["transaction_id"],
            value=txn
        )
        print(f"Publicado: {txn['transaction_id']} — R$ {txn['valor']}")
        time.sleep(0.5)
except KeyboardInterrupt:
    producer.flush()
    print("Producer encerrado.")
Consumer ML — inferência em tempo real
Python — consumer_ml.py
import json, pickle
import numpy as np
from kafka import KafkaConsumer, KafkaProducer

# Carregar modelo uma vez
with open("models/model.pkl", "rb") as f:
    MODEL = pickle.load(f)

consumer = KafkaConsumer(
    "transacoes",
    bootstrap_servers=["localhost:9092"],
    group_id="ml-fraude-grupo",
    value_deserializer=lambda m: json.loads(m.decode("utf-8")),
    auto_offset_reset="latest"     # processar apenas novos eventos
)

producer = KafkaProducer(
    bootstrap_servers=["localhost:9092"],
    value_serializer=lambda v: json.dumps(v).encode("utf-8")
)

def extrair_features(txn: dict) -> np.ndarray:
    """Transforma o evento em features para o modelo."""
    paises_risco = {"BR": 0, "US": 0, "NG": 1, "RU": 1}
    return np.array([[
        txn["valor"],
        txn["hora"],
        paises_risco.get(txn["pais_destino"], 0),
        txn["tentativas"],
    ]])

print("Consumer ML iniciado. Aguardando eventos...")

for msg in consumer:
    txn   = msg.value
    X     = extrair_features(txn)
    pred  = int(MODEL.predict(X)[0])
    proba = float(MODEL.predict_proba(X)[0][1])

    resultado = {
        "transaction_id": txn["transaction_id"],
        "cliente_id"    : txn["cliente_id"],
        "fraude"        : bool(pred),
        "probabilidade" : round(proba, 4),
        "alerta"        : proba > 0.80
    }

    # Publicar resultado no tópico de predições
    producer.send("predicoes-fraude", value=resultado)

    if resultado["alerta"]:
        print(f"🚨 ALERTA: {txn['transaction_id']} — fraude {proba:.1%}")
    else:
        print(f"✅ OK: {txn['transaction_id']} — {proba:.1%}")

Passo 04 — Segundo consumer group: auditoria

Crie um segundo consumer (consumer_auditoria.py) que também lê o tópico transacoes mas com group_id="auditoria-grupo". Este consumer não faz inferência — apenas salva cada evento em logs/transacoes.jsonl. Execute os dois consumers simultaneamente e confirme que ambos recebem os mesmos eventos (o Kafka entrega para grupos diferentes de forma independente).

Médio
Python — consumer_auditoria.py
import json, os
from kafka import KafkaConsumer

os.makedirs("logs", exist_ok=True)
consumer = KafkaConsumer(
    "transacoes",
    bootstrap_servers=["localhost:9092"],
    group_id="auditoria-grupo",   # group diferente!
    value_deserializer=lambda m: json.loads(m.decode())
)
with open("logs/transacoes.jsonl", "a") as f:
    for msg in consumer:
        f.write(json.dumps(msg.value) + "\n")
        f.flush()
        print(f"Auditoria: {msg.value['transaction_id']}")
Passo 05 — Desafio: adicionar SHAP ao consumer de alertas

Crie um terceiro consumer que lê predicoes-fraude e, para qualquer transação com alerta: true, busca o evento original (pelo transaction_id), calcula os valores SHAP e imprime as 3 principais features que contribuíram para o score de fraude. Integração completa: Kafka + inferência + explicabilidade em tempo real.

Desafio
Critério de conclusão: dois consumer groups rodando simultaneamente consumindo o mesmo tópico. O consumer ML publica predições em predicoes-fraude. O consumer de auditoria grava eventos em logs/transacoes.jsonl. Alertas aparecem no terminal quando proba > 80%.
