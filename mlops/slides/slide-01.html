O que é MLOps — Contexto e Motivação
Onde estamos e o que essa disciplina resolve

Você já sabe criar modelos de machine learning. Sabe treinar uma regressão, avaliar com métricas, ajustar hiperparâmetros num notebook. Mas o que acontece depois que o modelo performa bem no seu computador?

Na maioria das empresas: nada. O modelo fica no notebook. Nunca chega aos usuários. Nunca gera valor real. Esse é o problema que MLOps resolve.

Dado que define a disciplina: estudos da indústria apontam que cerca de 87% dos projetos de Machine Learning nunca saem do ambiente de experimento para produção. MLOps é o conjunto de práticas, ferramentas e processos que resolve isso.
A definição de MLOps

MLOps = Machine Learning + DevOps. É a aplicação dos princípios de DevOps — automação, versionamento, CI/CD, monitoramento — ao ciclo de vida de modelos de Machine Learning.

Mas MLOps não é só DevOps com modelos. Modelos têm características únicas que complicam tudo:

  • Modelos degradam com o tempo — os dados do mundo real mudam, e o modelo que funcionava bem em janeiro pode estar errado em julho sem que ninguém tenha tocado no código.
  • Dados são parte do sistema — no software tradicional, o comportamento é determinado pelo código. Em ML, o comportamento é determinado pelos dados de treino. Ambos precisam ser versionados.
  • Experimentos proliferam — sem controle, você rapidamente tem dezenas de versões de modelo sem saber qual gerou qual resultado, com quais dados e hiperparâmetros.
  • Reproduzir um resultado é difícil — "funcionou na minha máquina" é um problema antigo em software; em ML é ainda pior porque envolve dados, código, ambiente e aleatoriedade.

Notebook vs. Produção — e o Ciclo de Vida de Modelos
Por que o modelo que funciona no notebook quebra em produção

Essa é a pergunta mais comum de quem começa na área. O modelo tinha 94% de acurácia. Os testes passaram. Mas quando foi para produção, vieram os problemas. Por quê?

Notebook / Experimento

  • Dados fixos, estáticos, já limpos pelo cientista
  • Uma requisição de cada vez, sem concorrência
  • Ambiente: "funciona na minha máquina"
  • Sem versionamento de dados ou modelo
  • Sem tratamento de erros ou entradas inválidas
  • Resultado: arquivo model.pkl na pasta local

Produção

  • Dados chegam em tempo real, com ruído e variação
  • Centenas de requisições simultâneas
  • Ambiente reproduzível: Docker, requirements fixos
  • Modelo e dados versionados — rollback possível
  • Validação de input, logs, alertas de erros
  • API REST servindo predições com SLA definido
Resumo: notebook é um ambiente de exploração. Produção é um sistema com usuários reais, SLA, dados dinâmicos e necessidade de manutenção contínua. A distância entre os dois é o problema central de MLOps.
As 4 fases do ciclo de vida de um modelo

Todo modelo de ML em produção passa por quatro fases. Entender esse ciclo é o pré-requisito para qualquer prática de MLOps.

1 Definição e Dados Problema, coleta, exploração, limpeza, feature engineering
2 Experimentação Treinamento, avaliação, comparação de algoritmos, tuning
3 Deploy Empacotamento, API, containerização, testes de integração
4 Monitoramento Drift de dados, performance real, alertas, retreinamento

Note que a fase 4 leva de volta à fase 1. Não existe "modelo pronto para sempre". Um modelo em produção é um sistema vivo que precisa ser continuamente monitorado e atualizado.

Detalhando cada fase
FaseO que aconteceFerramentas típicasProblema sem MLOps
1 — Dados Coleta, EDA, limpeza, features Python, pandas, SQL Dados não versionados — impossível reproduzir
2 — Experimento Treino, avaliação, tuning scikit-learn, XGBoost, MLflow Dezenas de runs sem rastreamento — qual era o melhor?
3 — Deploy API, Docker, CI/CD FastAPI, Docker, GitHub Actions "Funciona na minha máquina" — ambiente incompatível
4 — Monitoramento Métricas, drift, alertas Evidently AI, Grafana, Prometheus Modelo degradando silenciosamente sem ninguém perceber
O modelo fantasma: um dos cenários mais comuns na indústria é descobrir que um modelo "em produção" há meses nunca foi monitorado e suas predições estão completamente erradas — mas como ninguém acompanhava, ninguém sabia. MLOps existe para tornar isso impossível.

Papéis, Responsabilidades e Estudo de Caso
Quem faz o quê — os três papéis centrais

Em empresas que levam ML a sério, existem papéis distintos com responsabilidades diferentes em torno do ciclo de vida do modelo. Entender onde cada um começa e termina evita confusão e lacunas de responsabilidade.

Data Scientist
  • Define o problema e as métricas de sucesso
  • Explora e processa os dados
  • Treina, avalia e seleciona o modelo
  • Responsável pela qualidade do modelo
  • Entrega: modelo treinado + código de experimentação
ML Engineer
  • Pega o modelo do cientista e leva a produção
  • Constrói a API de serving (FastAPI, etc)
  • Garante performance, latência e escalabilidade
  • Implementa testes e pipelines de CI/CD
  • Entrega: modelo servindo via API confiável
MLOps Engineer
  • Constrói e mantém a infraestrutura de ML
  • Configura rastreamento, registro e versionamento
  • Implementa monitoramento e alertas
  • Garante reprodutibilidade de ponta a ponta
  • Entrega: plataforma que acelera todos os outros
Na prática: em empresas menores, uma pessoa acumula dois ou três desses papéis. Mas entender a separação de responsabilidades é essencial para não deixar nada cair no vazio — especialmente monitoramento, que costuma ser a primeira vítima quando não há papel definido para isso.
Estudo de caso — sistema de recomendação de uma loja virtual

Vamos acompanhar o ciclo de vida de um modelo real: um sistema de recomendação de produtos para e-commerce. Cada fase vai ilustrar tanto as decisões técnicas quanto os problemas que surgem sem MLOps.

Cenário — contexto do problema 
──────────────────────────────────────────────────────────────
 Empresa:  loja virtual de eletrônicos, ~10k pedidos/dia
 Problema: "nossos clientes não encontram o que querem"
 Objetivo: recomendar produtos relevantes na página inicial
 Métrica:  aumentar a taxa de clique (CTR) em 15%
──────────────────────────────────────────────────────────────
Fase 1 — Dados: o time coleta 6 meses de histórico de compras do banco de dados. Um cientista faz EDA, remove outliers e cria features (última categoria comprada, frequência de compra, ticket médio). Os dados são salvos localmente. Problema: quando o experimento for reproduzido em 3 meses, quais dados foram usados? Ninguém sabe.
Fase 2 — Experimento: são testados 12 modelos diferentes (colaborativo, baseado em conteúdo, híbrido). O melhor tem AUC de 0.87 com um modelo de matrix factorization. O resultado está num notebook chamado final_v3_FINAL2.ipynb. Problema: os outros 11 experimentos não têm registro. Qual foi o segundo melhor? Com quais dados? Impossível dizer.
Fase 3 — Deploy: o ML Engineer serializa o modelo com pickle e escreve um script Flask. Funciona localmente. No servidor de produção, a versão do scikit-learn é diferente e o modelo não carrega. 2 dias de trabalho para resolver dependências. Problema: sem Docker e requirements fixos, ambiente de dev ≠ produção.
Fase 4 — Monitoramento: o modelo vai ao ar. A CTR sobe 18% no primeiro mês. Ninguém configura monitoramento. Quatro meses depois, durante a Black Friday, o perfil de compras muda completamente — e o modelo segue recomendando produtos do comportamento pré-Black Friday. CTR cai para -3%. A equipe só descobre 2 semanas depois. Problema: sem monitoramento de drift, o modelo degradou silenciosamente.
O mesmo cenário com MLOps

Vejamos como as mesmas fases funcionariam com práticas de MLOps aplicadas — que é exatamente o que vamos construir ao longo do semestre:

  • Dados versionados com DVC — cada experimento referencia exatamente qual versão do dataset foi usada. Reproduzir qualquer run é trivial. (Semana 3)
  • Experimentos rastreados com MLflow — todos os 12 modelos têm parâmetros, métricas e artefatos registrados. Comparar e voltar para o melhor é um clique. (Semana 4)
  • Ambiente containerizado com Docker — a imagem de produção é gerada a partir do mesmo Dockerfile testado em dev. "Funciona na minha máquina" deixa de existir. (Semana 8)
  • Monitoramento com Evidently AI + Grafana — data drift é detectado automaticamente. Um alerta é disparado quando a distribuição dos dados de entrada muda. O retreinamento pode ser automático. (Semanas 11–12)
