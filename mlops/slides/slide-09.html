Por que FastAPI — e Validação com Pydantic
O que torna FastAPI a escolha padrão para servir modelos

Na Semana 8 usamos FastAPI de forma mínima. Agora vamos entender por que ele se tornou o padrão da indústria para servir modelos de ML, superando Flask na maioria dos casos novos.

  • Tipagem nativa do Python: você declara os tipos esperados (float, str, List[float]) e o FastAPI valida automaticamente — sem escrever código de validação manual.
  • Documentação gerada sozinha: a partir dos tipos declarados, o FastAPI gera Swagger UI e ReDoc automaticamente — sem manter documentação separada que desatualiza.
  • Performance: construído sobre Starlette e Uvicorn (ASGI), é um dos frameworks Python mais rápidos — comparável a frameworks em Go ou Node.js em benchmarks.
  • Suporte a async/await: útil quando o endpoint precisa esperar I/O (banco de dados, chamada a outro serviço) sem bloquear outras requisições.
Para servir modelos de ML especificamente: a combinação de validação automática + documentação + performance resolve exatamente os três problemas mais comuns de APIs de inferência: receber dados malformados, falta de clareza sobre o contrato da API, e latência sob carga.
Pydantic — o motor de validação por trás do FastAPI

O Pydantic é a biblioteca que faz a validação de dados no FastAPI. Você define um modelo (uma classe Python) com os tipos esperados, e o Pydantic garante que qualquer dado recebido seja convertido ou rejeitado com erro claro.

Python — modelos Pydantic básicos
from pydantic import BaseModel, Field, field_validator
from typing import List, Optional

class PredictionInput(BaseModel):
    # Tipos básicos com validação automática
    idade: int = Field(gt=0, lt=120, description="Idade do cliente")
    saldo: float = Field(ge=0, description="Saldo em conta")
    tempo_cliente_meses: int = Field(ge=0)
    categoria: str = Field(pattern="^(bronze|prata|ouro)$")

    # Validação customizada
    @field_validator("saldo")
    @classmethod
    def saldo_razoavel(cls, v):
        if v > 10_000_000:
            raise ValueError("Saldo acima do limite esperado — verifique a entrada")
        return v

class PredictionOutput(BaseModel):
    prediction: int
    probability: float
    modelo_versao: str = "1.0"
Se a requisição enviar idade: -5 ou categoria: "platina", o FastAPI rejeita automaticamente com status 422 Unprocessable Entity e uma mensagem JSON explicando exatamente qual campo falhou e por quê — sem você escrever nenhum código de validação manual.
Como uma requisição é validada — o fluxo completo

1. Cliente envia JSON

POST com corpo JSON contendo os campos esperados pelo modelo Pydantic

2. Pydantic valida

Verifica tipos, ranges, regex. Se falhar, retorna 422 com detalhes do erro

3. Função executa

Só roda se a validação passou. Recebe objeto Python já tipado e validado

4. Resposta validada

response_model garante que a saída também segue o contrato declarado

Exemplo — erro 422 retornado automaticamente
// Requisição com idade negativa:
// POST /predict {"idade": -5, "saldo": 1000, ...}

// Resposta automática do FastAPI (HTTP 422):
{
  "detail": [
    {
      "type": "greater_than",
      "loc": ["body", "idade"],
      "msg": "Input should be greater than 0",
      "input": -5
    }
  ]
}

Endpoints de Predição — Single, Batch e Tratamento de Erros
API completa de inferência — estrutura

Uma API de inferência madura tem mais do que um único endpoint de predição. Vamos construir a estrutura completa: health check, predição individual, predição em lote e metadados do modelo.

GET/health — verifica se o serviço e o modelo estão operacionais
GET/model/info — retorna metadados do modelo carregado (versão, features esperadas)
POST/predict — predição de uma única instância
POST/predict/batch — predição de múltiplas instâncias em uma chamada
API completa — src/serve.py
Python — API de inferência completa
import pickle
import time
import numpy as np
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel, Field
from typing import List

app = FastAPI(
    title="API de Predição — Classificador de Churn",
    description="Serviço de inferência para modelo de churn de clientes",
    version="1.0.0"
)

# Carrega o modelo uma única vez, na inicialização
try:
    with open("models/model.pkl", "rb") as f:
        MODEL = pickle.load(f)
    MODEL_CARREGADO = True
except Exception as e:
    MODEL = None
    MODEL_CARREGADO = False
    print(f"ERRO ao carregar modelo: {e}")

N_FEATURES_ESPERADAS = 10


# ── Modelos Pydantic ─────────────────────────────────────────────
class PredictionInput(BaseModel):
    features: List[float] = Field(
        ..., min_length=N_FEATURES_ESPERADAS, max_length=N_FEATURES_ESPERADAS,
        description=f"Lista com exatamente {N_FEATURES_ESPERADAS} features numéricas"
    )

class BatchInput(BaseModel):
    instancias: List[PredictionInput]

class PredictionOutput(BaseModel):
    prediction: int
    probability: float
    tempo_inferencia_ms: float

class BatchOutput(BaseModel):
    resultados: List[PredictionOutput]
    total_processado: int


# ── Endpoints ────────────────────────────────────────────────────
@app.get("/health")
def health():
    if not MODEL_CARREGADO:
        raise HTTPException(status_code=503, detail="Modelo não carregado")
    return {"status": "healthy", "modelo_carregado": True}


@app.get("/model/info")
def model_info():
    return {
        "tipo": type(MODEL).__name__,
        "n_features_esperadas": N_FEATURES_ESPERADAS,
        "versao": "1.0.0"
    }


@app.post("/predict", response_model=PredictionOutput)
def predict(data: PredictionInput):
    if not MODEL_CARREGADO:
        raise HTTPException(status_code=503, detail="Modelo não disponível")

    inicio = time.perf_counter()
    try:
        X     = np.array(data.features).reshape(1, -1)
        pred  = int(MODEL.predict(X)[0])
        proba = float(MODEL.predict_proba(X)[0][pred])
    except Exception as e:
        raise HTTPException(status_code=400, detail=f"Erro na predição: {str(e)}")

    tempo_ms = (time.perf_counter() - inicio) * 1000
    return PredictionOutput(
        prediction=pred, probability=round(proba, 4), tempo_inferencia_ms=round(tempo_ms, 2)
    )


@app.post("/predict/batch", response_model=BatchOutput)
def predict_batch(data: BatchInput):
    if not MODEL_CARREGADO:
        raise HTTPException(status_code=503, detail="Modelo não disponível")

    inicio = time.perf_counter()
    X = np.array([inst.features for inst in data.instancias])
    preds  = MODEL.predict(X)
    probas = MODEL.predict_proba(X)
    tempo_total_ms = (time.perf_counter() - inicio) * 1000

    resultados = [
        PredictionOutput(
            prediction=int(p), probability=round(float(pr[p]), 4),
            tempo_inferencia_ms=round(tempo_total_ms / len(preds), 2)
        )
        for p, pr in zip(preds, probas)
    ]
    return BatchOutput(resultados=resultados, total_processado=len(resultados))
Tratamento de erros com HTTPException

O HTTPException do FastAPI permite retornar códigos de status HTTP específicos com mensagens claras. Isso é essencial para que sistemas clientes (e outros desenvolvedores) entendam exatamente o que deu errado.

CódigoSignificadoQuando usar em APIs de ML
200OKPredição realizada com sucesso
400Bad RequestErro ao processar dados válidos sintaticamente (ex: features fora do range do modelo)
422Unprocessable EntityGerado automaticamente pelo Pydantic — dado não corresponde ao schema esperado
503Service UnavailableModelo não foi carregado corretamente na inicialização
429Too Many RequestsRate limiting acionado (implementado na Semana 16)
Diferença importante: erros 422 acontecem antes da sua função executar — o Pydantic intercepta. Erros 400/503 você dispara manualmente dentro da função, com raise HTTPException(status_code=..., detail=...), quando a validação de schema passou mas algo de negócio falhou.

Documentação Automática e Atividade Prática
Swagger UI e ReDoc — documentação que nunca desatualiza

A partir dos tipos Pydantic e das anotações nas rotas, o FastAPI gera automaticamente duas interfaces de documentação interativa — sem você escrever uma linha de documentação manual.

  • /docs — Swagger UI: interface interativa onde é possível testar cada endpoint diretamente no navegador, preenchendo campos e vendo a resposta real.
  • /redoc — ReDoc: documentação em formato de leitura, mais adequada para compartilhar com outros times ou stakeholders não-técnicos.
  • /openapi.json: a especificação OpenAPI completa em JSON — pode ser importada em Postman, geradores de SDK client, ou ferramentas de teste automatizado.
Python — enriquecendo a documentação
from pydantic import BaseModel, Field

class PredictionInput(BaseModel):
    features: List[float] = Field(
        ...,
        description="10 features numéricas normalizadas do cliente",
        examples=[[0.5, -1.2, 0.8, 1.1, -0.3, 0.6, 1.5, -0.9, 0.2, 0.7]]
    )

    class Config:
        json_schema_extra = {
            "example": {
                "features": [0.5, -1.2, 0.8, 1.1, -0.3, 0.6, 1.5, -0.9, 0.2, 0.7]
            }
        }

@app.post(
    "/predict",
    response_model=PredictionOutput,
    summary="Realiza predição de churn",
    description="Recebe as features de um cliente e retorna a probabilidade de churn",
    tags=["predição"]
)
def predict(data: PredictionInput):
    """
    Esta docstring também aparece no Swagger UI como descrição detalhada.
    """
    pass
Por que isso importa em produção: em times que consomem APIs de ML construídas por outras equipes, a documentação automática elimina o ciclo de "pergunte no Slack qual o formato esperado". O contrato da API está sempre visível e sempre atualizado, porque é gerado do próprio código.
Middleware de logging e tempo de resposta

Para observabilidade básica (aprofundada na Semana 12), é útil registrar automaticamente o tempo de cada requisição. Um middleware do FastAPI faz isso sem modificar cada endpoint individualmente.

Python — middleware de logging
import time
import logging
from fastapi import Request

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("api-inferencia")

@app.middleware("http")
async def log_requests(request: Request, call_next):
    inicio = time.perf_counter()
    response = await call_next(request)
    duracao_ms = (time.perf_counter() - inicio) * 1000

    logger.info(
        f"{request.method} {request.url.path} "
        f"status={response.status_code} tempo={duracao_ms:.2f}ms"
    )
    response.headers["X-Process-Time-Ms"] = str(round(duracao_ms, 2))
    return response

Passo 02 — Endpoints /health e /predict

Implemente o endpoint GET /health que retorna o status do modelo carregado, e POST /predict que recebe PredictionInput, mede o tempo de inferência e retorna PredictionOutput. Use HTTPException(503, ...) se o modelo não tiver carregado.

Básico
Python
import time, pickle, numpy as np
from fastapi import FastAPI, HTTPException

app = FastAPI()

try:
    with open("models/model.pkl", "rb") as f:
        MODEL = pickle.load(f)
    OK = True
except:
    MODEL, OK = None, False

@app.get("/health")
def health():
    if not OK:
        raise HTTPException(503, "Modelo não carregado")
    return {"status": "healthy"}

@app.post("/predict", response_model=PredictionOutput)
def predict(data: PredictionInput):
    if not OK:
        raise HTTPException(503, "Modelo não disponível")
    inicio = time.perf_counter()
    X = np.array(data.features).reshape(1, -1)
    pred = int(MODEL.predict(X)[0])
    proba = float(MODEL.predict_proba(X)[0][pred])
    tempo = (time.perf_counter() - inicio) * 1000
    return PredictionOutput(prediction=pred, probability=round(proba,4), tempo_inferencia_ms=round(tempo,2))
Passo 03 — Endpoint de predição em batch

Implemente BatchInput com uma lista de PredictionInput, e o endpoint POST /predict/batch que processa todas as instâncias em uma única chamada ao model.predict() (não em loop chamando predict várias vezes). Retorne a lista de resultados e o total processado.

Médio
Python
class BatchInput(BaseModel):
    instancias: List[PredictionInput]

class BatchOutput(BaseModel):
    resultados: List[PredictionOutput]
    total_processado: int

@app.post("/predict/batch", response_model=BatchOutput)
def predict_batch(data: BatchInput):
    inicio = time.perf_counter()
    X = np.array([i.features for i in data.instancias])
    preds  = MODEL.predict(X)
    probas = MODEL.predict_proba(X)
    tempo_total = (time.perf_counter() - inicio) * 1000

    resultados = [
        PredictionOutput(
            prediction=int(p),
            probability=round(float(pr[p]), 4),
            tempo_inferencia_ms=round(tempo_total/len(preds), 2)
        )
        for p, pr in zip(preds, probas)
    ]
    return BatchOutput(resultados=resultados, total_processado=len(resultados))
