Docker Revisitado — Layers, Cache e Imagens para ML
Como Docker constrói imagens em camadas

Na Semana 2 você viu o Dockerfile básico. Agora vamos entender o mecanismo interno de layers — porque ele determina diretamente o tamanho da imagem, a velocidade de build e os problemas de reprodutibilidade.

Cada instrução no Dockerfile (FROM, COPY, RUN) cria uma nova camada imutável. As camadas são cacheadas: se uma camada e tudo que veio antes não mudou, o Docker reutiliza o cache em vez de re-executar.

CMD / ENTRYPOINTComando ao iniciar — muda a cada deploy
COPY models/ .Modelo serializado — muda quando retreina
COPY src/ .Código da API — muda com frequência
RUN pip install -r requirements.txtDependências — muda raramente · cacheável
COPY requirements.txt .Lockfile — deve vir antes do código
FROM python:3.11-slimImagem base — muda nunca (fixada)
Regra de ouro da ordem no Dockerfile: coloque o que muda menos frequentemente nas camadas inferiores. O pip install deve vir antes do COPY do código — se só o código mudou, o Docker reutiliza a camada de dependências sem reinstalar nada. Em projetos com 50+ pacotes, isso economiza vários minutos por build.
O problema das imagens gordas em ML

Imagens de ML tendem a ficar enormes se não houver cuidado — e isso tem custo real: mais tempo para fazer pull em produção, mais espaço em registry, mais superfície de ataque de segurança.

ProblemaCausaTamanho típicoSolução
Imagem base pesadaFROM python:3.11 (imagem completa)~900 MBFROM python:3.11-slim (~130 MB)
Dados de treino na imagemCOPY data/ . sem .dockerignore+GBAdicionar data/ ao .dockerignore
Cache do pip na imagempip install sem flags+100–300 MBpip install --no-cache-dir
Ferramentas de buildgcc, make, headers na imagem final+500 MBMulti-stage build (próxima seção)
Notebooks incluídosCOPY . . sem .dockerignore+MB a GB.dockerignore com notebooks/
Dockerfile de inferência — versão cuidadosa

Uma imagem de inferência só precisa rodar predições — não precisa de ferramentas de desenvolvimento, compiladores ou datasets. Cada linha abaixo tem um motivo específico.

Dockerfile — imagem de inferência
# slim: sem compiladores e ferramentas de build (~130MB vs ~900MB)
FROM python:3.11-slim

# Variáveis de ambiente de boas práticas para Python em containers
ENV PYTHONDONTWRITEBYTECODE=1   # não gera .pyc
ENV PYTHONUNBUFFERED=1          # logs aparecem imediatamente (não bufferiza)

WORKDIR /app

# 1. Dependências PRIMEIRO — camada cacheável
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 2. Código da API
COPY src/serve.py ./src/serve.py

# 3. Modelo serializado — camada separada do código
# (muda com retreinamento, independente do código da API)
COPY models/model.pkl ./models/model.pkl

# Usuário não-root por segurança
RUN useradd -m appuser
USER appuser

EXPOSE 8000

# Iniciar o servidor de inferência
CMD ["uvicorn", "src.serve:app", "--host", "0.0.0.0", "--port", "8000"]

Multi-stage Build e docker-compose para ML
Multi-stage build — o padrão profissional

Alguns pacotes Python precisam ser compilados durante a instalação (ex: numpy com BLAS, psycopg2, alguns pacotes de ML). Isso exige ferramentas de build (gcc, headers) que são pesadas e desnecessárias em runtime.

O multi-stage build resolve isso: você usa uma imagem "gorda" só para compilar, e copia apenas os artefatos finais para uma imagem "magra" de produção. O resultado é uma imagem final muito menor e mais segura.

Stage: builder

  • FROM python:3.11 (imagem completa)
  • gcc, g++, build-essential
  • pip install — compila pacotes
  • Wheels gerados em /wheels/

Stage: runtime

  • FROM python:3.11-slim (imagem enxuta)
  • Sem compiladores ou headers
  • Copia apenas os wheels compilados
  • Instala sem compilar (já compilado)
Dockerfile — multi-stage build
# ════ Stage 1: build ════════════════════════════════════════════
FROM python:3.11 AS builder

WORKDIR /build
COPY requirements.txt .

# Compila e empacota dependências como wheels
RUN pip install --no-cache-dir wheel && \
    pip wheel --no-cache-dir --wheel-dir /wheels -r requirements.txt


# ════ Stage 2: runtime ══════════════════════════════════════════
FROM python:3.11-slim

ENV PYTHONDONTWRITEBYTECODE=1 PYTHONUNBUFFERED=1
WORKDIR /app

# Copia apenas os wheels compilados do stage anterior
COPY --from=builder /wheels /wheels

# Instala a partir dos wheels (sem compilar, sem gcc necessário)
RUN pip install --no-cache-dir --no-index \
    --find-links=/wheels /wheels/*.whl && \
    rm -rf /wheels

# Código e modelo
COPY src/serve.py ./src/serve.py
COPY models/model.pkl ./models/model.pkl

RUN useradd -m appuser
USER appuser

EXPOSE 8000
CMD ["uvicorn", "src.serve:app", "--host", "0.0.0.0", "--port", "8000"]
O --from=builder copia arquivos do stage anterior. A imagem final nunca viu o gcc — ele existiu apenas no stage de build e é descartado. Para ML puro com scikit-learn/pandas, a diferença de tamanho costuma ser de 200–400 MB.
docker-compose — orquestrar serviços em desenvolvimento

Em desenvolvimento, um serviço de ML raramente existe sozinho. Ele precisa de um banco de dados, talvez um servidor MLflow, talvez um cache Redis. O docker-compose define todos esses serviços em um único arquivo YAML e os inicializa com um comando.

docker-compose.yml — modelo + MLflow server
services:

  # Servidor MLflow para rastreamento e Registry
  mlflow:
    image: ghcr.io/mlflow/mlflow:latest
    ports:
      - "5000:5000"
    volumes:
      - ./mlartifacts:/mlartifacts
      - ./mlflow.db:/mlflow.db
    command: >
      mlflow server
      --backend-store-uri sqlite:///mlflow.db
      --default-artifact-root /mlartifacts
      --host 0.0.0.0

  # Serviço de inferência — nossa imagem
  modelo:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8000:8000"
    environment:
      - MLFLOW_TRACKING_URI=http://mlflow:5000
    depends_on:
      - mlflow      # espera o MLflow subir antes
    volumes:
      - ./models:/app/models  # monta modelos sem rebuildar imagem

  # PostgreSQL opcional — para persistência de dados
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: mlops_db
      POSTGRES_USER: mlops
      POSTGRES_PASSWORD: ${DB_PASSWORD}   # vem do .env
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
Bash — comandos docker-compose essenciais
# Subir todos os serviços (build se necessário)
docker compose up --build

# Subir em segundo plano (detached)
docker compose up -d

# Ver logs de um serviço específico
docker compose logs -f modelo

# Reiniciar apenas o serviço do modelo (sem rebuildar)
docker compose restart modelo

# Parar tudo
docker compose down

# Parar e remover volumes (cuidado: apaga dados do BD)
docker compose down -v
O script de inferência — serve.py mínimo

Antes de empacotar, precisamos de um script que carrega o modelo e serve predições. Esta semana fazemos a versão mínima — na Semana 9 vamos aprofundar FastAPI com validação de input, documentação e testes.

Python — src/serve.py
import pickle
import numpy as np
from fastapi import FastAPI
from pydantic import BaseModel
from typing import List

app = FastAPI(title="Modelo de Inferência", version="1.0")

# Carrega o modelo na inicialização do servidor (uma vez só)
with open("models/model.pkl", "rb") as f:
    MODEL = pickle.load(f)

class InputData(BaseModel):
    features: List[float]

class Prediction(BaseModel):
    prediction: int
    probability: float

@app.get("/")
def health():
    return {"status": "ok", "modelo": str(type(MODEL).__name__)}

@app.post("/predict", response_model=Prediction)
def predict(data: InputData):
    X = np.array(data.features).reshape(1, -1)
    pred  = MODEL.predict(X)[0]
    proba = MODEL.predict_proba(X)[0][pred]
    return Prediction(prediction=int(pred), probability=float(proba))
Bash — testar com curl depois do docker run
# Build da imagem
docker build -t modelo-inferencia:1.0 .

# Rodar o container
docker run -p 8000:8000 --rm modelo-inferencia:1.0

# Testar health check

# Fazer predição
curl -X POST http://localhost:8000/predict \

  -d '{"features": [0.5, -1.2, 0.8, 1.1, -0.3, 0.6, 1.5, -0.9, 0.2, 0.7]}'

Estratégias de Atualização e Atividade Prática
Como atualizar o modelo sem downtime

Quando o modelo é retreinado, como você leva a nova versão para produção sem interromper o serviço? Existem três padrões principais, cada um com trade-offs diferentes.

EstratégiaComo funcionaVantagemRisco
Rebuildar imagem Novo modelo → novo Dockerfile → nova tag → trocar container Imagem 100% reproduzível, rollback por tag Downtime durante a troca (mitigado com orquestrador)
Volume mount Modelo em volume montado — trocar o arquivo sem rebuildar Atualização instantânea, sem rebuild Imagem não é mais autossuficiente — depende do volume
Carregar do Registry Container busca modelo do MLflow Registry na inicialização Desacoplado — trocar no Registry, restartar container Dependência de rede na inicialização
Padrão recomendado em produção: imagem com modelo embutido (opção 1) + tag semântica (v1.2.3) + rollback por tag. O volume mount é aceitável em desenvolvimento para agilizar iterações. O Registry serve bem para ambientes com Kubernetes (Semana 14).
Versionando imagens Docker de ML

Tags informativas são essenciais para rastrear qual modelo está em cada imagem — especialmente quando há rollbacks.

Bash — convenção de tags para imagens de ML
# Tag com versão semântica
docker build -t modelo-churn:1.2.0 .

# Tag com hash do commit Git — rastreável ao código exato
GIT_SHA=$(git rev-parse --short HEAD)
docker build -t modelo-churn:${GIT_SHA} .

# Tag com data — útil para modelos retreinados periodicamente
docker build -t modelo-churn:$(date +%Y%m%d) .

# Combinar: versão + sha
docker build \
  -t modelo-churn:1.2.0 \
  -t modelo-churn:1.2.0-${GIT_SHA} \
  -t modelo-churn:latest \
  .

# Rollback: basta trocar a tag no docker run / compose
docker run modelo-churn:1.1.0   # versão anterior
Nunca use :latest como única tag em produção. A tag latest é mutável — o mesmo nome pode apontar para imagens completamente diferentes em momentos diferentes. Se você fizer rollback para :latest, pode não saber qual versão realmente está rodando. Use sempre tags imutáveis (1.2.0, hash do commit) para produção.
Carregando modelo do MLflow Registry no container

Uma alternativa ao modelo embutido na imagem é fazer o container buscar o modelo do Registry na inicialização. A imagem fica menor e a atualização do modelo não exige rebuild — mas o container precisa de acesso à rede e ao servidor MLflow.

Python — src/serve_registry.py
import os
import mlflow.pyfunc
from fastapi import FastAPI
from pydantic import BaseModel
from typing import List

app = FastAPI()

# URI do servidor MLflow vem de variável de ambiente
TRACKING_URI = os.getenv("MLFLOW_TRACKING_URI", "http://localhost:5000")
MODEL_NAME   = os.getenv("MODEL_NAME", "classificador-churn")
MODEL_STAGE  = os.getenv("MODEL_STAGE", "Production")

mlflow.set_tracking_uri(TRACKING_URI)

print(f"Carregando {MODEL_NAME}/{MODEL_STAGE} de {TRACKING_URI}...")
MODEL = mlflow.pyfunc.load_model(f"models:/{MODEL_NAME}/{MODEL_STAGE}")
print("Modelo carregado com sucesso.")

class InputData(BaseModel):
    features: List[float]

@app.post("/predict")
def predict(data: InputData):
    import pandas as pd
    X = pd.DataFrame([data.features])
    pred = MODEL.predict(X)
    return {"prediction": int(pred[0])}

Passo 01 — Criar o serve.py

Crie src/serve.py com o código FastAPI mínimo da aula: endpoint GET / para health check e POST /predict que recebe uma lista de floats e retorna predição e probabilidade. O modelo deve ser carregado de models/model.pkl na inicialização.

Básico
Python — src/serve.py
import pickle
import numpy as np
from fastapi import FastAPI
from pydantic import BaseModel
from typing import List

app = FastAPI(title="Modelo de Inferência")

with open("models/model.pkl", "rb") as f:
    MODEL = pickle.load(f)

class InputData(BaseModel):
    features: List[float]

@app.get("/")
def health():
    return {"status": "ok", "modelo": type(MODEL).__name__}

@app.post("/predict")
def predict(data: InputData):
    X     = np.array(data.features).reshape(1, -1)
    pred  = MODEL.predict(X)[0]
    proba = MODEL.predict_proba(X)[0][int(pred)]
    return {"prediction": int(pred), "probability": round(float(proba), 4)}

Passo 04 — Inspecionar o tamanho da imagem

Use docker images para ver o tamanho da imagem gerada. Depois altere o Dockerfile para usar FROM python:3.11 (sem o -slim), rebuilde com a tag modelo-inferencia:gordinha e compare os tamanhos. Qual a diferença? O que justificaria usar a versão não-slim?

Médio
Bash
docker images | grep modelo-inferencia
# modelo-inferencia   1.0        ~350MB  (com slim)
# modelo-inferencia   gordinha   ~950MB  (sem slim)
A diferença típica é de 500–700 MB. A versão não-slim só se justifica quando um pacote do requirements precisa de compiladores em runtime — o que é raro para inferência com scikit-learn. Para pacotes que exigem compilação na instalação, o correto é usar multi-stage build (compilar no stage gordo, copiar para o slim).
