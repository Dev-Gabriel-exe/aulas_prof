Questão 1 · Integração DVC + MLflow (Bloco P1)

Um cientista de dados executa git checkout v1.0 && dvc checkout para voltar ao estado de 3 meses atrás. Em seguida, re-executa o pipeline de treino. Os experimentos agora aparecem no MLflow Tracking. Qual afirmação descreve corretamente o que aconteceu?

A
O dvc checkout substituiu o banco de dados do MLflow pela versão antiga, apagando experimentos recentes
B
O git checkout voltou o código e os arquivos .dvc ao estado da tag v1.0; o dvc checkout restaurou os arquivos de dados referenciados. O MLflow continua acumulando — o novo run aparece como um experimento adicional, não substitui os anteriores
C
O dvc checkout só funciona se o servidor MLflow estiver offline, caso contrário os dados são sobrescritos
D
O git checkout voltou apenas o código; os dados de treino permanecem na versão mais recente porque o DVC não versiona arquivos grandes

Resposta: B
Git e DVC trabalham em camadas separadas. O git checkout v1.0 restaura o código-fonte E os ponteiros .dvc (que apontam para hashes de dados). O dvc checkout usa esses ponteiros para baixar do storage remoto os arquivos de dados correspondentes à versão v1.0. O MLflow Tracking é um banco independente — acumula runs, nunca sobrescreve. O novo treino gerará um novo run_id com o timestamp atual, permitindo comparar diretamente com os runs históricos. Esse é o poder da combinação DVC + MLflow: reprodutibilidade total dos dados + rastreabilidade dos experimentos.

Questão 2 · Docker + FastAPI (Bloco P2)

Analise o trecho de Dockerfile abaixo e identifique os dois problemas que prejudicam performance e segurança:

Dockerfile com problemas
FROM python:3.11
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
USER root
EXPOSE 8000
CMD ["uvicorn", "src.serve:app", "--host", "0.0.0.0", "--port", "8000"]
A
Imagem base muito pesada (deveria ser slim) e COPY . . antes do pip install invalida cache de dependências a cada mudança de código
B
O CMD usa uvicorn em vez de gunicorn, que é mais adequado para produção
C
O WORKDIR /app está incorreto — deveria ser /usr/app para seguir o padrão Linux
D
O EXPOSE 8000 deveria ser EXPOSE 80 para seguir o padrão HTTP

Resposta: A
Dois problemas clássicos: (1) Imagem pesada: python:3.11 tem ~900MB com compiladores e ferramentas desnecessárias para inferência. A versão python:3.11-slim tem ~130MB — sem alterar funcionalidade para modelos sklearn. (2) Ordem de COPY: COPY . . antes do pip install invalida o cache da camada de dependências toda vez que qualquer arquivo do projeto muda — mesmo um comentário. A correção é: COPY requirements.txt . → RUN pip install → COPY src/ .. O USER root é também um problema de segurança (deveria ser um usuário sem privilégios), mas não foi listado nas opções — A ainda é a resposta mais completa.

Questão 3 · Evidently + Prometheus (Bloco P2)

Um time de MLOps usa Evidently para monitoramento de drift diário e Prometheus + Grafana para métricas em tempo real. Qual é a diferença fundamental entre o que cada ferramenta monitora?

A
O Evidently monitora métricas de infraestrutura (CPU, memória, latência); o Prometheus monitora qualidade dos dados do modelo
B
O Evidently monitora a distribuição dos dados de entrada/saída do modelo (data drift, concept drift) com comparação de períodos; o Prometheus monitora métricas de sistema em tempo real (latência, throughput, erros)
C
São ferramentas equivalentes — podem ser substituídas uma pela outra sem perda de funcionalidade
D
O Prometheus é mais adequado para drift porque coleta dados continuamente; o Evidently é apenas para relatórios históricos sem valor operacional

Resposta: B
São ferramentas complementares, não substitutas. O Evidently compara distribuições estatísticas entre dois períodos (treino vs produção, semana passada vs esta semana) usando testes como KS, PSI e chi-quadrado — responde "os dados estão mudando?". O Prometheus coleta métricas numéricas em tempo real — responde "a API está saudável agora?". Em uma stack madura, ambas coexistem: Prometheus alerta se a API caiu ou a latência disparou; Evidently alerta se o modelo está recebendo dados fora da distribuição de treino. A integração ideal é exportar métricas de PSI do Evidently para o Prometheus, permitindo visualizar drift no mesmo Grafana que mostra latência.

Questão 4 · Kubernetes + Celery (Bloco P3)

Uma empresa tem dois casos de uso: (1) clientes fazem predições individuais em tempo real via frontend, esperando resposta em <200ms; (2) analistas fazem uploads de CSVs com 200k registros para scoring semanal. Qual arquitetura atende melhor os dois casos simultaneamente?

A
FastAPI síncrona para ambos — o endpoint de batch deve ter timeout aumentado para 10 minutos
B
FastAPI síncrona para predições em tempo real (caso 1); FastAPI + Celery assíncrono para scoring em lote (caso 2), com filas separadas e workers dedicados para cada tipo de carga
C
Kafka para ambos os casos — o Kafka resolve tanto tempo real quanto lote com o mesmo consumer
D
BentoML com batching dinâmico para ambos — o max_latency_ms pode ser ajustado para acomodar os dois perfis

Resposta: B
Esta é a arquitetura híbrida correta. Para predições em tempo real (<200ms), o FastAPI síncrono é ideal — sem overhead de fila. Para scoring em lote de 200k registros, o Celery assíncrono evita bloquear a API durante o processamento longo: o analista recebe um task_id imediatamente e consulta o status depois. Filas separadas (realtime e batch) garantem que jobs pesados não atrapalham a fila de predições rápidas — workers dedicados para cada fila. No Kubernetes, os workers Celery podem ter HPA separado do FastAPI, escalando apenas quando há jobs de batch na fila.

Questão 5 · SHAP + Auditoria (Bloco P3)

Um banco usa um modelo de crédito e precisa atender ao Art. 20 da LGPD (direito à explicação de decisões automatizadas). Um cliente teve crédito negado. Qual implementação atende corretamente ao requisito legal?

A
Informar ao cliente a probabilidade retornada pelo modelo (ex: 0.78) e os pesos globais das features do modelo
B
Usar SHAP local para calcular a contribuição de cada feature para a predição específica deste cliente e apresentar as top-3 razões da negativa ("alto número de dívidas abertas", "histórico de crédito baixo")
C
Apresentar o código-fonte do modelo para que o cliente possa verificar como a decisão foi tomada
D
Enviar um e-mail genérico explicando que o modelo usa diversas variáveis financeiras para calcular o risco

Resposta: B
A LGPD exige explicação da decisão automatizada específica para aquele indivíduo — não uma explicação genérica do modelo. O SHAP global (importância média das features) não satisfaz esse requisito porque descreve o comportamento médio do modelo, não por que este cliente específico foi negado. O SHAP local calcula a contribuição de cada feature para esta predição individual — é exatamente o que o Art. 20 pede. O endpoint /explain que construímos na Semana 17 implementa isso: recebe os dados do cliente, retorna os valores SHAP locais com direção (aumenta/reduz risco) e pode ser traduzido em linguagem natural para o cliente.

Questão 6 · Kafka + Model Registry (Integração)

Uma equipe quer atualizar o modelo de detecção de fraude rodando em um consumer Kafka sem interromper o fluxo de eventos. O modelo atual está em produção no MLflow Registry como versão 3. A nova versão 4 foi treinada e está em Staging. Qual a sequência correta?

A
Parar o consumer Kafka, atualizar o código para carregar a versão 4, restartar — causando janela de indisponibilidade
B
Promover versão 4 para Production no Registry (com archive_existing_versions=True) → restartar o consumer graciosamente → o consumer carrega automaticamente models:/fraude/Production que agora aponta para v4 → versão 3 arquivada automaticamente
C
Publicar um evento especial no tópico Kafka com instrução para os consumers recarregarem o modelo
D
Criar um novo tópico Kafka e redirecionar o producer — o consumer antigo continua com o modelo v3 no tópico original

Resposta: B
Esta questão conecta o MLflow Registry (Semana 5) com o consumer Kafka (Semana 18). A URI models:/fraude/Production é dinâmica — sempre aponta para a versão atualmente em Production. Com archive_existing_versions=True, a v3 é arquivada automaticamente quando a v4 é promovida. O consumer precisa ser restartado para recarregar o modelo (o carregamento ocorre na inicialização). Em um ambiente Kubernetes, isso seria feito com kubectl rollout restart deployment/consumer-fraude que faz rolling restart dos pods — zero downtime se houver múltiplas réplicas. A opção A causa downtime desnecessário; a C não existe como mecanismo do Kafka; a D cria complexidade sem necessidade.

Questão 7 · Integração Completa — Diagnóstico de Produção

Um sistema de ML em produção apresenta: (1) Grafana mostra aumento gradual de latência P95 nas últimas 2 semanas; (2) Evidently reporta PSI = 0.23 na feature "valor_transacao"; (3) a acurácia do modelo caiu de 0.92 para 0.87; (4) nenhum alerta de infraestrutura disparou. Qual diagnóstico e ação são mais adequados?

A
Problema de infraestrutura — escalar horizontalmente os pods no Kubernetes para reduzir a latência
B
Data drift confirmado (PSI 0.23 > 0.2 = significativo) causando degradação do modelo. Ação: retreinar com dados recentes, promover no Registry e fazer rolling update. Investigar também se a mudança em valor_transacao é seasonal (temporária) ou estrutural (permanente)
C
A queda de acurácia é normal e esperada — reavaliar o threshold de alerta para 0.80
D
Problema no Prometheus — o aumento de latência é artefato da coleta de métricas, não real

Resposta: B
Diagnóstico claro integrando três semanas: o PSI > 0.2 (limiar de drift significativo da Semana 11) confirma que a distribuição de valor_transacao mudou substancialmente. Isso explica a queda de acurácia (o modelo foi treinado numa distribuição diferente da que está recebendo em produção) e pode explicar o aumento de latência (inputs anômalos podem estar causando lentidão em camadas de validação). A ação correta é retreinar com dados recentes, validar no MLflow, promover no Registry (Semana 5) e fazer rolling update no Kubernetes (Semana 14). A investigação seasonal/estrutural é importante: se for sazonalidade (ex: Black Friday), pode não ser necessário retreinar — apenas monitorar. Se for mudança estrutural, o retreinamento é obrigatório.

Instruções da Segunda Chamada e Encerramento da Disciplina

Formato da segunda chamada

  • 10 questões mistas (mais amplo que as provas regulares)
  • Duração: 120 minutos
  • Folha de cola A5 manuscrita (frente e verso)
  • Cobre todo o semestre — Semanas 1 a 19
  • Questões transversais (conectam múltiplos blocos)

Estratégia para a folha de cola

  • Priorize limiares e valores numéricos (PSI 0.1/0.2)
  • Códigos HTTP: 401/422/429/503 — quando cada um
  • Comandos kubectl essenciais e flags do Dockerfile
  • Diferença Kafka vs RabbitMQ em uma frase
  • Fluxo SHAP: base_value + Σshap = predição

Apresentação de Projeto pendente

  • Mesma rubrica da Semana 19 (100 pontos)
  • 10 minutos de apresentação + 5 de perguntas
  • Demo ao vivo obrigatória — sem slides apenas
  • Repositório Git deve estar acessível antes de começar
  • docker-compose up deve funcionar sem configuração manual

Atenção especial

  • Questões de código: leia linha a linha antes de responder
  • Questões de diagnóstico: identifique TODOS os sintomas antes de escolher
  • Não confunda Evidently (drift de dados) com Prometheus (métricas de infra)
  • Não confunda Kafka (stream) com Celery (tarefas assíncronas)
