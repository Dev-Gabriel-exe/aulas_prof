A Stack de Observabilidade e Métricas com prometheus_client
Como Prometheus e Grafana trabalham juntos

A stack de observabilidade padrão da indústria combina três peças que se complementam. Entender o papel de cada uma evita confusão sobre o que configurar onde.

1
Sua aplicação (FastAPI) + prometheus_client Expõe um endpoint /metrics com métricas no formato Prometheus (texto simples). Você define quais métricas coletar.
2
Prometheus — coleta e armazena Faz scraping periódico do /metrics de cada serviço. Armazena como séries temporais. Permite consultas com PromQL.
3
Grafana — visualiza e alerta Consulta o Prometheus com PromQL, exibe gráficos e dashboards em tempo real. Envia alertas via email, Slack, PagerDuty quando limiares são ultrapassados.
+
Evidently → Prometheus (integração opcional) Métricas de drift calculadas pelo Evidently podem ser exportadas para o Prometheus e exibidas no mesmo dashboard de observabilidade.
O modelo mental: a aplicação publica métricas. O Prometheus coleta. O Grafana visualiza. Essa separação de responsabilidades significa que você pode mudar a ferramenta de visualização sem tocar na aplicação.
Os 4 tipos de métricas Prometheus

Counter

Valor que só cresce (nunca diminui). Reseta ao reiniciar o serviço.

predicoes_total, erros_total, requisicoes_http_total

Gauge

Valor que pode subir e descer. Representa estado atual.

modelo_carregado, predicoes_em_andamento, psi_feature_1

Histogram

Distribui observações em buckets. Permite calcular percentis (p50, p95, p99).

latencia_inferencia_segundos, tamanho_payload_bytes

Summary

Semelhante ao Histogram mas calcula percentis no cliente. Menos flexível para agregação.

latencia_percentis — prefira Histogram na maioria dos casos
Instrumentando a API FastAPI com prometheus_client
Python — src/serve.py instrumentado com métricas
import time, pickle
import numpy as np
from fastapi import FastAPI, HTTPException, Request, Response
from pydantic import BaseModel
from typing import List
from prometheus_client import (
    Counter, Histogram, Gauge,
    generate_latest, CONTENT_TYPE_LATEST
)

app = FastAPI(title="API de Inferência com Métricas")

# ── Definir métricas ────────────────────────────────────────────
PREDICOES_TOTAL = Counter(
    "predicoes_total",
    "Total de predições realizadas",
    ["resultado"]  # label: "sucesso" ou "erro"
)
LATENCIA = Histogram(
    "latencia_inferencia_segundos",
    "Latência de inferência em segundos",
    buckets=[.001, .005, .01, .025, .05, .1, .25, .5, 1.0]
)
PREDICOES_ATIVAS = Gauge(
    "predicoes_em_andamento",
    "Predições sendo processadas agora"
)
REQUISICOES_HTTP = Counter(
    "requisicoes_http_total",
    "Total de requisições HTTP",
    ["metodo", "endpoint", "status"]
)

# ── Carregar modelo ─────────────────────────────────────────────
with open("models/model.pkl", "rb") as f:
    MODEL = pickle.load(f)

MODEL_CARREGADO = Gauge("modelo_carregado", "1 se o modelo está carregado")
MODEL_CARREGADO.set(1)

# ── Middleware para métricas HTTP ───────────────────────────────
@app.middleware("http")
async def middleware_metricas(request: Request, call_next):
    inicio = time.perf_counter()
    response = await call_next(request)
    duracao = time.perf_counter() - inicio
    REQUISICOES_HTTP.labels(
        metodo=request.method,
        endpoint=request.url.path,
        status=response.status_code
    ).inc()
    return response

# ── Endpoint de métricas (Prometheus faz scraping aqui) ─────────
@app.get("/metrics")
def metrics():
    return Response(generate_latest(), media_type=CONTENT_TYPE_LATEST)

# ── Endpoints da aplicação ──────────────────────────────────────
class InputData(BaseModel):
    features: List[float]

@app.get("/health")
def health():
    return {"status": "healthy"}

@app.post("/predict")
def predict(data: InputData):
    PREDICOES_ATIVAS.inc()
    inicio = time.perf_counter()
    try:
        X     = np.array(data.features).reshape(1, -1)
        pred  = int(MODEL.predict(X)[0])
        proba = float(MODEL.predict_proba(X)[0][pred])
        PREDICOES_TOTAL.labels(resultado="sucesso").inc()
        return {"prediction": pred, "probability": round(proba, 4)}
    except Exception as e:
        PREDICOES_TOTAL.labels(resultado="erro").inc()
        raise HTTPException(400, detail=str(e))
    finally:
        LATENCIA.observe(time.perf_counter() - inicio)
        PREDICOES_ATIVAS.dec()

Configurando o Prometheus e Consultando com PromQL
Como o Prometheus coleta métricas — scraping

O Prometheus usa o modelo pull: ele mesmo busca as métricas em cada serviço no intervalo configurado. O oposto do modelo push (onde a aplicação envia). O modelo pull é mais robusto porque o Prometheus detecta automaticamente quando um serviço cai (scraping falha).

prometheus.yml — configuração de scraping
global:
  scrape_interval: 15s       # coletar métricas a cada 15 segundos
  evaluation_interval: 15s   # avaliar regras de alerta a cada 15s

# Regras de alerta (arquivo separado)
rule_files:
  - "alerts.yml"

scrape_configs:

  # O próprio Prometheus se monitora
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]

  # Nossa API de inferência
  - job_name: "modelo-inferencia"
    static_configs:
      - targets: ["modelo:8000"]   # nome do serviço no docker-compose
    metrics_path: "/metrics"
    scrape_interval: 10s           # mais frequente para a API
PromQL — consultando métricas

PromQL (Prometheus Query Language) é a linguagem para consultar as métricas armazenadas. Parece complexo no início, mas alguns padrões cobrem 90% dos casos de ML.

O que você quer saberQuery PromQL
Taxa de predições por segundo (últimos 5 min)rate(predicoes_total[5m])
Taxa de erros por segundorate(predicoes_total{resultado="erro"}[5m])
Percentil 95 de latência (últimos 5 min)histogram_quantile(0.95, rate(latencia_inferencia_segundos_bucket[5m]))
Percentil 99 de latênciahistogram_quantile(0.99, rate(latencia_inferencia_segundos_bucket[5m]))
Taxa de erro como percentual do totalrate(predicoes_total{resultado="erro"}[5m]) / rate(predicoes_total[5m])
Modelo está carregado?modelo_carregado == 0
A função rate(): converte um Counter em taxa por segundo. Sem ela, o gráfico de um Counter mostra apenas o total acumulado (número sempre crescente). Com rate(counter[5m]) você vê "quantas vezes por segundo isso aconteceu nos últimos 5 minutos" — muito mais útil para análise de comportamento.
Regras de alerta no Prometheus

O Prometheus pode avaliar queries periodicamente e disparar alertas quando uma condição é verdadeira. Os alertas são enviados para o Alertmanager, que encaminha para email, Slack, PagerDuty etc.

alerts.yml — regras de alerta para o modelo
groups:
  - name: modelo-inferencia
    rules:

      # Alerta: taxa de erros acima de 5% por 2 minutos
      - alert: TaxaErroAlta
        expr: >
          rate(predicoes_total{resultado="erro"}[5m])
          /
          rate(predicoes_total[5m]) > 0.05
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "Taxa de erros de predição acima de 5%"
          description: "Taxa atual: {{ $value | humanizePercentage }}"

      # Alerta: latência p95 acima de 500ms por 5 minutos
      - alert: LatenciaAltaP95
        expr: >
          histogram_quantile(0.95,
            rate(latencia_inferencia_segundos_bucket[5m])
          ) > 0.5
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Latência P95 acima de 500ms"
          description: "P95 atual: {{ $value | humanizeDuration }}"

      # Alerta: modelo não está carregado
      - alert: ModeloNaoCarregado
        expr: modelo_carregado == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Modelo de inferência não está carregado"

Grafana e Stack Completa via docker-compose
docker-compose.yml — stack completa de observabilidade
docker-compose.yml
services:

  # API de inferência
  modelo:
    build: .
    ports:
      - "8000:8000"
    volumes:
      - ./models:/app/models

  # Prometheus — coleta métricas
  prometheus:
    image: prom/prometheus:v2.50.1
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
      - ./alerts.yml:/etc/prometheus/alerts.yml
      - prometheus_data:/prometheus
    command:
      - "--config.file=/etc/prometheus/prometheus.yml"
      - "--storage.tsdb.retention.time=15d"
    depends_on:
      - modelo

  # Grafana — dashboards
  grafana:
    image: grafana/grafana:10.4.0
    ports:
      - "3000:3000"
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=mlops123
      - GF_USERS_ALLOW_SIGN_UP="false"
    volumes:
      - grafana_data:/var/lib/grafana
      - ./grafana/provisioning:/etc/grafana/provisioning
    depends_on:
      - prometheus

volumes:
  prometheus_data:
  grafana_data:
Grafana — configurando dashboards

O Grafana pode ser configurado via interface web (porta 3000) ou via arquivos de provisionamento (recomendado para versionamento). Vamos ver os dois.

grafana/provisioning/datasources/prometheus.yml
apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    editable: false
Login Grafana: acesse http://localhost:3000 com usuário admin e senha mlops123 (definida no compose). O datasource Prometheus já está configurado automaticamente via provisionamento.

Painéis essenciais para um dashboard de modelo ML:

  • Throughput: rate(predicoes_total[5m]) — req/s ao longo do tempo
  • Latência P50/P95/P99: histogram_quantile(0.95, rate(latencia_inferencia_segundos_bucket[5m]))
  • Taxa de erros: rate(predicoes_total{resultado="erro"}[5m]) / rate(predicoes_total[5m]) * 100
  • Gauge de saúde: modelo_carregado — 1 (verde) ou 0 (vermelho)
  • Distribuição de latência: heatmap dos buckets do Histogram ao longo do tempo

Médio
Passo 04 — Criar dashboard no Grafana

Acesse o Grafana em localhost:3000. Crie um novo dashboard com 4 painéis: (1) Throughput em req/s, (2) Latência P95 em ms, (3) Taxa de erros em %, (4) Gauge de saúde do modelo. Para cada painel, use as queries PromQL da aula. Salve o dashboard com o nome "Modelo de Inferência".

Médio
Passo 05 — Desafio: criar regra de alerta e simular disparo

Crie o arquivo alerts.yml com a regra de alerta para latência P95 acima de 200ms. Referencie-o no prometheus.yml. Depois, modifique temporariamente o endpoint /predict para adicionar um time.sleep(0.3) artificial — gere tráfego e observe o alerta disparar na aba Alerts do Prometheus em localhost:9090/alerts.

Desafio
Critério de conclusão: o endpoint localhost:8000/metrics deve retornar texto com predicoes_total, latencia_inferencia_segundos_bucket e modelo_carregado. O Grafana deve mostrar ao menos dois painéis com dados reais após gerar tráfego.
