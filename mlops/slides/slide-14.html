Por que Kubernetes — e os Objetos Essenciais para ML
Kubernetes — o orquestrador de containers

Kubernetes (k8s) é um sistema de orquestração de containers que resolve os problemas que surgem quando você precisa rodar sua aplicação de forma confiável em produção: alta disponibilidade, escalonamento automático, auto-recuperação e deploys sem interrupção.

A ideia central é simples: você declara o estado desejado (quero 3 réplicas do meu modelo rodando, com 500m de CPU cada) e o Kubernetes trabalha continuamente para manter esse estado — reiniciando containers que morrem, distribuindo carga, agendando workloads em nodes disponíveis.

Docker Compose vs Kubernetes: o Compose é excelente para desenvolvimento local — simples, rápido, sem overhead. O Kubernetes é para produção — complexo, mas com garantias que o Compose não oferece: self-healing, rolling updates, escalonamento automático, gerenciamento de secrets, scheduling inteligente entre múltiplas máquinas.
Os objetos essenciais que você precisa conhecer

Pod

Menor unidade do k8s. Encapsula um ou mais containers que compartilham rede e storage. Efêmero — pode morrer e ser recriado a qualquer momento.

Deployment

Gerencia um conjunto de Pods idênticos (réplicas). Garante que o número desejado está sempre rodando. Controla rolling updates e rollbacks.

Service

Expõe Pods com um IP e DNS estáveis. Faz load balancing entre réplicas. Tipos: ClusterIP (interno), NodePort, LoadBalancer (externo).

HPA

HorizontalPodAutoscaler. Ajusta o número de réplicas do Deployment automaticamente com base em métricas (CPU, memória, métricas customizadas).

ConfigMap

Armazena configurações não-sensíveis (variáveis de ambiente, arquivos de config). Injetado nos Pods sem rebuildar a imagem.

Secret

Como ConfigMap, mas para dados sensíveis (senhas, tokens, chaves de API). Armazenado codificado em base64 (não criptografado por padrão — use Vault em prod).

Deployment para o modelo de inferência

O manifesto YAML do Deployment declara como o Kubernetes deve rodar seu modelo. Cada campo tem um propósito específico para garantir estabilidade e performance.

YAML — deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: modelo-inferencia
  labels:
    app: modelo-inferencia
spec:
  replicas: 3                    # 3 réplicas para alta disponibilidade
  selector:
    matchLabels:
      app: modelo-inferencia
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1               # máximo 1 pod extra durante update
      maxUnavailable: 0         # nunca derrubar pod antes de outro subir
  template:
    metadata:
      labels:
        app: modelo-inferencia
    spec:
      containers:
        - name: modelo
          image: modelo-inferencia:1.0
          ports:
            - containerPort: 8000
          resources:
            requests:
              cpu: "250m"          # 0.25 CPU garantido
              memory: "256Mi"      # 256MB RAM garantido
            limits:
              cpu: "500m"          # máximo 0.5 CPU
              memory: "512Mi"      # máximo 512MB RAM
          env:
            - name: MODEL_VERSION
              valueFrom:
                configMapKeyRef:
                  name: modelo-config
                  key: version
          readinessProbe:          # só recebe tráfego quando pronto
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 10
            periodSeconds: 5
          livenessProbe:           # reinicia se não responder
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 30
            periodSeconds: 10
readinessProbe vs livenessProbe: a readiness sinaliza "este Pod está pronto para receber tráfego" — enquanto não passar, o Service não envia requisições para ele (crucial durante a inicialização do modelo). A liveness sinaliza "este Pod ainda está vivo" — se falhar repetidamente, o k8s mata e reinicia o container. Para modelos ML, o endpoint /health que verifica se o modelo carregou é essencial para ambos.
Service e ConfigMap
YAML — service.yaml
apiVersion: v1
kind: Service
metadata:
  name: modelo-inferencia-svc
spec:
  selector:
    app: modelo-inferencia    # aponta para pods com este label
  ports:
    - port: 80               # porta exposta pelo Service
      targetPort: 8000       # porta do container
  type: LoadBalancer          # em cloud: IP externo | em kind: NodePort
YAML — configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: modelo-config
data:
  version: "1.2.0"
  mlflow_tracking_uri: "http://mlflow-svc:5000"
  n_features: "10"

HPA — Escalonamento Automático e Rolling Update sem Downtime
HorizontalPodAutoscaler — escalar conforme a demanda

O HPA monitora métricas do cluster e ajusta automaticamente o número de réplicas do Deployment. Para modelos de ML, os casos de uso mais comuns são: escalar quando a CPU está alta (picos de inferência) ou quando uma métrica customizada do Prometheus ultrapassa um limiar.

YAML — hpa.yaml baseado em CPU
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: modelo-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: modelo-inferencia
  minReplicas: 2            # mínimo 2 réplicas (sempre disponível)
  maxReplicas: 10           # máximo 10 réplicas
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70   # escalar quando CPU média > 70%
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80   # escalar quando memória > 80%
YAML — hpa com métrica customizada do Prometheus
# Requer prometheus-adapter instalado no cluster
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: modelo-hpa-custom
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: modelo-inferencia
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Pods
      pods:
        metric:
          name: predicoes_por_segundo   # métrica do Prometheus
        target:
          type: AverageValue
          averageValue: "50"           # escalar acima de 50 pred/s por pod
Como o HPA funciona: a cada 15 segundos, o HPA consulta as métricas. Se a CPU média das réplicas ultrapassar 70%, ele aumenta o número de réplicas (scale-out). Quando a carga diminui por um período configurável (padrão 5 minutos), reduz as réplicas (scale-in). O minReplicas: 2 garante que nunca haverá menos de 2 réplicas — importante para alta disponibilidade.
Rolling Update — atualizar o modelo sem derrubar o serviço

Quando você treina um novo modelo e quer colocá-lo em produção, o Kubernetes faz o Rolling Update: sobe novos Pods com a nova imagem enquanto mantém os antigos respondendo, e só remove os antigos quando os novos passam na readinessProbe.

Bash — fluxo completo de atualização do modelo
# 1. Retreinar e gerar nova versão do modelo
python pipeline.py run  # ou mlflow pipelines run

# 2. Buildar nova imagem Docker com o modelo atualizado
docker build -t modelo-inferencia:1.3.0 .

# Em ambiente com registry (kind usa imagem local):
# docker push meu-registry/modelo-inferencia:1.3.0

# 3. Atualizar o Deployment com a nova imagem
kubectl set image deployment/modelo-inferencia \
  modelo=modelo-inferencia:1.3.0

# 4. Acompanhar o rollout em tempo real
kubectl rollout status deployment/modelo-inferencia

# 5. Verificar histórico de versões
kubectl rollout history deployment/modelo-inferencia

# 6. Rollback para versão anterior se algo der errado
kubectl rollout undo deployment/modelo-inferencia

# Rollback para versão específica
kubectl rollout undo deployment/modelo-inferencia \
  --to-revision=2
A readinessProbe é o que garante zero downtime: com maxUnavailable: 0 no Deployment, o Kubernetes nunca remove um Pod antigo antes que o novo passe na readinessProbe. Como nosso /health verifica que o modelo está carregado, o tráfego só vai para o novo Pod depois que o modelo inicializou completamente.

Kind — Kubernetes Local e Atividade Prática
kind — Kubernetes in Docker

O kind (Kubernetes in Docker) cria um cluster Kubernetes completo usando containers Docker como nodes. É a forma mais simples de ter um k8s local para desenvolvimento e aprendizado — sem precisar de VMs ou cloud.

# Instalar kind e kubectl (macOS)
brew install kind kubectl

# Linux
curl -Lo ./kind https://kind.sigs.k8s.io/dl/v0.22.0/kind-linux-amd64
chmod +x ./kind && mv ./kind /usr/local/bin/kind

# (kind não acessa o Docker registry local automaticamente)

Bash — deploy completo no kind
# Criar namespace dedicado para o projeto
kubectl create namespace mlops

# Aplicar todos os manifestos
kubectl apply -f k8s/configmap.yaml  -n mlops
kubectl apply -f k8s/deployment.yaml -n mlops

Atividade · Deploy no Kubernetes Local
Do docker-compose ao cluster kind — com HPA e rolling update
Use a imagem Docker da Semana 8/9. Ao final, seu modelo deve estar rodando com 3 réplicas em um cluster kind, acessível via port-forward, com HPA configurado e um rolling update bem-sucedido demonstrado.
Passo 01 — Criar cluster kind e carregar a imagem

Instale o kind e kubectl. Crie um cluster chamado mlops com 1 control-plane e 2 workers. Carregue a imagem modelo-inferencia:1.0 para dentro do cluster com kind load docker-image. Verifique que os 3 nodes aparecem como Ready.

Básico
Bash
kind create cluster --name mlops
kind load docker-image modelo-inferencia:1.0 --name mlops
kubectl get nodes   # deve mostrar 1 control-plane + 2 workers
Passo 02 — Escrever e aplicar os manifestos

Crie a pasta k8s/ com os arquivos deployment.yaml, service.yaml e configmap.yaml. O Deployment deve ter 3 réplicas, requests de 250m CPU e 256Mi de memória, e as probes de readiness e liveness apontando para /health. Aplique com kubectl apply -f k8s/ e aguarde todos os Pods ficarem Running.

Básico
Passo 03 — Acessar a API via port-forward

Use kubectl port-forward svc/modelo-inferencia-svc 8080:80 para acessar o Service localmente. Em outro terminal, faça um GET /health e um POST /predict com curl. Verifique nos logs de qual Pod a requisição foi atendida usando kubectl logs.

Básico
Bash
kubectl port-forward svc/modelo-inferencia-svc 8080:80 &
curl http://localhost:8080/health
curl -X POST http://localhost:8080/predict \
  -H "Content-Type: application/json" \
  -d '{"features":[0.5,-1.2,0.8,1.1,-0.3,0.6,1.5,-0.9,0.2,0.7]}'

# Ver logs de todos os pods do deployment
kubectl logs -l app=modelo-inferencia --tail=5
Passo 04 — Configurar o HPA

Aplique o hpa.yaml com minReplicas: 2, maxReplicas: 8 e target de CPU em 70%. Execute kubectl get hpa e observe as colunas TARGETS e REPLICAS. Escale manualmente para 1 réplica com kubectl scale e veja o HPA corrigir para 2 automaticamente.

Médio
Bash
kubectl apply -f k8s/hpa.yaml
kubectl get hpa -w

# Tentar escalar abaixo do mínimo
kubectl scale deployment/modelo-inferencia --replicas=1
kubectl get pods -w  # HPA vai subir para 2 em ~30s
Passo 05 — Simular rolling update e rollback

Crie uma versão 1.1.0 da imagem (pode ser a mesma com tag diferente), carregue no kind com kind load, e execute o rolling update com kubectl set image. Acompanhe com kubectl rollout status. Enquanto o update acontece, faça requisições à API — confirme que não há interrupção. Depois, faça rollback para 1.0.

Desafio
Critério de conclusão: kubectl get pods deve mostrar 3 pods Running. kubectl get hpa deve mostrar o HPA ativo. Uma requisição POST /predict via port-forward deve retornar predição. kubectl rollout history deve mostrar pelo menos 2 revisões.
