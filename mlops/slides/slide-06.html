Revisão Express — Semanas 1 a 5
5–8 minutos por semana. O objetivo não é re-ensinar — é garantir que os conceitos-chave estão frescos antes do simulado.
Semana 1
O que é MLOps — Ciclo de Vida
  • ~87% dos projetos de ML nunca chegam a produção
  • MLOps = ML + Dev + Ops — não só DevOps com modelos
  • 4 fases: dados → experimentação → deploy → monitoramento
  • Ciclo contínuo — modelos degradam sem retreinamento
  • Data Scientist treina · ML Engineer faz o deploy · MLOps mantém a infraestrutura
  • Notebook: dados fixos, sem escala. Produção: dados dinâmicos, SLA, monitoramento
Semana 2
Reprodutibilidade — venv, Docker, Git
  • venv isola pacotes Python por projeto — nunca instale no Python global
  • conda: gerencia dependências de sistema (CUDA, C libs) além dos pacotes
  • requirements.txt com versões fixas garante mesmo ambiente em qualquer máquina
  • Docker: empacota OS + runtime + pacotes + código em imagem imutável
  • Copie requirements.txt antes do código no Dockerfile — aproveita cache de layers
  • Git: dados e modelos fora (.gitignore) · .env com segredos nunca entra no repo
Semana 3
DVC — Versionamento de Dados
  • Dados não entram no Git: tamanho, binários, dados sensíveis
  • dvc add cria ponteiro .dvc com hash MD5 — esse arquivo vai no Git
  • dvc add também adiciona o dado ao .gitignore automaticamente
  • dvc push envia dado ao remote · dvc pull baixa do remote
  • git checkout de ponteiro + dvc checkout = rollback de dados
  • dvc.yaml define pipeline · dvc repro só re-executa o que mudou (cache via dvc.lock)
Semana 4
MLflow Tracking — Experimentos
  • Experiment agrupa runs relacionados a um objetivo
  • Run = uma execução com parâmetros, métricas, tags e artefatos
  • Sempre use with mlflow.start_run(): — fecha o run corretamente
  • autolog() captura tudo automaticamente — sklearn, XGBoost, LightGBM…
  • log_metric com step= cria curva de aprendizado na UI
  • search_runs() retorna DataFrame — use order_by para achar o melhor
Semana 5
MLflow Model Registry — Estágios e Promoção
  • Tracking = laboratório (todos os runs) · Registry = prateleira aprovada (curado)
  • 4 estágios: None → Staging → Production → Archived
  • Registry precisa de backend DB: mlflow server --backend-store-uri sqlite:///mlflow.db
  • registered_model_name no log_model registra na hora
  • client.transition_model_version_stage() promove via código
  • archive_existing_versions=True arquiva versão anterior automaticamente
  • URI models:/nome/Production — código de inferência não precisa saber a versão
  • Rollback: arquivar versão nova + reativar versão anterior = 2 linhas de código

Q1
Conceito 1,5 pts

Explique com suas palavras por que um modelo de ML que atinge 94% de acurácia no notebook do cientista pode produzir resultados diferentes ao ser colocado em produção — mesmo sem nenhuma mudança no código de treino. Cite pelo menos dois fatores concretos.

Resposta esperada (ao menos 2 dos fatores abaixo):

1. Versões de pacotes diferentes: scikit-learn 1.2 e 1.5 podem implementar algoritmos com leves diferenças numéricas, parâmetros renomeados ou comportamento padrão alterado. O modelo serializado com uma versão pode se comportar diferente ao ser carregado em outra.

2. Dados de entrada dinâmicos: em produção os dados chegam em tempo real com ruído, formatos inesperados, campos nulos e distribuição diferente dos dados de treino. O notebook usava dados já limpos e estáticos.

3. Aleatoriedade e seeds: operações com random_state não definido podem gerar resultados diferentes entre execuções em ambientes distintos.

4. Diferenças de SO/bibliotecas de sistema: precisão de ponto flutuante pode variar entre arquiteturas, e bibliotecas como numpy dependem de BLAS/LAPACK do sistema.
Q2
Análise de Código 2,0 pts

O Dockerfile abaixo contém três problemas que afetam reprodutibilidade, desempenho de build ou segurança. Identifique cada um, explique o impacto e escreva a linha corrigida.

Dockerfile
FROM python:latest

COPY . .

RUN pip install scikit-learn pandas mlflow fastapi

ENV API_KEY="sk-prod-abc123secret"

CMD ["python", "serve.py"]
Problema 1 — FROM python:latest: a tag latest aponta para versões diferentes ao longo do tempo — build de hoje pode usar Python 3.11, build de amanhã Python 3.13. Não é reproduzível.
Correção: FROM python:3.11-slim

Problema 2 — COPY . . antes do pip install: qualquer mudança em qualquer arquivo do projeto invalida o cache Docker a partir daí, forçando reinstalação completa de todos os pacotes a cada build. Além disso, pacotes sem versão fixada (scikit-learn sem ==x.y.z) quebram a reprodutibilidade.
Correção: COPY requirements.txt . → RUN pip install --no-cache-dir -r requirements.txt → COPY . .

Problema 3 — ENV API_KEY="sk-prod-abc123secret": credenciais hardcodadas na imagem ficam expostas em docker inspect e no histórico de layers — qualquer pessoa com acesso à imagem vê o segredo.
Correção: remover do Dockerfile e injetar em runtime: docker run -e API_KEY=$API_KEY ...
Q3
DVC 1,5 pts

Um pipeline DVC tem três estágios: coletar → transformar → treinar. O time modifica apenas o script src/train.py e executa dvc repro. Descreva quais estágios são re-executados, quais são pulados, e qual mecanismo o DVC usa para tomar essa decisão.

Estágios pulados: "coletar" e "transformar" — nenhum arquivo nas suas deps mudou.
Estágio re-executado: "treinar" — src/train.py está listado nas suas deps e seu hash MD5 atual é diferente do registrado no dvc.lock.

O mecanismo: o arquivo dvc.lock armazena o hash MD5 de cada arquivo em deps e outs de cada estágio. Ao executar dvc repro, o DVC compara os hashes atuais com os do lockfile. Diferença = invalida o estágio e todos os estágios que dependem do seu output. Igualdade = usa cache.

Por que importa: se "coletar" leva 2 horas (ex: consulta a banco externo), re-executá-lo a cada mudança de código tornaria o ciclo inviável. O cache inteligente do DVC garante que só o necessário seja reprocessado.
Q4
Análise de Código — MLflow 2,0 pts

O script abaixo usa MLflow mas tem três erros ou omissões que deixam o rastreamento incompleto. Identifique cada problema e escreva a correção correspondente.

Python
import mlflow
from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import accuracy_score, f1_score

N_EST = 200
DEPTH = 10

model = RandomForestClassifier(n_estimators=N_EST, max_depth=DEPTH)
model.fit(X_train, y_train)
y_pred = model.predict(X_test)

mlflow.start_run()
mlflow.log_param("n_estimators", N_EST)
mlflow.log_metric("accuracy", accuracy_score(y_test, y_pred))
mlflow.log_metric("f1", f1_score(y_test, y_pred))
Problema 1 — treino fora do contexto de run: model.fit() ocorre antes de mlflow.start_run(). O autolog (se ativado) não capturaria o fit, e o tempo de treino não é rastreado.
Correção: usar with mlflow.start_run(): englobando todo o bloco, incluindo o fit.

Problema 2 — max_depth não é logado: DEPTH=10 nunca é registrado como parâmetro. Ao comparar runs na UI, não será possível saber qual max_depth gerou qual resultado.
Correção: adicionar mlflow.log_param("max_depth", DEPTH).

Problema 3 — run nunca é encerrado e modelo não é salvo: sem with start_run(): ou chamada a mlflow.end_run(), o run fica em estado RUNNING na UI. Além disso, o modelo não é salvo como artefato — impossível carregá-lo pelo Registry depois.
Correção: usar with mlflow.start_run(): (fecha automaticamente) e adicionar mlflow.sklearn.log_model(model, "modelo").
Q5
Análise 1,5 pts

Um modelo de recomendação de produtos foi treinado em março com acurácia de 91%. Em setembro, sem nenhuma mudança no código, a acurácia caiu para 68% e ninguém percebeu por três semanas. a) Explique a causa mais provável. b) Cite duas práticas de MLOps estudadas que teriam prevenido ou detectado o problema mais cedo.

a) Data drift / concept drift: o comportamento dos usuários muda sazonalmente. O modelo aprendeu o padrão de março (pré-verão) — em setembro o perfil de compras é diferente. O modelo continua fazendo predições baseadas em uma realidade que deixou de existir. Como ninguém monitorava as predições reais vs. os resultados, a degradação foi silenciosa.

b) Duas práticas:
— Monitoramento de drift (abordado na Semana 1 como fase do ciclo de vida, aprofundado na Semana 11): ferramentas como Evidently AI comparam continuamente a distribuição dos dados de entrada reais com a distribuição dos dados de treino. Um alerta seria disparado semanas antes da queda de acurácia se tornar crítica.
— Model Registry com versionamento (Semana 5): com o modelo de março versionado no Registry (Archived), o rollback seria imediato — uma chamada a transition_model_version_stage() para reativar a versão anterior enquanto o retreinamento é preparado.
Q6
Código — Registry 2,0 pts

Escreva o código Python completo que: (1) inicia um run MLflow, treina um RandomForestClassifier, loga accuracy e registra o modelo no Registry com o nome "classificador-churn"; (2) promove a versão 1 para Production arquivando qualquer versão anterior; (3) carrega o modelo em produção usando a URI correta sem hardcodar o número de versão.

Python — Gabarito Q6
import mlflow, mlflow.sklearn
from mlflow.tracking import MlflowClient
from sklearn.ensemble import RandomForestClassifier
from sklearn.metrics import accuracy_score

NOME = "classificador-churn"
mlflow.set_tracking_uri("http://localhost:5000")
mlflow.set_experiment("churn-prediction")

# 1. Treinar e registrar
with mlflow.start_run():
    model = RandomForestClassifier(n_estimators=200, random_state=42)
    model.fit(X_train, y_train)
    acc = accuracy_score(y_test, model.predict(X_test))
    mlflow.log_metric("accuracy", acc)
    mlflow.sklearn.log_model(
        model, "modelo",
        registered_model_name=NOME   # registra direto no Registry
    )

# 2. Promover v1 para Production
client = MlflowClient("http://localhost:5000")
client.transition_model_version_stage(
    name=NOME,
    version="1",
    stage="Production",
    archive_existing_versions=True   # arquiva versões anteriores
)

# 3. Carregar de Production — sem hardcodar versão
model_prod = mlflow.pyfunc.load_model(
    f"models:/{NOME}/Production"
)
predicoes = model_prod.predict(X_test)
Ponto crítico: a URI models:/classificador-churn/Production sempre aponta para a versão atual em Production. Quando o time promover a v2, o código de inferência carrega a nova versão automaticamente — sem deploy de código novo.
Q7
Prática 3,5 pts

Descreva, em sequência, todos os comandos Git e DVC necessários para que um novo membro do time, após clonar o repositório, possa: (a) ter o ambiente Python correto; (b) obter os dados da versão mais recente; (c) rodar o pipeline completo. Explique o que cada comando faz e por que a ordem importa.

Bash — sequência completa comentada
# 1. Clonar o repositório (código + ponteiros .dvc)
git clone https://github.com/time/projeto-ml.git
cd projeto-ml

# 2. Criar e ativar o ambiente virtual
python -m venv .venv
source .venv/bin/activate        # Linux/Mac

# 3. Instalar dependências com versões fixas
pip install -r requirements.txt  # inclui dvc[gdrive] ou dvc[s3]

# 4. Baixar os dados do remote storage
dvc pull
# Lê os ponteiros .dvc do repositório,
# consulta o remote configurado em .dvc/config
# e baixa os dados correspondentes ao commit atual

# 5. Executar o pipeline
dvc repro
# Executa apenas os estágios que não estão em cache
Por que a ordem importa: o git clone deve vir antes do dvc pull porque o DVC precisa dos ponteiros .dvc (que estão no Git) para saber quais arquivos buscar e de onde. Sem o clone não há ponteiros; sem os ponteiros o dvc pull não tem referência. O pip install deve vir antes do dvc repro porque os scripts Python do pipeline precisam das dependências instaladas para executar.
