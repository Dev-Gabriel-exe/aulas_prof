Atividade Prática

Construir uma API FastAPI que envia jobs de inferência em lote para uma fila RabbitMQ, processados por workers Celery, com polling de status e resultado.

Por que processamento assíncrono?
  • Inferência síncrona bloqueia o cliente durante todo o processamento
  • Grandes lotes (100k+ registros) não cabem em uma requisição HTTP
  • Filas desacoplam produtores de consumidores — escala independente
  • Retry automático em caso de falha — resiliência sem código extra
  • Workers podem rodar em máquinas diferentes, com GPU dedicada

O Problema do Blocking e a Arquitetura de Filas
Quando inferência síncrona falha

A API FastAPI que construímos nas semanas anteriores é síncrona: o cliente envia a requisição, espera o modelo rodar, recebe a resposta. Para predições individuais em tempo real (<100ms), isso é perfeito. Mas há cenários onde esse modelo quebra:

  • Scoring em lote: um analista precisa classificar 500.000 clientes. Uma única requisição HTTP com 500k registros excede limites de tamanho, timeout e memória da API.
  • Inferência lenta: modelos de visão computacional ou NLP pesados levam 5-30 segundos por exemplo. O cliente não pode ficar esperando — e a conexão HTTP pode cair.
  • Pico de demanda: às 23:59 de toda segunda-feira, 10.000 requisições chegam simultaneamente. A API fica sobrecarregada enquanto os workers de inferência ficam ociosos no resto do tempo.
  • Dependências externas: a inferência depende de buscar dados adicionais em um banco lento. O timeout da requisição HTTP expira antes de terminar.
A solução: separar a recepção do trabalho da execução do trabalho. A API recebe a solicitação instantaneamente e retorna um ID de job. Um worker processa em background, no seu próprio ritmo. O cliente consulta o status e busca o resultado quando pronto.
Arquitetura produtor–broker–consumidor

Três peças que se complementam, cada uma com responsabilidade clara:

FastAPI

Produtor — recebe a requisição e publica a tarefa na fila

RabbitMQ

Broker (fila) — armazena e distribui as tarefas para os workers

Celery Worker

Consumidor — pega a tarefa, executa o modelo e salva o resultado

Redis

Backend de resultados — armazena o resultado para o cliente buscar

Celery — Tasks, Workers, Retry e Resultados
Definindo tasks Celery

Uma task Celery é uma função Python decorada com @app.task. Quando chamada com .delay() ou .apply_async(), ela não executa localmente — é serializada e enviada para a fila do broker. O worker que consumir a fila executará a função.

Python — tasks.py
import os, pickle, time
import numpy as np
from celery import Celery

# Configurar o app Celery
app = Celery(
    "ml-tasks",
    broker=os.getenv("CELERY_BROKER_URL", "amqp://mlops:mlops123@localhost:5672//"),
    backend=os.getenv("CELERY_RESULT_BACKEND", "redis://localhost:6379/0"),
)

# Carregar modelo uma vez por worker (não a cada task)
_model = None

def get_model():
    global _model
    if _model is None:
        with open("models/model.pkl", "rb") as f:
            _model = pickle.load(f)
    return _model


# Task de predição individual
@app.task(bind=True, max_retries=3, default_retry_delay=5)
def predizer(self, features: list) -> dict:
    """Executa predição de uma instância. Retry automático em falha."""
    try:
        model = get_model()
        X     = np.array(features).reshape(1, -1)
        pred  = int(model.predict(X)[0])
        proba = float(model.predict_proba(X)[0][pred])
        return {"prediction": pred, "probability": round(proba, 4)}
    except Exception as exc:
        # Retry automático com backoff exponencial
        raise self.retry(exc=exc, countdown=2 ** self.request.retries)


# Task de scoring em lote
@app.task(bind=True, max_retries=2)
def predizer_lote(self, lista_features: list) -> dict:
    """Processa múltiplas instâncias em uma única chamada ao modelo."""
    model = get_model()
    X     = np.array(lista_features)
    preds  = model.predict(X).tolist()
    probas = model.predict_proba(X).max(axis=1).tolist()
    return {
        "predictions": preds,
        "probabilities": [round(p, 4) for p in probas],
        "total": len(preds)
    }
O padrão get_model(): o modelo é carregado uma única vez por processo worker (não a cada task). Isso é equivalente ao que fazemos na inicialização do FastAPI — carregar uma vez e reutilizar. Se cada task carregasse o modelo do disco, o throughput cairia drasticamente.
Executar e monitorar tasks
Python — enviar tasks e consultar resultados
from tasks import predizer, predizer_lote

# Enviar task assíncrona — retorna imediatamente com um AsyncResult
result = predizer.delay([0.5, -1.2, 0.8, 1.1, -0.3, 0.6, 1.5, -0.9, 0.2, 0.7])

print(f"Task ID: {result.id}")      # ID único da tarefa
print(f"Status: {result.status}")  # PENDING | STARTED | SUCCESS | FAILURE

# Verificar status sem bloquear
if result.ready():
    print(f"Resultado: {result.get()}")
else:
    print("Ainda processando...")

# Aguardar até terminar (bloqueante — usar só em scripts)
resultado = result.get(timeout=30)  # levanta exception se timeout

# Task de lote
features_lote = [[0.1*i for _ in range(10)] for i in range(1000)]
job = predizer_lote.delay(features_lote)
print(f"Lote enviado. ID: {job.id}")

Retry, prioridade e expiração
Python — configurações avançadas de task
# Enviar com prioridade alta e expiração
result = predizer.apply_async(
    args=[[0.5, -1.2, 0.8, 1.1, -0.3, 0.6, 1.5, -0.9, 0.2, 0.7]],
    priority=9,           # 0 (baixa) a 9 (alta)
    expires=300,          # expira em 300s se não executou
    queue="high_priority" # fila dedicada
)

# Configurar filas no app Celery
app.conf.task_routes = {
    "tasks.predizer"     : {"queue": "realtime"},
    "tasks.predizer_lote": {"queue": "batch"},
}

# Configurar TTL dos resultados no Redis (padrão: 1 dia)
app.conf.result_expires = 3600  # 1 hora

Integração FastAPI + Celery — API Assíncrona Completa
FastAPI como produtor de tasks

O padrão mais comum é a FastAPI receber a requisição, publicar a task na fila e retornar o task_id imediatamente. O cliente usa o task_id para fazer polling até obter o resultado.

Python — main.py — API com endpoints assíncronos
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import List, Optional
from celery.result import AsyncResult
from tasks import predizer, predizer_lote, app as celery_app

api = FastAPI(title="API Assíncrona de Inferência")

class PredictInput(BaseModel):
    features: List[float]

class BatchInput(BaseModel):
    instancias: List[List[float]]

class JobResponse(BaseModel):
    task_id: str
    status: str
    mensagem: str

class StatusResponse(BaseModel):
    task_id: str
    status: str
    resultado: Optional[dict] = None
    erro: Optional[str] = None


# Submeter predição individual
@api.post("/jobs/predict", response_model=JobResponse)
def submeter_predicao(data: PredictInput):
    task = predizer.delay(data.features)
    return JobResponse(
        task_id=task.id,
        status="PENDING",
        mensagem="Tarefa submetida. Use /jobs/{task_id} para consultar."
    )


# Submeter scoring em lote
@api.post("/jobs/batch", response_model=JobResponse)
def submeter_lote(data: BatchInput):
    if len(data.instancias) > 100_000:
        raise HTTPException(400, "Máximo 100.000 instâncias por lote")
    task = predizer_lote.delay(data.instancias)
    return JobResponse(
        task_id=task.id,
        status="PENDING",
        mensagem=f"Lote de {len(data.instancias)} instâncias submetido."
    )


# Consultar status e resultado de uma task
@api.get("/jobs/{task_id}", response_model=StatusResponse)
def consultar_job(task_id: str):
    result = AsyncResult(task_id, app=celery_app)

    if result.status == "PENDING":
        return StatusResponse(task_id=task_id, status="PENDING")

    if result.status == "STARTED":
        return StatusResponse(task_id=task_id, status="RUNNING")

    if result.status == "SUCCESS":
        return StatusResponse(
            task_id=task_id, status="SUCCESS", resultado=result.get()
        )

    if result.status == "FAILURE":
        return StatusResponse(
            task_id=task_id, status="FAILURE", erro=str(result.result)
        )

    return StatusResponse(task_id=task_id, status=result.status)


# Listar tasks ativas (útil para debug)
@api.get("/jobs")
def listar_jobs():
    inspect = celery_app.control.inspect()
    ativas  = inspect.active() or {}
    return {
        "workers_ativos": list(ativas.keys()),
        "tasks_em_execucao": sum(len(t) for t in ativas.values())
    }
O fluxo do cliente: POST /jobs/batch → recebe task_id → faz polling em /jobs/{task_id} a cada 2-5 segundos → quando status == "SUCCESS", busca o resultado. Esse padrão é chamado de polling assíncrono e é o mais simples de implementar. Para sistemas com muitos clientes, WebSockets ou Server-Sent Events são alternativas que evitam o polling.

# Submeter
RESP=$(curl -s -X POST http://localhost:8000/jobs/predict \
  -H "Content-Type: application/json" \
  -d '{"features":[0.5,-1.2,0.8,1.1,-0.3,0.6,1.5,-0.9,0.2,0.7]}')
echo $RESP

# Extrair task_id com jq (instalar se necessário)
TASK_ID=$(echo $RESP | python3 -c "import sys,json; print(json.load(sys.stdin)['task_id'])")

# Polling
while true; do
  STATUS=$(curl -s http://localhost:8000/jobs/$TASK_ID)
  echo $STATUS
  if echo $STATUS | grep -q SUCCESS; then break; fi
  sleep 1
done
Passo 04 — Submeter lote de 100 instâncias

Escreva um script Python que gera 100 instâncias de features aleatórias, submete via POST /jobs/batch, e faz polling em GET /jobs/{task_id} até o status ser SUCCESS. Imprima o total processado e a primeira predição do resultado.

Médio
Python — testar_lote.py
import time, requests, numpy as np

BASE = "http://localhost:8000"
instancias = np.random.randn(100, 10).tolist()

resp = requests.post(f"{BASE}/jobs/batch", json={"instancias": instancias})
task_id = resp.json()["task_id"]
print(f"Job submetido: {task_id}")

while True:
    status = requests.get(f"{BASE}/jobs/{task_id}").json()
    print(f"Status: {status['status']}")
    if status["status"] == "SUCCESS":
        res = status["resultado"]
        print(f"Total processado: {res['total']}")
        print(f"Primeira predição: {res['predictions'][0]}")
        break
    time.sleep(1)
Passo 05 — Desafio: filas separadas para realtime vs batch

Configure o Celery para usar duas filas: realtime para o endpoint /jobs/predict e batch para o /jobs/batch. Inicie dois workers: um consumindo apenas realtime com --concurrency=4 e outro consumindo apenas batch com --concurrency=1. Verifique no painel do RabbitMQ que as mensagens vão para as filas corretas.

Desafio
Critério de conclusão: um POST /jobs/batch com 100 instâncias deve retornar um task_id imediatamente (em <50ms). O polling em GET /jobs/{task_id} deve retornar status: "SUCCESS" com 100 predições dentro de 10 segundos. A API não deve ficar bloqueada durante o processamento — um GET /health simultâneo deve responder normalmente.
