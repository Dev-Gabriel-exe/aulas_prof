Questão 1 · Pipelines — Semana 7

Um cientista de dados mantém quatro scripts Python (01_limpar.py, 02_features.py, 03_treinar.py, 04_avaliar.py) para seu fluxo de ML. Após modificar apenas 03_treinar.py, ele precisa re-executar o processo. Qual é a principal vantagem de usar MLflow Pipelines nesse cenário?

A
O MLflow Pipelines converte automaticamente os scripts para notebooks Jupyter, facilitando a visualização
B
O cache inteligente detecta que apenas 03_treinar.py mudou e re-executa somente as etapas de treino e avaliação, pulando limpeza e features
C
O MLflow Pipelines executa todos os scripts em paralelo, reduzindo o tempo total pela metade
D
O uso de pipelines elimina a necessidade de versionamento com Git, pois o pipeline rastreia todas as mudanças automaticamente

Resposta: B
O cache inteligente do MLflow Pipelines (e do DVC) rastreia quais entradas de cada etapa foram modificadas. Como apenas o script de treino mudou, as etapas de limpeza e features têm as mesmas entradas de antes — então são puladas. Somente treino e avaliação são re-executados. A opção A é falsa (não converte para notebooks). A opção C é falsa (o pipeline é sequencial por padrão — paralelismo real requer Metaflow com foreach). A opção D é perigosamente falsa: pipelines e Git são complementares, não substitutos.

Questão 2 · Docker para ML — Semana 8

Analise o Dockerfile abaixo e identifique o principal problema de performance de build:

Dockerfile problemático
FROM python:3.11-slim
WORKDIR /app
COPY . .
RUN pip install --no-cache-dir -r requirements.txt
CMD ["uvicorn", "src.serve:app", "--host", "0.0.0.0", "--port", "8000"]
A
O uso de python:3.11-slim é inadequado para produção — deveria ser a imagem completa python:3.11
B
A flag --no-cache-dir está incorreta e faz o pip reinstalar todos os pacotes a cada execução
C
O COPY . . copia todo o projeto antes do pip install, fazendo o cache da camada de dependências invalidar toda vez que qualquer arquivo do projeto mudar
D
O WORKDIR /app deveria ser definido após o COPY para garantir que os arquivos sejam copiados para o local correto

Resposta: C
O problema clássico de ordem no Dockerfile. O COPY . . copia todos os arquivos — incluindo o código Python — antes do pip install. Isso significa que qualquer mudança em qualquer arquivo do projeto (até um comentário no código) invalida o cache da camada de dependências, forçando o pip install completo novamente. A solução correta é: COPY requirements.txt . → RUN pip install → COPY src/ .. Assim, a camada de dependências só é recriada quando o requirements.txt muda.

Questão 3 · FastAPI + Pydantic — Semana 9

Uma API de inferência recebe a seguinte requisição: POST /predict com corpo {"features": [1.0, 2.0]}. O modelo espera exatamente 10 features, definido com Field(min_length=10, max_length=10). O que acontece?

A
O FastAPI aceita a requisição e o modelo recebe os 2 valores; cabe ao modelo tratar a entrada incompleta
B
O FastAPI retorna HTTP 400 com a mensagem "features inválidas" configurada pelo desenvolvedor
C
O Pydantic rejeita automaticamente a requisição antes de executar a função, retornando HTTP 422 com detalhes do erro
D
O FastAPI completa a lista com zeros até atingir 10 features e executa a predição normalmente

Resposta: C
O Pydantic valida os dados antes de a função ser chamada. Com min_length=10, uma lista de 2 elementos falha na validação e o FastAPI retorna automaticamente HTTP 422 (Unprocessable Entity) com um JSON descrevendo exatamente qual campo falhou e por quê — sem nenhum código adicional do desenvolvedor. O 400 (opção B) é um código que o desenvolvedor dispara manualmente com HTTPException para erros de negócio. O 422 é o código automático do Pydantic para falhas de schema.

Questão 4 · BentoML — Semana 10

Um time configurou um runner BentoML com max_batch_size=32 e max_latency_ms=100. Em um teste de carga com 5 requisições simultâneas, as predições chegam em 10ms mas o usuário percebe latência de 100ms. Qual a explicação correta?

A
O BentoML tem um overhead fixo de 90ms para serializar e desserializar dados em JSON
B
O runner aguarda até 100ms para acumular mais requisições e formar um batch maior antes de processar, maximizando o throughput às custas da latência individual
C
O modelo está sobrecarregado e leva 100ms para processar cada requisição individualmente
D
Com apenas 5 requisições, o BentoML desativa o batching e processa sequencialmente, causando fila

Resposta: B
Este é o trade-off fundamental do batching dinâmico. O parâmetro max_latency_ms=100 significa: "espere até 100ms para ver se chegam mais requisições antes de processar". Com 5 requisições (bem abaixo de max_batch_size=32), o runner espera o tempo máximo configurado para tentar completar o batch. O resultado é que cada requisição individual tem latência próxima a 100ms, mesmo que a inferência em si seja rápida. A solução é reduzir max_latency_ms para sistemas com SLA de latência estrito, ou aceitar a latência em troca de maior throughput quando há alta concorrência.

Questão 5 · Evidently AI — Semana 11

Um modelo de crédito foi treinado antes de uma crise econômica. Após a crise, os clientes com perfil de renda idêntico ao dos dados de treino passaram a ter taxa de inadimplência muito maior. Os dados de entrada (features) não mudaram significativamente, mas as predições do modelo estão erradas. Que tipo de drift isso descreve?

A
Data drift — a distribuição das features mudou e precisa ser recalibrada com novos dados de treino
B
Prediction drift — as predições mudaram porque o modelo foi atualizado sem aviso
C
Concept drift — a relação entre as features e o target mudou; o mesmo perfil de cliente agora implica risco diferente
D
Label drift — os labels de treino estavam incorretos e precisam ser re-rotulados

Resposta: C
Concept drift é exatamente isso: a relação entre X (features) e y (target) muda, mesmo que a distribuição de X permaneça estável. Antes da crise, o modelo aprendeu que "renda X → inadimplência Y". Após a crise, a mesma renda X passou a implicar inadimplência Z (muito maior). O modelo usa o padrão antigo, logo erra. O data drift (A) seria se os perfis de clientes tivessem mudado. O prediction drift (B) é consequência de outros tipos de drift, não a causa. O label drift (D) não existe como categoria padrão — seria uma variante de concept drift.

Questão 6 · Prometheus + PromQL — Semana 12

Uma API de inferência usa um Counter chamado predicoes_total com o label resultado. Após 1 hora de operação, o valor acumulado é 10.000 para "sucesso" e 500 para "erro". Qual query PromQL fornece a taxa atual de erros por segundo (nos últimos 5 minutos)?

A
predicoes_total{resultado="erro"}
B
rate(predicoes_total{resultado="erro"}[5m])
C
sum(predicoes_total{resultado="erro"})
D
histogram_quantile(0.95, predicoes_total{resultado="erro"}[5m])

Resposta: B
A função rate(counter[janela]) é a forma correta de obter a taxa por segundo de um Counter. Sem rate(), o valor bruto (A) mostra apenas o total acumulado — que cresce sem parar e não é útil para monitoramento de "taxa atual". A opção C (sum()) também retorna o total, apenas somando todas as séries. A opção D usa histogram_quantile(), que é exclusiva para métricas do tipo Histogram — não faz sentido aplicar a um Counter.

Questão 7 · Integração — Semanas 7–12

Uma equipe construiu a seguinte stack: pipeline MLflow Pipelines que treina o modelo → MLflow Registry para versionamento → FastAPI com prometheus_client para servir → Prometheus + Grafana para observabilidade → Evidently AI em job diário para drift. Um alerta do Grafana dispara: "TaxaErroAlta — 12% das predições retornando erro". Qual sequência de investigação faz mais sentido?

A
Retreinar o modelo imediatamente via pipeline MLflow, pois taxa de erro alta indica degradação do modelo
B
Verificar os logs da API para entender o tipo de erro HTTP (400 = dado inválido do cliente vs 503 = modelo não carregado), depois checar o relatório Evidently para confirmar se há drift nos dados de entrada
C
Fazer rollback para a versão anterior do modelo no Registry, pois toda taxa de erro alta é causada por regressão do modelo
D
Escalar horizontalmente o serviço FastAPI com mais réplicas, pois a taxa de erro indica sobrecarga de capacidade

Resposta: B
Esta questão testa raciocínio de diagnóstico sistemático. Taxa de erro alta tem múltiplas causas possíveis — e a ordem correta é diagnosticar antes de agir. Os códigos HTTP revelam muito: erros 400 indicam clientes enviando dados malformados (solução: corrigir o cliente ou melhorar validação). Erros 503 indicam que o modelo não carregou (solução: verificar o modelo e reiniciar). Se os erros não são HTTP mas sim predições "erradas" semanticamente, aí sim o Evidently pode revelar drift como causa raiz. Retreinar (A) sem diagnóstico é precipitado. Rollback (C) assume causa que ainda não foi confirmada. Escalar (D) não resolve erros de lógica — apenas erros de capacidade, que seriam evidentes em métricas de latência e saturação de CPU, não só taxa de erro.

Aula 3 · Orientações Finais
Instruções da P2 — Formato, Critérios e Calendário

Formato da prova

  • 7 questões mistas (múltipla escolha + análise de código)
  • Duração total: 90 minutos
  • Questões de múltipla escolha: 4 alternativas, 1 correta
  • Questões de código: identificar erro ou completar trecho
  • Todas as questões valem o mesmo peso

O que é permitido

  • Folha de cola A5 — frente e verso, manuscrita
  • Caneta esferográfica azul ou preta
  • Régua (apenas para traçar linhas)
  • Nenhum dispositivo eletrônico ligado
  • Garrafa de água sem rótulo

O que não é permitido

  • Celular, tablet, notebook ou smartwatch
  • Consulta a apostilas ou livros (exceto a folha de cola)
  • Comunicação com outros alunos durante a prova
  • Saída da sala nos primeiros 20 minutos
  • Folha de cola digitada ou impressa

Estratégia recomendada

  • Leia todas as 7 questões antes de começar
  • Responda primeiro as que você tem certeza
  • Nas questões de código: leia linha a linha
  • Eliminate alternativas absurdas antes de escolher
  • Reserve 10 minutos no final para revisar
Atenção especial: questões que pedem identificação de código errado são as mais frequentes na P2. Treine ler Dockerfiles e código Python linha a linha. Na folha de cola, priorize: limiares do PSI, os 4 tipos de métricas Prometheus, a diferença entre rate() e o valor bruto, e a ordem correta das instruções no Dockerfile.
