MongoDB — operações fundamentais para a prova
# Conectar e selecionar coleção
from pymongo import MongoClient
client = MongoClient("mongodb://localhost:27017")
db = client["loja"]
col = db["pedidos"]

# INSERT — inserir documento
col.insert_one({"cliente": "Ana", "total": 4239.90, "status": "entregue"})

# FIND — buscar com filtro
entregues = list(col.find({"status": "entregue"}, {"_id": 0, "cliente": 1, "total": 1}))

# AGGREGATE — equivale a GROUP BY + SUM
pipeline = [
    {"$group": {"_id": "$status", "total_gasto": {"$sum": "$total"}}},
    {"$sort": {"total_gasto": -1}},
]
resultado = list(col.aggregate(pipeline))

# UPDATE — atualizar campo
col.update_one({"cliente": "Ana"}, {"$set": {"status": "arquivado"}})

# DELETE — remover documento
col.delete_one({"cliente": "Ana"})
Bancos vetoriais — o essencial

Conceitos obrigatórios:

  • Embedding: representação numérica de texto/imagem em alta dimensão (ex: 1536 dimensões)
  • Similaridade cosseno: mede o ângulo entre vetores — quanto mais próximo de 1, mais similares
  • pgvector: extensão do PostgreSQL — busca vetorial dentro do banco relacional
  • RAG: busca no banco vetorial → injeta contexto → LLM responde com base nos documentos

Fórmula da similaridade cosseno:

cos(θ) = (A · B) / (‖A‖ × ‖B‖)

Resultado entre -1 e 1. Em busca semântica, comparamos textos com scores próximos de 1 = muito similares, próximos de 0 = sem relação.
Python — pgvector: criar, inserir e buscar
# Pré-requisito: CREATE EXTENSION vector; no PostgreSQL
import psycopg2
from pgvector.psycopg2 import register_vector
import numpy as np

conn = psycopg2.connect(os.getenv("DATABASE_URL"))
register_vector(conn)
cur = conn.cursor()

# Criar tabela com coluna vetorial
cur.execute("""
    CREATE TABLE IF NOT EXISTS documentos (
        id SERIAL PRIMARY KEY,
        conteudo TEXT,
        embedding vector(3)   -- 3 dims para o exemplo; real = 1536
    )
""")

# Inserir documento com embedding
cur.execute("INSERT INTO documentos (conteudo, embedding) VALUES (%s, %s)",
            ("Delta Lake oferece ACID", np.array([0.1, 0.8, 0.3])))
conn.commit()

# Buscar os 3 mais similares por distância cosseno
query_vec = np.array([0.15, 0.75, 0.35])
cur.execute("""
    SELECT conteudo, 1 - (embedding <=> %s) AS similaridade
    FROM documentos
    ORDER BY embedding <=> %s
    LIMIT 3
""", (query_vec, query_vec))
for row in cur.fetchall():
    print(f"Similaridade: {row[1]:.3f} | {row[0]}")

Ponto de confusão frequente: _delta_log/ é uma pasta de arquivos JSON dentro do diretório da tabela Delta. Os dados continuam em arquivos .parquet normais — o Delta Lake apenas adiciona controle transacional por cima.

Zonas do Lakehouse:

  • raw / bronze: dados brutos — imutável, exatamente como chegaram
  • silver: limpos, tipados, particionados — podem ser consultados
  • gold: features prontas para ML ou BI — alta granularidade de negócio

Compressões — o que lembrar:

  • Snappy: mais rápida — padrão Spark/Athena
  • Zstd: melhor equilíbrio tamanho/velocidade — recomendada
  • Gzip: menor tamanho — mas lenta para leitura

Governança e Airflow — o que não confundir
Governança — pontos-chave consolidados

As 6 dimensões de qualidade:

  • Completude — sem nulos onde não pode haver
  • Acurácia — valor reflete a realidade (e-mail com @, preço positivo)
  • Unicidade — sem duplicatas indesejadas
  • Temporalidade — dado atualizado, sem datas no futuro
  • Consistência — mesmo valor em fontes diferentes
  • Conformidade — formato correto (CEP, CNPJ, UF)

Técnicas LGPD — reversibilidade:

  • Pseudonimização: reversível com chave — ex: UUID com mapeamento
  • Hash (SHA-256+salt): irreversível — ex: email_hash para joins
  • Generalização: perde precisão — ex: idade → faixa etária
  • Supressão: remove a coluna — ex: dropar CPF e nome
  • Dados sintéticos: gerar com Faker — sem dado real
Dado anonimizado não é dado pessoal segundo a LGPD. Portanto, a zona silver com colunas pessoais hasheadas ou suprimidas pode ser usada em modelos de ML sem as mesmas restrições da zona raw.

Python — estrutura mínima de DAG para consulta rápida
from airflow.decorators import dag, task
from datetime import datetime, timedelta

@dag(dag_id="minha_dag", schedule="0 2 * * *",
     start_date=datetime(2025,1,1), catchup=False,
     default_args={"retries":2, "retry_delay":timedelta(minutes=5)})
def minha_dag():

    @task
    def extrair(): return 100          # retorno vai para XCom

    @task
    def transformar(n: int): return n * 2

    @task
    def carregar(n: int): print(f"Carregados {n} registros")

    carregar(transformar(extrair()))   # define dependências E passa XCom

minha_dag()

Questão 01 · NoSQL

Um sistema de e-commerce precisa armazenar perfis de produtos onde cada produto pode ter atributos completamente diferentes (um livro tem ISBN e autor; uma TV tem polegadas e resolução). Qual tipo de banco NoSQL é mais adequado e por quê?

Banco de documentos (ex: MongoDB). Documentos JSON/BSON permitem schema flexível — cada documento pode ter campos diferentes sem a necessidade de alterar uma tabela rígida. Um livro pode ter o campo isbn e uma TV pode ter polegadas no mesmo banco, na mesma coleção, sem conflito. Bancos relacionais exigiriam uma tabela com todas as colunas possíveis (cheias de NULL) ou um design EAV complexo.
Questão 02 · NoSQL vs Relacional

Cite duas situações em que você deveria continuar usando banco de dados relacional (SQL) em vez de migrar para NoSQL.

Qualquer duas das seguintes são válidas:

(1) Quando os dados são altamente relacionais e há necessidade de JOINs frequentes entre múltiplas entidades (ex: pedido → cliente → produto → fornecedor).
(2) Quando integridade referencial é crítica — chaves estrangeiras garantidas pelo próprio banco.
(3) Quando transações ACID completas são necessárias (ex: sistema financeiro — débito e crédito na mesma transação).
(4) Quando a equipe precisa de queries ad hoc flexíveis com aggregações complexas.
Questão 03 · Bancos vetoriais

Explique o que é um embedding e qual é a diferença entre uma busca por igualdade exata (como em SQL) e uma busca por similaridade semântica (como em bancos vetoriais).

Embedding é uma representação numérica de um texto (ou imagem, áudio) em um espaço de alta dimensão — um vetor com centenas ou milhares de números. Textos semanticamente parecidos ficam próximos nesse espaço, mesmo usando palavras diferentes.

Busca por igualdade exata (SQL): WHERE conteudo = 'banco de dados' — só encontra documentos com essa string exata. Se o documento diz "SGBD" ou "armazenamento de dados", não é encontrado.

Busca por similaridade semântica: converte a query em embedding e busca os vetores mais próximos por distância cosseno. "Banco de dados" encontra "SGBD", "armazenamento relacional" etc., pois eles têm embeddings próximos no espaço vetorial.
Questão 04 · Parquet e particionamento

Um dataset de pedidos tem 50 colunas e 100 milhões de linhas particionado por ano e mes. Uma query precisa apenas do total dos pedidos de status = "entregue" do mês de março de 2025. Explique como predicate pushdown e column pruning juntos aceleram essa leitura.

Predicate pushdown: o filtro ano=2025, mes=3 é aplicado antes de abrir qualquer arquivo. O engine identifica as pastas ano=2025/mes=3/ e ignora completamente todas as outras partições — os arquivos de outros meses e anos nunca são lidos do disco.

Column pruning: o Parquet armazena colunas separadamente. Como a query só precisa de total e status, o engine lê apenas os blocos dessas 2 colunas, ignorando as outras 48 — mesmo dentro dos arquivos da partição de março.

O efeito combinado: de 100M linhas × 50 colunas, a query efetivamente lê apenas uma fração dos dados — potencialmente reduzindo de gigabytes para megabytes de I/O.
Questão 05 · Delta Lake

O que é o time travel no Delta Lake e cite um caso de uso real onde ele seria essencial.

Time travel é a capacidade de consultar versões anteriores de uma tabela Delta Lake, identificadas por número de versão (version=0, version=1…) ou por timestamp (version_at_timestamp("2025-01-14T12:00:00")). Isso é possível porque o _delta_log/ registra cada operação — os arquivos Parquet antigos só são removidos pelo vacuum.

Caso de uso real: um modelo de ML foi treinado com os dados da tabela gold na versão 5 e passou a produzir predições ruins após uma atualização. Com time travel, o engenheiro volta à versão 5 da tabela, re-treina o modelo e confirma que o problema está nos dados novos — sem precisar ter feito backup manual.
Questão 06 · Qualidade de dados

Um dataset de clientes tem as seguintes anomalias: (a) coluna email com 12% de valores nulos; (b) dois registros com o mesmo CPF mas nomes diferentes; (c) coluna cep com valores no formato 12345678 em vez de 12345-678. Qual dimensão de qualidade cada anomalia viola?

(a) Email com 12% de nulos → Completude — campo obrigatório com valores ausentes.
(b) Dois registros com mesmo CPF → Unicidade — CPF é identificador único; duplicata indica erro de carga ou deduplicação ausente.
(c) CEP sem hífen → Conformidade — o dado existe mas não segue o formato padrão esperado (00000-000).
Questão 07 · LGPD

Sua equipe precisa usar dados de clientes (nome, CPF, email, idade, cidade) para treinar um modelo de propensão de compra. Descreva como aplicar as técnicas de mascaramento adequadas para que o dataset de treino esteja em conformidade com a LGPD.

Nome e CPF: supressão — remover as colunas completamente. O modelo não precisa saber o nome da pessoa para prever propensão de compra.

Email: hash SHA-256 com salt (email_hash) — permite fazer join entre datasets sem expor o endereço real. Irreversível sem a chave.

Idade: generalização em faixas etárias (18–24, 25–34…) — mantém o sinal preditivo sem identificar individualmente.

Cidade: pode ser mantida — não é dado pessoal por si só (identifica um lugar, não uma pessoa).

Com essas transformações, o dataset resultante é anonimizado e, segundo a LGPD, não é mais considerado dado pessoal — portanto pode ser usado em modelos sem as restrições do Art. 7º.
Questão 08 · Airflow — conceitos

Explique a diferença entre DAG, Task e DAG Run no Apache Airflow, usando uma analogia ou exemplo concreto.

DAG é a receita — define quais tarefas existem, em que ordem e com que frequência. É o "plano" estático.

Task é cada passo da receita — extrair dados, limpar, validar, carregar. Cada task tem um operador (PythonOperator, BashOperator etc.).

DAG Run é a execução da receita para uma data específica — toda noite às 02h, o Airflow cria um novo DAG Run do dia e executa todas as tasks para aquela data lógica.

Analogia: DAG = receita de bolo escrita no caderno. Task = cada etapa (misturar, assar, decorar). DAG Run = a vez que você fez o bolo na quarta-feira às 15h.
Questão 09 · Airflow — XCom e retry

No Airflow com TaskFlow API, uma task extrair retorna o número de linhas extraídas (return 5000). A task seguinte validar recebe esse valor e levanta um erro se for menor que 1000. Configure também retries=2 e retry_delay de 3 minutos. Escreva o código completo dessa DAG.

Python
from airflow.decorators import dag, task
from datetime import datetime, timedelta

@dag(dag_id="dag_q9", schedule="@daily",
     start_date=datetime(2025,1,1), catchup=False,
     default_args={"retries":2,"retry_delay":timedelta(minutes=3)})
def dag_q9():

    @task
    def extrair():
        return 5000   # na prática: len(df) após read_sql

    @task
    def validar(n: int):
        if n < 1000:
            raise ValueError(f"Esperado >= 1000 linhas, recebido {n}")
        print(f"✓ {n} linhas — validação OK")

    validar(extrair())

dag_q9()
O return 5000 da task extrair é salvo no XCom automaticamente pela TaskFlow API e passado como argumento n para validar.
Questão 10 · Integração — tudo junto

Descreva em 5 etapas um pipeline de dados completo para alimentar um modelo de ML com dados de pedidos, mencionando: zona onde cada dado fica, formato de arquivo, ferramenta de qualidade, técnica de mascaramento LGPD e como a orquestração é feita.

Etapa 1 — Extração (raw): Airflow → PythonOperator extrai a tabela pedidos do PostgreSQL. Salva em Parquet snappy na zona lake/raw/, particionado por status. Sem alterações nos dados.

Etapa 2 — Validação de qualidade: Great Expectations verifica que id é único e não nulo, total é positivo. Se falhar, a DAG para e dispara callback de alerta — dado ruim não avança.

Etapa 3 — Limpeza e mascaramento (silver): tipos corrigidos, status normalizado, email substituído por hash SHA-256+salt, nome e CPF suprimidos. Salvo em Parquet zstd particionado por ano/mes na zona lake/silver/.

Etapa 4 — Feature engineering (gold): agrega por cliente_id — ticket médio, dias inativo, total pedidos. Salvo em Delta Lake na zona lake/gold/ com mode="overwrite" e linhagem registrada em JSON.

Etapa 5 — Orquestração: toda a sequência é uma DAG no Airflow (schedule="0 2 * * *", catchup=False, retries=2). O modelo de ML lê diretamente da zona gold, que é atualizada diariamente.
