Questão A2
Lakehouse — Parquet e Delta Lake
10 pts
(a) Explique o que é predicate pushdown e como ele difere de column pruning no contexto de leitura de arquivos Parquet. Use um exemplo concreto com um dataset de pedidos.

(b) Um engenheiro afirmou que "Delta Lake substitui o Parquet". Essa afirmação está correta ou incorreta? Justifique explicando a relação real entre os dois.
Espaço para resposta do aluno...
(a) Predicate pushdown é a capacidade de filtrar dados no nível do sistema de arquivos, sem ler arquivos que não atendem ao predicado. Em um dataset particionado por ano/mes, a query WHERE ano=2025 AND mes=3 lê apenas a pasta ano=2025/mes=3/ — os demais arquivos nem são abertos. Column pruning opera dentro de cada arquivo Parquet: como o formato é colunar, a query SELECT total lê apenas o bloco da coluna total, ignorando as demais 49 colunas do arquivo. Ambos juntos: poucas partições + poucas colunas = fração mínima do I/O total.

(b) Incorreto. Delta Lake não substitui o Parquet — ele usa Parquet como formato de armazenamento físico. O que o Delta Lake adiciona é uma camada de controle transacional: a pasta _delta_log/ com arquivos JSON registra cada operação (insert, update, delete), permitindo ACID, time travel e schema evolution. Os dados continuam em arquivos .parquet normais — o Delta Lake é uma camada de metadados sobre eles, não um formato alternativo.
  • (a) 6 pts: definição correta de pushdown (2 pt) + definição correta de pruning (2 pt) + exemplo concreto (2 pt)
  • (b) 4 pts: posição correta: "incorreto" (1 pt) + explicação da relação real (3 pt)

Um dataset de clientes contém: id, nome, cpf, email, idade, cidade, total_gasto.

(a) Identifique quais colunas são dados pessoais segundo a LGPD e quais não são.

(b) A equipe precisa usar esse dataset para treinar um modelo de machine learning. Descreva quais técnicas de mascaramento você aplicaria em cada coluna pessoal, justificando a escolha (supressão, hash, generalização ou pseudonimização).
Espaço para resposta do aluno...
(a) Dados pessoais: nome, cpf, email, idade (podem identificar ou contribuir para identificar individualmente). cidade sozinha geralmente não é pessoal (identifica um lugar). id interno e total_gasto não são dados pessoais por si só.

(b) Técnicas por coluna:
— nome e cpf: supressão — remover completamente. O modelo não precisa dessas colunas para prever comportamento de compra; mantê-las, mesmo mascaradas, aumenta o risco sem benefício.
— email: hash SHA-256 + salt (email_hash) — permite fazer joins entre datasets sem expor o endereço real. Irreversível sem a chave secreta.
— idade: generalização em faixas etárias (18–24, 25–34, 35–44, 45–59, 60+) — mantém o sinal preditivo para o modelo (faixa etária influencia comportamento) sem identificar individualmente.
— cidade: pode ser mantida sem mascaramento.

Questão A4
Airflow — conceitos e funcionamento
10 pts
(a) Explique a diferença entre DAG, Task e DAG Run no Airflow usando uma analogia ou exemplo do cotidiano.

(b) O que é XCom e qual é a diferença de uso correto entre XCom e armazenamento em arquivo/banco? Dê um exemplo de quando usar cada um.

(c) Por que é recomendado usar catchup=False ao criar uma DAG nova em produção?
Espaço para resposta do aluno...
(a) DAG é o plano completo — como uma receita de bolo: define quais etapas existem, em qual ordem e com qual frequência. Task é cada etapa individual — misturar, assar, decorar. DAG Run é a execução concreta da receita em uma data específica — a vez que você fez o bolo na quinta-feira às 14h. O mesmo plano pode gerar múltiplos runs em datas diferentes.

(b) XCom (Cross-Communication) é o mecanismo do Airflow para passar pequenos valores entre tasks — o retorno de uma função vira XCom automaticamente na TaskFlow API. Deve ser usado para metadados: número de linhas processadas, caminho de um arquivo, status de uma operação. Para DataFrames ou grandes volumes de dados, use armazenamento externo (arquivo Parquet, S3, banco) e passe apenas o caminho via XCom. Exemplo: return len(df) → XCom; o próprio df deve ser salvo em disco.

(c) catchup=True (padrão) faz o Airflow executar automaticamente todas as DAG Runs desde o start_date até hoje ao ativar a DAG. Se o start_date for 6 meses atrás, o Airflow vai tentar executar ~180 runs em paralelo, sobrecarregando o banco, os workers e potencialmente duplicando dados. catchup=False evita esse efeito — e quando necessário, o backfill é feito manualmente e de forma controlada.
  • (a) 4 pts: os três conceitos definidos corretamente com distinção clara
  • (b) 4 pts: definição de XCom (1 pt) + distinção correto/incorreto de uso (2 pt) + exemplo (1 pt)
  • (c) 2 pts: risco correto explicado (runs em cascata / sobrecarga)

Python — gabarito B1
import pandas as pd
from deltalake import DeltaTable, write_deltalake

df = pd.DataFrame({
    "id":         [1, 2, 3, 4, 5],
    "cliente_id": [10, 20, 10, 30, 20],
    "total":      [4239.90, 1200.00, 399.90, 4150.00, 89.90],
    "status":     ["entregue", "entregue", "cancelado", "entregue", "pendente"],
    "criado_em":  pd.to_datetime(["2025-01-10", "2025-01-15",
                                   "2024-12-20", "2025-01-22", "2024-11-05"]),
})

# (1) Criar colunas de partição e salvar Parquet particionado
df["ano"] = df["criado_em"].dt.year
df["mes"] = df["criado_em"].dt.month

df.to_parquet(
    "silver/pedidos/",
    partition_cols=["ano", "mes"],
    index=False,
    compression="zstd",
    engine="pyarrow",
)
print("Parquet silver escrito com particionamento ano/mes")

# (2) Ler com predicate pushdown + column pruning
df_jan = pd.read_parquet(
    "silver/pedidos/",
    filters=[("ano", "=", 2025), ("mes", "=", 1)],
    columns=["id", "cliente_id", "total"],
)
print(f"Pedidos jan/2025 lidos: {len(df_jan)}")

# (3) Salvar como Delta Lake
write_deltalake("gold/pedidos_jan25/", df_jan, mode="overwrite")

# (4) Versão e contagem
dt = DeltaTable("gold/pedidos_jan25/")
print(f"Delta versão: {dt.version()} | registros: {len(dt.to_pandas())}")
  • 3 pts — criação correta das colunas ano/mes e escrita particionada com zstd
  • 4 pts — leitura com filtros corretos no predicate pushdown E column pruning
  • 4 pts — escrita correta em Delta Lake com mode overwrite
  • 4 pts — impressão de versão e contagem via DeltaTable

Python — gabarito B2
import pandas as pd, hashlib

df = pd.DataFrame({
    "id":    [1, 2, 3, 4],
    "nome":  ["Ana Silva", "Bruno Costa", None, "Diego"],
    "cpf":   ["111.222.333-44", "555.666.777-88",
              "111.222.333-44", "999.000.111-22"],
    "email": ["ana@x.com", "bruno_invalido", "carla@z.com", "diego@w.com"],
    "idade": [28, 45, 33, 62],
    "cidade":["SP", "RJ", "BH", "PI"],
})

# (1) Diagnóstico de qualidade
print("=== Diagnóstico de Qualidade ===")
pct_nulos = (df.isnull().mean() * 100).round(2)
print("% nulos:\n", pct_nulos[pct_nulos > 0])
print(f"CPFs duplicados: {df['cpf'].duplicated().sum()}")
print(f"E-mails sem '@': {(~df['email'].str.contains('@', na=False)).sum()}")

# (2) Mascaramento LGPD
SALT = "p2_2025"
df_masked = df.copy()

# Supressão
colunas_removidas = ["nome", "cpf"]
df_masked = df_masked.drop(columns=colunas_removidas)

# Hash do email
df_masked["email_hash"] = df["email"].apply(
    lambda v: hashlib.sha256(f"{SALT}{v}".encode()).hexdigest()[:16]
)
df_masked = df_masked.drop(columns=["email"])
colunas_removidas.append("email")

# Generalização da idade
def faixa(i):
    if   i <= 29: return "18–29"
    elif i <= 44: return "30–44"
    elif i <= 59: return "45–59"
    else:          return "60+"

df_masked["faixa_etaria"] = df["idade"].apply(faixa)
df_masked = df_masked.drop(columns=["idade"])
colunas_removidas.append("idade")

# (3) Imprimir resultado
print("\n=== Dataset Anonimizado ===")
print(df_masked.to_string(index=False))
print(f"\nColunas removidas: {colunas_removidas}")
  • 5 pts — diagnóstico correto: nulos, duplicatas CPF e e-mails inválidos
  • 4 pts — supressão correta de nome e cpf
  • 3 pts — hash SHA-256 com salt correto e drop do email original
  • 3 pts — generalização de idade em faixas e drop da coluna original
Questão B3
Airflow — DAG com TaskFlow API
15 pts
Escreva uma DAG completa do Airflow chamada dag_p2 usando a TaskFlow API com as seguintes especificações:

(1) Agendamento diário às 03h30, sem catchup, com retries=2 e retry_delay de 5 minutos.
(2) Task extrair: cria um DataFrame com 3 pedidos fictícios (id, cliente_id, total, status) e retorna a contagem de linhas via XCom.
(3) Task validar: recebe a contagem e levanta ValueError se for menor que 1. Imprime "✓ N registros válidos" se passar.
(4) Task reportar: recebe a contagem e imprime "Relatório: pipeline concluído com N registros em [data de hoje]".
(5) As tasks devem executar em sequência: extrair → validar → reportar.
Espaço para o código do aluno...
Python — gabarito B3
from airflow.decorators import dag, task
from datetime import datetime, timedelta, date

@dag(
    dag_id="dag_p2",
    schedule="30 3 * * *",
    start_date=datetime(2025, 1, 1),
    catchup=False,
    default_args={
        "retries": 2,
        "retry_delay": timedelta(minutes=5),
    },
    tags=["p2", "banco-de-dados"],
)
def dag_p2():

    @task
    def extrair():
        import pandas as pd
        df = pd.DataFrame({
            "id":         [1, 2, 3],
            "cliente_id": [10, 20, 30],
            "total":      [4239.90, 1200.00, 399.90],
            "status":     ["entregue", "entregue", "cancelado"],
        })
        return len(df)   # XCom automático

    @task
    def validar(n: int):
        if n < 1:
            raise ValueError(f"Pipeline sem registros: {n}")
        print(f"✓ {n} registros válidos")
        return n

    @task
    def reportar(n: int):
        hoje = date.today().isoformat()
        print(f"Relatório: pipeline concluído com {n} registros em {hoje}")

    n = extrair()
    n_v = validar(n)
    reportar(n_v)

dag_p2()

  • 4 pts — task extrair com DataFrame e return correto para XCom
  • 4 pts — task validar recebendo XCom, ValueError correto e mensagem de sucesso
  • 2 pts — task reportar com data e mensagem corretas
  • 2 pts — encadeamento sequencial correto com passagem de valores

Questão C1
Projeto de arquitetura — decisão técnica fundamentada
15 pts
Uma empresa de saúde digital quer construir um sistema de dados para alimentar um modelo de ML que prevê risco de abandono de tratamento. Os dados disponíveis são:

— Registros de consultas médicas (PostgreSQL relacional existente)
— Histórico de mensagens de suporte (texto livre, ~500 GB, crescendo 5 GB/dia)
— Perfil dos pacientes: nome, CPF, data de nascimento, condição clínica, cidade

Responda:
(a) Qual banco você usaria para armazenar e buscar as mensagens de suporte para um sistema de recuperação semântica (RAG)? Justifique.
(b) Descreva as 3 zonas do Lakehouse para esse cenário — o que entra em cada zona e em qual formato.
(c) Quais colunas do perfil do paciente precisam de mascaramento LGPD antes de chegar à zona gold? Qual técnica para cada uma?
(d) Escreva a estrutura (não precisa de código completo, mas precisa de pseudocódigo ou estrutura de tasks) de uma DAG Airflow para orquestrar esse pipeline, com pelo menos 4 tasks e suas dependências.
Espaço para resposta do aluno...
(a) Para busca semântica em texto livre, a escolha ideal é um banco vetorial. As mensagens são convertidas em embeddings (ex: via OpenAI ou modelo local) e armazenadas com pgvector (PostgreSQL + extensão vector) ou Pinecone/Chroma. Quando o modelo de ML ou um sistema RAG precisa contexto, a query é vetorizada e os documentos mais similares são recuperados por distância cosseno. Um banco relacional com LIKE ou full-text search não captura semântica — "paciente melhorou" e "tratamento evoluiu positivamente" têm sentidos similares mas palavras diferentes.

(b) Zonas do Lakehouse:
— Raw/bronze: dados exatamente como chegaram do PostgreSQL e do sistema de mensagens. Arquivos Parquet snappy, sem nenhuma transformação. Contém CPF, nome, condição clínica — zona de acesso restrito.
— Silver: dados limpos e mascarados. CPF e nome suprimidos, email hasheado, data de nascimento generalizada em faixa etária. Parquet zstd particionado por data. Textos de mensagens tokenizados. Sem dados pessoais identificáveis.
— Gold: features agregadas para o modelo — score de engajamento, dias desde última consulta, frequência de mensagens negativas, faixa etária, condição clínica anonimizada. Delta Lake para rastreabilidade e time travel.

(c) Mascaramento por coluna:
— nome: supressão
— cpf: supressão
— data_nascimento: generalização → faixa etária
— condição_clínica: é dado sensível (saúde) → supressão ou generalização em categorias amplas (ex: "doença crônica", "saúde mental") sem diagnóstico específico
— cidade: pode ser mantida

(d) Estrutura da DAG: @dag dag_saude_digital, schedule="0 3 * * *", catchup=False
  @task extrair_consultas() → n_consultas
  @task extrair_mensagens() → n_msgs
  @task [extrair_consultas, extrair_mensagens] → transformar_silver(n_consultas, n_msgs)
  @task transformar_silver → validar_qualidade(n_silver)
  @task validar_qualidade → gerar_features_gold(n_silver)
  @task gerar_features_gold → registrar_linhagem(n_gold)
  • (a) 3 pts — banco vetorial correto com justificativa semântica
  • (b) 4 pts — 3 zonas descritas corretamente com formato e conteúdo (1,3 pt cada)
  • (c) 4 pts — identificação e técnica correta por coluna pessoal/sensível
  • (d) 4 pts — pelo menos 4 tasks com dependências lógicas corretas e estrutura DAG válida
