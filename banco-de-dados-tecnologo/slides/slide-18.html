Replicação — copiar dados para múltiplos servidores

Replicação é o processo de manter cópias sincronizadas do banco em diferentes servidores. Resolve dois problemas: disponibilidade (se o servidor primário cair, uma réplica assume) e escalabilidade de leitura (distribuir queries SELECT entre réplicas).

Replicação síncrona vs assíncrona:

  • Síncrona: COMMIT só confirma quando a réplica confirma recepção. RPO = 0 (zero perda de dados). Mais lenta.
  • Assíncrona: COMMIT confirma imediatamente no primário. Réplica pode estar alguns milissegundos atrás. Padrão do PostgreSQL — mais rápida.

Tipos de réplica no PostgreSQL:

  • Streaming Replication: réplica recebe WAL (Write-Ahead Log) em tempo real. Réplica é read-only.
  • Logical Replication: replica tabelas específicas, permite writes na réplica para outras tabelas.
  • Patroni: ferramenta que gerencia failover automático em cluster PostgreSQL.
Replicação ≠ Backup. Se você deletar acidentalmente uma tabela, a deleção é replicada para todas as réplicas em segundos. Replicação protege contra falha de hardware — backup protege contra erro humano e corrupção lógica.
SQL — configurar replicação e inspecionar status
-- No postgresql.conf do servidor primário:
-- wal_level = replica
-- max_wal_senders = 3
-- wal_keep_size = 64MB

-- Criar usuário de replicação
CREATE ROLE replicador WITH REPLICATION LOGIN PASSWORD 'senha_replicacao';

-- Ver réplicas conectadas (no primário)
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    write_lag,
    flush_lag,
    replay_lag,
    sync_state
FROM pg_stat_replication;

-- Ver se sou réplica ou primário
SELECT pg_is_in_recovery();
-- TRUE = sou réplica (standby) | FALSE = sou primário

-- Lag de replicação em bytes
SELECT
    application_name,
    pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS lag_bytes
FROM pg_stat_replication;
Sharding — escalar horizontalmente

Sharding distribui os dados de uma tabela entre múltiplos servidores (shards) com base em uma chave. Ao contrário da replicação (mesmos dados em todos), no sharding cada servidor tem uma fração dos dados.

ConceitoDefiniçãoExemplo
Shard keyColuna usada para distribuir os dados entre shardscliente_id % N distribui entre N shards
Range shardingShard por intervalo de valores da chaveIDs 1–1M → shard 1; 1M–2M → shard 2
Hash shardingShard por hash da chave — distribuição uniformehash(cliente_id) % 4 → shard 0,1,2,3
Consistent hashingPermite adicionar/remover shards com mínimo re-balanceamentoCassandra, Redis Cluster
Cross-shard queryQuery que precisa acessar múltiplos shards — lenta e complexaJOIN entre duas tabelas em shards diferentes
No PostgreSQL: sharding nativo via pg_partman + postgres_fdw (Foreign Data Wrappers). Ferramentas dedicadas: Citus, Vitess, PlanetScale. Para a maioria dos sistemas, particionamento de tabela (semana 14) é suficiente antes de precisar de sharding real.
Backup e restore — estratégias e comandos
pg_dump — backup lógico

Exporta SQL ou formato binário. Portátil — restaura em versões diferentes. Mais lento para tabelas grandes.

pg_basebackup — físico

Cópia binária dos arquivos do banco. Muito rápido para bancos grandes. Deve ser restaurado na mesma versão do PostgreSQL.

PITR — Point-in-Time Recovery

Combina pg_basebackup + WAL arquivado. Permite restaurar para qualquer momento no passado. Padrão para produção.

Bash — backup e restore com pg_dump
# Backup completo em formato custom (comprimido, permite restore seletivo)
pg_dump -U postgres -d meu_banco -F c -f backup_$(date +%Y%m%d).dump

# Backup apenas do schema (sem dados)
pg_dump -U postgres -d meu_banco --schema-only -f schema.sql

# Backup de uma tabela específica
pg_dump -U postgres -d meu_banco -t pedidos -F c -f pedidos.dump

# Restore completo
pg_restore -U postgres -d meu_banco_novo -F c backup_20250115.dump

# Restore de uma tabela específica de um backup completo
pg_restore -U postgres -d meu_banco -t pedidos -F c backup_20250115.dump

# pg_dumpall — backup de todos os bancos + roles + tablespaces
pg_dumpall -U postgres -f todos_os_bancos.sql

# Verificar integridade do arquivo de backup
pg_restore --list backup_20250115.dump | head -20
MétricaDefiniçãoExemplo
RPO (Recovery Point Objective)Máxima perda de dados tolerável — "quanto de dado posso perder?"RPO = 1h → backup a cada hora; perco no máx. 1h de dados
RTO (Recovery Time Objective)Tempo máximo para recuperar o sistema — "quanto tempo posso ficar fora?"RTO = 15min → sistema deve estar operacional em 15 minutos
SLA (Service Level Agreement)Acordo de disponibilidade — "99.9% uptime = 8.7h de downtime/ano"99.99% = 52min downtime/ano
Alta disponibilidade — Patroni e pgBouncer

Patroni — failover automático:

  • Gerencia cluster PostgreSQL com primary + standbys
  • Detecta falha do primário e promove automaticamente uma réplica
  • Usa etcd, ZooKeeper ou Consul como DCS (distributed configuration store)
  • Expõe endpoint REST para verificar quem é o primário atual

pgBouncer — connection pooling:

  • Proxy entre a aplicação e o PostgreSQL
  • Mantém poucas conexões reais ao banco mas aceita milhares da aplicação
  • 3 modos: session pool, transaction pool (mais comum), statement pool
  • Reduz overhead de conexão — cada conexão PG custa ~5MB de RAM
Bash — verificar saúde do cluster com Patroni
# Ver status do cluster (qual é o primário?)
patronictl -c /etc/patroni/config.yml list

# Saída esperada:
# + Cluster: postgres-cluster ----+----+-----------+
# | Member   | Host        | Role   | State   | TL |
# +----------+-------------+--------+---------+----+
# | pg-node1 | 10.0.0.1:5432 | Leader | running |  1 |
# | pg-node2 | 10.0.0.2:5432 | Replica| running |  1 |

# Verificar via API REST (útil para load balancers)
curl -s http://10.0.0.1:8008/primary   # 200 = é o primário
curl -s http://10.0.0.2:8008/replica   # 200 = é réplica

# Failover manual controlado
patronictl -c /etc/patroni/config.yml failover postgres-cluster

Questão 01
Indexação — B-tree e seletividade
Uma tabela pedidos tem 2 milhões de linhas e uma coluna status com apenas 3 valores distintos: "pendente", "entregue", "cancelado". Um desenvolvedor criou um índice B-tree em status para acelerar a query WHERE status = 'entregue', mas o PostgreSQL continua fazendo Seq Scan. Por que isso acontece? Em que situação o índice em status seria útil?
Por que o Seq Scan persiste: "entregue" representa ~33% das linhas (1 de 3 valores). O planner estima que um Index Scan para buscar 666k linhas seria mais caro que um Seq Scan: cada linha encontrada no índice exige um heap fetch (acesso ao bloco da tabela) — para 666k acessos aleatórios, o custo supera o de ler a tabela sequencialmente. O planner escolhe Seq Scan racionalmente.

Quando seria útil: com um índice composto — ex: (status, criado_em) e uma query WHERE status = 'pendente' AND criado_em > '2025-01-01'. A combinação é muito mais seletiva (ex: 0,1% das linhas) e o Index Scan compensa. Ou um covering index com INCLUDE que evita o heap fetch completamente (Index Only Scan).
Questão 02
Indexação — regra do prefixo
Existe um índice composto (cidade, data_criacao, status) na tabela clientes. Diga se cada query abaixo usa ou não usa o índice, justificando: (a) WHERE cidade = 'SP'; (b) WHERE data_criacao > '2025-01-01'; (c) WHERE cidade = 'SP' AND data_criacao > '2025-01-01'; (d) WHERE cidade = 'SP' AND status = 'ativo'.
(a) ✅ Usa — filtra pela 1ª coluna do prefixo (cidade). O índice é útil para encontrar todas as linhas de SP.
(b) ❌ Não usa — pula a 1ª coluna (cidade) e vai direto para a 2ª. Sem cidade no filtro, o banco não consegue navegar pelo prefixo da B-tree e faz Seq Scan.
(c) ✅ Usa com máxima eficiência — filtra pela 1ª coluna (cidade = SP, igualdade) e depois restringe por range na 2ª (data). O índice elimina a maioria das linhas.
(d) ✅ Usa parcialmente — filtra por cidade (1ª coluna, igualdade) mas pula data_criacao (2ª coluna) e vai para status (3ª). O índice é usado para encontrar clientes de SP, mas status precisa ser verificado linha a linha dentro dessa partição — menos eficiente que (c).
Questão 03
Transações — ACID e anomalias
Explique com um exemplo concreto o que é uma Non-Repeatable Read e qual nível de isolamento do PostgreSQL a previne. Por que o nível padrão (Read Committed) não a previne?
Non-Repeatable Read: a Transação A lê o saldo da conta 1 e obtém R$1.000. Antes de A terminar, a Transação B atualiza o saldo para R$500 e dá COMMIT. Quando A relê o saldo da mesma linha, obtém R$500 — o mesmo dado retornou valor diferente dentro da mesma transação.

Por que Read Committed não previne: neste nível, cada statement lê um snapshot atualizado no momento em que ele é executado. Entre dois SELECTs na mesma transação, COMMITs de outras transações se tornam visíveis. A transação A não tem garantia de que dados não mudarão entre leituras.

Solução — Repeatable Read: o PostgreSQL cria um snapshot no início da transação. Todos os SELECTs dentro da transação enxergam o mesmo estado do banco — COMMITs feitos por outras transações após o início de A não são visíveis para A.
Questão 04
Transações — deadlock
Descreva a sequência exata de eventos que causa um deadlock entre duas transações acessando as contas de IDs 10 e 20. Em seguida, descreva duas estratégias para prevenir esse deadlock.
Sequência do deadlock:
t1: Tx A faz UPDATE em conta 10 → adquire lock em 10
t2: Tx B faz UPDATE em conta 20 → adquire lock em 20
t3: Tx A tenta UPDATE em conta 20 → aguarda Tx B liberar
t4: Tx B tenta UPDATE em conta 10 → aguarda Tx A liberar → ciclo!
O PostgreSQL detecta o ciclo e cancela uma das transações com "deadlock detected".

Estratégia 1 — Ordenação de locks: Tx A e Tx B devem sempre bloquear o menor ID primeiro. Se ambas usam ORDER BY id FOR UPDATE, nunca formarão um ciclo de espera — a primeira que chegar ao menor ID terá o lock, a outra esperará sem criar deadlock.

Estratégia 2 — Retry com backoff: capturar psycopg2.errors.DeadlockDetected, fazer ROLLBACK e retentar a transação com espera exponencial (0.1s, 0.2s, 0.4s…). Não previne o deadlock, mas torna o sistema resiliente — a transação vítima re-executa e geralmente encontra o caminho livre.
Questão 05
Integração Python — segurança
O código abaixo tem uma vulnerabilidade grave. Identifique o problema, explique o risco e reescreva o trecho de forma segura:

status = request.args.get("status")
cur.execute(f"SELECT * FROM pedidos WHERE status = '{status}'")
Problema: SQL Injection. A variável status é interpolada diretamente na string SQL com f-string. Um atacante pode passar status = "' OR '1'='1" para retornar todos os pedidos, ou status = "x'; DROP TABLE pedidos; --" para apagar a tabela inteira.

Versão segura com psycopg2:
Python
# psycopg2 — parâmetros como tupla
cur.execute("SELECT * FROM pedidos WHERE status = %s", (status,))

# SQLAlchemy — parâmetros nomeados com text()
conn.execute(text("SELECT * FROM pedidos WHERE status = :status"),
             {"status": status})
O driver escapa automaticamente os valores, tornando qualquer tentativa de injeção inofensiva.
Questão 06
Integração Python — performance
Explique a diferença de performance entre: (a) inserir 50.000 linhas com um loop Python chamando cur.execute() individualmente; (b) usando execute_values do psycopg2 com page_size=1000; (c) usando df.to_sql(..., method="multi", chunksize=1000). Por que existe essa diferença?
(a) Loop individual: cada execute() gera um round-trip cliente→servidor→cliente: 50.000 round-trips para 50.000 linhas. O overhead de rede e de parse SQL domina. Muito lento (pode levar minutos).

(b) execute_values: agrupa as inserções em batches de 1.000 linhas. Em vez de 50.000 statements, envia 50 statements com 1.000 valores cada. O PostgreSQL recebe e processa muito mais eficientemente. Tipicamente 20–100× mais rápido que o loop.

(c) to_sql com method="multi" e chunksize=1000: similar ao execute_values — pandas gera INSERTs com múltiplos VALUES por statement. Levemente diferente internamente mas mesma ordem de magnitude de performance. A vantagem do pandas é a integração direta com DataFrames.

Por que a diferença: o custo dominante em inserções em loop é o round-trip de rede + parsing SQL + overhead de transação por statement. Batch insert amortiza esses custos por N linhas em vez de 1.

Questão 08
Replicação e Backup
Uma empresa tem um banco PostgreSQL em produção com replicação assíncrona para um servidor standby. Um desenvolvedor apaga acidentalmente a tabela pedidos com DROP TABLE pedidos. (a) O standby será afetado? (b) Como recuperar os dados? (c) Como evitar esse problema no futuro?
(a) Sim, o standby será afetado. A replicação replica todas as operações WAL, incluindo DDL como DROP TABLE. Em segundos (ou imediatamente em replicação síncrona), a tabela também desaparece do standby. A replicação não protege contra erro humano — é uma cópia fiel, inclusive dos erros.

(b) Recuperação: usar backup. Se a empresa tem PITR (Point-in-Time Recovery) configurado com WAL archiving, é possível restaurar o banco para o momento exato antes do DROP. Alternativamente, restaurar o backup lógico mais recente (pg_restore) e depois aplicar os WALs até o ponto do acidente. Sem backup, os dados estão perdidos.

(c) Prevenção: (1) Política de backup regular — pg_dump diário ou pg_basebackup + WAL archiving para PITR. (2) Permissões por papel — usuário da aplicação não deve ter DROP TABLE; apenas o DBA deve ter. (3) Proteção de tabelas críticas: REVOKE DROP ON TABLE pedidos FROM app_user. (4) Teste periódico de restore — backup não testado não é backup confiável.
Questão 09
RPO, RTO e Alta Disponibilidade
Uma fintech define em seu SLA: RPO = 5 minutos e RTO = 2 minutos. (a) O que significa cada métrica neste contexto? (b) Qual estratégia de backup e replicação atenderia esses requisitos? (c) O que é Patroni e como ele contribui para o RTO de 2 minutos?
(a) RPO = 5 minutos: se o servidor cair, a empresa aceita perder no máximo 5 minutos de transações. Dados mais antigos que 5 minutos devem estar preservados. RTO = 2 minutos: o sistema deve estar operacional novamente em até 2 minutos após a falha. Cada minuto além disso representa prejuízo financeiro.

(b) Estratégia: replicação síncrona (RPO = 0, zero perda) ou assíncrona com lag < 5 min + WAL archiving a cada 5 minutos. Cluster Patroni com pelo menos 1 réplica hot standby que pode assumir imediatamente. Backups diários com pg_basebackup para recuperação de desastres.

(c) Patroni: é uma ferramenta de gestão de clusters PostgreSQL de alta disponibilidade. Monitora continuamente o primário. Se detectar falha (timeout de heartbeat), automaticamente elege e promove a réplica mais atualizada como novo primário em segundos — sem intervenção manual. O failover automático é o que torna o RTO de 2 minutos alcançável; sem ele, seria necessário intervenção humana que tipicamente leva 10–30 minutos.
Questão 10 — Integradora
Pipeline completo — todos os temas do bloco
Você foi contratado para construir um sistema de dados para uma empresa de saúde. Descreva as decisões técnicas para: (a) Qual índice criar para acelerar buscas por paciente_id e data_consulta? (b) Qual nível de isolamento usar para transações de registro de consultas e por quê? (c) Como estruturar a conexão Python ao banco com segurança? (d) Como alimentar um modelo de ML de risco de internação com dados do banco e gravar as predições de volta com rastreabilidade? (e) Qual estratégia de backup para RPO = 1 hora?
(a) Índice: CREATE INDEX idx_consultas_comp ON consultas (paciente_id, data_consulta DESC). Índice composto com paciente primeiro (igualdade) e data segundo (range/ordenação). Adicionar INCLUDE (diagnostico, medico_id) se a query precisar dessas colunas — evita heap fetch com Index Only Scan.

(b) Isolamento: Read Committed (padrão) é suficiente para registro de consultas. Cada INSERT/UPDATE em consulta é autocontido. Se houver relatório que lê múltiplas tabelas e precisa de snapshot consistente (ex: prontuário completo), usar Repeatable Read para aquela transação específica.

(c) Conexão segura: credenciais em .env carregado com python-dotenv. SQLAlchemy engine com pool_size=5, pool_recycle=1800, sslmode=require. Usuário de banco com mínimas permissões (SELECT/INSERT/UPDATE nas tabelas necessárias — sem DROP). Parâmetros SQL sempre com text(:param) — nunca f-string.

(d) Pipeline IA: read_sql extrai histórico de consultas e exames → groupby(paciente_id).agg() gera features (freq. de consultas, diagnósticos prévios, idade, comorbidades) → treino com RandomForest → joblib.dump salva modelo → predict_proba gera scores → to_sql("predicoes_internacao", if_exists="append") com modelo_versao e gerado_em → VIEW SQL para consumo pelo time clínico.

(e) Backup RPO=1h: pg_basebackup diário para backup físico completo + archive_mode = on no PostgreSQL arquivando WAL segments a cada minuto (garantindo RPO << 1h). PITR permite restaurar para qualquer ponto. Armazenar backups em storage externo (S3). Testar restore mensalmente.

Dicas finais para a prova
  • Em EXPLAIN, Seq Scan com Rows Removed alto = índice faltando na coluna do filtro
  • Index Only Scan é melhor que Index Scan — significa que o índice tem tudo que a query precisa
  • Read Committed previne dirty read mas não non-repeatable. Repeatable Read previne ambos
  • Deadlock = ciclo de espera. Prevenção = mesma ordem de locks sempre
  • psycopg2 usa %s. SQLAlchemy usa :nome. Nunca f-string em SQL.
  • to_sql method="multi" é muito mais rápido que o padrão — use sempre com chunksize
  • No pipeline IA: features via groupby+agg → treino → predict_proba[:,1] → to_sql append
  • RPO = perda de dados. RTO = tempo de recuperação. São diferentes!
  • Replicação protege contra falha de hardware. Backup protege contra erro humano.
  • Sharding distribui dados horizontalmente. Particionamento de tabela é local em um servidor.
