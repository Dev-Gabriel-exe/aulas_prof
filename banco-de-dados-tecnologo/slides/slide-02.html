Subqueries — consultas dentro de consultas
O que é uma subquery?

Uma subquery é uma consulta SELECT dentro de outra consulta. O resultado interno é usado como valor, filtro ou tabela pela consulta externa.

Quando usar: quando você precisa de um valor calculado dinamicamente — como a média de todos os preços — sem precisar rodá-lo separadamente e substituir manualmente.
Subquery Escalar — retorna um único valor

A subquery retorna exatamente uma linha e uma coluna. Pode aparecer onde um valor literal seria usado.

SQL — Subquery Escalar
-- Produtos com preço ACIMA da média geral
SELECT nome, preco
FROM produtos
WHERE preco > (
  SELECT AVG(preco) FROM produtos   -- retorna um número: 745.65
)
ORDER BY preco DESC;

-- No SELECT: mostrar quanto cada produto está acima da média
SELECT
  nome,
  preco,
  ROUND(AVG(preco) OVER (), 2)      AS media_geral,
  ROUND(preco - (SELECT AVG(preco) FROM produtos), 2) AS diferenca_media
FROM produtos
ORDER BY diferenca_media DESC;
Subquery no FROM — como se fosse uma tabela

A subquery vira uma tabela temporária (derivada) que pode ser consultada normalmente. Obrigatório dar um alias.

SQL — Subquery no FROM
-- Total gasto por cliente → filtrar só quem gastou acima da média
SELECT cliente_nome, total_gasto
FROM (
  SELECT
    c.nome          AS cliente_nome,
    SUM(p.total)    AS total_gasto
  FROM clientes c
  INNER JOIN pedidos p ON p.cliente_id = c.id
  WHERE p.status = 'entregue'
  GROUP BY c.id, c.nome
) AS resumo_clientes   -- alias obrigatório
WHERE total_gasto > (
  SELECT AVG(total) FROM pedidos WHERE status = 'entregue'
)
ORDER BY total_gasto DESC;
Subquery Correlacionada — referencia a query externa

A subquery referencia uma coluna da query externa. É executada uma vez para cada linha da query externa — pode ser lenta em tabelas grandes, mas é muito expressiva.

SQL — Correlacionada
-- Clientes que têm pelo menos um pedido com total > 2000
SELECT c.nome, c.cidade
FROM clientes c
WHERE EXISTS (
  SELECT 1
  FROM pedidos p
  WHERE p.cliente_id = c.id   -- ← referencia c da query externa
    AND p.total > 2000
);

-- Clientes que NUNCA fizeram pedido
SELECT c.nome
FROM clientes c
WHERE NOT EXISTS (
  SELECT 1 FROM pedidos p WHERE p.cliente_id = c.id
);
EXISTS vs IN: use EXISTS quando a subquery pode retornar muitas linhas — ele para na primeira correspondência. Use IN quando a lista é pequena e estática.
Quando usar subquery
  • Valor calculado dinamicamente (média, máximo)
  • Filtrar por resultado de agregação sem GROUP BY
  • EXISTS para verificar existência com eficiência
  • Tabela derivada no FROM para pré-filtrar
Limitações
  • Subquery correlacionada em tabela grande = lento
  • Subqueries aninhadas demais = difícil de manter
  • Solução: substituir por CTE (próximo bloco)

CTEs com WITH — modularidade e legibilidade
O que é uma CTE?

CTE (Common Table Expression) é uma consulta nomeada temporária definida com WITH antes do SELECT principal. É como dar um nome a uma subquery para reutilizá-la.

WITH nome AS ( query ) → SELECT * FROM nome → resultado final
Vantagem real: uma CTE com 4 partes encadeadas é muito mais legível do que 4 subqueries aninhadas. Em pipelines de BI e feature engineering, CTEs são padrão de mercado.
CTE simples — uma etapa
SQL — CTE básica
-- Clientes com total gasto acima da média
WITH total_por_cliente AS (
  SELECT
    c.id,
    c.nome,
    SUM(p.total) AS total_gasto
  FROM clientes c
  INNER JOIN pedidos p ON p.cliente_id = c.id
  WHERE p.status = 'entregue'
  GROUP BY c.id, c.nome
)
SELECT nome, total_gasto
FROM total_por_cliente
WHERE total_gasto > (SELECT AVG(total_gasto) FROM total_por_cliente)
ORDER BY total_gasto DESC;
CTEs encadeadas — múltiplos WITH

Você pode definir várias CTEs em sequência, onde cada uma pode referenciar as anteriores. Isso cria um pipeline de transformação legível.

SQL — CTEs encadeadas
WITH

-- Etapa 1: pedidos entregues por cliente
pedidos_entregues AS (
  SELECT cliente_id, COUNT(*) AS qtd, SUM(total) AS total_gasto
  FROM pedidos
  WHERE status = 'entregue'
  GROUP BY cliente_id
),

-- Etapa 2: classificar clientes como VIP ou regular
clientes_classificados AS (
  SELECT
    c.nome,
    c.cidade,
    pe.qtd,
    pe.total_gasto,
    CASE
      WHEN pe.total_gasto > 3000 THEN 'VIP'
      WHEN pe.total_gasto > 500  THEN 'Regular'
      ELSE 'Ocasional'
    END AS categoria
  FROM clientes c
  INNER JOIN pedidos_entregues pe ON pe.cliente_id = c.id  -- referencia CTE anterior
),

-- Etapa 3: só os VIP
clientes_vip AS (
  SELECT * FROM clientes_classificados WHERE categoria = 'VIP'
)

-- Query final: usar a CTE do passo 3
SELECT nome, cidade, qtd AS pedidos, total_gasto
FROM clientes_vip
ORDER BY total_gasto DESC;
Cada CTE existe apenas dentro da query que a define. Ela não é uma VIEW permanente — é descartada assim que a query termina.

CritérioSubqueryCTE

Window Functions — análise sem perder as linhas
O problema que window functions resolvem

Com GROUP BY, você agrega e perde as linhas originais. Com window functions, você calcula valores analíticos mantendo todas as linhas.

GROUP BY — perde as linhas

SQL
SELECT categoria, AVG(preco)
FROM produtos
GROUP BY categoria;
-- retorna 2 linhas (uma por categoria)
-- perdemos os produtos individuais

Window Function — mantém as linhas

SQL
SELECT nome, categoria, preco,
  AVG(preco) OVER (PARTITION BY categoria)
             AS media_categoria
FROM produtos;
-- retorna TODOS os produtos + a média
ROW_NUMBER, RANK e DENSE_RANK

As três funções de ranking — diferem apenas quando há empates:

FunçãoComportamento no empateExemplo (empate na posição 2)
ROW_NUMBER()Numera sem repetir — ordem arbitrária no empate1, 2, 3, 4
RANK()Pula posições após empate1, 2, 2, 4
DENSE_RANK()Não pula posições1, 2, 2, 3
SQL — Ranking de produtos por preço dentro da categoria
SELECT
  nome,
  categoria,
  preco,
  ROW_NUMBER() OVER (PARTITION BY categoria ORDER BY preco DESC) AS row_num,
  RANK()       OVER (PARTITION BY categoria ORDER BY preco DESC) AS rank,
  DENSE_RANK() OVER (PARTITION BY categoria ORDER BY preco DESC) AS dense_rank
FROM produtos
ORDER BY categoria, preco DESC;
SUM, AVG e COUNT com OVER — acumulados e particionados
SQL — Agregações com janela
-- Total acumulado de vendas por data (running total)
SELECT
  criado_em::date                          AS data,
  SUM(total)                               AS vendas_dia,
  SUM(SUM(total)) OVER (
    ORDER BY criado_em::date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  )                                        AS total_acumulado
FROM pedidos
WHERE status = 'entregue'
GROUP BY criado_em::date
ORDER BY data;

-- % que cada pedido representa do total do cliente
SELECT
  c.nome                                   AS cliente,
  p.id                                     AS pedido,
  p.total,
  SUM(p.total) OVER (PARTITION BY c.id)    AS total_cliente,
  ROUND(
    p.total / SUM(p.total) OVER (PARTITION BY c.id) * 100, 1
  )                                        AS pct_do_cliente
FROM pedidos p
INNER JOIN clientes c ON c.id = p.cliente_id
WHERE p.status = 'entregue'
ORDER BY c.nome, p.total DESC;
LAG e LEAD — comparando com linhas vizinhas

LAG(col, n) acessa n linhas antes. LEAD(col, n) acessa n linhas depois. Essencial para séries temporais e feature engineering.

SQL — LAG para variação diária
WITH vendas_diarias AS (
  SELECT
    criado_em::date    AS data,
    SUM(total)         AS vendas
  FROM pedidos
  WHERE status = 'entregue'
  GROUP BY criado_em::date
)
SELECT
  data,
  vendas,
  LAG(vendas, 1) OVER (ORDER BY data)             AS vendas_dia_anterior,
  vendas - LAG(vendas, 1) OVER (ORDER BY data)     AS variacao_abs,
  ROUND(
    (vendas - LAG(vendas, 1) OVER (ORDER BY data))
    / NULLIF(LAG(vendas, 1) OVER (ORDER BY data), 0) * 100, 1
  )                                                   AS variacao_pct
FROM vendas_diarias
ORDER BY data;
NULLIF(expr, 0): evita divisão por zero — retorna NULL quando o divisor é 0, em vez de lançar erro.
NTILE, FIRST_VALUE e LAST_VALUE
SQL — Outros window functions úteis
-- NTILE(4): dividir clientes em quartis de gasto
WITH gastos AS (
  SELECT c.nome, SUM(p.total) AS total
  FROM clientes c JOIN pedidos p ON p.cliente_id = c.id
  GROUP BY c.id, c.nome
)
SELECT nome, total,
  NTILE(4) OVER (ORDER BY total) AS quartil  -- 1=menor, 4=maior
FROM gastos;

-- FIRST_VALUE / LAST_VALUE: produto mais barato e mais caro da categoria
SELECT
  nome, categoria, preco,
  FIRST_VALUE(nome) OVER (PARTITION BY categoria ORDER BY preco)     AS mais_barato,
  FIRST_VALUE(nome) OVER (PARTITION BY categoria ORDER BY preco DESC) AS mais_caro
FROM produtos;

6 Exercícios — Subqueries, CTEs e Window Functions
Use o banco loja_exemplo criado na Semana 1. Tente resolver antes de ver o gabarito.
Exercício 01

Liste os produtos cujo preço está acima da média geral de preços. Mostre nome, preço e a diferença em relação à média. Ordene do maior para o menor preço.

Básico — Subquery Escalar
SQL
SELECT
  nome,
  preco,
  ROUND(preco - (SELECT AVG(preco) FROM produtos), 2) AS acima_da_media
FROM produtos
WHERE preco > (SELECT AVG(preco) FROM produtos)
ORDER BY preco DESC;
Exercício 02

Liste os clientes que nunca fizeram um pedido. Use NOT EXISTS.

Básico — NOT EXISTS
SQL
SELECT c.nome, c.cidade
FROM clientes c
WHERE NOT EXISTS (
  SELECT 1 FROM pedidos p WHERE p.cliente_id = c.id
);
Exercício 03

Usando uma CTE, calcule o total gasto por cada cliente em pedidos entregues. Em seguida, classifique como 'VIP' (acima de R$ 3.000), 'Regular' (entre R$ 500 e R$ 3.000) ou 'Ocasional' (abaixo de R$ 500).

Médio — CTE + CASE WHEN
SQL
WITH gastos_cliente AS (
  SELECT
    c.nome,
    SUM(p.total) AS total_gasto
  FROM clientes c
  INNER JOIN pedidos p ON p.cliente_id = c.id
  WHERE p.status = 'entregue'
  GROUP BY c.id, c.nome
)
SELECT
  nome,
  total_gasto,
  CASE
    WHEN total_gasto > 3000 THEN 'VIP'
    WHEN total_gasto > 500  THEN 'Regular'
    ELSE 'Ocasional'
  END AS categoria
FROM gastos_cliente
ORDER BY total_gasto DESC;
Exercício 04

Para cada produto que já foi vendido, mostre o nome, a quantidade total vendida e o ranking dentro da sua categoria (do mais vendido para o menos). Use RANK() com PARTITION BY.

Médio — RANK + PARTITION BY
SQL
WITH vendas_produto AS (
  SELECT
    pr.nome        AS produto,
    pr.categoria,
    SUM(i.quantidade) AS qtd_vendida
  FROM produtos pr
  INNER JOIN itens_pedido i ON i.produto_id = pr.id
  GROUP BY pr.id, pr.nome, pr.categoria
)
SELECT
  produto,
  categoria,
  qtd_vendida,
  RANK() OVER (PARTITION BY categoria ORDER BY qtd_vendida DESC) AS rank_categoria
FROM vendas_produto
ORDER BY categoria, rank_categoria;
Exercício 05

Mostre cada pedido entregue com o seu total, o total acumulado (running sum) até aquele pedido (ordenado por data) e o percentual que ele representa do total geral de pedidos entregues.

Médio — SUM OVER + Running Total
SQL
SELECT
  p.id,
  c.nome                                                        AS cliente,
  p.criado_em::date                                             AS data,
  p.total,
  SUM(p.total) OVER (ORDER BY p.criado_em
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)          AS total_acumulado,
  ROUND(
    p.total / SUM(p.total) OVER () * 100, 1
  )                                                             AS pct_total
FROM pedidos p
INNER JOIN clientes c ON c.id = p.cliente_id
WHERE p.status = 'entregue'
ORDER BY p.criado_em;
Exercício 06 — Desafio

Usando CTEs encadeadas: 1) calcule o total vendido por produto; 2) rankei por categoria com DENSE_RANK(); 3) retorne apenas os top 1 de cada categoria (o produto mais vendido em cada categoria).

Desafio — CTEs + DENSE_RANK + Filtro de ranking
SQL
WITH

vendas_produto AS (
  SELECT
    pr.nome        AS produto,
    pr.categoria,
    SUM(i.quantidade) AS qtd_vendida,
    SUM(i.quantidade * i.preco_unit) AS receita
  FROM produtos pr
  INNER JOIN itens_pedido i ON i.produto_id = pr.id
  GROUP BY pr.id, pr.nome, pr.categoria
),

rankeados AS (
  SELECT *,
    DENSE_RANK() OVER (PARTITION BY categoria ORDER BY qtd_vendida DESC) AS pos
  FROM vendas_produto
)

SELECT categoria, produto, qtd_vendida, receita
FROM rankeados
WHERE pos = 1    -- só o top 1 de cada categoria
ORDER BY categoria;
