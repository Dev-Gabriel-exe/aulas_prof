Aula 1 · ~50 minutos
Revisão Express — Semanas 1 a 5
5–8 minutos por semana. O objetivo não é re-ensinar — é garantir que os conceitos-chave estão frescos na memória antes do simulado.
Semana 1
Modelo Relacional e SQL Básico
  • PK identifica unicamente cada linha — não pode ser nula
  • FK cria vínculo entre tabelas — garante integridade referencial
  • INNER JOIN: só linhas com correspondência nos dois lados
  • LEFT JOIN: todos da tabela esquerda, mesmo sem match
  • WHERE filtra antes do GROUP BY · HAVING filtra depois
Semana 2
Subqueries, CTEs e Window Functions
  • Subquery escalar retorna um único valor — usada no WHERE
  • EXISTS para verificar existência — para na primeira linha
  • WITH cria CTEs nomeadas — mais legível que subquery aninhada
  • Window function: calcula sem colapsar linhas (≠ GROUP BY)
  • RANK pula posições no empate · DENSE_RANK não pula
  • LAG acessa linha anterior · LEAD acessa linha seguinte
Semana 3
Normalização 1FN → FNBC
  • 1FN: atributos atômicos, sem grupos repetidos
  • 2FN: 1FN + sem dependência parcial da PK composta
  • 3FN: 2FN + sem dependência transitiva entre não-chaves
  • FNBC: todo determinante deve ser superchave
  • Anomalias: inserção, atualização e exclusão
  • OLTP normaliza · OLAP/ML pode desnormalizar com justificativa
Semana 4
Pipelines de Dados para IA
  • ETL: transforma antes de carregar · ELT: carrega bruto, transforma depois
  • Raw/Bronze → Silver (limpo) → Gold (pronto para ML)
  • Parquet: binário colunar, 3–10x menor que CSV, schema embutido
  • FILTER (WHERE ...) no PostgreSQL: condicional dentro de agregação
  • COALESCE e NULLIF para tratar nulos com segurança
  • pd.read_sql + to_sql: Python ↔ PostgreSQL via SQLAlchemy
Semana 5
Bancos NoSQL — MongoDB e Redis
  • NoSQL = "Not Only SQL" — projetado para casos específicos
  • Tipos: documento, chave-valor, colunar, grafos
  • CAP: Consistência, Disponibilidade, Partição — escolha 2
  • MongoDB: coleção = tabela · documento = linha · field = coluna
  • find() com projeção · $match, $group, $lookup no aggregate
  • Redis: in-memory → 10–100x mais rápido que BD em disco
  • TTL: chaves expiram automaticamente — ideal para cache
  • Cache-Aside: leia cache → miss → busque fonte → grave cache

Q1
SQL 2,0 pts

Usando o banco loja_exemplo, escreva uma única query que retorne: nome do cliente, cidade, quantidade de pedidos entregues e ticket médio. Mostre apenas clientes com 2 ou mais pedidos entregues, ordenados pelo ticket médio decrescente.

SQL — Gabarito Q1
SELECT
  c.nome,
  c.cidade,
  COUNT(p.id)               AS qtd_pedidos,
  ROUND(AVG(p.total)::numeric, 2) AS ticket_medio
FROM clientes c
INNER JOIN pedidos p ON p.cliente_id = c.id
WHERE p.status = 'entregue'
GROUP BY c.id, c.nome, c.cidade
HAVING COUNT(p.id) >= 2
ORDER BY ticket_medio DESC;
Pontos críticos: WHERE p.status = 'entregue' filtra antes do GROUP BY. HAVING COUNT(p.id) >= 2 filtra depois. Sem o WHERE, contaria pedidos cancelados no ticket médio.
Q2
SQL — CTE 2,5 pts

Usando CTEs encadeadas: (1) calcule a receita total por produto; (2) calcule o rank de receita dentro de cada categoria usando DENSE_RANK(); (3) retorne apenas o produto com maior receita de cada categoria.

SQL — Gabarito Q2
WITH

receita_produto AS (
  SELECT
    pr.id,
    pr.nome,
    pr.categoria,
    SUM(i.quantidade * i.preco_unit) AS receita
  FROM produtos pr
  INNER JOIN itens_pedido i ON i.produto_id = pr.id
  GROUP BY pr.id, pr.nome, pr.categoria
),

rankeados AS (
  SELECT *,
    DENSE_RANK() OVER (
      PARTITION BY categoria
      ORDER BY receita DESC
    ) AS pos
  FROM receita_produto
)

SELECT categoria, nome, receita
FROM rankeados
WHERE pos = 1
ORDER BY receita DESC;
Ponto crítico: o filtro WHERE pos = 1 não pode estar dentro da CTE rankeados — window functions são calculadas depois do WHERE. Por isso precisamos de uma CTE extra ou subquery para filtrar pelo resultado do rank.
Q3
Window Function 2,0 pts

Escreva uma query que mostre cada pedido entregue com: id do pedido, nome do cliente, data, total do pedido, total acumulado (running sum ordenado por data) e percentual que cada pedido representa do total geral de pedidos entregues.

SQL — Gabarito Q3
SELECT
  p.id,
  c.nome                                                     AS cliente,
  p.criado_em::date                                          AS data,
  p.total,
  SUM(p.total) OVER (
    ORDER BY p.criado_em
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  )                                                          AS total_acumulado,
  ROUND(p.total / SUM(p.total) OVER () * 100, 1)           AS pct_total
FROM pedidos p
INNER JOIN clientes c ON c.id = p.cliente_id
WHERE p.status = 'entregue'
ORDER BY p.criado_em;
Ponto crítico: SUM(...) OVER () sem nenhuma cláusula calcula o total de todas as linhas do resultado — é o denominador do percentual. O frame ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW garante o acumulado até a linha atual.
Q4
Normalização 2,0 pts

A tabela abaixo tem PK composta (projeto_id, funcionario_id). Identifique todas as violações de forma normal, especifique qual forma normal cada uma viola, e escreva o SQL corrigido até a 3FN.

Esquema
alocacao(projeto_id, funcionario_id, horas_alocadas,
         nome_funcionario, cargo_funcionario, salario_funcionario,
         nome_projeto, cliente_projeto, orcamento_projeto)
Violações identificadas:
• nome_funcionario, cargo_funcionario, salario_funcionario → dependem só de funcionario_id — viola 2FN (dependência parcial)
• nome_projeto, cliente_projeto, orcamento_projeto → dependem só de projeto_id — viola 2FN (dependência parcial)
• horas_alocadas → depende da PK inteira ✅
SQL — Gabarito Q4 (corrigido até 3FN)
CREATE TABLE funcionarios (
  id     SERIAL PRIMARY KEY,
  nome   VARCHAR(100) NOT NULL,
  cargo  VARCHAR(80),
  salario NUMERIC(10,2)
);

CREATE TABLE projetos (
  id        SERIAL PRIMARY KEY,
  nome      VARCHAR(120) NOT NULL,
  cliente   VARCHAR(100),
  orcamento NUMERIC(12,2)
);

CREATE TABLE alocacoes (
  projeto_id     INT REFERENCES projetos(id),
  funcionario_id INT REFERENCES funcionarios(id),
  horas_alocadas NUMERIC(6,1) NOT NULL,
  PRIMARY KEY (projeto_id, funcionario_id)
);
Q5
Normalização — Conceito 1,5 pts

Explique com suas palavras a diferença entre dependência parcial (viola 2FN) e dependência transitiva (viola 3FN). Dê um exemplo de cada uma com uma tabela hipotética.

Dependência parcial (viola 2FN): ocorre quando um atributo não-chave depende de parte da chave primária composta — não dela inteira.

Exemplo: tabela matricula(turma_id, aluno_id, nota, nome_aluno) com PK (turma_id, aluno_id). nome_aluno depende só de aluno_id — dependência parcial.

Dependência transitiva (viola 3FN): ocorre quando um atributo não-chave determina outro atributo não-chave — criando uma cadeia indireta.

Exemplo: tabela pedido(id, cliente_id, cliente_cidade). id → cliente_id → cliente_cidade. cliente_cidade depende de cliente_id, que não é a PK — dependência transitiva.
Q6
Pipeline SQL 2,0 pts

Escreva uma query SQL que gere uma tabela gold de features de cliente para um modelo de propensão à compra. Inclua: cliente_id, total_pedidos, ticket_medio, dias_desde_ultimo_pedido, qtd_cancelados e pct_pedidos_entregues. Use CTEs para organizar.

SQL — Gabarito Q6
WITH metricas AS (
  SELECT
    cliente_id,
    COUNT(*)                                             AS total_pedidos,
    ROUND(AVG(total)::numeric, 2)                      AS ticket_medio,
    CURRENT_DATE - MAX(criado_em::date)               AS dias_desde_ultimo,
    COUNT(*) FILTER (WHERE status = 'cancelado')      AS qtd_cancelados,
    COUNT(*) FILTER (WHERE status = 'entregue')       AS qtd_entregues
  FROM pedidos
  GROUP BY cliente_id
)
SELECT
  cliente_id,
  total_pedidos,
  ticket_medio,
  dias_desde_ultimo,
  qtd_cancelados,
  ROUND(
    qtd_entregues::numeric / NULLIF(total_pedidos, 0) * 100, 1
  )                                                      AS pct_entregues
FROM metricas
ORDER BY ticket_medio DESC;
Q7
NoSQL — MongoDB 2,0 pts

No MongoDB, escreva um pipeline de agregação que: (1) filtre pedidos com status "entregue"; (2) agrupe por cliente_id calculando total de pedidos e soma do total; (3) ordene pelo maior total; (4) limite aos 5 maiores clientes. Depois, reescreva a consulta equivalente em SQL.

MongoDB
db.pedidos.aggregate([
  { $match:  { status: "entregue" } },
  { $group:  {
      _id:           "$cliente_id",
      total_pedidos: { $sum: 1 },
      soma_total:    { $sum: "$total" }
  }},
  { $sort:   { soma_total: -1 } },
  { $limit:  5 }
])
SQL — equivalente
SELECT
  cliente_id,
  COUNT(*) AS total_pedidos,
  SUM(total) AS soma_total
FROM pedidos
WHERE status = 'entregue'
GROUP BY cliente_id
ORDER BY soma_total DESC
LIMIT 5;
Q8
NoSQL — Conceito 2,0 pts

Você está projetando o backend de um sistema de recomendação de produtos para um e-commerce de alta escala. O sistema precisa: (a) armazenar o perfil de cada usuário com campos variáveis; (b) servir features pré-calculadas em menos de 5ms para o modelo de ML; (c) registrar o histórico de todas as interações (cliques, compras) com volume de 50k eventos/segundo. Para cada necessidade, indique qual banco usar e justifique a escolha.

(a) Perfis com campos variáveis → MongoDB
Esquema flexível: cada usuário pode ter campos diferentes sem necessidade de ALTER TABLE. Documentos JSON nativos para dados semi-estruturados.

(b) Features em menos de 5ms → Redis
In-memory garante latência sub-milissegundo. Hash por chave features:usuario:{id} com TTL. Features pré-calculadas em batch e servidas do cache sem tocar o banco relacional.

(c) 50k eventos/segundo → Cassandra
Banco colunar projetado para writes massivos e escala horizontal. Particionamento por usuario_id + timestamp distribui a carga. Eventual consistency (AP no CAP) é aceitável para histórico de cliques.
