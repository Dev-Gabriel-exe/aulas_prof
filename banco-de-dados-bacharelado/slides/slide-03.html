Por que normalizar? Anomalias e 1FN
O problema da redundância

Imagine uma tabela que armazena pedidos assim — tudo em uma linha só:

pedido_idcliente_nomecliente_cidadeprodutos_compradosfornecedorfornecedor_cidade
1Ana LimaSão PauloNotebook, MouseDellAustin
2Ana LimaSão PauloMonitorLGSeoul
3Bruno CostaRioTeclado, MouseLogitechLausanne

Parece simples — mas esse esquema tem três tipos de anomalias esperando para acontecer.

Anomalia de Inserção

Não consigo cadastrar um fornecedor sem que ele tenha pelo menos um pedido associado.

Anomalia de Atualização

Ana Lima mudou para Campinas — preciso atualizar N linhas. Se esquecer uma, os dados ficam inconsistentes.

Anomalia de Exclusão

Se deletar o único pedido do fornecedor Dell, perco todos os dados desse fornecedor.

Normalização é o processo de organizar as tabelas para eliminar redundância e evitar anomalias, aplicando regras chamadas Formas Normais.
Primeira Forma Normal (1FN)

Uma tabela está na 1FN quando:

  • Todos os atributos são atômicos (valor único, indivisível)
  • Não há grupos repetidos (colunas como produto1, produto2, produto3)
  • Cada linha é única (tem uma chave primária)
Viola a 1FN: a coluna produtos_comprados com valor "Notebook, Mouse" não é atômica — contém múltiplos valores numa única célula.

❌ Violação — não atômico

SQL — estrutura ruim
-- produtos_comprados não é atômico
pedido_id | produtos_comprados
    1     | 'Notebook, Mouse'
    2     | 'Monitor'

-- grupos repetidos — igualmente ruim
pedido_id | produto_1   | produto_2
    1     | 'Notebook'  | 'Mouse'
    2     | 'Monitor'   | NULL

✅ Correto — 1FN

SQL — correto
-- Uma linha por produto por pedido
pedido_id | produto_id | quantidade
    1     |     1      |     1      -- Notebook
    1     |     2      |     1      -- Mouse
    2     |     4      |     1      -- Monitor

-- PK composta: (pedido_id, produto_id)

Segunda e Terceira Formas Normais
Segunda Forma Normal (2FN)

Uma tabela está na 2FN quando está na 1FN e não tem dependências parciais — todo atributo não-chave depende da chave inteira, não de uma parte dela.

Só se aplica a tabelas com PK composta. Se a PK tem apenas uma coluna, a tabela já está automaticamente na 2FN (se estiver na 1FN).

Considere a tabela itens_pedido_ruim com PK composta (pedido_id, produto_id):

pedido_idproduto_idquantidadenome_produtocategoria
111Notebook DellEletrônicos
122Mouse LogitechEletrônicos
211Notebook DellEletrônicos
nome_produto e categoria dependem só de produto_id — não do pedido inteiro. Isso é uma dependência parcial: viola a 2FN.
SQL — Corrigindo para 2FN
-- Separar o que depende do produto inteiro
CREATE TABLE produtos (
  id        SERIAL PRIMARY KEY,
  nome      VARCHAR(120) NOT NULL,   -- depende de id (produto)
  categoria VARCHAR(60)               -- depende de id (produto)
);

-- Só fica o que depende da PK composta inteira
CREATE TABLE itens_pedido (
  pedido_id  INT REFERENCES pedidos(id),
  produto_id INT REFERENCES produtos(id),
  quantidade INT NOT NULL,             -- depende do pedido + produto
  preco_unit NUMERIC(10,2) NOT NULL,   -- depende do pedido + produto
  PRIMARY KEY (pedido_id, produto_id)
);
Terceira Forma Normal (3FN)

Uma tabela está na 3FN quando está na 2FN e não tem dependências transitivas — nenhum atributo não-chave determina outro atributo não-chave.

A → B → C é uma dependência transitiva: C depende de A indiretamente, via B. Isso cria redundância.

pedido_id (PK)cliente_idcliente_nomecliente_cidadecep_cliente
110Ana LimaSão Paulo01310-100
210Ana LimaSão Paulo01310-100
311Bruno CostaRio20040-020
cliente_nome, cliente_cidade e cep_cliente dependem de cliente_id — que não é a PK. Isso é dependência transitiva: viola a 3FN. Dados do cliente repetidos em todo pedido.
SQL — Corrigindo para 3FN
-- Dados do cliente ficam na tabela de clientes
CREATE TABLE clientes (
  id     SERIAL PRIMARY KEY,
  nome   VARCHAR(100) NOT NULL,
  cidade VARCHAR(80),
  cep    VARCHAR(10)
);

-- Pedido referencia o cliente pela FK — sem repetição
CREATE TABLE pedidos (
  id         SERIAL PRIMARY KEY,
  cliente_id INT NOT NULL REFERENCES clientes(id),
  status     VARCHAR(20) DEFAULT 'pendente',
  total      NUMERIC(10,2),
  criado_em  TIMESTAMP DEFAULT NOW()
  -- sem cliente_nome, cliente_cidade — estão na tabela clientes
);
Forma Normal de Boyce-Codd (FNBC)

A FNBC é uma versão mais estrita da 3FN: todo determinante deve ser uma superchave. Na prática, a diferença entre 3FN e FNBC aparece em tabelas com múltiplas chaves candidatas que se sobrepõem.

Regra resumida: se X → Y (X determina Y), então X deve ser superchave da tabela. Ou seja: só a(s) chave(s) podem determinar outros atributos.
Forma NormalO que eliminaPré-requisito
1FNAtributos não-atômicos e grupos repetidos—
2FNDependências parciais da PK compostaEstar na 1FN
3FNDependências transitivas entre não-chavesEstar na 2FN
FNBCDeterminantes que não são superchaveEstar na 3FN
Passo a passo para normalizar
1
Identifique a chave primária
Qual combinação de atributos identifica unicamente cada linha?
2
Verifique a 1FN — atributos atômicos
Há células com múltiplos valores? Listas separadas por vírgula? Grupos repetidos? → Decompor.
3
Verifique a 2FN — dependências parciais (só se PK for composta)
Algum atributo não-chave depende de apenas parte da PK? → Mover para tabela própria com a parte da PK que o determina.
4
Verifique a 3FN — dependências transitivas
Algum atributo não-chave determina outro atributo não-chave? → Mover o determinado para tabela própria com o determinante como PK.
5
Crie as FKs entre as tabelas
Cada tabela nova é referenciada pela anterior via chave estrangeira — mantendo o relacionamento sem repetir dados.

Desnormalização Controlada + Prática
Quando desnormalizar é correto

Normalização até 3FN é o padrão para sistemas transacionais (OLTP). Em sistemas analíticos (OLAP), Data Warehouses e pipelines de ML, desnormalizar pode ser a decisão certa.

OLTP — normalizado

  • Muitos INSERTs, UPDATEs, DELETEs
  • Integridade crítica — bancos de e-commerce, ERP
  • Poucos dados lidos por vez
  • JOINs no momento da consulta são aceitáveis

OLAP / ML — pode desnormalizar

  • Leituras massivas, poucos writes
  • Relatórios, dashboards, feature stores
  • JOINs pesados prejudicam performance
  • Redundância aceitável se documentada
Regra de ouro: normalize primeiro, desnormalize depois — com justificativa técnica documentada. Nunca desnormalize por preguiça de modelar corretamente.
SQL — Exemplo de tabela desnormalizada para analytics
-- Tabela "wide" para feature engineering em ML
-- Viola 3FN de propósito — justificativa: leitura em batch por modelo
CREATE TABLE features_cliente_v1 (
  cliente_id        INT PRIMARY KEY,
  cliente_nome      VARCHAR(100),     -- desnorm: redundante com tabela clientes
  total_pedidos     INT,
  ticket_medio      NUMERIC(10,2),
  total_gasto       NUMERIC(10,2),
  dias_desde_ultimo INT,
  categorias_compradas TEXT,         -- viola 1FN de propósito — array para ML
  atualizado_em     TIMESTAMP DEFAULT NOW()
);

-- Popular via pipeline (Airflow/Spark) — não por trigger

Nos exercícios 1–3, analise o esquema proposto e identifique as violações. Nos exercícios 4–5, escreva o SQL corrigido.
Exercício 01

A tabela abaixo viola a 1FN? Justifique e proponha a correção.

Esquema
funcionario(id, nome, telefones, habilidades)
-- Exemplo de dado:
-- id=1, nome='Ana', telefones='(85)9999-1111,(85)8888-2222', habilidades='Python,SQL,Docker'
Básico — 1FN
Violação: telefones e habilidades são multivalorados — armazenam múltiplos valores numa célula. Viola a 1FN (atributos devem ser atômicos).
SQL — Corrigido (1FN)
CREATE TABLE funcionarios (
  id   SERIAL PRIMARY KEY,
  nome VARCHAR(100) NOT NULL
);
CREATE TABLE funcionario_telefones (
  funcionario_id INT REFERENCES funcionarios(id),
  telefone       VARCHAR(20) NOT NULL,
  PRIMARY KEY (funcionario_id, telefone)
);
CREATE TABLE funcionario_habilidades (
  funcionario_id INT REFERENCES funcionarios(id),
  habilidade     VARCHAR(60) NOT NULL,
  PRIMARY KEY (funcionario_id, habilidade)
);
Exercício 02

A tabela abaixo tem PK composta (turma_id, aluno_id). Ela está na 2FN? Identifique as dependências parciais.

Esquema
matricula(turma_id, aluno_id, nota_final, nome_aluno, email_aluno, nome_turma, professor)
Básico — 2FN
Dependências parciais identificadas:
• nome_aluno, email_aluno → dependem só de aluno_id
• nome_turma, professor → dependem só de turma_id
• nota_final → depende da PK inteira ✅
SQL — Corrigido (2FN)
CREATE TABLE alunos (
  id    SERIAL PRIMARY KEY,
  nome  VARCHAR(100),
  email VARCHAR(150)
);
CREATE TABLE turmas (
  id        SERIAL PRIMARY KEY,
  nome      VARCHAR(100),
  professor VARCHAR(100)
);
CREATE TABLE matriculas (
  turma_id   INT REFERENCES turmas(id),
  aluno_id   INT REFERENCES alunos(id),
  nota_final NUMERIC(4,1),
  PRIMARY KEY (turma_id, aluno_id)
);
Exercício 03

A tabela abaixo está na 3FN? Identifique a dependência transitiva.

Esquema
produto(id, nome, categoria_id, categoria_nome, categoria_descricao)
Básico — 3FN
Dependência transitiva: categoria_nome e categoria_descricao dependem de categoria_id — que não é a PK. Ou seja: id → categoria_id → categoria_nome. Viola a 3FN.
SQL — Corrigido (3FN)
CREATE TABLE categorias (
  id        SERIAL PRIMARY KEY,
  nome      VARCHAR(60) NOT NULL,
  descricao TEXT
);
CREATE TABLE produtos (
  id           SERIAL PRIMARY KEY,
  nome         VARCHAR(120) NOT NULL,
  categoria_id INT REFERENCES categorias(id)
  -- sem categoria_nome e categoria_descricao aqui
);
Exercício 04

A tabela vendas_loja abaixo viola múltiplas formas normais. Normalize até a 3FN e escreva o SQL completo das tabelas resultantes.

Tabela inicial
vendas_loja(venda_id, data_venda, cliente_cpf, cliente_nome,
            cliente_cidade, vendedor_id, vendedor_nome, vendedor_regiao,
            produtos_vendidos, total)
Médio — 1FN + 2FN + 3FN
Violações identificadas:
• produtos_vendidos → não atômico (1FN)
• cliente_nome, cliente_cidade → transitivo via cliente_cpf (3FN)
• vendedor_nome, vendedor_regiao → transitivo via vendedor_id (3FN)
SQL — Normalizado até 3FN
CREATE TABLE clientes (
  cpf    CHAR(11) PRIMARY KEY,
  nome   VARCHAR(100) NOT NULL,
  cidade VARCHAR(80)
);
CREATE TABLE vendedores (
  id     SERIAL PRIMARY KEY,
  nome   VARCHAR(100) NOT NULL,
  regiao VARCHAR(60)
);
CREATE TABLE produtos (
  id   SERIAL PRIMARY KEY,
  nome VARCHAR(120) NOT NULL
);
CREATE TABLE vendas (
  id           SERIAL PRIMARY KEY,
  data_venda   DATE NOT NULL,
  cliente_cpf  CHAR(11) REFERENCES clientes(cpf),
  vendedor_id  INT REFERENCES vendedores(id),
  total        NUMERIC(10,2)
);
CREATE TABLE itens_venda (
  venda_id   INT REFERENCES vendas(id),
  produto_id INT REFERENCES produtos(id),
  quantidade INT NOT NULL,
  preco_unit NUMERIC(10,2) NOT NULL,
  PRIMARY KEY (venda_id, produto_id)
);
Exercício 05 — Desafio

Dado o esquema normalizado do banco loja_exemplo (Semana 1), escreva uma query SQL que reconstituí a visão desnormalizada de uma venda — mostrando numa única linha: nome do cliente, cidade, nome do produto, categoria, quantidade, preço unitário e subtotal. Use CTEs para organizar as etapas.

Desafio — JOINs + CTE + Desnormalização via Query
SQL
WITH

itens_detalhados AS (
  SELECT
    p.id           AS pedido_id,
    p.cliente_id,
    p.criado_em,
    pr.nome        AS produto,
    pr.categoria,
    i.quantidade,
    i.preco_unit,
    (i.quantidade * i.preco_unit) AS subtotal
  FROM pedidos p
  INNER JOIN itens_pedido i ON i.pedido_id  = p.id
  INNER JOIN produtos pr    ON pr.id       = i.produto_id
)

SELECT
  c.nome          AS cliente,
  c.cidade,
  id.pedido_id,
  id.criado_em::date AS data_pedido,
  id.produto,
  id.categoria,
  id.quantidade,
  id.preco_unit,
  id.subtotal
FROM itens_detalhados id
INNER JOIN clientes c ON c.id = id.cliente_id
ORDER BY id.pedido_id, id.produto;
