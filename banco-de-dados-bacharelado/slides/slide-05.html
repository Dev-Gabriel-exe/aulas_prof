Por que NoSQL? Os quatro tipos
O que o relacional não resolve bem

Bancos relacionais são excelentes para dados estruturados com relacionamentos claros. Mas existem cenários onde eles criam fricção:

  • Esquema variável: cada usuário de uma rede social tem campos diferentes — forçar um schema fixo gera colunas nulas em excesso
  • Escala horizontal: um banco relacional escala verticalmente (servidor maior); NoSQL foi projetado para distribuir dados em múltiplos nós
  • Alto volume de writes: 100k inserções/segundo em logs de IoT — o overhead transacional do relacional é impeditivo
  • Dados não-tabulares: documentos JSON, grafos de relacionamentos, séries temporais — forçar em tabelas é trabalhoso e ineficiente
NoSQL não significa "sem SQL" — significa "Not Only SQL". São bancos projetados para casos específicos onde o modelo relacional não é o mais adequado.
Os quatro tipos de banco NoSQL
Documento
MongoDB, Firestore, CouchDB

Armazena documentos JSON/BSON. Esquema flexível — cada documento pode ter campos diferentes. Ideal para perfis, catálogos e conteúdo.

Chave-Valor
Redis, DynamoDB, Memcached

Acesso ultra-rápido por chave única. Estrutura mais simples. Ideal para cache, sessões, filas e feature stores em memória.

Colunar
Cassandra, HBase, ScyllaDB

Armazena dados por coluna em vez de linha. Altamente eficiente para séries temporais, IoT e analytics em escala massiva.

Grafos
Neo4j, Amazon Neptune, ArangoDB

Nós e arestas como estrutura nativa. Ideal para redes sociais, sistemas de recomendação, grafos de conhecimento e RAG.

Teorema CAP — o trade-off fundamental

Todo sistema distribuído precisa escolher dois dos três:

PropriedadeSignificadoQuem prioriza
ConsistencyTodos os nós veem os mesmos dados ao mesmo tempoHBase, Zookeeper
AvailabilityO sistema sempre responde, mesmo com falhasCassandra, CouchDB
Partition toleranceFunciona mesmo com falha de comunicação entre nósTodos os distribuídos
Na prática: Partition tolerance é obrigatório em sistemas distribuídos. A escolha real é entre Consistência e Disponibilidade. MongoDB escolhe CP; Cassandra escolhe AP.

MongoDB — FIND (SELECT)
// find() equivale ao SELECT * FROM + WHERE
db.usuarios.find({ cidade: "Fortaleza" })

// Projeção: só retornar campos específicos (1=incluir, 0=excluir)
db.usuarios.find(
  { cidade: "Fortaleza" },
  { nome: 1, email: 1, _id: 0 }
)

// Operadores de comparação
db.produtos.find({ preco: { $gt: 100, $lt: 2000 } })  // $gt, $lt, $gte, $lte
db.produtos.find({ categoria: { $in: ["Eletrônicos", "Móveis"] } })

// Busca em array: todos que têm "Python" no array interesses
db.usuarios.find({ interesses: "Python" })

// Ordenar, limitar e pular
db.produtos.find().sort({ preco: -1 }).limit(5).skip(0)
// -1 = descendente, 1 = ascendente
MongoDB — UPDATE e DELETE
// updateOne: atualiza o primeiro documento que bater
db.produtos.updateOne(
  { nome: "Mouse Logitech" },
  { $set: { preco: 99.90, atualizado_em: new Date() } }
)

// updateMany: atualiza todos que baterem
db.produtos.updateMany(
  { categoria: "Eletrônicos" },
  { $inc: { estoque: -1 } }   // $inc decrementa o estoque de todos
)

// deleteOne / deleteMany
db.usuarios.deleteOne({ email: "ana@email.com" })
db.pedidos.deleteMany({ status: "cancelado", criado_em: { $lt: new Date("2024-01-01") } })
Aggregation Pipeline — o GROUP BY do MongoDB

O pipeline de agregação processa documentos em estágios sequenciais. Cada estágio transforma os documentos e passa para o próximo.

MongoDB — Aggregate
// Total gasto por cidade — equivalente ao GROUP BY + SUM do SQL
db.pedidos.aggregate([
  // Estágio 1: filtrar pedidos entregues
  { $match: { status: "entregue" } },

  // Estágio 2: join com coleção de clientes (equivalente ao LEFT JOIN)
  { $lookup: {
    from: "clientes",
    localField: "cliente_id",
    foreignField: "_id",
    as: "cliente"
  }},

  // Estágio 3: desempilhar o array resultado do lookup
  { $unwind: "$cliente" },

  // Estágio 4: agrupar por cidade e calcular métricas
  { $group: {
    _id: "$cliente.cidade",
    total_pedidos: { $sum: 1 },
    receita_total: { $sum: "$total" },
    ticket_medio:  { $avg: "$total" }
  }},

  // Estágio 5: ordenar pela receita total
  { $sort: { receita_total: -1 } },

  // Estágio 6: limitar ao top 10
  { $limit: 10 }
])
Os estágios mais comuns: $match (filtrar), $group (agregar), $sort (ordenar), $project (selecionar campos), $lookup (join), $unwind (desempilhar array), $limit e $skip.

Redis — Cache e Estruturas de Dados em Memória
O que é Redis e por que ele é tão rápido

Redis é um banco de dados in-memory — todos os dados ficam na RAM. Isso o torna 10–100x mais rápido que um banco em disco para leituras e escritas simples.

Casos de uso principais

  • Cache de resultados de queries caras
  • Sessões de usuários autenticados
  • Feature store — features pré-calculadas em memória
  • Rate limiting de APIs
  • Cache de respostas de LLMs (semantic cache)

Estruturas de dados suportadas

  • String — valor simples, counter, flag
  • List — fila FIFO/LIFO
  • Hash — mapa chave-valor (como um objeto)
  • Sorted Set — ranking com score
  • Set — conjunto sem duplicatas
Redis — Comandos essenciais
Redis CLI — Strings e TTL
# SET / GET — chave-valor básico
SET usuario:1001:nome "Ana Lima"
GET usuario:1001:nome          # → "Ana Lima"

# TTL (Time To Live) — expira automaticamente em segundos
SET cache:query:clientes_sp "[{...}]" EX 300   # expira em 5 minutos
TTL cache:query:clientes_sp                    # → 298 (segundos restantes)

# Incremento atômico — útil para contadores e rate limiting
INCR contador:api:usuario:1001
INCRBY contador:api:usuario:1001 10
EXPIRE contador:api:usuario:1001 60   # reset a cada 60s
Redis CLI — Hash (perfil de usuário)
# HSET armazena múltiplos campos de uma vez (como um objeto)
HSET usuario:1001 nome "Ana Lima" cidade "Fortaleza" score 750

# HGET — buscar campo específico
HGET usuario:1001 nome         # → "Ana Lima"

# HGETALL — buscar todos os campos
HGETALL usuario:1001
# → { nome: "Ana Lima", cidade: "Fortaleza", score: "750" }

# HINCRBY — incrementar campo numérico
HINCRBY usuario:1001 score 50   # score agora = 800
Redis CLI — Sorted Set (ranking)
# ZADD — adicionar ao ranking com score
ZADD ranking:produtos 152 "Notebook Dell"
ZADD ranking:produtos 89  "Mouse Logitech"
ZADD ranking:produtos 43  "Cadeira Gamer"

# ZREVRANGE — top N em ordem decrescente de score
ZREVRANGE ranking:produtos 0 2 WITHSCORES
# → Notebook Dell 152 / Mouse Logitech 89 / Cadeira Gamer 43

# ZINCRBY — incrementar score (ex: nova venda)
ZINCRBY ranking:produtos 1 "Mouse Logitech"
Redis com Python — redis-py
Python — Padrão Cache-Aside com Redis
import redis, json, os
from sqlalchemy import create_engine
import pandas as pd

# Conectar ao Redis e ao PostgreSQL
cache = redis.Redis(host="localhost", port=6379, decode_responses=True)
engine = create_engine(os.getenv("DATABASE_URL"))

def get_features_cliente(cliente_id: int) -> dict:
    # 1. Tentar buscar do cache primeiro
    cache_key = f"features:cliente:{cliente_id}"
    cached = cache.get(cache_key)

    if cached:
        print("✅ Cache hit")
        return json.loads(cached)

    # 2. Cache miss — buscar do banco
    print("⚠️  Cache miss — buscando do banco")
    df = pd.read_sql(f"""
        SELECT * FROM gold_features_cliente
        WHERE cliente_id = {cliente_id}
    """, engine)

    if df.empty:
        return {}

    features = df.iloc[0].to_dict()

    # 3. Armazenar no cache com TTL de 10 minutos
    cache.setex(cache_key, 600, json.dumps(features, default=str))

    return features

# Uso
f = get_features_cliente(1)
print(f["ticket_medio"], f["dias_inativo"])
Padrão Cache-Aside: leia do cache primeiro → se não existir (miss), busque da fonte e grave no cache com TTL. Reduz carga no banco e latência para o modelo.

Exercício 06 — Desafio

Implemente em Python o padrão Cache-Aside para a função get_produto(produto_id): (1) tente buscar o produto do Redis (Hash); (2) se não existir, busque do MongoDB e armazene no Redis com TTL de 5 minutos usando HSET + EXPIRE; (3) retorne o produto. Teste chamando a função duas vezes para ver o cache hit.

Desafio — Cache-Aside Redis + MongoDB
Python
import redis, os
from pymongo import MongoClient
from bson import ObjectId

cache = redis.Redis(host="localhost", port=6379, decode_responses=True)
mongo = MongoClient(os.getenv("MONGODB_URI", "mongodb://localhost:27017"))
db = mongo["loja"]

def get_produto(nome_produto: str) -> dict:
    cache_key = f"produto:{nome_produto.lower().replace(' ', '_')}"

    # 1. Tentar do Redis
    cached = cache.hgetall(cache_key)
    if cached:
        print(f"✅ Cache HIT para '{nome_produto}'")
        cached["preco"] = float(cached["preco"])
        cached["estoque"] = int(cached["estoque"])
        return cached

    # 2. Cache miss — buscar do MongoDB
    print(f"⚠️  Cache MISS — buscando '{nome_produto}' do MongoDB")
    produto = db.produtos.find_one(
        { "nome": nome_produto },
        { "_id": 0 }
    )
    if not produto:
        return {}

    # 3. Armazenar no Redis como Hash
    cache.hset(cache_key, mapping={
        "nome":      produto["nome"],
        "categoria": produto["categoria"],
        "preco":     str(produto["preco"]),
        "estoque":   str(produto["estoque"])
    })
    cache.expire(cache_key, 300)  # TTL de 5 minutos

    return produto

# Teste: 1ª chamada = miss, 2ª = hit
print(get_produto("Notebook Dell"))
print(get_produto("Notebook Dell"))
