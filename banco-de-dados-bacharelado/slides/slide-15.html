ACID — as quatro propriedades que tornam bancos confiáveis
As quatro propriedades ACID

Uma transação é uma sequência de operações tratada como uma unidade indivisível. O banco garante que ela respeita as quatro propriedades ACID — mesmo em caso de falha de hardware, queda de energia ou múltiplos usuários simultâneos.

A
Atomicidade

Tudo ou nada. Se qualquer passo falhar, o banco desfaz todas as operações da transação.

C
Consistência

A transação leva o banco de um estado válido a outro estado válido — nunca viola constraints ou regras de negócio.

I
Isolamento

Transações concorrentes não se interferem. Cada uma enxerga o banco como se fosse a única executando.

D
Durabilidade

Uma vez confirmada (COMMIT), a transação persiste mesmo que o servidor caia logo depois.

SQL — estrutura básica de transação
-- Transferência bancária: deve ser atômica
BEGIN;  -- inicia a transação

UPDATE contas SET saldo = saldo - 500 WHERE id = 1;  -- debitar Ana
UPDATE contas SET saldo = saldo + 500 WHERE id = 2;  -- creditar Bruno

-- Se ambos funcionaram:
COMMIT;

-- Se algo falhou:
ROLLBACK;

-- SAVEPOINT: ponto de retorno parcial dentro da transação
BEGIN;
INSERT INTO log VALUES ('inicio');
SAVEPOINT antes_debit;
UPDATE contas SET saldo = saldo - 500 WHERE id = 1;
-- Se o débito falhou, volta só até o savepoint:
ROLLBACK TO antes_debit;
-- o INSERT no log ainda está ativo!
COMMIT;
Anomalias de concorrência — o que pode dar errado

Sem controle adequado de isolamento, transações concorrentes podem produzir resultados incorretos. Existem três anomalias clássicas:

AnomaliaO que aconteceExemplo
Dirty Read
leitura suja		 Tx B lê dados que Tx A escreveu mas ainda não confirmou. Se A fizer ROLLBACK, B leu um dado que nunca existiu. Ana transfere R$500 mas ainda não deu COMMIT. Bruno lê o saldo atualizado e toma decisão baseada num dado que pode ser desfeito.
Non-Repeatable Read
leitura não repetível		 Tx A lê uma linha, Tx B atualiza e dá COMMIT, Tx A lê a mesma linha de novo e obtém valor diferente. Relatório lê saldo = 1000, outro processo paga uma conta (saldo → 500 + COMMIT), relatório lê saldo de novo = 500. Mesmo relatório, valores diferentes.
Phantom Read
leitura fantasma		 Tx A executa uma query com filtro, Tx B insere novas linhas que satisfazem o filtro e dá COMMIT, Tx A repete a query e encontra linhas novas ("fantasmas"). Auditoria conta pedidos do dia = 150. Novo pedido é inserido + COMMIT. Auditoria recontará = 151. Mesma query, resultados diferentes.

Dirty Read — linha do tempo:

TempoTransação ATransação B
t1 BEGIN
t2 UPDATE saldo → 500
t3 (ainda não fez COMMIT) SELECT saldo → lê 500 ⚠️
t4 ROLLBACK — saldo volta a 1000 B usou dado inválido!

Níveis de isolamento, MVCC e deadlock
Os 4 níveis de isolamento do SQL padrão

O SQL padrão define quatro níveis de isolamento que controlam quais anomalias são permitidas. Mais isolamento = mais segurança, mas menos concorrência (mais contenção de locks).

Nível 1
Read Uncommitted

Permite ler dados não confirmados de outras transações.

Permite: dirty read, non-repeatable, phantom
Nível 2 · padrão PG
Read Committed

Só lê dados já confirmados. Padrão do PostgreSQL.

Previne: dirty read
Permite: non-repeatable, phantom
Nível 3
Repeatable Read

Garante que releituras na mesma transação retornam os mesmos dados.

Previne: dirty read, non-repeatable
Permite: phantom (em alguns SGBDs; PG previne)
Nível 4 · máximo
Serializable

Transações executam como se fossem em série, uma de cada vez.

Previne: todas as anomalias
Custo: maior contenção
SQL — configurar nível de isolamento
-- Definir isolamento para a próxima transação
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT saldo FROM contas WHERE id = 1;
-- Mesmo que outra transação altere e dê COMMIT,
-- esta transação continuará vendo o valor original
COMMIT;

-- Serializable — máxima segurança (relatórios de auditoria, transações financeiras)
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
SELECT count(*) FROM pedidos WHERE status = 'pendente';
-- Nenhum novo pedido pendente pode ser inserido e confirmado
-- enquanto esta transação estiver aberta
COMMIT;

-- Verificar nível de isolamento atual
SHOW transaction_isolation;

-- Alterar padrão da sessão (raro — use com cuidado)
SET SESSION CHARACTERISTICS AS TRANSACTION ISOLATION LEVEL SERIALIZABLE;

MVCC — Multi-Version Concurrency Control

O PostgreSQL não usa locks para leituras — usa MVCC. Cada linha tem versões com timestamps de validade. Quando uma transação lê, enxerga um snapshot do banco no momento em que a transação começou. Escritas criam novas versões sem sobrescrever as antigas — leitores e escritores não se bloqueiam mutuamente.

Como o MVCC funciona:

  • Cada linha armazena xmin (txid que criou) e xmax (txid que deletou)
  • UPDATE = INSERT de nova versão + marcar xmax na antiga
  • Cada transação enxerga apenas versões cuja xmin foi confirmada antes do início
  • Leituras nunca bloqueiam escritas — e vice-versa

Consequências do MVCC:

  • Alta concorrência — reads e writes coexistem sem espera
  • Time travel natural — snapshot consistente da transação
  • Dead tuples — versões antigas acumulam em disco
  • VACUUM limpa dead tuples periodicamente (autovacuum é automático)
SQL — inspecionar versões MVCC e dead tuples
-- Ver xmin e xmax das linhas (versões MVCC)
SELECT xmin, xmax, id, saldo FROM contas;

-- Ver dead tuples acumuladas (candidatas ao VACUUM)
SELECT
    relname,
    n_live_tup   AS linhas_vivas,
    n_dead_tup   AS linhas_mortas,
    last_vacuum,
    last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;

-- Forçar limpeza manual
VACUUM contas;
VACUUM ANALYZE contas;  -- também atualiza estatísticas
Deadlock — quando duas transações se bloqueiam mutuamente

Um deadlock ocorre quando a Transação A aguarda um lock que a Transação B possui, e B aguarda um lock que A possui. Nenhuma pode avançar — impasse eterno. O PostgreSQL detecta automaticamente e cancela uma das transações (a "vítima") para desbloquear a outra.

Transação A
🔒 tem lock em conta_id=1
⏳ aguardando conta_id=2
⇄
Transação B
🔒 tem lock em conta_id=2
⏳ aguardando conta_id=1
SQL — sequência que provoca deadlock
-- Sessão A (executa na ordem: conta 1 → conta 2)
BEGIN;
UPDATE contas SET saldo = saldo - 100 WHERE id = 1;  -- adquire lock em id=1
-- aguarda Sessão B liberar lock em id=2...
UPDATE contas SET saldo = saldo + 100 WHERE id = 2;
COMMIT;

-- Sessão B (executa na ordem INVERSA: conta 2 → conta 1)
BEGIN;
UPDATE contas SET saldo = saldo - 200 WHERE id = 2;  -- adquire lock em id=2
-- aguarda Sessão A liberar lock em id=1... → DEADLOCK!
UPDATE contas SET saldo = saldo + 200 WHERE id = 1;
COMMIT;

-- O PostgreSQL detectará e cancelará uma das transações com:
-- ERROR: deadlock detected
-- DETAIL: Process 1234 waits for ShareLock on transaction 5678
-- HINT: See server log for query details.

-- SOLUÇÃO: sempre acessar recursos na mesma ordem
-- Sessão A e B devem ambas fazer: conta menor id ANTES do maior
UPDATE contas SET saldo = saldo - 100 WHERE id = LEAST(1,2);
UPDATE contas SET saldo = saldo + 100 WHERE id = GREATEST(1,2);
Regra de ouro para evitar deadlocks: sempre acesse múltiplos recursos (linhas, tabelas) na mesma ordem em todas as transações. Se A sempre bloqueia conta 1 antes de conta 2, e B também, eles nunca formarão um ciclo de espera.

Simulando concorrência com threads Python
Python — simulação de transferência concorrente
import psycopg2, threading, time, os

DB = os.getenv("DATABASE_URL")

def transferir(de_id, para_id, valor, nome_tx, nivel="READ COMMITTED"):
    """Realiza uma transferência dentro de uma transação."""
    conn = psycopg2.connect(DB)
    conn.autocommit = False
    cur = conn.cursor()
    try:
        cur.execute(f"SET TRANSACTION ISOLATION LEVEL {nivel}")
        # Lê saldo atual
        cur.execute("SELECT saldo FROM contas WHERE id = %s", (de_id,))
        saldo_atual = cur.fetchone()[0]
        print(f"[{nome_tx}] saldo de conta {de_id} = R${saldo_atual}")

        if saldo_atual < valor:
            print(f"[{nome_tx}] saldo insuficiente — ROLLBACK")
            conn.rollback()
            return

        time.sleep(0.1)  # simular latência

        cur.execute("UPDATE contas SET saldo = saldo - %s WHERE id = %s", (valor, de_id))
        cur.execute("UPDATE contas SET saldo = saldo + %s WHERE id = %s", (valor, para_id))
        conn.commit()
        print(f"[{nome_tx}] transferência R${valor} de {de_id}→{para_id} — COMMIT ✓")
    except Exception as e:
        conn.rollback()
        print(f"[{nome_tx}] ERRO: {e} — ROLLBACK")
    finally:
        cur.close(); conn.close()

# Duas transferências simultâneas do mesmo saldo (Ana → Bruno e Ana → Carla)
print("=== Simulando transferências concorrentes ===")
t1 = threading.Thread(target=transferir, args=(1, 2, 4000, "Tx-A"))
t2 = threading.Thread(target=transferir, args=(1, 3, 4000, "Tx-B"))
t1.start(); t2.start()
t1.join(); t2.join()

# Verificar saldo final — o CHECK constraint garante que nunca fica negativo
conn = psycopg2.connect(DB)
cur = conn.cursor()
cur.execute("SELECT id, nome, saldo FROM contas")
print("\nSaldos finais:")
for row in cur.fetchall():
    print(f"  conta {row[0]} ({row[1]}): R${row[2]}")
cur.close(); conn.close()
Provocando e capturando deadlock em Python
Python — simulação de deadlock e tratamento
import psycopg2, threading, time, os
from psycopg2 import errors as pg_errors

DB = os.getenv("DATABASE_URL")
resultado = {}

def tx_a():
    """Acessa conta 1 → conta 2"""
    conn = psycopg2.connect(DB); conn.autocommit = False
    cur = conn.cursor()
    try:
        cur.execute("UPDATE contas SET saldo = saldo - 100 WHERE id = 1")
        print("[Tx-A] lock em conta 1 adquirido")
        time.sleep(0.2)  # deixa Tx-B adquirir lock em conta 2
        cur.execute("UPDATE contas SET saldo = saldo + 100 WHERE id = 2")
        conn.commit()
        resultado["A"] = "COMMIT"
    except pg_errors.DeadlockDetected:
        conn.rollback()
        resultado["A"] = "DEADLOCK → ROLLBACK"
        print("[Tx-A] 💀 DEADLOCK detectado — rollback automático")
    finally:
        cur.close(); conn.close()

def tx_b():
    """Acessa conta 2 → conta 1 (ordem inversa = deadlock!)"""
    conn = psycopg2.connect(DB); conn.autocommit = False
    cur = conn.cursor()
    try:
        time.sleep(0.05)  # garante que Tx-A começa primeiro
        cur.execute("UPDATE contas SET saldo = saldo - 200 WHERE id = 2")
        print("[Tx-B] lock em conta 2 adquirido")
        cur.execute("UPDATE contas SET saldo = saldo + 200 WHERE id = 1")
        conn.commit()
        resultado["B"] = "COMMIT"
    except pg_errors.DeadlockDetected:
        conn.rollback()
        resultado["B"] = "DEADLOCK → ROLLBACK"
        print("[Tx-B] 💀 DEADLOCK detectado — rollback automático")
    finally:
        cur.close(); conn.close()

print("=== Simulando DEADLOCK ===")
ta = threading.Thread(target=tx_a)
tb = threading.Thread(target=tx_b)
ta.start(); tb.start()
ta.join(); tb.join()

print(f"\nResultados: Tx-A={resultado.get('A')} | Tx-B={resultado.get('B')}")
print("Uma transação foi cancelada pelo PostgreSQL — a outra completou com sucesso.")

# ── SOLUÇÃO: mesma ordem de acesso ───────────────────────────────
def transferir_seguro(de_id, para_id, valor, nome):
    """Sempre acessa o menor ID primeiro — evita deadlock."""
    primeiro  = min(de_id, para_id)
    segundo   = max(de_id, para_id)
    conn = psycopg2.connect(DB); conn.autocommit = False
    cur = conn.cursor()
    try:
        cur.execute("SELECT id FROM contas WHERE id = ANY(%s) ORDER BY id FOR UPDATE",
                    ([primeiro, segundo],))
        cur.execute("UPDATE contas SET saldo = saldo - %s WHERE id = %s", (valor, de_id))
        cur.execute("UPDATE contas SET saldo = saldo + %s WHERE id = %s", (valor, para_id))
        conn.commit()
        print(f"[{nome}] transferência segura: COMMIT ✓")
    except Exception as e:
        conn.rollback()
        print(f"[{nome}] ERRO: {e}")
    finally:
        cur.close(); conn.close()
SELECT FOR UPDATE adquire um lock exclusivo nas linhas selecionadas, garantindo que nenhuma outra transação as modifique enquanto a sua ainda está aberta. Use com ORDER BY id para garantir ordem consistente de aquisição de locks.
Inspecionar locks ativos no PostgreSQL
SQL — ver locks e transações em espera
-- Ver todos os locks ativos
SELECT
    pid,
    locktype,
    relation::regclass AS tabela,
    mode,
    granted,
    wait_event_type,
    wait_event
FROM pg_locks l
LEFT JOIN pg_stat_activity a USING (pid)
WHERE relation IS NOT NULL
ORDER BY granted, pid;

-- Transações bloqueadas esperando lock
SELECT
    pid,
    now() - pg_stat_activity.query_start AS duracao,
    query,
    state
FROM pg_stat_activity
WHERE (now() - pg_stat_activity.query_start) > INTERVAL '5 seconds'
  AND state = 'active'
ORDER BY duracao DESC;

-- Cancelar transação que está bloqueando outra (emergência)
SELECT pg_cancel_backend(1234);  -- cancela a query, mantém a conexão
SELECT pg_terminate_backend(1234); -- encerra a conexão

Exercício 01

Escreva um script Python que realiza uma transferência de R$ 2.000 da conta 1 (Ana) para a conta 2 (Bruno) dentro de uma transação explícita. O script deve: (a) imprimir os saldos antes da transferência; (b) executar o débito e o crédito; (c) verificar se o saldo de Ana ficaria negativo — se sim, fazer ROLLBACK e imprimir mensagem de erro; (d) caso contrário, fazer COMMIT e imprimir os saldos após a transferência.

Básico — BEGIN / COMMIT / ROLLBACK
Python
import psycopg2, os

conn = psycopg2.connect(os.getenv("DATABASE_URL"))
conn.autocommit = False
cur = conn.cursor()

VALOR = 2000
DE_ID, PARA_ID = 1, 2

try:
    # Saldos antes
    cur.execute("SELECT nome, saldo FROM contas WHERE id = ANY(%s)", ([DE_ID, PARA_ID],))
    print("Antes:", {r[0]: r[1] for r in cur.fetchall()})

    # Verificar saldo suficiente
    cur.execute("SELECT saldo FROM contas WHERE id = %s", (DE_ID,))
    saldo = cur.fetchone()[0]
    if saldo < VALOR:
        raise ValueError(f"Saldo insuficiente: R${saldo} < R${VALOR}")

    # Transferência
    cur.execute("UPDATE contas SET saldo = saldo - %s WHERE id = %s", (VALOR, DE_ID))
    cur.execute("UPDATE contas SET saldo = saldo + %s WHERE id = %s", (VALOR, PARA_ID))
    conn.commit()

    # Saldos depois
    cur.execute("SELECT nome, saldo FROM contas WHERE id = ANY(%s)", ([DE_ID, PARA_ID],))
    print("Depois:", {r[0]: r[1] for r in cur.fetchall()})
    print(f"✓ Transferência de R${VALOR} realizada com sucesso")

except ValueError as e:
    conn.rollback()
    print(f"✗ ROLLBACK: {e}")
finally:
    cur.close(); conn.close()

Exercício 03

Demonstre a diferença entre Read Committed e Repeatable Read: use duas threads Python com conexões separadas. A Thread 1 inicia uma transação com Repeatable Read e lê o saldo da conta 1. A Thread 2 atualiza o saldo da conta 1 e dá COMMIT. A Thread 1 relê o saldo. Mostre que com Repeatable Read o valor não muda para a Thread 1, e repita o experimento com Read Committed para mostrar que o valor muda.

Médio — Níveis de isolamento na prática
Python
import psycopg2, threading, time, os

DB = os.getenv("DATABASE_URL")
eventos = []

def leitor(nivel):
    conn = psycopg2.connect(DB); conn.autocommit = False; cur = conn.cursor()
    cur.execute(f"SET TRANSACTION ISOLATION LEVEL {nivel}")
    cur.execute("SELECT saldo FROM contas WHERE id = 1")
    v1 = cur.fetchone()[0]
    eventos.append(f"[{nivel}] Leitura 1: R${v1}")
    time.sleep(0.4)  # escritor faz COMMIT nesse intervalo
    cur.execute("SELECT saldo FROM contas WHERE id = 1")
    v2 = cur.fetchone()[0]
    eventos.append(f"[{nivel}] Leitura 2: R${v2} {'← MUDOU!' if v1!=v2 else '← mesmo valor ✓'}")
    conn.rollback(); cur.close(); conn.close()

def escritor():
    time.sleep(0.15)  # garante que leitura 1 já aconteceu
    conn = psycopg2.connect(DB); conn.autocommit = False; cur = conn.cursor()
    cur.execute("UPDATE contas SET saldo = saldo + 999 WHERE id = 1")
    conn.commit(); eventos.append("[Escritor] UPDATE + COMMIT")
    cur.close(); conn.close()

for nivel in ["READ COMMITTED", "REPEATABLE READ"]:
    # Resetar saldo
    conn = psycopg2.connect(DB)
    conn.autocommit = True
    conn.cursor().execute("UPDATE contas SET saldo = 5000 WHERE id = 1")
    conn.close()

    eventos.clear()
    threads = [threading.Thread(target=leitor, args=(nivel,)),
               threading.Thread(target=escritor)]
    for t in threads: t.start()
    for t in threads: t.join()
    print(f"\n=== {nivel} ===")
    for e in sorted(eventos): print(" ", e)
Exercício 04

Escreva um script Python que provoca um deadlock intencionalmente entre duas threads (Thread A: bloqueia conta 1 → aguarda conta 2; Thread B: bloqueia conta 2 → aguarda conta 1), captura a exceção psycopg2.errors.DeadlockDetected em ambas as threads, imprime qual transação foi a vítima e qual completou. Em seguida, reimplemente as mesmas threads usando a estratégia de ordenação de locks (sempre bloquear o menor ID primeiro) e confirme que o deadlock não ocorre mais.

Médio — Deadlock e prevenção
Python
import psycopg2, threading, time, os
from psycopg2 import errors as pge

DB = os.getenv("DATABASE_URL")
log = {}

def sem_ordenacao(nome, primeiro, segundo, valor):
    conn = psycopg2.connect(DB); conn.autocommit = False; cur = conn.cursor()
    try:
        cur.execute("UPDATE contas SET saldo=saldo-%s WHERE id=%s",(valor,primeiro))
        time.sleep(0.15)
        cur.execute("UPDATE contas SET saldo=saldo+%s WHERE id=%s",(valor,segundo))
        conn.commit(); log[nome] = "COMMIT ✓"
    except pge.DeadlockDetected:
        conn.rollback(); log[nome] = "💀 DEADLOCK vítima → ROLLBACK"
    finally: cur.close(); conn.close()

def com_ordenacao(nome, de_id, para_id, valor):
    """Bloqueia sempre pelo menor ID primeiro"""
    ids = sorted([de_id, para_id])
    conn = psycopg2.connect(DB); conn.autocommit = False; cur = conn.cursor()
    try:
        cur.execute("SELECT id FROM contas WHERE id=ANY(%s) ORDER BY id FOR UPDATE",(ids,))
        cur.execute("UPDATE contas SET saldo=saldo-%s WHERE id=%s",(valor,de_id))
        cur.execute("UPDATE contas SET saldo=saldo+%s WHERE id=%s",(valor,para_id))
        conn.commit(); log[nome] = "COMMIT ✓ (sem deadlock)"
    except pge.DeadlockDetected:
        conn.rollback(); log[nome] = "💀 DEADLOCK (não deveria acontecer!)"
    finally: cur.close(); conn.close()

# Sem ordenação — deve dar deadlock
print("=== SEM ordenação de locks ===")
log.clear()

tb = threading.Thread(target=sem_ordenacao, args=("B",2,1,200))
ta.start(); tb.start(); ta.join(); tb.join()
for k,v in log.items(): print(f"  Tx-{k}: {v}")

Exercício 05 — Desafio

Implemente uma função Python transferir_robusto(de_id, para_id, valor, max_tentativas=3) que: (1) executa a transferência dentro de uma transação com nível Read Committed; (2) usa SELECT ... FOR UPDATE para adquirir locks na ordem correta (menor ID primeiro); (3) em caso de deadlock ou serialization failure, faz ROLLBACK e reexecuta automaticamente (retry com backoff exponencial: 0.1s, 0.2s, 0.4s…); (4) registra em um log cada tentativa, sucesso ou falha definitiva; (5) teste a função com 10 threads simultâneas fazendo transferências aleatórias entre 3 contas e verifique que o saldo total permanece constante (invariante de conservação).

Desafio — Transferência robusta com retry e invariante
Python
import psycopg2, threading, time, random, os
from psycopg2 import errors as pge

DB = os.getenv("DATABASE_URL")
log_lock = threading.Lock()
log_msgs = []

def log(msg):
    with log_lock:
        log_msgs.append(msg)
        print(msg)

def transferir_robusto(de_id, para_id, valor, max_tentativas=3):
    ids = sorted([de_id, para_id])
    for tentativa in range(1, max_tentativas + 1):
        conn = psycopg2.connect(DB)
        conn.autocommit = False
        cur = conn.cursor()
        try:
            # Lock na ordem correta
            cur.execute("SELECT saldo FROM contas WHERE id=ANY(%s) ORDER BY id FOR UPDATE", (ids,))
            rows = {r[0]: r[1] for r in cur.fetchall()}

            if rows[de_id] < valor:
                conn.rollback()
                log(f"[{de_id}→{para_id}] saldo insuficiente — abortado")
                return False

            cur.execute("UPDATE contas SET saldo=saldo-%s WHERE id=%s",(valor,de_id))
            cur.execute("UPDATE contas SET saldo=saldo+%s WHERE id=%s",(valor,para_id))
            conn.commit()
            log(f"[{de_id}→{para_id}] R${valor} — COMMIT (tentativa {tentativa})")
            return True

        except (pge.DeadlockDetected, pge.SerializationFailure) as e:
            conn.rollback()
            espera = 0.1 * (2 ** (tentativa - 1))
            log(f"[{de_id}→{para_id}] {type(e).__name__} — retry {tentativa}/{max_tentativas} em {espera:.1f}s")
            time.sleep(espera)
        except Exception as e:
            conn.rollback()
            log(f"[{de_id}→{para_id}] ERRO: {e}")
            return False
        finally:
            cur.close(); conn.close()

    log(f"[{de_id}→{para_id}] FALHA definitiva após {max_tentativas} tentativas")
    return False

# Verificar saldo total inicial
conn = psycopg2.connect(DB)
cur = conn.cursor()
cur.execute("SELECT SUM(saldo) FROM contas")
total_inicial = cur.fetchone()[0]
cur.close(); conn.close()
print(f"Saldo total inicial: R${total_inicial}")

# 10 threads fazendo transferências aleatórias
threads = []
for _ in range(10):
    ids = random.sample([1,2,3], 2)
    valor = random.randint(100, 500)
    t = threading.Thread(target=transferir_robusto, args=(*ids, valor))
    threads.append(t)

for t in threads: t.start()
for t in threads: t.join()

# Verificar invariante: saldo total não mudou

cur.execute("SELECT SUM(saldo) FROM contas")
total_final = cur.fetchone()[0]
