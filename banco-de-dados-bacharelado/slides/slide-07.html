Embeddings — representando significado como números
O problema que embeddings resolvem

Modelos de ML trabalham com números — mas dados reais são texto, imagem, áudio. A pergunta é: como representar "gato" e "felino" de forma que o modelo saiba que são parecidos?

A resposta é o embedding: uma função que transforma qualquer dado em um vetor de números reais em um espaço de alta dimensão, onde itens semanticamente similares ficam próximos.

Intuição: "gato" e "felino" produzem vetores próximos. "gato" e "banco de dados" produzem vetores distantes. A distância no espaço vetorial representa a diferença semântica.
O que é um vetor de embedding

Um modelo de embedding transforma texto em um vetor de N dimensões (típico: 384, 768, 1536). Cada dimensão captura algum aspecto semântico — nenhum dimensão isolada tem significado claro, mas o conjunto codifica o sentido.

Exemplo: vetor de 8 dimensões para "banco de dados relacional" (simplificado)

# Modelo multilíngue — funciona bem com português
modelo = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
# Gratuito, roda local, 384 dimensões

textos = [
    "Banco de dados relacional usa tabelas e SQL",
    "PostgreSQL é um sistema gerenciador de banco de dados",
    "Python é uma linguagem de programação",
    "Futebol é o esporte mais popular do Brasil",
]

embeddings = modelo.encode(textos)
print(embeddings.shape)  # (4, 384) — 4 textos, 384 dimensões cada

# Ver o vetor do primeiro texto
print(embeddings[0][:8])  # primeiras 8 dimensões
Medidas de similaridade — cosseno e euclidiana

Dado dois vetores, precisamos medir o quão parecidos eles são. As duas métricas mais usadas:

Similaridade Cosseno

Mede o ângulo entre os vetores. Varia de -1 a 1, onde 1 = idênticos, 0 = sem relação, -1 = opostos. Não depende do tamanho do vetor — "banco" e "banco de dados relacional" ficam próximos mesmo tendo comprimentos diferentes.

Distância Euclidiana

Mede a distância geométrica entre os pontos no espaço. Menor = mais similar. Sensível à magnitude dos vetores — geralmente menos usada para texto.

Python — calcular similaridade
from sentence_transformers import SentenceTransformer, util
import numpy as np

modelo = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")

frases = [
    "Banco de dados relacional usa SQL",          # referência
    "PostgreSQL é um SGBD relacional",             # muito similar
    "Python é uma linguagem de programação",       # pouco similar
    "O gato dormiu na cadeira",                    # não relacionado
]

embs = modelo.encode(frases)

# Similaridade cosseno entre a frase 0 e todas as outras
sims = util.cos_sim(embs[0], embs)
for i, (frase, sim) in enumerate(zip(frases, sims[0])):
    print(f"{sim:.3f}  →  {frase}")

# Saída esperada:
# 1.000  →  Banco de dados relacional usa SQL
# 0.812  →  PostgreSQL é um SGBD relacional
# 0.341  →  Python é uma linguagem de programação
# 0.089  →  O gato dormiu na cadeira
pgvector usa três operadores: <-> distância euclidiana · <=> distância cosseno · <#> produto interno negativo. Para texto, use sempre <=> (cosseno).

pgvector — busca vetorial no PostgreSQL
Por que pgvector e não Pinecone / Weaviate?
CritériopgvectorPinecone / Weaviate
InfraestruturaMesmo PostgreSQL que você já temServiço separado para manter
SQL junto✅ JOIN vetorial + SQL na mesma query❌ API separada, sem SQL nativo
EscalaAté ~1M vetores com bom desempenhoEscala massiva (bilhões)
CustoGratuito — extensão open sourcePago acima do free tier
Quando usarMaioria dos projetos reaisEscala de produto em produção
Regra prática: comece com pgvector. Migre para um banco vetorial dedicado somente quando o volume de vetores ou a latência de busca se tornarem um gargalo real.
Instalação e configuração do pgvector
SQL — Ativar extensão e criar tabela vetorial
-- 1. Ativar a extensão (uma vez por banco)
CREATE EXTENSION IF NOT EXISTS vector;

-- 2. Criar tabela com coluna do tipo vector(N)
CREATE TABLE documentos (
  id          SERIAL PRIMARY KEY,
  titulo      TEXT NOT NULL,
  conteudo    TEXT NOT NULL,
  fonte       VARCHAR(200),
  embedding   vector(384),          -- 384 dim do MiniLM
  criado_em   TIMESTAMP DEFAULT NOW()
);

-- 3. Criar índice HNSW para busca aproximada rápida (recomendado)
CREATE INDEX ON documentos
  USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 64);
-- HNSW = Hierarchical Navigable Small World
-- muito mais rápido que busca exata para N > 10k
Neon.tech (sandbox gratuito que usamos) já tem pgvector disponível — é só executar o CREATE EXTENSION. Para instalação local: apt install postgresql-16-pgvector no Ubuntu.
Inserir e buscar vetores com SQL
SQL — Inserir vetor manualmente
-- Inserir um documento com embedding (valores fictícios abreviados)
INSERT INTO documentos (titulo, conteudo, embedding) VALUES (

SQL — Busca por similaridade cosseno
-- Buscar os 5 documentos mais similares a um vetor de query
-- O vetor da query vem do Python (substituir pelos valores reais)
SELECT
  id,
  titulo,
  conteudo,
  1 - (embedding <=> '[0.38, -0.22, 0.71, ...]'::vector) AS similaridade
FROM documentos
ORDER BY embedding <=> '[0.38, -0.22, 0.71, ...]'::vector
LIMIT 5;
-- <=> = distância cosseno (menor = mais similar)
-- 1 - distância = similaridade (maior = mais similar)

-- Combinar busca vetorial com filtro SQL — o poder do pgvector
SELECT
  titulo,
  conteudo,
  1 - (embedding <=> '[...]'::vector) AS similaridade
FROM documentos
WHERE fonte = 'manual_bd'                    -- filtro SQL normal
  AND criado_em > '2025-01-01'               -- outro filtro
ORDER BY embedding <=> '[...]'::vector
LIMIT 3;

Python — popular tabela com embeddings reais
import psycopg2, os
from sentence_transformers import SentenceTransformer

modelo = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
conn = psycopg2.connect(os.getenv("DATABASE_URL"))
cur = conn.cursor()

# Documentos a indexar
documentos = [
    ("1FN — Primeira Forma Normal",
     "Uma tabela está na 1FN quando todos os atributos são atômicos e não há grupos repetidos.",
     "apostila_normalizacao"),
    ("INNER JOIN vs LEFT JOIN",
     "INNER JOIN retorna apenas linhas com correspondência nos dois lados. LEFT JOIN retorna todos da tabela esquerda.",
     "apostila_sql"),
    ("Window Functions",
     "Window functions calculam valores sobre um conjunto de linhas sem colapsar o resultado como o GROUP BY faz.",
     "apostila_sql"),
    ("Redis — Padrão Cache-Aside",
     "No padrão Cache-Aside, a aplicação lê do cache primeiro. Se não encontrar, busca na fonte e armazena no cache com TTL.",
     "apostila_nosql"),
]

# Gerar embeddings em batch (eficiente)
textos = [titulo + " " + conteudo for titulo, conteudo, _ in documentos]
embeddings = modelo.encode(textos, show_progress_bar=True)

# Inserir no banco
for (titulo, conteudo, fonte), emb in zip(documentos, embeddings):
    cur.execute(
        "INSERT INTO documentos (titulo, conteudo, fonte, embedding) VALUES (%s, %s, %s, %s)",
        (titulo, conteudo, fonte, emb.tolist())
    )

conn.commit()
cur.close()
conn.close()
print(f"{len(documentos)} documentos indexados.")

Arquitetura RAG — Retrieval-Augmented Generation
Por que RAG existe — o problema dos LLMs

LLMs como GPT-4 e Llama são poderosos, mas têm limitações críticas para uso empresarial:

  • Conhecimento estático: dados de treino têm data de corte — não sabe de eventos recentes
  • Alucinação: inventa fatos plausíveis mas incorretos com confiança
  • Sem dados privados: não conhece seus documentos internos, manuais, contratos
  • Janela de contexto limitada: não dá para enviar mil páginas de documentos
RAG resolve tudo isso: em vez de tentar colocar toda a base de conhecimento no prompt, o sistema recupera dinamicamente apenas os trechos mais relevantes para aquela pergunta específica.
As duas fases do RAG

Fase 1 — Indexação (feita uma vez, offline)

1
Carregar documentos
PDFs, Markdown, banco de dados, APIs — qualquer fonte de texto
2
Chunking — dividir em pedaços
Documentos grandes são divididos em chunks de ~300–500 tokens com sobreposição
chunk_size=500, overlap=50
3
Gerar embeddings de cada chunk
Modelo de embedding transforma cada chunk em vetor de 384 ou 1536 dimensões
modelo.encode(chunks)
4
Armazenar no banco vetorial
pgvector, Pinecone, Weaviate — junto com metadados (título, fonte, página)
INSERT INTO documentos (conteudo, embedding, ...)

Fase 2 — Retrieval + Generation (em tempo real, por pergunta)

5
Usuário faz uma pergunta
"O que é a 2FN e como ela difere da 3FN?"
6
Gerar embedding da pergunta
Mesma função de embedding usada na indexação
query_emb = modelo.encode(pergunta)
7
Busca vetorial — recuperar top-K chunks
Encontrar os N chunks mais similares à pergunta por distância cosseno
ORDER BY embedding <=> query_emb LIMIT 5
8
Montar o prompt com contexto
Pergunta + chunks recuperados → enviados ao LLM
"Contexto: {chunks}\n\nPergunta: {query}"
9
LLM gera a resposta final
Resposta fundamentada nos documentos recuperados — sem alucinação
Pipeline RAG completo em Python + pgvector
Python — RAG com pgvector (sem API externa)
import psycopg2, os
from sentence_transformers import SentenceTransformer

modelo = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
conn = psycopg2.connect(os.getenv("DATABASE_URL"))

def buscar_contexto(pergunta: str, top_k: int = 3) -> list[dict]:
    """Fase de Retrieval: busca os chunks mais similares à pergunta."""
    # 1. Embed da pergunta
    query_emb = modelo.encode(pergunta).tolist()

    # 2. Busca vetorial com filtro opcional
    cur = conn.cursor()
    cur.execute("""
        SELECT
          titulo,
          conteudo,
          fonte,
          1 - (embedding <=> %s::vector) AS similaridade
        FROM documentos
        ORDER BY embedding <=> %s::vector
        LIMIT %s
    """, (query_emb, query_emb, top_k))

    resultados = []
    for row in cur.fetchall():
        resultados.append({
            "titulo":      row[0],
            "conteudo":    row[1],
            "fonte":       row[2],
            "similaridade": round(row[3], 3),
        })
    cur.close()
    return resultados


def responder(pergunta: str) -> str:
    """Pipeline RAG completo — sem API de LLM, só retrieval."""
    contextos = buscar_contexto(pergunta, top_k=3)

    if not contextos:
        return "Nenhum documento relevante encontrado."

    # Montar contexto para o prompt
    contexto_str = "\n\n---\n\n".join(
        f"[{c['titulo']}] {c['conteudo']}" for c in contextos
    )

    print(f"\n📚 Contexto recuperado (top {len(contextos)}):")
    for c in contextos:
        print(f  "  [{c['similaridade']:.3f}] {c['titulo']}")

    # Sem LLM: retorna o contexto diretamente
    # Com LLM (ex: OpenAI), enviaria:
    # prompt = f"Contexto:\n{contexto_str}\n\nPergunta: {pergunta}"
    # response = openai.chat.completions.create(...)
    return contexto_str


# Teste
resposta = responder("Como funciona a normalização em banco de dados?")
print("\n💬 Resposta:\n", resposta)
Para adicionar a etapa de geração com um LLM real, substitua o return final por uma chamada à API da OpenAI, Anthropic ou um modelo local via Ollama. O retrieval já está completo.

from sentence_transformers import SentenceTransformer, util

modelo = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")

frases = {
    "A": "Normalização remove redundância em banco de dados",
    "B": "A terceira forma normal elimina dependências transitivas",
    "C": "Window functions calculam ranking sem GROUP BY",
    "D": "Receita de bolo de chocolate com cobertura",
}

embs = modelo.encode(list(frases.values()))
ref = embs[0]  # frase A como referência

resultados = []
for (label, frase), emb in zip(frases.items(), embs):
    sim = util.cos_sim(ref, emb).item()
    resultados.append((sim, label, frase))

resultados.sort(reverse=True)
for sim, label, frase in resultados:
    print(f"{sim:.3f}  [{label}] {frase}")
# Esperado: A=1.0, B > C >> D

Exercício 04

Escreva uma query SQL que combine busca vetorial com filtro: busque os 3 documentos mais similares a "normalização de banco de dados", mas apenas entre os documentos cuja coluna fonte seja 'semanas_1_6' e que foram criados nos últimos 7 dias.

Médio — Busca vetorial + filtro SQL
SQL
-- Substitua [VETOR] pelo embedding real gerado em Python
SELECT
  titulo,
  conteudo,
  1 - (embedding <=> '[VETOR]'::vector) AS similaridade
FROM documentos
WHERE fonte = 'semanas_1_6'
  AND criado_em > NOW() - INTERVAL '7 days'
ORDER BY embedding <=> '[VETOR]'::vector
LIMIT 3;
Ponto importante: o filtro WHERE é processado antes da ordenação vetorial — isso é muito mais eficiente do que buscar tudo e filtrar depois. É a principal vantagem do pgvector sobre bancos vetoriais externos.

Desafio — Pipeline RAG completo
Python
import psycopg2, os
from sentence_transformers import SentenceTransformer

modelo = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
conn = psycopg2.connect(os.getenv("DATABASE_URL"))

def retrieval(pergunta: str, top_k: int = 3) -> list[dict]:
    q_emb = modelo.encode(pergunta).tolist()
    cur = conn.cursor()
    cur.execute("""
        SELECT titulo, conteudo,
               1-(embedding<=>%s::vector) AS sim
        FROM documentos
        ORDER BY embedding<=>%s::vector
        LIMIT %s
    """, (q_emb, q_emb, top_k))
    rows = cur.fetchall()
    cur.close()
    return [{"titulo": r[0], "conteudo": r[1], "sim": round(r[2], 3)} for r in rows]

def montar_prompt(pergunta: str) -> str:
    chunks = retrieval(pergunta)
    print(f"\n🔍 Chunks recuperados para: '{pergunta}'")
    for c in chunks:
        print(f"  [{c['sim']}] {c['titulo']}")

    contexto = "\n\n".join(
        f"Título: {c['titulo']}\n{c['conteudo']}" for c in chunks
    )
    return (
        f"Contexto:\n{contexto}\n\n"
        f"Pergunta: {pergunta}\n\n"
        f"Responda com base no contexto acima."
    )

# Testar com 3 perguntas
perguntas = [
    "Como funciona o LEFT JOIN?",
    "O que é a Segunda Forma Normal?",
    "Para que serve o Redis no contexto de IA?",
]
for p in perguntas:
    prompt = montar_prompt(p)
    print("\n📝 Prompt:\n", prompt[:300], "...\n")
