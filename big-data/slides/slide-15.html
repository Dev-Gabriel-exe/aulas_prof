Batch vs streaming — a "tabela infinita"
Semana 14

Você aprendeu a garantir ACID em dados que já chegaram ao Data Lake (Delta Lake). Mas e os dados que estão chegando agora mesmo, continuamente? É disso que trata o processamento de streaming — a extensão natural de tudo que você já sabe sobre Spark.

Por que streaming não é só "batch mais rápido"

Até agora, todo processamento que você fez com Spark foi batch: um conjunto de dados finito e conhecido, processado do início ao fim. Streaming é fundamentalmente diferente — os dados nunca terminam de chegar.

📦 Batch
  • Dataset finito, com início e fim conhecidos
  • Job roda, processa tudo, termina
  • Latência de minutos a horas é aceitável
🌊 Streaming
  • Dataset infinito — dados continuam chegando indefinidamente
  • Job roda continuamente, processando novos dados assim que chegam
  • Latência de segundos é o objetivo (dashboards, alertas, fraude)
A grande ideia do Structured Streaming: tabela infinita

O Structured Streaming trata um stream como uma tabela que cresce infinitamente — cada novo evento que chega é como uma nova linha sendo inserida nessa tabela. Você escreve a mesma query DataFrame/SQL que já conhece do batch, e o motor cuida de aplicá-la incrementalmente a cada novo lote de dados.

Isso significa que quase tudo que você já aprendeu se aplica aqui: select, filter, groupBy, joins — a sintaxe é praticamente idêntica ao DataFrame batch das Semanas 8-9. A diferença está em como o motor executa por baixo dos panos.

Por baixo dos panos, o Structured Streaming não processa evento por evento (isso seria streaming "puro", como no Kafka Streams) — ele usa um modelo de micro-batch: a cada intervalo curto (ex: 1 segundo), ele coleta os novos dados chegados, roda o mesmo pipeline Spark de sempre sobre esse pequeno lote, e produz o resultado incremental.

🙋 Quiz — qual é a principal diferença conceitual entre processamento batch e streaming?
A
Streaming usa uma linguagem de programação diferente do batch
B
Batch processa um dataset finito e conhecido; streaming processa um dataset que nunca termina de chegar
C
Streaming não pode usar DataFrames, apenas RDDs
D
Streaming só funciona com dados vindos do Kafka

Lendo, processando e escrevendo um stream
readStream e writeStream — a API básica
Python — pipeline de streaming mínimo
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, window

spark = SparkSession.builder.appName("streaming-demo").getOrCreate()

# LEITURA — readStream em vez de read
eventos = (spark.readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "localhost:9092")
    .option("subscribe", "cliques")
    .load())

# PROCESSAMENTO — mesma sintaxe DataFrame do batch!
cliques_processados = (eventos
    .selectExpr("CAST(value AS STRING) as evento_json")
    .select(from_json(col("evento_json"), schema).alias("dados"))
    .select("dados.*")
    .filter(col("pagina") != ""))

# ESCRITA — writeStream em vez de write
query = (cliques_processados.writeStream
    .format("console")
    .outputMode("append")
    .start())

query.awaitTermination()   # mantém o processo rodando continuamente
Repare: a única diferença sintática real é read → readStream, write → writeStream, e o uso de .start() em vez de .save() (que retorna uma query em execução contínua, não uma ação que termina).
Output modes — como o resultado é entregue a cada micro-batch
Output modeComportamentoQuando usar
appendSó emite linhas NOVAS desde o último micro-batchPadrão. Não funciona com agregações sem watermark
completeReemite a tabela de resultado INTEIRA a cada micro-batchAgregações pequenas (ex: contagem total por categoria)
updateEmite apenas as linhas que MUDARAM desde o último micro-batchAgregações onde só interessa o que mudou (dashboards)
Armadilha comum: usar outputMode("append") em uma agregação sem watermark gera erro — o Spark não sabe quando uma linha agregada está "definitivamente pronta" para ser emitida sem descartar a possibilidade dela ainda mudar.
Sinks e checkpointing — tolerância a falhas
console
Imprime o resultado no terminal
Só para debug e desenvolvimento local
delta / parquet
Grava o resultado como tabela Delta ou Parquet
O sink mais comum para pipelines de produção
kafka
Republica o resultado processado em outro tópico
Comum em arquiteturas de streaming encadeado
memory
Mantém o resultado em uma tabela temporária em memória
Útil para testes e notebooks interativos
Python — writeStream com checkpoint
query = (cliques_processados.writeStream
    .format("delta")
    .outputMode("append")
    .option("checkpointLocation", "s3://meu-bucket/checkpoints/cliques/")
    .trigger(processingTime="30 seconds")   # roda um micro-batch a cada 30s
    .start("s3://meu-bucket/delta/cliques_processados/"))
checkpointLocation é obrigatório em produção: o Spark grava ali o progresso da leitura (offsets já processados) e o estado das agregações. Se o job cair e reiniciar, ele retoma exatamente de onde parou — sem duplicar nem perder dados (garantia exactly-once).
🙋 Quiz — por que o checkpointLocation é essencial em um job de streaming em produção?
A
Ele acelera o processamento de cada micro-batch
B
Ele registra o progresso e o estado, permitindo retomar exatamente de onde parou após uma falha
C
Ele define o formato de saída do sink (console, delta, kafka)
D
Ele é necessário apenas em output mode "complete"
Aula 3 · Window Operations e Watermarks

Aula 3 · ~50 minutos
Agregando por janelas de tempo e lidando com atraso
Tumbling windows — janelas fixas e não sobrepostas

Quando você quer agregar eventos em blocos de tempo fixos (ex: "quantos cliques por minuto"), usa uma tumbling window — janelas de tamanho fixo que não se sobrepõem, cobrindo a linha do tempo de ponta a ponta.

[10:00–10:05)
[10:05–10:10)
[10:10–10:15)
tumbling window de 5 minutos — cada evento cai em exatamente UMA janela
Python — tumbling window
resultado = (eventos
    .groupBy(window(col("timestamp_evento"), "5 minutes"))
    .count())

# Cada linha do resultado terá: window.start, window.end, count
Sliding windows — janelas sobrepostas

Quando você quer uma métrica mais suave, atualizada com mais frequência (ex: "média móvel dos últimos 10 minutos, recalculada a cada 5 minutos"), usa uma sliding window — as janelas se sobrepõem, e cada evento pode contribuir para múltiplas janelas.

[10:00–10:10)
[10:05–10:15)
[10:10–10:20)
[10:15–10:25)
sliding window de 10 minutos, deslizando a cada 5 minutos — um mesmo evento pode cair em até 2 janelas
Python — sliding window
resultado = (eventos
    .groupBy(window(col("timestamp_evento"), "10 minutes", "5 minutes"))
    #                                       ^tamanho da janela  ^intervalo de deslizamento
    .agg(sum("valor").alias("total")))
window(coluna, "10 minutes") = tumbling (o segundo parâmetro de deslizamento é omitido, então equivale ao próprio tamanho da janela). window(coluna, "10 minutes", "5 minutes") = sliding.
Watermarks — o que fazer com eventos atrasados

Em sistemas distribuídos, eventos podem chegar fora de ordem ou atrasados (problema de rede, app offline, etc). Sem um limite, o Spark teria que manter estado infinito na memória, esperando por eventos atrasados de qualquer janela, para sempre.

O watermark define até quanto tempo de atraso o Spark tolera antes de considerar uma janela "fechada" e liberar o estado da memória:

Python — window + watermark
resultado = (eventos
    .withWatermark("timestamp_evento", "10 minutes")   # tolera até 10min de atraso
    .groupBy(window(col("timestamp_evento"), "5 minutes"))
    .count())

# Um evento com timestamp de 10:03 só é aceito na janela [10:00-10:05)
# se chegar até 10:05 + 10min de watermark = 10:15.
# Depois disso, a janela é fechada e o evento atrasado é DESCARTADO.
Watermark curto: libera memória mais rápido, resultados ficam disponíveis antes, mas descarta mais eventos atrasados.
Watermark longo: aceita mais eventos atrasados, mas consome mais memória e atrasa a finalização das janelas.
Watermark é o que viabiliza outputMode("append") com agregações: só quando o Spark tem certeza (via watermark) que uma janela não vai mais receber dados, ele pode emiti-la de forma definitiva no modo append.
🙋 Quiz — qual é a função do watermark em uma agregação por janela?
A
Acelerar o processamento de cada micro-batch
B
Definir até quanto tempo de atraso é tolerado antes de fechar uma janela e liberar o estado da memória
C
Definir o formato do sink de saída
D
Garantir que o job nunca falhe

Aula 4 · ~50 minutos
Construindo um pipeline de streaming completo
Pipeline final — eventos, janela e watermark juntos
Python — pipeline de streaming completo
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, window, count

spark = SparkSession.builder.appName("pipeline-eventos").getOrCreate()

# Fonte simulada — gera 1 evento por segundo (ótima para testes locais)
eventos_raw = (spark.readStream
    .format("rate")
    .option("rowsPerSecond", 1)
    .load())

# Transformação: simula uma categoria e um timestamp de evento
eventos = eventos_raw.selectExpr(
    "timestamp as timestamp_evento",
    "CAST(value % 3 AS STRING) as categoria"
)

# Agregação: contagem de eventos por categoria, em janelas de 1 minuto,
# tolerando até 2 minutos de atraso
resultado = (eventos
    .withWatermark("timestamp_evento", "2 minutes")
    .groupBy(
        window(col("timestamp_evento"), "1 minute"),
        col("categoria")
    )
    .agg(count("*").alias("total_eventos")))

query = (resultado.writeStream
    .format("console")
    .outputMode("update")
    .option("truncate", "false")
    .trigger(processingTime="10 seconds")
    .start())

query.awaitTermination()
Fonte "rate": gera dados sintéticos automaticamente (um contador incremental + timestamp), sem precisar configurar Kafka — ideal para testar conceitos de streaming localmente antes de conectar a uma fonte real.

Watermark + Append
Comportamento esperado
  • Com update (Exercício 2): cada janela aparece MÚLTIPLAS vezes no console, com o total crescendo a cada micro-batch, até a janela ser considerada completa.
  • Com append + watermark (Exercício 3): cada janela aparece UMA ÚNICA vez no console — só depois que o watermark garante que ela não vai mais receber dados.
  • Sem watermark, tentar usar append com agregação geraria um erro do Spark, pois ele não teria como saber quando uma janela está "definitivamente pronta".

O que você precisa ter gravado
Tabela infinita

Structured Streaming trata um stream como uma tabela que cresce continuamente. Motor micro-batch aplica a mesma query incrementalmente.

Output modes

append (só novo), complete (tudo de novo) e update (só o que mudou). Cada um tem restrições sobre agregações.

Windows

Tumbling (fixas, sem sobreposição) e sliding (sobrepostas). window(col, tamanho[, deslizamento]).

Watermark

Define a tolerância a atraso, libera memória e viabiliza outputMode append em agregações.
