Por que o Spark — o problema com o MapReduce
Semana 4

Na semana 4 vimos que o HDFS armazena dados em blocos distribuídos. Mas como processar esses dados? A primeira resposta foi o MapReduce — funcional, mas lento. O Spark nasceu para resolver exatamente isso.

O problema do MapReduce — o disco como gargalo

O MapReduce (base do Hadoop 1.x) processa dados em duas fases: Map (transforma dados em pares chave-valor) e Reduce (agrega os pares). Mas entre cada fase, os resultados intermediários são escritos no disco (HDFS). Para pipelines com múltiplas etapas, isso significa:

🐌 MapReduce — ciclo por etapa
Lê HDFS
→
Map
→
Escreve disco
→
Lê disco
→
Reduce
→
Escreve disco
Cada etapa adicional = mais leituras e escritas em disco. Pipeline com 10 etapas = 20 ciclos de I/O.
⚡ Spark — in-memory pipeline
Lê HDFS/S3
→
RAM
→
Etapa 1
→
Etapa 2...N
→
Escreve final
Dados ficam em memória entre etapas. Pipeline com 10 etapas = apenas 1 leitura + 1 escrita em disco.
Resultado: o Spark é 10x–100x mais rápido que MapReduce para a maioria dos workloads. Para algoritmos iterativos de ML (que passam pelos dados dezenas de vezes), a diferença pode ser 1000x.
O ecossistema Spark — uma plataforma unificada

Além da velocidade, o Spark trouxe algo que o MapReduce não tinha: uma plataforma unificada para diferentes tipos de processamento.

RDD — Resilient Distributed Dataset e Lazy Evaluation
O que é um RDD — os três adjetivos que definem tudo

RDD é a abstração fundamental do Spark. Entender o nome é entender o conceito:

🛡 Resilient
Tolerante a falhas. Se uma partição se perde (nó cai), o Spark reconstrói a partir do lineage — o histórico de operações que criou aquela partição. Não precisa de réplica explícita.
🌐 Distributed
Particionado entre múltiplos executors no cluster. Cada executor processa sua partição em paralelo. Mais partições = mais paralelismo (até o limite de cores).
📦 Dataset
Coleção de objetos Python/Scala/Java. Pode ser qualquer tipo: strings, tuplas, dicionários. Sem schema definido — ao contrário do DataFrame.
Lazy Evaluation — o segredo da performance

O conceito mais importante do Spark é a avaliação preguiçosa (lazy evaluation). Quando você chama uma transformação (map, filter, groupBy…), o Spark não executa nada imediatamente. Ele apenas registra a operação no DAG (Directed Acyclic Graph). Somente quando uma action é chamada (collect, count, show, save…) é que o Spark otimiza e executa toda a cadeia.

Transformaçãordd2 = rdd.filter(lambda x: x['total'] > 100)❌ Não executa
Transformaçãordd3 = rdd2.map(lambda x: (x['category'], x['total']))❌ Não executa
Transformaçãordd4 = rdd3.reduceByKey(lambda a, b: a + b)❌ Não executa
Action ⚡resultado = rdd4.collect()✅ Executa TUDO agora
Por que lazy é melhor? O Spark pode otimizar o DAG inteiro antes de executar. Se você filtra 90% dos dados na primeira etapa e depois faz um join pesado, o Catalyst pode reorganizar para fazer o join com menos dados — impossível se cada operação executasse imediatamente.
Transformações vs Actions — o guia completo
Transformações (lazy)
# Cada linha retorna um NOVO RDD — lazy
rdd.map(lambda x: x * 2)
rdd.filter(lambda x: x > 0)
rdd.flatMap(lambda x: x.split(" "))
rdd.distinct()
rdd.sortBy(lambda x: x[1])
rdd.reduceByKey(lambda a, b: a + b)
rdd.groupByKey()   # evitar!
rdd1.union(rdd2)
rdd1.join(rdd2)
rdd.repartition(8)
rdd.coalesce(4)    # reduz partições
Actions (trigger execution) ⚡
# Cada uma dessas EXECUTA o DAG completo
rdd.collect()        # ⚠ traz tudo pro driver
rdd.count()          # conta elementos
rdd.first()          # primeiro elemento
rdd.take(10)         # primeiros 10
rdd.takeSample(False, 100) # amostra
rdd.reduce(lambda a,b: a+b)
rdd.sum()
rdd.max() / rdd.min()
rdd.foreach(print)   # executa f em cada
rdd.saveAsTextFile("/path")
Cuidado com collect(): traz todos os dados para a memória do driver (seu notebook/script). Em um RDD de 10 GB, isso vai crashar o driver. Use take(n) ou show(n) para inspecionar amostras.

DataFrame e Spark SQL — a API moderna do Spark
RDD vs DataFrame — por que o DataFrame ganhou
🔴 RDD (modo antigo)
Sem schema — você manipula objetos Python genéricos
Sem otimização automática — você é responsável pela eficiência
API verbosa — muito código para operações simples
Python lento: cada elemento passa pelo Python GIL
Difícil de debugar — erros de tipo aparecem em runtime
rdd.map(lambda r: (r['cat'], r['total'])).reduceByKey(lambda a,b: a+b)
VS
🟢 DataFrame (API moderna)
Schema definido — tipos validados, autocomplete no IDE
Catalyst Optimizer otimiza automaticamente o plano de execução
API expressiva — SQL ou Python fluente
Tungsten executa em JVM — muito mais rápido que RDD Python
Erros de tipo em tempo de compilação (schema validation)
df.groupBy("cat").agg(sum("total"))
Regra hoje: use DataFrame (ou Dataset em Scala). RDD é legado — você vai encontrar em código antigo, mas para código novo, DataFrame é sempre a escolha certa.
Operações essenciais com DataFrame
PySpark — operações DataFrame
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, sum, count, avg, max, min, round, when, lit, to_date

spark = SparkSession.builder.appName("Vendas").getOrCreate()
df = spark.read.parquet("s3://bucket/silver/orders/")

# ── Exploração ────────────────────────────────────────────────
df.show(5)              # primeiras 5 linhas
df.printSchema()       # estrutura e tipos
df.describe().show()  # estatísticas básicas
df.count()             # total de linhas
df.columns             # lista de colunas
df.dtypes              # lista de (coluna, tipo)

# ── Seleção e filtragem ───────────────────────────────────────
df.select("order_id", "customer_id", "total")
df.filter(col("total") > 100)
df.filter((col("status") == "paid") & (col("total") > 50))
df.where("total > 100 AND status = 'paid'")  # sintaxe SQL

# ── Transformações ───────────────────────────────────────────
df.withColumn("total_com_frete", col("total") + col("shipping"))
df.withColumn("categoria_preco",
    when(col("total") < 100, "baixo")
    .when(col("total") < 500, "medio")
    .otherwise("alto"))
df.withColumnRenamed("total", "valor_pedido")
df.drop("coluna_desnecessaria")
df.dropDuplicates(["order_id"])
df.fillna({"shipping": 0, "discount": 0})

# ── Agregações ───────────────────────────────────────────────
df.groupBy("category").agg(
    sum("total").alias("receita_total"),
    count("order_id").alias("qtd_pedidos"),
    avg("total").alias("ticket_medio"),
    max("total").alias("maior_pedido")
).orderBy(col("receita_total").desc()).show()

# ── Joins ────────────────────────────────────────────────────
df_clientes = spark.read.parquet("s3://bucket/silver/customers/")
df_completo = df.join(df_clientes, on="customer_id", how="left")

# ── Escrita ──────────────────────────────────────────────────
df.write.mode("overwrite").parquet("s3://bucket/gold/revenue/")
Spark SQL — SQL sobre DataFrames distribuídos

Uma das funcionalidades mais poderosas do Spark é a integração com SQL. Você pode registrar um DataFrame como uma view temporária e usar SQL puro para consultá-la — exatamente a mesma sintaxe que você usaria no PostgreSQL ou BigQuery.

PySpark — Spark SQL
# Registrar DataFrame como view temporária (só dura enquanto a sessão existir)
df.createOrReplaceTempView("orders")
df_clientes.createOrReplaceTempView("customers")

# Usar SQL puro — mesma sintaxe do PostgreSQL
resultado = spark.sql("""
    SELECT
        c.state,
        o.category,
        COUNT(o.order_id)      AS qtd_pedidos,
        SUM(o.total)           AS receita,
        ROUND(AVG(o.total), 2) AS ticket_medio,
        MAX(o.total)           AS maior_pedido
    FROM orders o
    LEFT JOIN customers c ON o.customer_id = c.id
    WHERE o.status = 'paid'
      AND o.order_date >= '2024-01-01'
    GROUP BY c.state, o.category
    HAVING COUNT(o.order_id) > 10
    ORDER BY receita DESC
    LIMIT 20
""")

resultado.show()

# Resultado é um DataFrame normal — pode continuar transformando
resultado.write.mode("overwrite").parquet("s3://bucket/gold/ranking_estado/")
DataFrame API vs SQL: ambas produzem exatamente o mesmo plano de execução após o Catalyst Optimizer. A escolha é de preferência pessoal — analistas tendem a preferir SQL; engenheiros de dados tendem a preferir a API encadeada. Em times mistos, Spark SQL facilita a colaboração.

PySpark na prática — carregar, explorar e transformar
SparkSession — o ponto de entrada de tudo

Toda aplicação PySpark começa com a criação de uma SparkSession. É ela que gerencia a conexão com o cluster, a configuração e o acesso a todas as funcionalidades do Spark.

PySpark — SparkSession e configurações essenciais
from pyspark.sql import SparkSession

# Criação básica — local (usa todos os cores da máquina)
spark = (SparkSession.builder
    .appName("AnaliseVendas")
    .master("local[*]")               # local[*] = todos os cores
    .config("spark.sql.shuffle.partitions", "8")  # padrão=200, reduzir p/ dev
    .config("spark.driver.memory", "2g")
    .getOrCreate()
)

# Verificar a versão
print(spark.version)            # ex: 3.5.0
print(spark.sparkContext.master) # local[*]

# Ao terminar (libera recursos)
spark.stop()
spark.sql.shuffle.partitions: valor padrão é 200 — excelente para produção com grandes clusters, mas muito lento para desenvolvimento local com datasets pequenos. Reduza para 8 ou 16 ao desenvolver localmente.
Pipeline completo — do CSV ao Parquet
PySpark — pipeline de exploração completo
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, sum, count, avg, round, to_date, year, month, when

spark = SparkSession.builder.appName("Vendas").master("local[*]").getOrCreate()

# ── 1. CARREGAR ──────────────────────────────────────────────
df = (spark.read
    .option("header", "true")        # primeira linha = cabeçalho
    .option("inferSchema", "true")   # detectar tipos automaticamente
    .option("sep", ";")              # separador ponto-e-vírgula
    .option("encoding", "UTF-8")
    .csv("data/vendas_2024.csv")
)

# ── 2. EXPLORAR ──────────────────────────────────────────────
print(f"Total de linhas: {df.count()}")
print(f"Total de colunas: {len(df.columns)}")
df.printSchema()
df.show(5, truncate=False)
df.describe(["total", "quantidade"]).show()

# Verificar nulos
from pyspark.sql.functions import isnull
df.select([count(when(isnull(c), c)).alias(c) for c in df.columns]).show()

# ── 3. LIMPAR (Raw → Silver) ──────────────────────────────────
df_silver = (df
    .dropDuplicates(["order_id"])
    .dropna(subset=["order_id", "total"])
    .withColumn("total", col("total").cast("double"))
    .withColumn("order_date", to_date(col("order_date"), "dd/MM/yyyy"))
    .withColumn("ano",  year("order_date"))
    .withColumn("mes",  month("order_date"))
    .filter(col("total") > 0)
    .fillna({"shipping": 0.0, "discount": 0.0})
)

print(f"Após limpeza: {df_silver.count()} linhas")

# ── 4. AGREGAR (Silver → Gold) ────────────────────────────────
df_gold = (df_silver
    .groupBy("ano", "mes", "category")
    .agg(
        sum("total").alias("receita"),
        count("order_id").alias("pedidos"),
        round(avg("total"), 2).alias("ticket_medio")
    )
    .orderBy("ano", "mes", col("receita").desc())
)

df_gold.show(20)

# ── 5. SALVAR ────────────────────────────────────────────────
(df_silver.write
    .mode("overwrite")
    .partitionBy("ano", "mes")
    .option("compression", "snappy")
    .parquet("output/silver/orders/")
)

(df_gold.write
    .mode("overwrite")
    .parquet("output/gold/revenue_monthly/")
)
