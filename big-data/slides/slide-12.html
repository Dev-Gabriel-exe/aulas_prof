Por que precisamos de um orquestrador — o problema do cron
Semana 11

Você aprendeu a transformar dados com dbt — mas quem executa dbt run todo dia às 3h da manhã? E se a ingestão do dia anterior ainda não terminou? E se um modelo falhar — quem avisa alguém? É disso que trata a orquestração.

O que cron não resolve

A solução mais simples para "rodar algo todo dia" é o cron do Linux. Mas pipelines de dados têm necessidades que o cron simplesmente não atende:

❌ O que o cron NÃO faz
  • Não entende dependências — não sabe que a Tarefa B precisa esperar a A terminar
  • Não tem retry automático — se falhar, simplesmente não roda
  • Não tem visibilidade — você não sabe se rodou, quanto demorou, ou onde falhou
  • Não alerta ninguém quando algo dá errado
  • Não escala — cada servidor tem seu próprio cron, sem coordenação central
✅ O que o Airflow resolve
  • Dependências explícitas — define quem espera por quem
  • Retries configuráveis com backoff exponencial
  • UI completa: histórico, duração, logs de cada execução
  • Alertas automáticos via e-mail, Slack, PagerDuty
  • Escala horizontalmente com Celery/Kubernetes Executor
Origem: o Airflow foi criado pelo Airbnb em 2014 para gerenciar a complexidade crescente de pipelines de dados internos. Em 2016 se tornou projeto Apache, e hoje é o padrão de mercado para orquestração de dados — usado por Netflix, Spotify, Lyft, Adobe e milhares de empresas.

DAGs e Operadores — definindo o "o quê" e a ordem
DAG — Directed Acyclic Graph

Assim como o dbt (Semana 11), o Airflow organiza o trabalho em um DAG — grafo direcionado e acíclico. Direcionado significa que as setas têm um sentido único (A → B, nunca os dois ao mesmo tempo). Acíclico significa que não pode haver loops — uma tarefa nunca pode depender (direta ou indiretamente) de si mesma.

Python — estrutura básica de uma DAG Airflow
from airflow import DAG
from airflow.operators.bash import BashOperator
from datetime import datetime, timedelta

# Configurações padrão aplicadas a todas as tasks da DAG
default_args = {
    "owner": "engenharia_dados",
    "retries": 3,
    "retry_delay": timedelta(minutes=5),
}

with DAG(
    dag_id="pipeline_vendas_diario",
    default_args=default_args,
    start_date=datetime(2024, 1, 1),
    schedule_interval="0 3 * * *",   # todo dia às 3h da manhã
    catchup=False,
) as dag:

    extract = BashOperator(
        task_id="extract_orders",
        bash_command="python /scripts/extract_orders.py",
    )

    transform = BashOperator(
        task_id="dbt_run",
        bash_command="cd /dbt_project && dbt run",
    )

    test = BashOperator(
        task_id="dbt_test",
        bash_command="cd /dbt_project && dbt test",
    )

    # O operador >> define a ordem: extract → transform → test
    extract >> transform >> test
O operador >> (bitshift) define dependências: tarefa_a >> tarefa_b significa "tarefa_b só roda depois que tarefa_a terminar com sucesso". Você também pode usar tarefa_b << tarefa_a (sentido contrário) ou listas para paralelismo: extract >> [transform_a, transform_b] >> test.
Operadores — os blocos de construção de uma task

Um Operator define o que uma task realmente faz. O Airflow tem centenas de operadores prontos (chamados "providers") para integrar com praticamente qualquer ferramenta.

BashOperator
Executa qualquer comando shell
Ideal para chamar scripts, CLIs (dbt, Spark-submit)
Simples mas "caixa preta" — Airflow não vê o que acontece dentro
PythonOperator
Executa uma função Python diretamente
Acesso total ao contexto da execução (datas, parâmetros)
Mais flexível para lógica customizada
Sensors
Espera até uma condição ser verdadeira
FileSensor, S3KeySensor, ExternalTaskSensor
Modo "poke" (verifica periodicamente) ou "reschedule"
Operadores de Provider
SparkSubmitOperator, PostgresOperator, S3Hook
Conectam diretamente com cada ferramenta do seu pipeline
Centenas disponíveis: Kafka, Snowflake, BigQuery, dbt Cloud...
TaskFlow API — a forma moderna de escrever DAGs

Desde o Airflow 2.0, existe uma forma mais "Pythônica" de escrever DAGs usando decorators (@dag e @task), eliminando boa parte do código repetitivo dos Operators tradicionais.

Python — TaskFlow API (Airflow 2.0+)
from airflow.decorators import dag, task
from datetime import datetime

@dag(
    schedule_interval="0 3 * * *",
    start_date=datetime(2024, 1, 1),
    catchup=False,
)
def pipeline_vendas():

    @task()
    def extrair_pedidos():
        # lógica de extração
        return {"linhas_extraidas": 15000}

    @task()
    def transformar(metadados: dict):
        # XCom automático: 'metadados' já vem com o retorno da task anterior!
        print(f"Transformando {metadados['linhas_extraidas']} linhas")

    @task()
    def validar():
        # dbt test ou Great Expectations aqui
        pass

    # Dependências ficam implícitas pela passagem de dados!
    dados = extrair_pedidos()
    transformar(dados) >> validar()

pipeline_vendas()
XCom automático: no exemplo, o retorno de extrair_pedidos() é passado automaticamente para transformar() via XCom (Cross-Communication) — o mecanismo do Airflow para tasks compartilharem pequenos dados entre si. Na API tradicional, isso exigiria código explícito de xcom_push/xcom_pull.
🙋 Quiz — no código extract >> [transform_a, transform_b] >> load, qual é o comportamento esperado?
A
transform_a e transform_b rodam em sequência, uma depois da outra
B
transform_a e transform_b rodam em paralelo após extract; load espera AMBAS terminarem
C
Apenas uma das duas (a mais rápida) é executada
D
load roda independente, sem esperar transform_a ou transform_b

Sensores, agendamento e monitoramento — controlando o tempo e a saúde do pipeline
Sensores — esperando por condições externas

Às vezes uma task não deve começar em um horário fixo, mas sim quando algo específico acontecer — um arquivo chegar no S3, uma outra DAG terminar, uma tabela ser atualizada. Para isso existem os Sensors.

Python — Sensores comuns
from airflow.sensors.filesystem import FileSensor
from airflow.providers.amazon.aws.sensors.s3 import S3KeySensor
from airflow.sensors.external_task import ExternalTaskSensor

# Espera um arquivo aparecer no sistema de arquivos local
esperar_arquivo = FileSensor(
    task_id="wait_for_file",
    filepath="/data/raw/orders_{{ ds }}.csv",   # {{ ds }} = data de execução (Jinja)
    poke_interval=60,     # verifica a cada 60 segundos
    timeout=3600,         # desiste depois de 1 hora
    mode="reschedule",    # libera o worker entre verificações
)

# Espera um objeto aparecer no S3
esperar_s3 = S3KeySensor(
    task_id="wait_for_s3_file",
    bucket_name="meu-datalake",
    bucket_key="raw/orders/{{ ds }}/data.parquet",
    poke_interval=120,
)

# Espera outra DAG terminar com sucesso
esperar_outra_dag = ExternalTaskSensor(
    task_id="wait_for_ingestion_dag",
    external_dag_id="pipeline_ingestao",
    external_task_id="extract_complete",
)
poke vs reschedule: no modo "poke" (padrão), o sensor ocupa um worker inteiro enquanto espera — desperdiçando recursos em esperas longas. No modo "reschedule", o sensor libera o worker entre verificações, sendo muito mais eficiente para esperas de horas.

Pegadinha clássica: catchup e o "logical date"

Quando você cria uma DAG com start_date=2024-01-01 e ela só é ativada hoje, o Airflow por padrão tenta executar TODAS as execuções "perdidas" desde a data de início — isso se chama catchup. Para a maioria dos pipelines de produção, você quer catchup=False para evitar uma avalanche de execuções históricas desnecessárias.

Outra confusão comum: a execução rotulada como "2024-01-15" no Airflow representa o período que termina em 2024-01-15 — ou seja, ela só dispara depois que esse dia passar (no dia 16). Isso é chamado de "logical date" e existe porque o Airflow foi desenhado para processar dados de períodos já completos.

Python — Retries, alertas e SLA
from datetime import timedelta
from airflow.providers.slack.operators.slack_webhook import SlackWebhookOperator

default_args = {
    "retries": 3,                              # tenta 3 vezes antes de marcar como failed
    "retry_delay": timedelta(minutes=5),      # espera 5 min entre tentativas
    "retry_exponential_backoff": True,        # 5min, 10min, 20min...
    "sla": timedelta(hours=2),               # alerta se task demorar mais que 2h
    "email_on_failure": True,
    "email": ["dados@empresa.com"],
}

# Callback customizado disparado em caso de falha
def notificar_slack_falha(context):
    SlackWebhookOperator(
        task_id="slack_alert",
        slack_webhook_conn_id="slack_default",
        message=f"🔴 Pipeline falhou: {context['task_instance'].task_id}",
    ).execute(context=context)

# Aplicado na DAG: on_failure_callback=notificar_slack_falha
SLA (Service Level Agreement): diferente de retry (que lida com falhas), o SLA monitora demora. Se uma task que normalmente leva 10 minutos passar de 2 horas, o Airflow dispara um alerta — mesmo que a task ainda esteja "rodando" e não tenha falhado tecnicamente. Essencial para detectar gargalos antes que afetem relatórios do dia seguinte.
🙋 Quiz — uma DAG tem start_date de 6 meses atrás e acabou de ser ativada com catchup=True. O que vai acontecer?
A
Nada — catchup só funciona se a DAG já tiver rodado antes
B
O Airflow vai disparar todas as execuções "perdidas" dos últimos 6 meses de uma vez
C
A DAG só executa a partir de hoje, ignorando o histórico
D
O Airflow pede confirmação manual antes de rodar qualquer execução

Python — DAG completa do pipeline de vendas
from airflow.decorators import dag, task
from airflow.operators.bash import BashOperator
from airflow.sensors.filesystem import FileSensor
from datetime import datetime, timedelta

default_args = {
    "retries": 2,
    "retry_delay": timedelta(minutes=5),
    "email_on_failure": True,
}

@dag(
    dag_id="pipeline_vendas_completo",
    default_args=default_args,
    start_date=datetime(2024, 1, 1),
    schedule_interval="0 3 * * *",
    catchup=False,
    tags=["vendas", "producao"],
)
def pipeline_vendas_completo():

    # 1. INGESTÃO — extrai dados da fonte (Semana 7)
    extract = BashOperator(
        task_id="extract_orders",
        bash_command="python /scripts/extract_orders.py --date={{ ds }}",
    )

    # 2. SENSOR — espera o arquivo ficar disponível no Data Lake
    wait_for_data = FileSensor(
        task_id="wait_for_raw_file",
        filepath="/datalake/raw/orders_{{ ds }}.parquet",
        poke_interval=30,
        timeout=1800,
        mode="reschedule",
    )

    # 3. TRANSFORMAÇÃO — roda os modelos dbt (Semana 11)
    dbt_run_staging = BashOperator(
        task_id="dbt_run_staging",
        bash_command="cd /dbt_project && dbt run --select staging",
    )

    dbt_run_marts = BashOperator(
        task_id="dbt_run_marts",
        bash_command="cd /dbt_project && dbt run --select marts",
    )

    # 4. VALIDAÇÃO — testes de qualidade (Semana 11)
    dbt_test = BashOperator(
        task_id="dbt_test",
        bash_command="cd /dbt_project && dbt test",
    )

    # 5. NOTIFICAÇÃO — sucesso ou falha
    @task(trigger_rule="all_success")
    def notificar_sucesso():
        print("✅ Pipeline concluído com sucesso!")
        # Aqui: enviar Slack, atualizar dashboard, etc.

    # Definindo a ordem: extração → espera → staging → marts → teste → notificação
    extract >> wait_for_data >> dbt_run_staging >> dbt_run_marts >> dbt_test >> notificar_sucesso()

pipeline_vendas_completo()
trigger_rule: por padrão, uma task só roda se TODAS as tasks anteriores tiverem sucesso (all_success). Mas você pode mudar esse comportamento: all_failed (só roda se tudo falhou — útil para rollback), one_success (basta uma ter sucesso), none_failed (roda mesmo se alguma foi "skipped").

Exercício 3 — Desafio: simular falha e retry (15 min)

Crie uma task que falha propositalmente (ex: exit 1 no BashOperator) e configure retries=2 com retry_delay de 1 minuto. Observe na Airflow UI: (a) quantas tentativas o Airflow fez, (b) o status final da task após esgotar os retries, (c) o que acontece com as tasks downstream que dependiam dela (devem aparecer como upstream_failed).

Retry + Status
Comportamento esperado
  • (a) O Airflow tenta a execução original + 2 retries = 3 tentativas totais, todas falhando, com 1 minuto de espera entre cada.
  • (b) Após esgotar os retries, a task fica com status failed permanentemente para aquela execução (a menos que você a limpe manualmente para forçar novo retry).
  • (c) Qualquer task que dependia dela (downstream) recebe o status upstream_failed e não é executada — o Airflow propaga a falha automaticamente para proteger a integridade do pipeline.
