Revisão rápida — cumulativa
Os 5 blocos do semestre — pontos-chave e armadilhas
Bloco 1 — Fundamentos (S1-5)
5 Vs: volume, velocidade, variedade, veracidade, valor
Lambda: batch + speed + serving. Kappa: streaming puro
Zonas do Data Lake: raw/bronze → silver → gold
HDFS: blocos replicados, NameNode/DataNode
Kafka: tópicos, partições, producers/consumers, offset, consumer groups
Armadilha: confundir Lakehouse (arquitetura) com Data Lake (armazenamento bruto).
Bloco 2 — Ingestão e Processamento (S7-10)
Full Load/Incremental/CDC: Incremental não captura DELETEs; CDC lê o log de transações
Spark: in-memory, lazy evaluation, transformação vs action
PySpark: narrow (sem shuffle) vs wide (com shuffle); broadcast na tabela PEQUENA
Catalyst: 4 fases; explain() mostra PushedFilters e tipo de join
Armadilha: UDF bloqueia predicate pushdown — prefira funções nativas.
Bloco 3 — Transformação e Orquestração (S11-12)
ETL vs ELT: ETL transforma antes (dados sensíveis); ELT transforma depois (padrão moderno)
dbt: staging → intermediate → marts; ref() cria dependência, source() aponta pra fonte bruta
Testes dbt: genéricos (unique, not_null...) e singulares (SQL customizado)
Airflow: DAG, operador >> define dependência; sensor espera, operator executa
Armadilha: catchup=True dispara todas as execuções perdidas — use catchup=False em produção.
Bloco 4 — Armazenamento e Streaming (S14-15)
Delta Lake: _delta_log garante ACID; MERGE INTO = upsert atômico; time travel
OPTIMIZE+ZORDER aceleram leitura; VACUUM remove órfãos (cuidado com retenção curta)
Structured Streaming: tabela infinita, micro-batch
Windows: tumbling (sem sobreposição) vs sliding (com sobreposição)
Armadilha: outputMode append em agregação SEM watermark gera erro.
Bloco 5 — Governança e IA (S16-17)
Data Catalog: metadados técnicos, de negócio, operacionais e de governança
Linhagem: upstream (origem/debug) vs downstream (impacto)
LGPD: classificação PII + linhagem viabilizam o direito ao esquecimento
Feature Store: elimina treino-serving skew; offline (treino) vs online (serving)
Armadilha: usar online store para montar dataset de treino perde o point-in-time correctness.

Bloco 1
Uma empresa precisa de uma arquitetura que processe tanto dados históricos em lote quanto eventos em tempo real, mantendo uma camada de serving unificada para consultas. Qual arquitetura é essa?
A
Kappa
B
Lambda
C
Lakehouse
D
ETL clássico
✅ B. A arquitetura Lambda combina explicitamente uma camada batch (dados históricos), uma camada speed (tempo real) e uma camada serving que unifica as duas para consultas — exatamente o que o enunciado descreve. Kappa usa apenas streaming, sem uma camada batch separada.

Bloco 1
Em um Data Lake organizado em zonas, onde os dados já foram limpos e padronizados, mas ainda não estão agregados para consumo de negócio?
A
Raw/Bronze
B
Silver
C
Gold
D
Staging externo
✅ B. A zona Silver contém dados já limpos, tipados e padronizados (equivalente à camada staging do dbt), mas ainda não agregados em métricas de negócio — isso é responsabilidade da zona Gold.

Bloco 2
Um sistema legado deleta fisicamente registros cancelados. A empresa precisa capturar esses cancelamentos no Data Lake. Qual estratégia de ingestão é obrigatória?
A
Incremental com cursor em updated_at
B
CDC, que captura eventos DELETE via log de transações
C
Airbyte em modo incremental dedup
D
Full Load semanal
✅ B. Incremental não captura DELETEs — o registro simplesmente desaparece silenciosamente. CDC lê o log de transações do banco e emite um evento explícito de DELETE, sendo o único método que resolve esse requisito nativamente.

Bloco 2
Um join entre uma tabela de 2 bilhões de linhas e uma de 100 linhas está usando SortMergeJoin e demorando muito. Qual é a correção mais impactante?
A
repartition() na tabela grande
B
Forçar broadcast() na tabela pequena
C
Converter para RDD
D
coalesce() antes do join
✅ B. Forçar broadcast na tabela de 100 linhas copia essa tabela inteira para cada executor, eliminando completamente o shuffle da tabela de 2 bilhões de linhas — a otimização mais impactante disponível nesse cenário clássico.

Bloco 2
No explain() de um DataFrame aparece o nó BatchEvalPython e PushedFilters: []. O que isso indica?
A
O plano está totalmente otimizado
B
Uma UDF Python está bloqueando o predicate pushdown
C
O arquivo Parquet não foi encontrado
D
O DataFrame está cacheado corretamente
✅ B. BatchEvalPython aparece quando há chamada a uma UDF Python. PushedFilters: [] confirma que o Catalyst não conseguiu empurrar filtros até o FileScan porque não "enxerga" o que a UDF faz — solução: substituir por funções nativas do pyspark.sql.functions.

Bloco 3
Uma empresa de saúde precisa anonimizar CPFs antes que os dados cheguem a um Data Warehouse na nuvem, por exigência da LGPD. Qual abordagem é obrigatória?
A
ETL — anonimizar antes de carregar
B
ELT — carregar bruto e anonimizar com dbt depois
C
Streaming, para reduzir a exposição
D
Tanto faz — a ordem não importa
✅ A. Por exigência da LGPD, dados sensíveis como CPF devem ser anonimizados ANTES de trafegarem para sistemas de terceiros/cloud. No ELT, os CPFs brutos chegariam ao Data Warehouse — violando o princípio de minimização de dados.

Bloco 3
O que acontece quando um modelo dbt usa {{ ref('stg_orders') }}?
A
Apenas substitui pelo nome literal da tabela
B
Registra a dependência no DAG, garantindo que stg_orders rode antes
C
Executa testes de qualidade automaticamente
D
Aponta para a zona Raw
✅ B. ref() sinaliza que o modelo DEPENDE de stg_orders, fazendo o dbt construir o DAG de dependências e garantir a ordem de execução correta — eliminando a necessidade de gerenciar isso manualmente.

Bloco 3
Uma DAG Airflow tem start_date de 6 meses atrás e é ativada hoje com catchup=True. O que acontece?
A
Nada — catchup só funciona se já tiver rodado antes
B
Dispara todas as execuções perdidas dos últimos 6 meses de uma vez
C
Executa apenas a partir de hoje
D
Pede confirmação manual antes de cada execução
✅ B. catchup=True faz o Airflow disparar uma execução para cada intervalo de agendamento entre start_date e agora — com 6 meses e schedule diário, isso significa dezenas de execuções de uma vez. Por isso catchup=False é recomendado em produção.

Bloco 4
Um job Spark falha no meio de uma escrita em tabela Delta. O que garante que a tabela não fique com dados parciais visíveis?
A
O formato Parquet
B
A atomicidade garantida pelo _delta_log — a transação só é commitada por completo, ou não é commitada
C
O particionamento da tabela
D
O comando VACUUM
✅ B. A atomicidade do Delta Lake, garantida pelo _delta_log, assegura que uma transação só é considerada válida quando o arquivo JSON de versão é commitado com sucesso — se o job falhar antes, a versão anterior íntegra continua sendo a visível.

Bloco 4
Uma agregação por groupBy usa outputMode("append") sem nenhum watermark definido. O que acontece?
A
Roda normalmente, emitindo cada linha assim que chega
B
Gera erro — o Spark não sabe quando uma janela está definitivamente pronta sem watermark
C
Converte automaticamente para complete mode
D
Ignora silenciosamente o groupBy
✅ B. Sem watermark, o Spark não tem como garantir que uma janela agregada nunca mais vai mudar — condição necessária para emiti-la em modo append. Essa combinação gera erro de análise antes do job rodar.

Bloco 5
Um engenheiro precisa saber se pode remover com segurança uma coluna de uma tabela staging. Qual tipo de linhagem ele deve consultar?
A
Upstream
B
Downstream
C
Nenhuma — não é resolvido por linhagem

Apenas o Data Catalog, sem linhagem

Bloco 5
Um modelo de fraude tem ótima performance no treino, mas se comporta inconsistentemente em produção porque uma feature é calculada com lógicas diferentes em cada ambiente. Qual ferramenta resolve isso estruturalmente?
A
Airflow, agendando o retraining
B
Feature Store, com uma única definição de feature usada em treino e produção
C
OpenMetadata, catalogando as features
D
Delta Lake, com time travel
✅ B. Esse é o clássico treino-serving skew, resolvido estruturalmente por um Feature Store (como o Feast), que centraliza a definição da feature em um único FeatureView consumido tanto no treino (offline) quanto em produção (online).

A

Arquitetura mínima esperada

O projeto final deve demonstrar, com um caso de uso real (ainda que pequeno), a capacidade de conectar as ferramentas estudadas ao longo do semestre em um pipeline coeso:

Ingestão
Airbyte/Kafka/CDC
Armazenamento
Delta Lake
Transformação
Spark + dbt
Orquestração
Airflow
Governança
Catálogo + LGPD
Análise Final
Dashboard / Feature Store
Não é obrigatório usar TODAS as tecnologias do semestre — o essencial é justificar as escolhas feitas para o cenário do seu projeto, mostrando que você entende os trade-offs discutidos em cada semana.
Rubrica sugerida de avaliação
CritérioO que é avaliado
IngestãoEstratégia de ingestão escolhida (Full Load/Incremental/CDC/streaming) é justificada pelo cenário
ProcessamentoUso correto de Spark/PySpark, evitando armadilhas discutidas (UDF desnecessária, shuffle evitável)
TransformaçãoModelos organizados em camadas (staging/marts), com testes de qualidade configurados
OrquestraçãoPipeline automatizado via Airflow, com dependências e tratamento de falha (retries)
ArmazenamentoUso de Delta Lake (ou justificativa da ausência) e garantias de qualidade dos dados
GovernançaDocumentação mínima dos dados (mesmo que manual) e atenção a dados sensíveis, se houver
ApresentaçãoClareza ao explicar decisões de arquitetura e responder perguntas técnicas da banca
Estrutura sugerida para a apresentação
  • 1. Contexto (1-2 min): qual problema de negócio o pipeline resolve, e por que Big Data é necessário aqui.
  • 2. Arquitetura (3-4 min): diagrama do pipeline completo, explicando cada etapa e a tecnologia escolhida.
  • 3. Demonstração (3-5 min): rodar (ou mostrar gravação) do pipeline funcionando, do dado bruto até o resultado final.
  • 4. Decisões técnicas (2-3 min): pelo menos uma decisão de arquitetura justificada com trade-offs (ex: "escolhi ELT em vez de ETL porque...").
  • 5. Perguntas: esteja pronto para justificar qualquer escolha tecnológica com os conceitos vistos nas semanas 1-17.
Erro comum: apresentar só o resultado final (um dashboard bonito) sem conseguir explicar o pipeline por trás. A banca tende a perguntar "por que você escolheu X e não Y?" — prepare-se para justificar, não só descrever.
