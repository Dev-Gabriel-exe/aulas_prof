Você já sabe como os dados chegam (ingestão, Semana 7) e como o Spark processa grandes volumes (Semanas 8-10). Agora a pergunta é: onde e como organizamos as transformações de negócio de forma sustentável, testável e versionada?

ETL — Extract, Transform, Load (o modelo clássico)

Por décadas, o padrão foi ETL: extrair os dados da fonte, transformá-los fora do destino (em um servidor de processamento separado) e só então carregar o resultado já limpo no Data Warehouse.

🔵 ETL — transforma ANTES de carregar
Extract→ Transform→ Load
Transformação roda em servidor de ETL dedicado (Informatica, Talend, SSIS)
O Data Warehouse só recebe dados já limpos e modelados
Surgiu quando storage e compute eram caros — economiza espaço no DW
Mudar a lógica de transformação exige reprocessar a pipeline inteira
🟢 ELT — transforma DEPOIS de carregar
Extract→ Load→ Transform
Dados brutos são carregados primeiro (igual à zona Raw que você já conhece!)
Transformação roda DENTRO do próprio Data Warehouse (SQL)
Surgiu com cloud warehouses baratos e poderosos (Snowflake, BigQuery)
Mudar a lógica é só mudar uma query SQL — reprocessamento rápido
Conexão com o que você já sabe: ELT é, na prática, o mesmo princípio das zonas Raw → Silver → Gold do Data Lake (Semana 3)! Você carrega os dados brutos primeiro (Raw/Load) e depois transforma dentro do mesmo ambiente (Silver/Gold = Transform). dbt é a ferramenta que organiza exatamente essa etapa "T" do ELT.
Por que o ELT dominou — três forças que mudaram o jogo
💰 Storage barato
S3/GCS custam centavos por GB. Não há mais razão para evitar guardar dados brutos.
⚡ Compute elástico
Snowflake, BigQuery e Databricks processam terabytes em SQL com paralelismo automático.
🔄 Iteração rápida
Analistas e engenheiros podem ajustar a lógica de negócio direto em SQL, sem reingerir nada.
Quando ainda usar ETL tradicional

Resumo prático: hoje a maioria dos times de dados usa ELT — Airbyte/Kafka para o "EL" (Extract + Load para o Data Lake/Warehouse) e dbt para o "T" (Transform). É exatamente a stack que você está aprendendo neste curso!

dbt — transformação como software de engenharia
O que é o dbt (data build tool)

dbt é uma ferramenta que permite escrever transformações de dados em SQL puro (com um pouco de templating Jinja), versioná-las em Git, testá-las automaticamente e documentá-las — trazendo práticas de engenharia de software para o mundo de analytics que antes era dominado por scripts SQL avulsos e sem controle.

❌ Antes do dbt
  • Scripts SQL soltos em pastas, sem versionamento
  • Sem testes — descobrir erro só quando o relatório quebra
  • Dependências entre tabelas só na cabeça de quem escreveu
  • Documentação inexistente ou desatualizada
✅ Com dbt
  • Cada transformação é um arquivo .sql versionado em Git
  • Testes automáticos (unique, not_null, relationships)
  • DAG de dependências gerado automaticamente via ref()
  • Documentação e linhagem geradas automaticamente
A grande ideia do dbt: "Engineers shouldn't write pipelines, they should write SQL — and let the tool handle the pipeline." Você escreve apenas o que cada tabela deve conter (a query SELECT); o dbt resolve a ordem de execução, materializa as tabelas e roda os testes.
Estrutura em camadas — Staging, Intermediate e Marts

A convenção da comunidade dbt organiza os modelos em três camadas, espelhando exatamente o que você já conhece das zonas Raw/Silver/Gold:

staging — 1:1 com a fonte, limpeza mínima (≈ Silver)
stg_orders
stg_customers
stg_products
intermediate — joins e lógica de negócio intermediária
int_orders_joined
int_customer_metrics
marts — modelos finais por área de negócio (≈ Gold)
fct_revenue_daily
dim_customers
fct_customer_ltv

Stagingstg_*Renomeia colunas, converte tipos, filtra nulos óbvios. Espelha a fonte 1:1.
Intermediateint_*Joins entre staging models, cálculos intermediários reutilizáveis.

Anatomia de um modelo dbt — ref() e source()
SQL — models/staging/stg_orders.sql
-- models/staging/stg_orders.sql
-- source() aponta para a tabela BRUTA (zona Raw) declarada em sources.yml

SELECT
    order_id,
    customer_id,
    CAST(total AS DECIMAL(10,2)) AS total,
    CAST(order_date AS DATE) AS order_date,
    LOWER(TRIM(status)) AS status
FROM {{ source('raw', 'orders') }}
WHERE order_id IS NOT NULL
SQL — models/marts/fct_revenue_daily.sql
-- models/marts/fct_revenue_daily.sql
-- ref() aponta para OUTRO MODELO dbt — cria a dependência no DAG automaticamente

SELECT
    order_date,
    COUNT(order_id)      AS total_pedidos,
    SUM(total)            AS receita_total,
    ROUND(AVG(total), 2) AS ticket_medio
FROM {{ ref('stg_orders') }}   -- ← dbt sabe que precisa rodar stg_orders ANTES
WHERE status = 'paid'
GROUP BY order_date
Por que ref() é revolucionário: ao usar ref('stg_orders') em vez do nome literal da tabela, o dbt constrói automaticamente o DAG de dependências. Ele sabe que precisa rodar stg_orders antes de fct_revenue_daily — sem você escrever nenhum orquestrador manual. Compare isso com escrever scripts SQL soltos: lá, você teria que controlar manualmente a ordem de execução.
🙋 Quiz — qual é a função do ref() dentro de um modelo dbt?
A
Apontar para uma tabela bruta externa ao dbt (zona Raw)
B
Referenciar outro modelo dbt, criando uma dependência automática no DAG
C
Executar testes automáticos de qualidade nos dados

Testes de qualidade — confiança automatizada nos dados
Os 4 testes genéricos nativos do dbt

O dbt vem com quatro testes prontos para uso — basta declará-los em um arquivo YAML, sem escrever SQL. Eles cobrem os problemas de qualidade mais comuns de qualquer pipeline de dados.

unique
Garante que não há valores duplicados na coluna. Essencial em chaves primárias (order_id, customer_id).
not_null
Garante que a coluna nunca está vazia. Útil em chaves e campos obrigatórios para o negócio.
accepted_values
Garante que a coluna só contém valores de uma lista pré-definida (ex: status só pode ser "paid", "pending", "cancelled").
relationships
Garante integridade referencial — todo customer_id em orders deve existir na tabela customers (como uma FK).
YAML — models/staging/schema.yml
version: 2

models:
  - name: stg_orders
    description: "Pedidos limpos e padronizados"
    columns:
      - name: order_id
        description: "Chave primária do pedido"
        tests:
          - unique
          - not_null

      - name: customer_id
        tests:
          - not_null
          - relationships:
              to: ref('stg_customers')
              field: customer_id

      - name: status
        tests:
          - accepted_values:
              values: ['paid', 'pending', 'cancelled', 'refunded']

      - name: total
        tests:
          - not_null
          - dbt_utils.accepted_range:    # pacote dbt_utils — teste extra
              min_value: 0
Testes customizados (singular tests) — SQL que vira asserção

Quando os testes genéricos não cobrem uma regra de negócio específica, você escreve um teste singular: uma query SQL que deve retornar zero linhas. Se retornar alguma linha, o teste falha.

SQL — tests/assert_no_negative_revenue.sql
-- tests/assert_no_negative_revenue.sql
-- Regra de negócio: receita diária nunca pode ser negativa
-- Se esta query retornar QUALQUER linha, o teste FALHA

SELECT
    order_date,
    receita_total
FROM {{ ref('fct_revenue_daily') }}
WHERE receita_total < 0
Quando usar cada tipo: testes genéricos (unique, not_null) para validações estruturais simples e reutilizáveis. Testes singulares para regras de negócio específicas daquele projeto — "receita nunca negativa", "estoque nunca abaixo de zero", "data de entrega sempre depois da data de pedido".
Rodando o dbt — comandos essenciais
Bash — comandos dbt
# Rodar TODOS os modelos (na ordem correta do DAG)
dbt run

# Rodar apenas um modelo específico e suas dependências
dbt run --select fct_revenue_daily

# Rodar um modelo e tudo que depende dele (+) ou que ele depende (+...)
dbt run --select stg_orders+      # stg_orders + tudo que depende dele
dbt run --select +fct_revenue_daily  # tudo que fct_revenue_daily precisa

# Rodar TODOS os testes
dbt test

# Rodar testes de um modelo específico
dbt test --select stg_orders

# Gerar documentação + visualizar o DAG no navegador
dbt docs generate
dbt docs serve

# Rodar tudo de uma vez: modelos + testes
dbt build
dbt build é o comando mais usado em produção: ele roda cada modelo e, imediatamente após, os testes daquele modelo — interrompendo o pipeline se um teste falhar, antes de prosseguir para os modelos que dependem dele. Isso evita propagar dados ruins pelo DAG inteiro.
🙋 Quiz — você precisa garantir que a coluna "discount_percent" nunca seja maior que 100. Qual abordagem é mais adequada?
A
Usar o teste genérico "unique" na coluna
B
Usar o teste genérico "relationships" apontando para a tabela de descontos
C
Escrever um teste singular — uma query SQL que retorna linhas onde discount_percent > 100, esperando zero resultados
D
Não é possível testar isso no dbt — precisa de validação manual

Estrutura de diretórios — projeto dbt
meu_projeto_dbt/
├── dbt_project.yml          # configuração principal
├── profiles.yml             # credenciais de conexão (fora do projeto, em ~/.dbt/)
├── models/
│   ├── staging/
│   │   ├── stg_orders.sql
│   │   ├── stg_customers.sql
│   │   ├── stg_products.sql
│   │   └── schema.yml        # testes e descrições da camada staging
│   ├── intermediate/
│   │   └── int_orders_enriched.sql
│   └── marts/
│       ├── fct_revenue_daily.sql
│       ├── dim_customers.sql
│       └── schema.yml        # testes e descrições da camada marts
├── tests/
│   └── assert_no_negative_revenue.sql   # teste singular customizado
└── sources.yml              # declaração das tabelas brutas (zona Raw)
Declarando as fontes (sources.yml)
YAML — models/sources.yml
version: 2

sources:
  - name: raw
    description: "Dados brutos da zona Raw do Data Lake"
    tables:
      - name: orders
        description: "Pedidos brutos, sem transformação"
      - name: customers
        description: "Cadastro de clientes"
      - name: products
        description: "Catálogo de produtos"
Por que declarar sources? Isso permite usar {{ source('raw', 'orders') }} em vez do nome literal da tabela. Se a tabela física mudar de nome ou esquema, você só ajusta em um lugar — e o dbt também rastreia a "freshness" (atualidade) dos dados de origem automaticamente.

O que você precisa ter gravado
ETL vs ELT

ETL transforma antes de carregar (servidor dedicado). ELT carrega bruto e transforma dentro do warehouse. ELT domina hoje.

Camadas dbt

Staging (1:1 com fonte) → Intermediate (joins) → Marts (fato/dimensão). Espelha Raw/Silver/Gold.

ref() e source()

source() aponta para tabela bruta. ref() aponta para outro modelo dbt — gera o DAG de dependências automaticamente.

Testes

Genéricos: unique, not_null, accepted_values, relationships. Singulares: SQL customizado que deve retornar zero linhas.
