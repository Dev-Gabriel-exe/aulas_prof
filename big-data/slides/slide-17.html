O problema que os Feature Stores resolvem
Semana 16

Você aprendeu a catalogar e rastrear dados de forma geral. Agora vamos ao caso específico de dados usados para treinar e servir modelos de Machine Learning — onde surgem problemas únicos que um Data Catalog comum não resolve sozinho.

Treino-serving skew — o bug mais silencioso de ML em produção

Imagine um modelo de detecção de fraude treinado com uma feature "número de transações nas últimas 24h", calculada em um notebook Python com pandas. Em produção, a mesma feature precisa ser calculada em tempo real, em uma API — muitas vezes reimplementada do zero em outra linguagem ou lógica.

❌ Sem Feature Store
  • Lógica da feature duplicada: uma versão no notebook de treino, outra na API de serving
  • Pequenas diferenças de implementação geram "treino-serving skew" — o modelo se comporta diferente em produção
  • Sem reuso: cada time de ML recalcula as mesmas features do zero
✅ Com Feature Store
  • Uma ÚNICA definição de feature, usada tanto no treino quanto no serving
  • Garantia de consistência entre o valor histórico (treino) e o valor em tempo real (produção)
  • Features viram ativos reutilizáveis entre times e projetos de ML
Definição: um Feature Store é uma camada de armazenamento e serviço que centraliza a definição, o cálculo e o acesso a features de Machine Learning, garantindo que o mesmo valor esteja disponível tanto para treinar modelos (em lote, no passado) quanto para servir predições (em tempo real, no presente).
Por que isso é "Big Data para IA"

Calcular features como "total gasto pelo cliente nos últimos 90 dias" ou "número de cliques na última hora" sobre milhões de clientes exige exatamente as ferramentas que você já domina: Spark para processamento em larga escala (Semanas 8-10), Delta Lake para armazenamento confiável (Semana 14) e, opcionalmente, Structured Streaming para features em tempo real (Semana 15).

O Feature Store não substitui nada que você aprendeu — ele organiza e serve o que o Spark já calcula, adicionando a camada de consistência entre treino e produção que faltava.
🙋 Quiz — o que é "treino-serving skew"?
A
Quando o modelo demora mais para treinar do que para servir predições
B
Quando a lógica de cálculo de uma feature difere entre o ambiente de treino e o de produção, causando comportamento inconsistente do modelo
C
Um tipo de overfitting causado por poucos dados de treino
D
A diferença de custo entre treinar na nuvem e servir localmente

Feast — offline store, online store e feature views
A arquitetura de duas camadas do Feast

Feast (Feature Store) é uma ferramenta open-source que implementa a ideia de Feature Store com duas camadas de armazenamento complementares:

Fonte de dados
Delta Lake / Data Warehouse
↓ features calculadas com Spark ↓
Offline Store
histórico completo
→
Online Store
último valor, baixa latência
Treino de modelo
consome Offline Store
Predição em produção
consome Online Store
CamadaPropósitoTecnologia típica
Offline StoreHistórico completo de features, para gerar datasets de treino (point-in-time correct)Delta Lake, BigQuery, Parquet
Online StoreApenas o valor MAIS RECENTE de cada feature, consultado com latência de milissegundosRedis, DynamoDB, SQLite (dev)
Point-in-time correctness: ao montar o dataset de treino, o Feast garante que o valor da feature usado para cada exemplo é exatamente o valor que existia NAQUELE momento no passado — evitando "vazamento de futuro" (usar informação que só existiria depois do evento que você está tentando prever).
Definindo uma feature view no Feast
Python — feature_repo/features.py
from feast import FeatureView, Field, FileSource
from feast.types import Float32, Int64
from datetime import timedelta

# Fonte: onde o Feast busca o histórico das features (offline)
fonte_pedidos = FileSource(
    path="s3://meu-bucket/delta/features_cliente/",
    timestamp_field="event_timestamp",
)

# FeatureView: agrupa features relacionadas de uma mesma entidade
cliente_features = FeatureView(
    name="cliente_features",
    entities=["cliente_id"],
    ttl=timedelta(days=90),
    schema=[
        Field(name="total_gasto_90d", dtype=Float32),
        Field(name="qtd_pedidos_90d", dtype=Int64),
        Field(name="ticket_medio_90d", dtype=Float32),
    ],
    source=fonte_pedidos,
)
Python — consumindo features (treino e serving)
from feast import FeatureStore

store = FeatureStore(repo_path="feature_repo/")

# TREINO — busca histórico com point-in-time correctness (offline store)
dataset_treino = store.get_historical_features(
    entity_df=df_com_timestamps_e_labels,
    features=["cliente_features:total_gasto_90d", "cliente_features:qtd_pedidos_90d"],
).to_df()

# SERVING — busca o valor mais recente com baixa latência (online store)
features_producao = store.get_online_features(
    features=["cliente_features:total_gasto_90d", "cliente_features:qtd_pedidos_90d"],
    entity_rows=[{"cliente_id": 12345}],
).to_dict()
Repare: a mesma definição de feature (cliente_features) é usada nas duas chamadas — get_historical_features para treino e get_online_features para produção. É essa reutilização que elimina o treino-serving skew.

Features em larga escala com Spark, MLlib e MLflow
Calculando features agregadas com PySpark

O cálculo das features em si usa exatamente as ferramentas das Semanas 8-10 — groupBy, window functions, joins. A diferença é que o resultado é desenhado para alimentar um Feature Store, com uma coluna de timestamp do evento.

Python — cálculo de features com PySpark
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, sum, count, avg, current_timestamp

spark = SparkSession.builder.appName("feature-engineering").getOrCreate()

pedidos = spark.read.format("delta").load("s3://.../delta/pedidos/")

# Filtra últimos 90 dias e agrega por cliente
features_cliente = (pedidos
    .filter(col("data_pedido") >= date_sub(current_timestamp(), 90))
    .groupBy("cliente_id")
    .agg(
        sum("total").alias("total_gasto_90d"),
        count("pedido_id").alias("qtd_pedidos_90d"),
        avg("total").alias("ticket_medio_90d"),
    )
    .withColumn("event_timestamp", current_timestamp()))

# Grava como tabela Delta — esta é a "fonte" que o Feast vai ler
features_cliente.write.format("delta").mode("overwrite") \
    .save("s3://meu-bucket/delta/features_cliente/")
Conexão direta: este job de features roda exatamente como qualquer outro job Spark — pode ser orquestrado pelo Airflow (Semana 12) para recalcular diariamente, e gravado em Delta Lake (Semana 14) para ganhar as garantias ACID que você já conhece.
Spark MLlib — treinando modelos no mesmo cluster dos dados

MLlib é a biblioteca de Machine Learning distribuído do Spark. A ideia central: em vez de extrair dados do cluster para treinar em um notebook local (o que não escala para bilhões de linhas), o treino acontece diretamente onde os dados já estão, em paralelo.

Python — pipeline MLlib simples
from pyspark.ml.feature import VectorAssembler
from pyspark.ml.classification import LogisticRegression
from pyspark.ml import Pipeline

# VectorAssembler junta as colunas de features em um único vetor
assembler = VectorAssembler(
    inputCols=["total_gasto_90d", "qtd_pedidos_90d", "ticket_medio_90d"],
    outputCol="features"
)

modelo = LogisticRegression(featuresCol="features", labelCol="churn")

pipeline = Pipeline(stages=[assembler, modelo])
pipeline_ajustado = pipeline.fit(dataset_treino_spark)

predicoes = pipeline_ajustado.transform(dataset_teste_spark)
MLlib é mais indicado quando o VOLUME de dados de treino é o gargalo (bilhões de linhas). Para datasets que cabem em uma máquina, bibliotecas como scikit-learn continuam sendo mais simples e com mais recursos disponíveis.
Integração com MLflow — rastreando o experimento completo

Combinando tudo: o MLflow (rastreamento de experimentos) registra não só métricas e hiperparâmetros do modelo, mas também qual versão das features do Feast foi usada no treino — fechando o ciclo de reprodutibilidade.

Python — MLflow + Feast + MLlib
import mlflow

with mlflow.start_run():
    dataset_treino = store.get_historical_features(
        entity_df=entity_df, features=lista_features
    ).to_df()

    modelo = pipeline.fit(dataset_treino)

    mlflow.log_param("features_usadas", lista_features)
    mlflow.log_metric("auc", auc_score)
    mlflow.spark.log_model(modelo, "modelo_churn")
Fechando o ciclo: Spark calcula as features (Semanas 8-10) → Delta Lake armazena com ACID (Semana 14) → Feast serve com consistência treino/produção → MLflow rastreia qual combinação de features + hiperparâmetros gerou qual modelo. Esse é o pipeline completo de "Big Data para IA".

Revisão rápida — os 4 temas da P3
Semana 14 — Delta Lake
ACID via _delta_log; leitura sempre passa pelo transaction log
MERGE INTO = upsert atômico; time travel via versionAsOf/timestampAsOf
Schema enforcement é padrão; mergeSchema=true habilita evolution
OPTIMIZE+ZORDER aceleram leitura; VACUUM remove arquivos órfãos (cuidado com o histórico)
Semana 15 — Structured Streaming
Stream = tabela infinita processada em micro-batches
Output modes: append (só novo), complete (tudo), update (mudanças)
Tumbling window = sem sobreposição; sliding window = com sobreposição
Watermark tolera atraso e viabiliza append em agregações
Semana 16 — Governança
Data Catalog guarda metadados técnicos, de negócio, operacionais e de governança
Linhagem upstream = de onde vem; downstream = o que depende
Classificação identifica PII antes de aplicar qualquer proteção
LGPD exige minimização, anonimização e rastreabilidade para direito ao esquecimento
Semana 17 — Big Data para IA
Feature Store elimina o treino-serving skew com uma única definição de feature
Offline store = histórico (treino); online store = valor mais recente (serving)
Point-in-time correctness evita vazamento de dados do futuro
MLlib treina distribuído; MLflow rastreia features + modelo

O que você precisa ter gravado antes da P3
Treino-serving skew

Inconsistência entre a feature calculada no treino e a calculada em produção. Feature Store elimina isso com definição única.

Offline vs Online

Offline store guarda histórico completo (treino, point-in-time correct). Online store guarda só o valor mais recente (serving, baixa latência).

Spark + MLlib

Features calculadas em larga escala com o mesmo Spark das Semanas 8-10. MLlib treina modelos distribuídos quando o dado não cabe em uma máquina.

MLflow
