Por que precisamos de arquiteturas específicas para Big Data?
Semana 1

Na semana passada entendemos que Big Data nasce quando os dados ultrapassam a capacidade de uma máquina. Agora a pergunta é: como você desenha um sistema que processa bilhões de eventos com confiabilidade, baixa latência e custo razoável? É disso que tratam as arquiteturas de Big Data.

O problema que precisava ser resolvido

Imagine que você é engenheiro de dados no Twitter em 2010. Seu sistema precisa responder a uma pergunta simples a qualquer momento: "quantos tweets sobre o assunto X foram feitos nas últimas 24 horas?".

Parece fácil — mas há um problema duplo. Se você calcular em tempo real, qualquer falha no sistema perde eventos para sempre e os resultados ficam imprecisos. Se você calcular em batch, o resultado fica desatualizado por horas. Como ter os dois ao mesmo tempo?

O dilema fundamental do Big Data: precisão histórica (que exige reprocessar todos os dados) versus baixa latência (que exige processar em tempo real). Durante anos, acreditava-se que era impossível ter os dois. As arquiteturas que veremos hoje são a solução para esse dilema.
CAP Theorem — a lei física do Big Data

Antes de entender as arquiteturas, é essencial conhecer o Teorema CAP (Brewer, 2000). Ele afirma que qualquer sistema de dados distribuído pode garantir no máximo dois dos três atributos ao mesmo tempo:

🔒
Consistency (C)
Toda leitura recebe o dado mais atualizado, ou um erro. Todos os nós veem os mesmos dados ao mesmo tempo.
⚡
Availability (A)
Todo pedido recebe uma resposta (não necessariamente a mais atualizada). O sistema nunca fica indisponível.
🔀
Partition Tolerance (P)
O sistema continua funcionando mesmo quando partes da rede falham ou ficam isoladas.
Na prática: em sistemas distribuídos, falhas de rede (P) são inevitáveis — você precisa tolerar partições. Então a escolha real é entre CP (consistência + tolerância, como HBase e Zookeeper) ou AP (disponibilidade + tolerância, como Cassandra e DynamoDB). Essa escolha guia o design das arquiteturas Lambda e Kappa.

Arquitetura Lambda — precisão histórica com latência real
Origem: Nathan Marz e o problema do Twitter

Em 2011, o engenheiro Nathan Marz — trabalhando no Twitter — publicou a solução para o dilema batch vs. real-time. A ideia central era elegante: não escolha entre os dois — use os dois ao mesmo tempo.

A Arquitetura Lambda divide o sistema em três camadas independentes que trabalham em paralelo. O nome vem da letra grega λ (lambda), que visualmente representa a bifurcação do fluxo de dados em dois caminhos que se reencontram no final.

Fórmula Lambda: resultado da query = batch view + real-time view. A camada batch garante precisão histórica; a camada de velocidade preenche o gap dos dados recentes.

Vantagens da Lambda
  • Tolerância a falhas: se o speed layer cair, o batch ainda cobre tudo
  • Precisão histórica: o batch reprocessa dados com lógica atualizada a qualquer momento
  • Baixa latência: o speed layer entrega resultados em segundos
  • Reprocessamento fácil: basta rodar novamente o job batch com nova lógica
  • Escalabilidade independente: batch e speed escalam separadamente
Desvantagens da Lambda
  • Duplicação de código: a mesma lógica precisa ser implementada em batch E em streaming
  • Alta complexidade: dois sistemas completamente diferentes para manter e debugar
  • Sincronização difícil: garantir que batch view e speed view retornam resultados consistentes é desafiador
  • Custo elevado: infraestrutura dobrada (cluster Hadoop + cluster Storm/Spark Streaming)
Quem usa Lambda na prática
🐦 Twitter
Análise de trending topics: batch diário para histórico completo + speed layer para "trending agora" em tempo real.
🛒 Alibaba
Recomendação de produtos: batch recalcula perfis de usuário toda noite + speed layer atualiza preferências durante a sessão.
📊 LinkedIn
Analytics de engajamento: batch gera relatórios diários de alcance + speed layer mostra visualizações em tempo real do perfil.

Aula 3 · ~50 minutos
Arquitetura Kappa — tudo é um stream
Origem: Jay Kreps e a crítica à Lambda

Em 2014, Jay Kreps — co-criador do Apache Kafka e fundador da Confluent — publicou um artigo provocador: "Por que eu não gosto da Arquitetura Lambda". A crítica central era simples e cirúrgica:

Argumento de Kreps: "Manter dois sistemas (batch e streaming) que fazem a mesma coisa é o pior dos dois mundos: dobro do código, dobro dos bugs, dobro da complexidade operacional. E se você tem Kafka, pode fazer tudo com streaming — inclusive reprocessar dados históricos."

A proposta da Arquitetura Kappa é radical na sua simplicidade: elimine completamente o batch layer. Tudo — dados históricos e dados novos — é processado como um stream. O Kafka armazena o histórico completo e você pode "rebobinar" e reprocessar quando precisar.

Arquitetura Kappa — visão geral
ENTRADA
Fonte de dados (todos os eventos)
APIs · Sensores · Bancos · Logs · Aplicações
Apache Kafka — o Log Imutável Universal
Armazena TODOS os eventos em ordem cronológica (log imutável)
Retenção configurável: de horas a anos (ou infinita com tiered storage)
Para reprocessar: leia o tópico desde o offset 0 com nova versão do código
Múltiplos consumidores independentes podem ler o mesmo tópico
Stream Processing — UMA única camada
Processa eventos em tempo real continuamente
Para atualizar lógica: sobe nova versão, rebobina Kafka
Mantém estado (agregações, joins) em memória ou RocksDB
Apache Flink Spark Structured Streaming Kafka Streams
SERVING Banco de dados de resultados
Cassandra · Redis · DynamoDB · Druid
Query em tempo real — único ponto de resposta
Como funciona o reprocessamento na Kappa

A grande dúvida sobre a Kappa é sempre: "e se eu precisar corrigir um bug na lógica de processamento e recalcular dados históricos?". Na Lambda isso é fácil — basta rodar o batch novamente. Na Kappa, a resposta está no Kafka:

Reprocessamento na Arquitetura Kappa
# Situação: bug descoberto na lógica de cálculo de receita
# Kafka retém TODOS os eventos dos últimos 90 dias

# Passo 1: subir a versão nova do processador em paralelo
# (sem derrubar a versão antiga ainda)
job_v2 = StreamingJob(
  kafka_topic="transactions",
  start_offset=0,          # ← começa do início do histórico!
  output_table="revenue_v2",  # ← tabela separada para não afetar prod
  logic="fixed_revenue_calculation"
)

# Passo 2: aguardar o job v2 alcançar o tempo real (catch-up)
# Kafka processa 90 dias de eventos em ~2-4 horas com cluster adequado

# Passo 3: quando v2 alcançar o tempo real, fazer o swap
swap_traffic(old="revenue_v1", new="revenue_v2")
decommission(job_v1)

# Resultado: dados históricos reprocessados com lógica correta
# Sem downtime. Sem perda de dados. Sem sistema batch separado.
A chave do reprocessamento Kappa: o Kafka é um log imutável e ordenado. Você pode "rebobinar" para qualquer ponto no tempo e reproduzir todos os eventos. É como ter um VCR dos seus dados — você pode assistir do começo quantas vezes quiser.

Aula 4 · ~50 minutos
Quando usar cada arquitetura — e o que veio depois
Guia de decisão: Lambda, Kappa ou nenhuma?
Use Lambda quando:
Dados precisam ser corrigidos retroativamente (ex: preços de produtos mudam e afetam cálculos passados)
Precisão histórica absoluta é crítica (ex: compliance financeiro, dados fiscais)
Você precisa reprocessar anos de dados com frequência
Equipe tem expertise em Hadoop/Spark batch separado
Dados chegam em formatos muito diferentes (batch files + streaming)
Use Kappa quando:
Eventos são imutáveis por natureza (cliques, transações, logs de sistema)
Equipe é pequena e complexidade operacional é preocupação real
Kafka já está na stack e o volume de retenção é gerenciável
Latência muito baixa é obrigatória em todo o sistema
Startup ou empresa que ainda está escalando a operação de dados
Use Lakehouse quando:
Quer unificar Data Lake e Data Warehouse em uma só camada
Precisa de ACID transactions no Data Lake (Delta Lake, Iceberg)
Quer eliminar a complexidade das duas arquiteturas acima
Equipe usa Databricks, Snowflake ou está no ecossistema moderno
Projetos novos (2020+) — o padrão de mercado atual
Atenção: essas arquiteturas são para problemas de escala real. Se sua empresa tem menos de 1–2 TB de dados e uma equipe pequena, um Data Warehouse comum (BigQuery, Redshift, Snowflake) provavelmente resolve com menos custo e complexidade.
Arquiteturas modernas — o que vem depois de Lambda e Kappa

Em 2019, a Databricks propôs o conceito de Data Lakehouse — uma arquitetura que tenta combinar as melhores características do Data Lake (barato, flexível, suporta todos os formatos) com as do Data Warehouse (ACID, performance, schema forte). O Delta Lake é a tecnologia que torna isso possível.

🏞️ Data Lake (tradicional)
  • Barato — armazenamento em S3/HDFS
  • Flexível — qualquer formato (Parquet, JSON, CSV)
  • Sem ACID — dados inconsistentes são comuns
  • Sem schema enforcement — "Data Swamp" é um risco real
🏗️ Data Lakehouse (Delta Lake)
  • Barato — ainda usa S3/HDFS por baixo
  • ACID transactions — updates e deletes confiáveis
  • Schema enforcement + time travel
  • Batch e streaming na mesma tabela
Tendência atual: a maioria dos novos projetos de Big Data em 2024 começa com uma arquitetura Lakehouse (Delta Lake ou Apache Iceberg) em vez de Lambda ou Kappa puras. Vamos explorar Delta Lake em profundidade na Semana 14.

Atividade · Diagrama + Análise em grupo
Escolha justificada de arquitetura para cenários reais
Atividade em duplas. Cada dupla recebe os três cenários abaixo e deve: (1) desenhar em papel o fluxo simplificado da arquitetura escolhida e (2) justificar a escolha com base nos critérios discutidos em aula. Ao final, cada dupla apresenta um dos cenários.
Cenário A — Plataforma de Streaming de Música

Uma empresa do porte do Spotify tem 100 milhões de usuários ativos. O sistema precisa: (1) recomendar músicas em tempo real durante a sessão do usuário, (2) gerar relatórios semanais de artistas mais ouvidos por região, e (3) atualizar o modelo de recomendação com novos dados de escuta a cada hora. Qual arquitetura você escolheria e por quê?

Análise técnica
Solução: Kappa ou Lakehouse
  • Por que não Lambda? Os eventos de escuta são imutáveis — uma música ouvida não muda retroativamente. Não precisa de batch reprocessamento frequente.
  • Kappa é adequada: Kafka retém histórico de escutas, Flink/Spark Streaming atualiza preferências em tempo real, servindo recomendações instantâneas.
  • Para relatórios semanais: a mesma camada de streaming pode materializar views diárias em um banco analítico (Druid ou BigQuery) para os relatórios.
  • Alternativa moderna: Lakehouse com Delta Lake para os dados históricos + Spark Streaming para o tempo real — arquitetura atual do Spotify.
Cenário B — Sistema Bancário de Detecção de Fraude

Um banco processa 2 milhões de transações por dia. O modelo de fraude precisa: (1) analisar cada transação em menos de 150ms antes de aprovar, (2) recalcular scores de risco quando as regras de compliance mudam (o que acontece mensalmente), e (3) manter histórico de 5 anos para auditorias regulatórias. Qual arquitetura e por quê?

Análise técnica
Solução: Lambda ou Lakehouse
  • Por que Lambda faz sentido aqui? As regras de compliance mudam e precisam ser aplicadas retroativamente ao histórico de 5 anos. Reprocessamento batch é necessário e simples na Lambda.
  • Speed Layer: Kafka + Flink analisam cada transação em <150ms usando o modelo mais recente carregado em memória.
  • Batch Layer: re-roda mensalmente quando as regras mudam, recalculando scores históricos para conformidade regulatória.
  • Alternativa moderna: Lakehouse com Delta Lake (time travel para auditorias + ACID) + Flink para o tempo real. Mais simples de operar que Lambda clássica.
Cenário C — Startup de E-commerce em Crescimento

Uma startup de e-commerce tem 500 mil pedidos por mês, espera crescer 10x em 2 anos e precisa de: (1) dashboard de vendas atualizado a cada 5 minutos para o time de operações, (2) análise de cohort de clientes (comportamento ao longo do tempo) para o time de marketing, e (3) budget de engenharia limitado com equipe de 3 engenheiros de dados. O que você recomenda?

Decisão estratégica
Solução: Nenhuma das duas — ainda
  • Nem Lambda nem Kappa são necessárias: com 500k pedidos/mês, um PostgreSQL bem configurado ou um Data Warehouse gerenciado (BigQuery free tier) resolve tudo com folga.
  • Recomendação real: dbt + BigQuery (ou Redshift) para transformações, Airbyte para ingestão, Metabase para dashboards. Stack moderna, custo baixo, 1 engenheiro consegue manter.
  • Dashboard a cada 5 min: BigQuery Scheduled Queries ou Airbyte sync incremental resolve sem Kafka.
  • Quando migrar para Lakehouse/Kappa? Quando atingir ~50TB de dados ou quando o time crescer para 8+ engenheiros de dados. Antes disso, é overengineering.
Lição principal deste cenário: a melhor arquitetura é aquela que resolve o problema atual com o menor custo e complexidade possíveis. Big Data é ferramenta para problema de Big Data — não para todo problema.

CAP Theorem

Sistemas distribuídos garantem no máximo dois de três: Consistency, Availability, Partition Tolerance.

Lambda

3 camadas: Batch (histórico preciso) + Speed (tempo real) + Serving (merge). Complexo, poderoso para reprocessamento.

Kappa

Tudo é stream. Kafka como log imutável. Mais simples que Lambda. Reprocessa rebobinando o Kafka.

Lakehouse

Arquitetura moderna (Delta Lake / Iceberg). ACID no Data Lake. Tende a substituir Lambda e Kappa em novos projetos.
