Por que um sistema de arquivos comum não aguenta Big Data
Semana 3

Na semana passada vimos que um Data Lake armazena dados em zonas (Raw/Silver/Gold), em formatos como Parquet e Avro. Mas surgiu uma pergunta implícita: onde fisicamente esses arquivos ficam? A resposta para escala de TB e PB é o HDFS — ou sua equivalente em nuvem.

O problema que precisava ser resolvido

Imagine que você tem 500 TB de dados de logs para analisar. Cada servidor de alta performance em 2024 tem no máximo 20–30 TB de disco. Você poderia comprar um servidor maior, mas há um limite físico — e o custo cresce exponencialmente.

A solução óbvia é: distribuir os dados entre muitos servidores comuns e baratos. Mas isso cria novos problemas que o sistema de arquivos tradicional (ext4, NTFS) não resolve:

❓ Como localizar um arquivo?
Se um arquivo está dividido entre 50 servidores, quem sabe onde cada pedaço está? Como você monta o arquivo completo?
❓ E se um servidor cair?
Em um cluster de 200 servidores, é estatisticamente certo que algum vai falhar. Como não perder dados?
❓ Como processar eficientemente?
Ler 500 TB pela rede é lento. Como fazer o processamento acontecer próximo dos dados?
O HDFS responde às três perguntas: um servidor central (NameNode) sabe onde cada pedaço de cada arquivo está; replicação em 3 cópias garante que a falha de um servidor não perde dados; e o Spark consegue processar os dados onde eles estão, sem mover tudo para uma máquina central.

Arquitetura HDFS — NameNode, DataNode, Blocos e Replicação
Visão geral: arquitetura master-worker

O HDFS segue um modelo master-worker (também chamado de master-slave): um nó central coordena e muitos nós de trabalho armazenam os dados. Essa separação é fundamental para entender como o sistema funciona.

NameNode — o índice do sistema

O NameNode não armazena dados. Ele armazena apenas metadados — informações sobre os arquivos. Todo esse mapeamento fica em memória RAM para acesso ultrarrápido.

  • FsImage: snapshot completo do sistema de arquivos no disco
  • EditLog: log de todas as operações desde o último snapshot
  • Sabe: "arquivo /user/dados.csv tem blocos B001, B002, B003"
  • Sabe: "bloco B001 está nos DataNodes DN1, DN2, DN4"
  • NÃO armazena o conteúdo dos arquivos
Single Point of Failure: se o NameNode cair sem proteção, o cluster inteiro para — os DataNodes têm os dados mas ninguém sabe onde. Solução: Secondary NameNode (checkpoint) ou HA NameNode (active/standby com ZooKeeper).
DataNode — onde os dados vivem

Os DataNodes são os servidores de armazenamento. Eles armazenam os blocos físicos e reportam sua saúde ao NameNode continuamente.

  • Heartbeat: envia sinal ao NameNode a cada 3 segundos
  • Block report: lista todos os blocos que possui periodicamente
  • Se o NameNode não recebe heartbeat em 10 minutos → considera o DataNode morto e inicia re-replicação
  • Escala horizontal: adicionar DataNodes aumenta capacidade do cluster sem downtime
Commodity hardware: DataNodes são intencionalmente servidores comuns e baratos. A falha é esperada e tratada pelo sistema — não é um problema, é uma feature.
Blocos — por que 128 MB?

Um arquivo de 1 GB no HDFS é dividido em 8 blocos de 128 MB (ou 16 blocos de 64 MB na configuração antiga). Mas por que 128 MB e não 4 KB como nos sistemas de arquivos tradicionais?

❌ Blocos pequenos (4 KB como ext4)
  • 1 GB de arquivo = 262.144 blocos
  • NameNode precisa de 262.144 entradas de metadados em RAM
  • Para 500 TB: bilhões de entradas de metadados → NameNode não cabe em RAM
✅ Blocos grandes (128 MB)
  • 1 GB de arquivo = 8 blocos
  • 500 TB = ~4 milhões de blocos — cabe em RAM no NameNode
  • Leitura sequencial eficiente (disco adora ler grandes blocos contíguos)
Replicação — rack awareness e tolerância a falhas

Por padrão, o HDFS mantém 3 cópias de cada bloco (fator de replicação = 3). Mas a distribuição dessas cópias não é aleatória — o HDFS usa Rack Awareness para garantir que uma falha de rack inteiro não cause perda de dados.

Réplica 1 (Primary)
DataNode 1 — Rack A
Mesma máquina do cliente que escreveu (data locality)
Réplica 2
DataNode 3 — Rack B
Rack diferente — se o Rack A cair, essa cópia sobrevive
Réplica 3
DataNode 4 — Rack B
Mesmo rack da réplica 2 — reduz tráfego inter-rack
Lógica do Rack Awareness: 1ª réplica no nó local (ou aleatório no Rack A), 2ª réplica em um DataNode de outro rack (Rack B), 3ª réplica no mesmo rack da 2ª (Rack B, DataNode diferente). Isso garante sobrevivência a falha de um nó OU de um rack inteiro completo.
🙋 Quiz — um DataNode falha e o NameNode não recebe heartbeat por 10 minutos. O que o HDFS faz automaticamente?
A
Suspende todas as operações do cluster até o DataNode ser reiniciado
B
Re-replica os blocos que estavam naquele DataNode para outros nós do cluster
C
Apaga os dados que estavam no DataNode falho permanentemente
D

Aula 3 · ~50 minutos
Como os dados fluem — leitura, escrita e cloud storage
Pipeline de escrita — como um arquivo entra no HDFS
1
Cliente contata o NameNode: "quero escrever o arquivo /dados/vendas.parquet de 384 MB"
2
NameNode responde: "seu arquivo terá 3 blocos. Escreva o Bloco 1 em DN1→DN3→DN4, Bloco 2 em DN2→DN3→DN1, Bloco 3 em DN1→DN4→DN2". Atualiza o EditLog.
3
Cliente escreve em pipeline: envia o Bloco 1 para DN1. DN1 simultaneamente encaminha para DN3. DN3 encaminha para DN4. As 3 réplicas são escritas em paralelo.
4
Acknowledgment reverso: DN4 confirma para DN3 → DN3 confirma para DN1 → DN1 confirma para o Cliente. Só quando o cliente recebe a confirmação dos 3 DataNodes é que o bloco é considerado persistido.
5
Processo se repete para os Blocos 2 e 3. Ao final, o cliente avisa o NameNode: "escrita concluída". NameNode fecha o arquivo nos metadados.
Data locality: note que o Spark, ao processar o arquivo, solicita ao NameNode onde estão os blocos e tenta agendar as tarefas nos próprios DataNodes que hospedam os dados. Isso elimina tráfego de rede e é um dos maiores ganhos de performance do HDFS sobre soluções de rede compartilhada.
Pipeline de leitura — como um arquivo sai do HDFS
1
Cliente solicita ao NameNode: "quero ler /dados/vendas.parquet"
2
NameNode retorna localização dos blocos: "Bloco 1 está em DN1, DN3, DN4 — mais próximo de você é DN1. Bloco 2 está em DN2, DN3, DN1 — mais próximo é DN2." etc.
3
Cliente lê diretamente dos DataNodes: sem passar pelo NameNode. Lê Bloco 1 de DN1, Bloco 2 de DN2, Bloco 3 de DN1. Os DataNodes lidos são os mais próximos da rede (rack awareness).
4
Se um DataNode falhar durante a leitura: o cliente usa a próxima réplica disponível na lista fornecida pelo NameNode. Transparente para a aplicação.
HDFS na nuvem — S3, GCS e Azure ADLS

Desde 2015, uma tendência clara se consolidou: empresas estão substituindo HDFS por object storage em nuvem (AWS S3, Google Cloud Storage, Azure Data Lake Storage). O Apache Spark lê e escreve no S3 exatamente como leria no HDFS, via o conector s3a://.

Por que o mercado está migrando para S3? Custo e operação. Com HDFS você paga o cluster 24/7 mesmo que o Spark só rode 2h por dia. Com S3 + Spark no EMR/Databricks, você paga o cluster só quando ele roda e paga storage separado. Para a maioria das empresas, é 3–10x mais barato.

Comandos HDFS na prática — subindo o cluster com Docker
Comandos essenciais — hdfs dfs

Todos os comandos HDFS seguem o padrão hdfs dfs -comando /caminho. A sintaxe é intencionalmente parecida com o shell Unix.

HDFS — Comandos essenciais
# ── Navegação e listagem ────────────────────────────────────
hdfs dfs -ls /                          # listar diretório raiz
hdfs dfs -ls -R /user/dados             # listar recursivamente
hdfs dfs -du -s -h /datalake/silver    # tamanho total (human-readable)
hdfs dfs -df -h                          # espaço livre no cluster

# ── Criar e remover diretórios ─────────────────────────────
hdfs dfs -mkdir -p /datalake/raw/orders   # criar (com -p cria pais)
hdfs dfs -rm -r /datalake/raw/tmp        # remover recursivamente

# ── Upload e download ──────────────────────────────────────
hdfs dfs -put vendas.csv /datalake/raw/   # upload local → HDFS
hdfs dfs -put -f vendas.csv /datalake/raw/ # forçar sobrescrita
hdfs dfs -get /datalake/silver/orders.parquet ./  # download HDFS → local
hdfs dfs -copyToLocal /hdfs/file.txt ./  # equivalente ao -get

# ── Leitura e manipulação ─────────────────────────────────
hdfs dfs -cat /datalake/raw/config.txt    # ler conteúdo
hdfs dfs -tail /datalake/raw/events.log   # últimas linhas
hdfs dfs -cp /datalake/raw/ /backup/raw/   # copiar dentro do HDFS
hdfs dfs -mv /datalake/raw/tmp/ /datalake/archive/  # mover

# ── Verificação e administração ───────────────────────────
hdfs fsck /datalake -files -blocks       # verificar saúde do FS
hdfs dfsadmin -report                   # status do cluster
hdfs dfsadmin -safemode get             # verificar modo seguro
Subindo HDFS local com Docker Compose

Vamos criar um cluster HDFS mínimo (1 NameNode + 2 DataNodes) em Docker para praticar os comandos sem precisar de infraestrutura real.

docker-compose.yml — cluster HDFS mínimo
version: '3.8'
services:

  namenode:
    image: apache/hadoop:3
    container_name: namenode
    hostname: namenode
    environment:
      - ENSURE_NAMENODE_DIR=/tmp/hadoop-root/dfs/name
    command: ["hdfs", "namenode"]
    ports:
      - "9870:9870"   # Web UI do NameNode
      - "9000:9000"   # RPC port
    volumes:
      - namenode_data:/tmp/hadoop-root/dfs/name
      - ./hadoop.env:/opt/hadoop/etc/hadoop/core-site.xml

  datanode1:
    image: apache/hadoop:3
    container_name: datanode1
    command: ["hdfs", "datanode"]
    depends_on: [namenode]
    volumes:
      - datanode1_data:/tmp/hadoop-root/dfs/data

  datanode2:
    image: apache/hadoop:3
    container_name: datanode2
    command: ["hdfs", "datanode"]
    depends_on: [namenode]
    volumes:
      - datanode2_data:/tmp/hadoop-root/dfs/data

volumes:
  namenode_data:
  datanode1_data:
  datanode2_data:

Exercício 2 — Upload e verificação de blocos (12 min)

Análise
Por que o tamanho aparece multiplicado por 3

O HDFS reporta o espaço consumido no disco levando em conta as réplicas. Se seu arquivo tem 10 MB e o fator de replicação é 3, o HDFS ocupa 30 MB no cluster (10 MB × 3 réplicas). O comando hdfs dfs -du -s /path mostra dois valores: o tamanho real do arquivo e o tamanho total no disco (incluindo réplicas). Isso é intencional — o HDFS está sendo transparente sobre o custo real de armazenamento.

Exercício 3 — Desafio: simular falha de DataNode (10 min)

Faça o upload de um arquivo e anote em quais DataNodes seus blocos estão (use hdfs fsck -locations). Em seguida, pare um dos DataNodes com docker stop datanode1. Espere 2 minutos e verifique com hdfs dfsadmin -report. Responda: (a) o que aconteceu com os blocos que estavam no datanode1? (b) o arquivo ainda pode ser lido? Tente com hdfs dfs -cat. (c) suba o datanode1 novamente — o que o cluster faz?

Tolerância a falhas
Comportamento esperado
  • (a) Após ~10 minutos (ou mais rápido em ambiente de teste), o NameNode detecta o DataNode como dead. Inicia re-replicação automática dos blocos que estavam nele para outros DataNodes disponíveis. O fator de replicação volta para 3.
  • (b) Sim! O arquivo ainda pode ser lido imediatamente — as outras 2 réplicas estão em datanode2 (e possivelmente datanode3 se tiver). O hdfs dfs -cat lê das réplicas disponíveis sem interrupção.
  • (c) Quando o datanode1 volta, o NameNode o reconhece pelo heartbeat. Recebe o block report com os blocos que ele ainda tem. Se o fator de replicação foi restaurado durante a ausência, alguns blocos no datanode1 podem ser marcados como "excess" e removidos. O cluster se auto-balanceia.
Conclusão prática: este exercício demonstra o pilar mais importante do HDFS — fault tolerance automático. Nenhuma intervenção humana foi necessária para manter os dados disponíveis durante a falha do DataNode.

NameNode

Só metadados. Todo em RAM. Single point of failure. Conhece onde cada bloco está.

DataNode

Armazena blocos reais. Heartbeat a cada 3s. Se ficar 10 min sem heartbeat → considerado morto.

Blocos + Replicação

128 MB por bloco. Fator 3: réplica 1 local, réplica 2 em outro rack, réplica 3 no mesmo rack da 2ª.

HDFS vs S3

HDFS: compute+storage acoplados, data locality. S3: mais barato, desacoplado, durabilidade 11 9s.
