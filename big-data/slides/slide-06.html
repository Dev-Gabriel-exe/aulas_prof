Os 5 temas — pontos-chave e armadilhas comuns
Semana 1 — Os 5 Vs
Volume: escala que ultrapassa uma máquina → escalonamento horizontal
Velocidade: batch (horas), micro-batch (min), streaming (ms) — latência define a arquitetura
Variedade: estruturado (SQL), semi (JSON/Avro), não estruturado (imagens/texto)
Veracidade: qualidade dos dados → garbage in, garbage out
Valor: o objetivo final — insights, predições, ações
BD Tradicional: até ~TB, schema fixo, ACID. Big Data: TB+, qualquer tipo, escala horizontal
Armadilha: Big Data não é "melhor" que BD tradicional — é para problemas de escala real. 5.000 usuários não justifica Kafka + Spark.
Semana 2 — Lambda e Kappa
Lambda: Batch Layer (histórico) + Speed Layer (tempo real) + Serving Layer (merge)
Fórmula Lambda: resultado = batch view + real-time view
Lambda ✅: reprocessamento fácil, precisão histórica alta
Lambda ❌: dois codebases, complexidade dobrada, custo alto
Kappa: tudo é stream via Kafka, um único codebase
Kappa reprocessa: rebobinando o Kafka (offset=0), nova versão em paralelo, swap
Lakehouse: ACID no Data Lake via Delta Lake — arquitetura moderna
Armadilha: Lambda não é "melhor" que Kappa. Lambda para dados que mudam retroativamente; Kappa para eventos imutáveis.
Semana 3 — Data Lake
DW: schema-on-write, estruturado, ACID, caro — para BI
Data Lake: schema-on-read, qualquer tipo, barato — para ML e analytics
Raw/Bronze: dado bruto, imutável, NUNCA modificar — fonte da verdade
Silver: limpa, validada, deduplicada, Parquet — confusão zero
Gold: agregada, pronta para BI e ML, por caso de uso
Parquet: colunar, compressão excelente, ideal para analytics (Silver/Gold)
Avro: row-based, schema embutido, schema evolution — ideal para Kafka/Raw
Armadilha: bug na Silver? Corrija a lógica e reprocesse a partir da Raw — NUNCA modifique a Raw.
Semana 4 — HDFS
NameNode: só metadados em RAM — sabe onde cada bloco está. SPOF sem HA.
DataNode: armazena blocos. Heartbeat a cada 3s. 10 min sem heartbeat → considerado morto
Bloco: 128 MB — reduz entradas de metadados no NameNode
Replicação (fator 3): réplica 1 no nó local, réplica 2 em outro rack, réplica 3 no mesmo rack da 2ª
Falha de DataNode: NameNode detecta, inicia re-replicação automática
HDFS vs S3: data locality vs custo/operação/durabilidade (11 9s)
Armadilha: se o NameNode cair sem HA, o cluster para — DataNodes têm os dados mas ninguém sabe onde estão.
Semana 5 — Apache Kafka
Broker: servidor Kafka do cluster
Topic: canal nomeado, append-only, imutável
Partition: log ordenado, unidade de paralelismo
Offset: posição da mensagem na partição (começa em 0)
Producer: publica; mesma chave → mesma partição
Consumer: lê, mantém offset, pode rebobinar
Consumer Group: cada partição = 1 consumer por grupo
Grupos diferentes: leem tópico independentemente
At-least-once: commit após processar (padrão)
LAG: mensagens pendentes (LOG-END - CURRENT-OFFSET)
Armadilha: partições > consumers → alguns consumers leem mais de uma. Consumers > partições → consumers extras ficam ociosos.

Uma fintech decide migrar de Lambda para Kappa. O time alega que o Kafka retém 90 dias de eventos. Qual é a principal limitação dessa decisão?
A
Kafka não suporta reprocessamento — é impossível rebobinar o offset
B
A Kappa não garante a ordem das mensagens dentro de uma partição

Semana 3
Um pipeline Kafka ingere eventos de clique de usuários que serão armazenados na zona Raw. Qual formato é mais adequado e por quê?
A
Parquet — colunar, melhor compressão, ideal para todos os contextos
B
Avro — row-based, schema embutido, suporte a schema evolution, padrão do ecossistema Kafka
C
ORC — otimizado para Hive, melhor para streaming

Semana 4
Um cluster HDFS tem 1 NameNode e 6 DataNodes. O DataNode 3 falha. Quais blocos correm risco de perda?
A
Todos os blocos do DataNode 3 — sem o nó, os dados são perdidos
B
Nenhum — com replication-factor=3, cada bloco tem 2 réplicas em outros DataNodes
C
Os blocos cujo Leader estava no DataNode 3 — as réplicas não são suficientes
D
Apenas os blocos que estavam sendo escritos no momento da falha
✅ Alternativa B. Com replication-factor=3 padrão, cada bloco tem 3 cópias em DataNodes diferentes. A falha de 1 DataNode deixa os blocos com 2 réplicas — ainda disponíveis para leitura. O NameNode detecta a falha pelo heartbeat, identifica os blocos que ficaram com apenas 2 réplicas e inicia re-replicação para restaurar o fator 3. Nenhum dado é perdido.

Semana 4
Por que o HDFS usa blocos de 128 MB em vez de blocos de 4 KB como sistemas de arquivos tradicionais?
A
Para melhorar a segurança — blocos grandes são mais difíceis de interceptar
B
Para reduzir o número de entradas de metadados no NameNode (que fica em RAM) e otimizar leituras sequenciais
C
Para facilitar a replicação — blocos grandes são mais fáceis de copiar entre DataNodes
D
Porque os discos dos DataNodes são mais lentos e blocos grandes compensam a latência
✅ Alternativa B. O NameNode mantém todos os metadados em RAM. Com blocos de 4 KB, um arquivo de 1 GB teria 262.144 entradas; com 128 MB, apenas 8. Para 500 TB de dados, blocos pequenos tornariam o NameNode impraticável. Além disso, blocos grandes permitem leitura sequencial eficiente — discos HDD são muito mais rápidos para leituras contínuas do que para muitos acessos aleatórios.

Semana 5
Um tópico Kafka tem 4 partições. O Consumer Group "analytics" tem 6 consumers. Como as partições são distribuídas?
A
Todas as 4 partições são lidas por todos os 6 consumers simultaneamente
B
4 consumers ficam com 1 partição cada; 2 consumers ficam ociosos (idle)
C
Cada partição é dividida igualmente entre os 6 consumers
D
O Kafka cria 2 partições extras para atender os 2 consumers excedentes
✅ Alternativa B. A regra fundamental dos Consumer Groups: cada partição é atribuída a exatamente um consumer dentro do grupo ao mesmo tempo. Com 4 partições e 6 consumers, 4 consumers ficam ativos (um por partição) e 2 ficam completamente ociosos. Para máximo paralelismo, o número de consumers deve ser igual ao número de partições.

Semana 5
Um producer Kafka envia eventos de pedido usando o campo customer_id como chave da mensagem. Qual garantia isso oferece?
A
Todos os pedidos são processados mais rapidamente por ter chave definida
B
Pedidos de diferentes clientes nunca chegam ao mesmo consumer
C
Todos os eventos do mesmo cliente vão para a mesma partição, garantindo a ordem temporal dos eventos por cliente
D
A chave criptografa os dados do pedido para maior segurança
✅ Alternativa C. No Kafka, a chave da mensagem determina a partição de destino (via hash da chave). A mesma chave sempre resulta na mesma partição. Como cada partição é um log ordenado, todos os eventos do cliente C001 (criado → pago → enviado → entregue) chegam na ordem correta ao consumer. Sem chave, os eventos de um mesmo cliente poderiam ir para partições diferentes e chegar fora de ordem.

Semana 5
O Consumer Group "fraud-detector" mostra LAG=500 no tópico "transactions". O que isso significa?
A
O grupo já processou 500 mensagens com sucesso
B
O tópico tem apenas 500 mensagens no total
C
O grupo tem 500 mensagens pendentes de processamento — está 500 offsets atrás do final do log
D
O Kafka está 500ms mais lento do que o normal
✅ Alternativa C. LAG = LOG-END-OFFSET − CURRENT-OFFSET. Um LAG de 500 significa que o grupo está 500 mensagens atrás do final do log — há 500 transações que chegaram ao Kafka mas ainda não foram processadas pelo grupo. Em detecção de fraude, um LAG alto é crítico: significa que transações fraudulentas podem estar passando sem análise. A solução é adicionar mais consumers (até o limite de partições).

Questão Dissertativa 1 — Semanas 2 e 5
Compare as Arquiteturas Lambda e Kappa em termos de: (a) estrutura de camadas, (b) como cada uma realiza o reprocessamento de dados históricos, e (c) em que cenários você escolheria cada uma. Use exemplos concretos.
(a) Estrutura: A Lambda possui três camadas: Batch Layer (processa todos os dados históricos periodicamente, gera "batch views" precisas), Speed Layer (processa apenas dados recentes em tempo real, gera "real-time views") e Serving Layer (combina as duas views para responder queries). A fórmula é: resultado = batch view + real-time view. A Kappa possui uma única camada de streaming — o Kafka funciona como log imutável central e um único job de streaming (Flink/Spark Streaming) processa tanto dados históricos quanto novos eventos.

(b) Reprocessamento: Na Lambda, o reprocessamento é simples — corrige-se o código do job batch e ele é re-executado sobre os dados imutáveis do Batch Layer (HDFS/S3). Na Kappa, o reprocessamento exige: (1) subir nova versão do job consumindo o tópico Kafka desde o offset 0 em uma tabela temporária, (2) aguardar o job atingir o tempo real (catch-up), (3) fazer o swap de tráfego para a nova versão, (4) desligar a versão antiga.

(c) Quando usar: Lambda é preferível quando os dados precisam ser corrigidos retroativamente (preços que mudam e afetam cálculos passados, regras de compliance que se aplicam ao histórico) ou quando a retenção do Kafka não cobre o horizonte histórico necessário. Exemplo: sistema bancário que precisa recalcular scoring de crédito dos últimos 5 anos com nova regulação. Kappa é preferível quando os eventos são imutáveis por natureza, a equipe é pequena e complexidade é uma preocupação real. Exemplo: sistema de rastreamento de entregas — um pacote saiu às 14h e chegou às 18h; esses fatos não mudam retroativamente.
Questão Dissertativa 2 — Semana 4
Explique o papel do NameNode e dos DataNodes no HDFS. O que acontece quando um DataNode falha? E o que acontece quando o NameNode falha? Por que o NameNode é considerado um Single Point of Failure (SPOF)?
NameNode: é o master do cluster HDFS. Armazena exclusivamente metadados em RAM: o mapeamento de cada arquivo para seus blocos (FsImage) e o registro de onde cada bloco está armazenado (block location). Ele NÃO armazena os dados dos arquivos. Todos os clientes consultam o NameNode para saber onde ler ou onde escrever antes de contatar os DataNodes diretamente.

DataNodes: são os workers do cluster. Armazenam fisicamente os blocos de dados. Enviam heartbeat ao NameNode a cada 3 segundos indicando que estão ativos, e um block report periódico listando todos os blocos que possuem.

Falha de DataNode: O NameNode detecta a ausência de heartbeat após 10 minutos. Ele verifica quais blocos que estavam naquele DataNode ficaram com menos de 3 réplicas (fator padrão). Em seguida, instrui outros DataNodes ativos a criar cópias adicionais desses blocos para restaurar o fator de replicação. Nenhum dado é perdido e o sistema continua funcionando transparentemente.

Falha do NameNode: É catastrófica sem HA. Os DataNodes continuam vivos com todos os dados físicos intactos, mas nenhum cliente sabe onde os blocos estão — o único índice existia na memória do NameNode. O cluster fica completamente inacessível. Por isso o NameNode é um SPOF: uma única falha derruba todo o sistema. A solução é HA NameNode (Active/Standby com ZooKeeper e JournalNodes compartilhados), garantindo failover automático em segundos.
Questão Dissertativa 3 — Semana 5
Uma empresa tem um tópico Kafka "transactions" com 6 partições. Três sistemas precisam processar cada transação: detecção de fraude (crítico, baixa latência), analytics (não crítico) e geração de notificações. Como você configuraria os Consumer Groups? Explique o comportamento esperado e como garantir que cada sistema processa 100% das transações.
Configuração: Criar 3 Consumer Groups independentes: "fraud-detector", "analytics" e "notifications". Cada grupo se inscreve no tópico "transactions" de forma totalmente independente.

Comportamento: Como o Kafka mantém o offset por grupo, cada um lê o tópico inteiro de forma independente — como se cada grupo tivesse sua própria cópia dos dados. O grupo fraud-detector no offset 1.000 não interfere no analytics no offset 980 nem em notifications no offset 1.050. Uma mensagem publicada no tópico é consumida pelos 3 grupos separadamente.

Distribuição de partições: Com 6 partições e 3 consumers por grupo (ideal), cada consumer do grupo lê 2 partições. Para fraud-detector com baixa latência, seria adequado ter 6 consumers (1 por partição) para máximo paralelismo.

Garantia de 100% das transações: O Kafka retém as mensagens independentemente do consumo. Cada grupo commita seus próprios offsets. Se o analytics cair por 2 horas, ao voltar, ele retoma do último offset commitado e processa todas as mensagens que chegaram durante a indisponibilidade — sem perda, graças ao LAG. O grupo fraud-detector não é afetado pela indisponibilidade do analytics.
Questão Dissertativa 4 — Semanas 1, 3 e 4
Projete a arquitetura de armazenamento para uma plataforma de streaming de vídeo (estilo Netflix). Considere: vídeos, metadados de filmes, histórico de visualização dos usuários, logs de evento (play/pause/stop) e modelos de recomendação treinados. Para cada tipo de dado, indique: zona do Data Lake (Raw/Silver/Gold), formato de arquivo e justificativa.
Vídeos (arquivos .mp4): Raw — armazenados como recebidos, não estruturados. Formato original (.mp4, .mkv). Sem processamento até que um job de transcodificação precise deles. Não passam pela Silver/Gold — são processados por um pipeline separado de mídia.

Metadados de filmes (título, gênero, elenco, duração): Raw → CSV/JSON; Silver → Parquet (limpo, deduplicado, padronizado em schema fixo); Gold → Parquet (tabela de features para o modelo de recomendação, agregações por gênero/ano). Parquet na Silver/Gold por ser colunar — queries como "todos os filmes de ação de 2020" leem apenas as colunas relevantes.

Histórico de visualização: Raw → Avro (streaming via Kafka); Silver → Parquet (sessões completas, duração calculada, usuário validado); Gold → Parquet (features: % assistido, géneros preferidos, horário de pico). Avro na Raw por chegar via streaming Kafka com schema que pode evoluir.

Logs de evento (play/pause/stop): Raw → Avro (streaming, imutável, alto volume); Silver → Parquet (parseado, sessões identificadas, bots filtrados); Gold → Parquet agregado por feature de engajamento. O volume é enorme (bilhões/dia) — Parquet com compressão Snappy na Silver reduz em 5-8x.

Modelos de recomendação treinados: Ficam em um diretório separado (ex: s3://models/recommender/v1.2/) no formato nativo do framework (pickle, ONNX, TFLite). Versionados separadamente do Data Lake. Promovidos pelo MLflow Model Registry.
