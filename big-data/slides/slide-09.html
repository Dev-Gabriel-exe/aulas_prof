Narrow vs Wide Transformations — a distinção mais importante

Toda transformação do Spark se encaixa em uma de duas categorias, e essa categoria determina se a operação é rápida ou cara:

🟢 Narrow Transformation

Cada partição de entrada contribui para no máximo uma partição de saída. Não precisa mover dados entre máquinas.

  • map(), filter(), select(), withColumn()
  • Executa dentro do mesmo executor, sem rede
  • Muito rápido — sem shuffle
🔴 Wide Transformation

Uma partição de saída depende de múltiplas partições de entrada — possivelmente de outras máquinas. Exige shuffle.

  • groupBy(), join(), distinct(), orderBy()
  • Move dados pela rede entre executors
  • Caro — é a principal causa de jobs lentos
Por que isso importa? Um filter() antes de um join() reduz o volume de dados que precisa ser embaralhado (shuffled) pela rede. Essa é a otimização mais básica e mais impactante em Spark: filtre o máximo possível antes de qualquer wide transformation.
map() vs select() — quando usar cada um

Em DataFrames modernos, você raramente usa map() diretamente (isso é mais comum em RDD). Mas é importante saber a diferença conceitual:

PySpark — map em RDD vs select em DataFrame
# ── RDD: map() aplica função Python a CADA elemento ──────────
# Lento: serializa/desserializa entre JVM e processo Python
rdd2 = rdd.map(lambda row: (row[0], row[1] * 1.1))

# ── DataFrame: select()/withColumn() usa funções nativas do Spark
# Rápido: executa dentro da JVM com Tungsten (sem Python overhead)
df2 = df.withColumn("total_ajustado", col("total") * 1.1)

# Se PRECISAR de lógica Python customizada em DataFrame, use UDF
# (mas UDFs também têm overhead — evite quando possível)
from pyspark.sql.functions import udf
from pyspark.sql.types import StringType

@udf(returnType=StringType())
def classificar_cliente(total_gasto):
    if total_gasto > 10000: return "VIP"
    elif total_gasto > 1000: return "Regular"
    else: return "Novo"

df.withColumn("segmento", classificar_cliente(col("total_gasto")))
Cuidado com UDFs: funções Python customizadas (UDF) quebram a otimização do Catalyst e forçam serialização entre JVM e Python a cada linha. Prefira sempre as funções nativas (pyspark.sql.functions) — elas cobrem 95% dos casos sem precisar de UDF.
groupBy() — o que acontece por trás

Quando você chama df.groupBy("categoria").agg(sum("total")), o Spark precisa garantir que todas as linhas da mesma categoria estejam na mesma partição antes de somar. Isso exige um shuffle: os dados são redistribuídos pela rede com base no hash da chave de agrupamento.

PySpark — groupBy com múltiplas agregações
from pyspark.sql.functions import sum, count, avg, countDistinct, collect_list

resultado = (df
    .groupBy("categoria", "regiao")
    .agg(
        sum("total").alias("receita"),
        count("order_id").alias("pedidos"),
        countDistinct("customer_id").alias("clientes_unicos"),
        avg("total").alias("ticket_medio")
    )
)
# Por trás: Spark faz "partial aggregation" em cada partição primeiro
# (soma local), DEPOIS faz shuffle dos resultados parciais — não dos
# dados brutos. Isso é chamado de "map-side combine" e reduz MUITO
# o volume de dados transferidos pela rede.
Otimização automática: diferente do groupByKey() em RDD (que embaralha todos os dados brutos), o groupBy().agg() em DataFrame faz uma agregação parcial local antes do shuffle — reduzindo drasticamente o tráfego de rede. É uma das razões pelas quais DataFrame supera RDD.

Joins e Shuffle — a operação que mais derruba clusters
Tipos de Join no Spark
Inner Join

Retorna apenas linhas que têm correspondência em ambos os DataFrames. O mais comum e mais eficiente.

df1.join(df2, on="id", how="inner")
Left Join

Retorna todas as linhas do DataFrame esquerdo, com NULL onde não há correspondência à direita.

df1.join(df2, on="id", how="left")
Broadcast Join

Quando uma tabela é pequena (<10MB padrão), o Spark a copia inteira para TODOS os executors — evita shuffle completamente.

from pyspark.sql.functions import broadcast
df1.join(broadcast(df_pequena), "id")
Shuffle Hash Join

Quando ambas as tabelas são grandes: ambas são particionadas pelo hash da chave de join e embaralhadas pela rede.

# Acontece automaticamente quando
# ambas tabelas são grandes
df_grande1.join(df_grande2, "id")
Broadcast Join — a otimização mais impactante que existe

Imagine juntar uma tabela de 500 milhões de pedidos com uma tabela de 50 categorias de produto. Sem otimização, o Spark faria shuffle dos 500M de pedidos pela rede. Com Broadcast Join, ele copia as 50 categorias (poucos KB) para cada executor — e o join acontece localmente, sem mover os pedidos.

PySpark — Broadcast Join explícito
from pyspark.sql.functions import broadcast

# df_pedidos tem 500M linhas | df_categorias tem 50 linhas
df_pedidos = spark.read.parquet("s3://bucket/silver/orders/")
df_categorias = spark.read.parquet("s3://bucket/silver/categories/")

# Forçar broadcast da tabela pequena — evita shuffle de 500M linhas
resultado = df_pedidos.join(
    broadcast(df_categorias),
    on="category_id",
    how="left"
)

# O Spark também faz isso automaticamente se a tabela for menor
# que spark.sql.autoBroadcastJoinThreshold (padrão: 10MB)
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", 50 * 1024 * 1024)  # 50MB
Regra prática: sempre que você fizer join entre uma tabela "fato" (grande) e uma "dimensão" (pequena, tipo categoria, país, status), use broadcast explicitamente. Não confie só no automático — em produção, force com broadcast() para garantir previsibilidade.

Stages, Shuffle e Particionamento — o que acontece dentro do cluster
Jobs, Stages e Tasks — a hierarquia de execução

Quando você chama uma action, o Spark divide o trabalho em uma hierarquia: Job (toda a execução de uma action) → Stages (cada stage termina onde há um shuffle) → Tasks (uma task por partição dentro do stage).

Stage 1 — Narrow transformations8 tasks (8 partições)
read.parquet() → filter() → withColumn() → select()
⚡ SHUFFLE — groupBy() exige reembaralhar dados
Stage 2 — Agregação pós-shuffle200 tasks (200 partições)
groupBy("categoria").agg(sum("total"))
⚡ SHUFFLE — orderBy() exige ordem global
Stage 3 — Ordenação final1 task (resultado final)
orderBy(col("receita").desc()) → collect()
Regra de ouro: cada vez que você vê o número de Stages aumentar na Spark UI, significa que houve um shuffle. Menos stages geralmente significa um job mais rápido — minimizar shuffles é a otimização mais valiosa em Spark.
repartition() vs coalesce() — controlando o número de partições
repartition(n)
  • Pode aumentar OU diminuir partições
  • Faz full shuffle — redistribui tudo igualmente
  • Resultado: partições de tamanho uniforme
  • Caro — sempre causa shuffle
coalesce(n)
  • Só DIMINUI partições (não aumenta)
  • Combina partições vizinhas — evita shuffle completo
  • Mais barato que repartition
  • Pode gerar partições desbalanceadas
PySpark — quando usar cada um
# Cenário 1: você fez um filter() pesado e sobrou pouco dado
# mas ainda tem 200 partições (herdadas do shuffle anterior)
# → use coalesce para reduzir sem gastar com shuffle
df_filtrado = df.filter(col("total") > 10000).coalesce(4)

# Cenário 2: seus dados estão desbalanceados entre partições
# (data skew) e você precisa redistribuir igualmente
# → use repartition mesmo pagando o custo do shuffle
df_balanceado = df.repartition(16, "customer_id")  # particiona por chave

# Cenário 3: antes de escrever em Parquet, controlar nº de arquivos
df.coalesce(1).write.parquet("output/")  # 1 arquivo só (cuidado com volume!)
Data Skew — quando uma partição vira o gargalo

Data skew acontece quando os dados não estão distribuídos uniformemente entre as partições. Se você particiona por customer_id e um cliente (ex: uma conta de teste) tem 40% de todos os pedidos, essa partição vai demorar muito mais que as outras — e o job inteiro espera por ela.

PySpark — habilitar Adaptive Query Execution
# AQE é o recurso mais importante do Spark 3.x para skew
# Habilitado por padrão no Spark 3.2+, mas vale confirmar:
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# O AQE detecta partições muito maiores que a média durante
# a execução e as divide automaticamente em sub-tarefas menores

B

C

Pipeline completo em dataset de 1M+ linhas — diagnosticando com a Spark UI
Spark UI — seu painel de diagnóstico

Toda aplicação Spark expõe uma interface web em localhost:4040 (modo local) com informações detalhadas sobre Jobs, Stages, Tasks e uso de memória. É a primeira ferramenta para entender por que um job está lento.

Aba Jobs
Lista todas as actions executadas, duração total e número de stages.
Aba Stages
Mostra cada stage, quantas tasks, e se houve shuffle read/write.
Aba SQL
Plano de execução visual (DAG) gerado pelo Catalyst Optimizer.
Sinal de alerta na Spark UI: se uma task demora muito mais que as outras no mesmo stage (visível no gráfico de duração por task), você provavelmente tem data skew. Se o "Shuffle Read" de um stage é gigante, considere broadcast join ou reduzir dados antes do shuffle.
Pipeline completo — análise de 1M+ pedidos com join
PySpark — pipeline otimizado de ponta a ponta
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, sum, count, avg, round, broadcast, year, month

spark = (SparkSession.builder
    .appName("PipelineOtimizado")
    .config("spark.sql.adaptive.enabled", "true")
    .config("spark.sql.shuffle.partitions", "16")
    .getOrCreate()
)

# ── 1. CARREGAR (1M+ linhas) ──────────────────────────────────
df_pedidos = spark.read.parquet("data/pedidos/")   # ~1.2M linhas
df_clientes = spark.read.parquet("data/clientes/")  # ~50K linhas
df_categorias = spark.read.parquet("data/categorias/") # ~80 linhas

print(f"Pedidos: {df_pedidos.count():,} | Partições: {df_pedidos.rdd.getNumPartitions()}")

# ── 2. FILTRAR PRIMEIRO (reduz volume antes do shuffle) ────────
df_pedidos_validos = (df_pedidos
    .filter(col("status") == "paid")
    .filter(col("total") > 0)
    .filter(col("order_date") >= "2024-01-01")
)
# Filtrar ANTES do join reduz o que precisa ser embaralhado

# ── 3. JOIN com broadcast nas tabelas pequenas ─────────────────
df_completo = (df_pedidos_validos
    .join(broadcast(df_categorias), "category_id", "left")  # 80 linhas → broadcast
    .join(broadcast(df_clientes), "customer_id", "left")    # 50K linhas → broadcast
)

# ── 4. AGREGAÇÃO (único shuffle necessário) ────────────────────
df_resultado = (df_completo
    .withColumn("ano", year("order_date"))
    .withColumn("mes", month("order_date"))
    .groupBy("ano", "mes", "category_name", "customer_state")
    .agg(
        round(sum("total"), 2).alias("receita"),
        count("order_id").alias("pedidos"),
        round(avg("total"), 2).alias("ticket_medio")
    )
)

# ── 5. ESCREVER (coalesce para evitar muitos arquivos pequenos) ─
(df_resultado
    .coalesce(4)
    .write.mode("overwrite")
    .partitionBy("ano", "mes")
    .parquet("output/gold/revenue_by_region/")
)

print("Pipeline concluído! Verifique localhost:4040 para análise de performance.")
Comparação: antes e depois da otimização
# SEM otimização (join na ordem errada, sem broadcast, sem filtro antes)
Stage 0: read parquet — 1,2M linhas — 2.1s
Stage 1: shuffle join pedidos x clientes — 847MB shuffle — 38.4s
Stage 2: shuffle join + categorias — 623MB shuffle — 22.1s
Stage 3: groupBy + write — 412MB shuffle — 19.8s
TOTAL: 82.4 segundos
# COM otimização (filter antes, broadcast nas tabelas pequenas)
Stage 0: read parquet + filter — 1,2M → 340K linhas — 2.3s
Stage 1: broadcast join x2 (sem shuffle) — 0MB shuffle — 4.1s
Stage 2: groupBy + write — 89MB shuffle — 6.2s
TOTAL: 12.6 segundos (6.5x mais rápido)

Reescrita completa
Checklist de otimização Spark
  • 1. Filtre o máximo possível antes de qualquer join ou groupBy
  • 2. Use broadcast() em tabelas pequenas (dimensões) antes de juntar com tabelas grandes (fatos)
  • 3. Selecione apenas as colunas necessárias antes de operações pesadas (reduz volume do shuffle)
  • 4. Habilite Adaptive Query Execution para correção automática de skew
  • 5. Use coalesce() (não repartition) ao reduzir partições antes de escrever
  • 6. Evite UDFs Python — prefira funções nativas do pyspark.sql.functions
  • 7. Sempre confira a Spark UI após mudanças — meça, não suponha

O que você precisa ter gravado
Narrow vs Wide

Narrow (map, filter) não exige shuffle. Wide (groupBy, join, orderBy) exige — é onde os jobs ficam lentos.

Broadcast Join

Tabela pequena copiada para todos os executors. Evita shuffle da tabela grande. Use sempre que possível.

Partições

repartition() faz shuffle completo. coalesce() só diminui, mais barato. Data skew = uma partição vira gargalo.
