Revisão rápida
Os 6 temas — pontos-chave e armadilhas comuns
Semana 7 — Ingestão
Full Load: copia tudo. Simples, pesado, sem cursor necessário
Incremental: cursor em updated_at. Eficiente. Não captura DELETEs
CDC: lê WAL do banco. Captura INSERT/UPDATE/DELETE. Latência segundos. Sem cursor necessário
Streaming: eventos de app → Kafka direto. Produtor publica, consumer lê com offset
Airbyte: conectores prontos para SaaS/APIs. Batch. Airbyte ≠ CDC
Armadilha: Incremental não vê DELETEs. Sem updated_at → só Full Load ou CDC.
Semana 8 — Apache Spark
In-memory: Spark mantém dados em RAM entre etapas — 10-100x mais rápido que MapReduce
RDD: Resilient (lineage), Distributed, Dataset. Lazy evaluation
Transformação (lazy): map, filter, select — não executam
Action (eager): count, collect, show — disparam execução do DAG
DataFrame: schema definido + Catalyst Optimizer. Supera RDD
Spark SQL: createOrReplaceTempView + spark.sql() — resultado é DataFrame
Armadilha: collect() traz TUDO para o driver — crashes em datasets grandes. Use take(n).
Semana 9 — PySpark Avançado
Narrow: map, filter, select — sem shuffle, rápido
Wide: groupBy, join, orderBy, distinct — exige shuffle, caro
Broadcast Join: tabela pequena → copiada para todos executors → sem shuffle
Shuffle Hash Join: ambas tabelas grandes → ambas são redistribuídas pela rede
repartition(n): shuffle completo, pode aumentar partições
coalesce(n): só diminui, sem shuffle completo, mais barato
Armadilha: broadcast() na tabela PEQUENA, não na grande. Skew = uma partição vira gargalo.
Semana 10 — Spark SQL + Otimização
Catalyst: 4 fases: Análise → Otimiz. Lógica → Planej. Físico → Code Gen
Predicate Pushdown: filtros empurrados até o FileScan — lê menos dados
Column Pruning: lê só as colunas necessárias
explain(): mostra o plano sem executar. PushedFilters + BroadcastHashJoin = bom
cache()/persist(): evita recalcular quando DataFrame é reutilizado em múltiplas actions
UDF: opaca para o Catalyst → PushedFilters: [] → evitar
Armadilha: cache() é lazy — só materializa na 1ª action após o persist. Sempre unpersist() ao final.
Semana 11 — ETL vs ELT e dbt
ETL: transforma antes de carregar — para dados sensíveis/regulados
ELT: carrega bruto, transforma dentro do warehouse. Padrão moderno
Camadas dbt: staging (stg_*) → intermediate (int_*) → marts (fct_*, dim_*)
source(): aponta para tabela bruta externa ao dbt (zona Raw)
ref(): aponta para outro modelo dbt — cria dependência no DAG
Testes: genéricos (unique, not_null, accepted_values, relationships) e singulares (SQL custom)
Armadilha: ref() ≠ source(). Teste singular = SQL que deve retornar ZERO linhas para passar.
Semana 12 — Apache Airflow
DAG: grafo direcionado acíclico. Operator >> define dependência
BashOperator: executa shell. PythonOperator: executa função Python
Sensor: espera condição (arquivo, S3, outra DAG). Não confundir com Operator
poke vs reschedule: poke ocupa worker, reschedule libera entre verificações
catchup=True: dispara TODAS execuções perdidas desde start_date
retries: tentativas antes de marcar failed. SLA: alerta por demora (não falha)
Armadilha: catchup=False em produção para evitar avalanche. Sensor ≠ Operator — sensor espera, operator executa.

Semana 7
Um sistema legado (Oracle, anos 2000) não tem coluna updated_at em suas tabelas de faturamento. A empresa precisa sincronizar esses dados mensalmente para analytics. Qual estratégia de ingestão é obrigatória?
A
Incremental com cursor em ID sequencial
B
CDC com Debezium — não precisa de cursor
C
Full Load — sem coluna de cursor e sem necessidade de tempo real, é a única opção viável
D
Streaming via Kafka — mais eficiente para grandes volumes
✅ C. Sem coluna updated_at e sem ID sequencial confiável, o Incremental com cursor não é possível. O CDC seria tecnicamente viável (lê o Redo Log do Oracle), mas a empresa pediu sincronização mensal — não há necessidade de latência de segundos. Para cargas mensais de um sistema legado sem cursor, Full Load é a solução mais simples e adequada.

Semana 7
Um sistema de pedidos deleta registros cancelados da tabela de produção. O Data Lake precisa rastrear esses cancelamentos para relatórios de devolução. Qual estratégia OBRIGATORIAMENTE deve ser usada?
A
Full Load diário — a comparação entre os snapshots detecta o que foi deletado
B
Incremental com cursor em updated_at — eficiente para detectar mudanças
C
CDC — único método nativo que captura eventos DELETE com before/after state em tempo real
D
Airbyte com modo incremental dedup — detecta registros removidos automaticamente
✅ C. O Incremental (Airbyte ou cursor) não captura DELETEs — o registro simplesmente desaparece silenciosamente. O Full Load por comparação funcionaria mas é caro e complexo. O CDC lê diretamente o transaction log do banco e emite um evento explícito de DELETE com o estado anterior do registro (before state), sendo o único método que resolve esse problema nativamente.

Semana 8
Um engenheiro escreveu: df2 = df1.filter(col("total") > 100) seguido de df3 = df2.groupBy("cat").agg(sum("total")). Quantas vezes o Spark leu os dados do disco ao executar essas duas linhas?
A
Duas vezes — uma para o filter e uma para o groupBy
B
Nenhuma — o Spark armazena tudo em memória automaticamente
C
Nenhuma — essas linhas são transformações lazy, o Spark não executou nada ainda
D
Uma vez — o filter executa imediatamente ao ser chamado
✅ C. filter() e groupBy().agg() são transformações lazy — elas apenas adicionam operações ao DAG, sem executar nada. Nenhuma leitura de disco acontece nessas duas linhas. O Spark só lê os dados e executa o pipeline inteiro quando uma action é chamada (count, show, collect, write). Esse é o conceito central de lazy evaluation.

Semana 9
Você está fazendo join entre uma tabela fato de 2 bilhões de linhas e uma tabela de 150 países (com ~150 linhas). O Spark está usando SortMergeJoin e o job demora 40 minutos. Qual é a correção mais impactante?
A
Aumentar o número de partições com repartition() para ter mais paralelismo
B
Forçar broadcast() na tabela de países — elimina o shuffle dos 2 bilhões de linhas
C
Converter o DataFrame para RDD antes do join para mais controle
D
Usar coalesce() para reduzir o número de partições antes do join
✅ B. 150 linhas é uma tabela minúscula. O Catalyst não usou broadcast automático possivelmente porque a tabela excede o limite padrão de 10MB (ou porque o tamanho não foi inferido corretamente). Forçar broadcast(df_paises) copia as 150 linhas inteiras para cada executor, eliminando completamente o shuffle de 2 bilhões de linhas. Essa é a otimização mais impactante disponível neste cenário.

Semana 9
Você tem um DataFrame com 200 partições após um groupBy() pesado, mas o resultado tem apenas 5 MB de dados. Você precisa salvá-lo em Parquet com no máximo 2 arquivos. Qual comando é mais adequado?
A
df.repartition(2).write.parquet("output/")
B
df.coalesce(2).write.parquet("output/")
C
São equivalentes — qualquer um resolve
D
df.persist(StorageLevel.DISK_ONLY).write.parquet("output/")
✅ B. coalesce(2) é mais eficiente que repartition(2) quando o objetivo é REDUZIR o número de partições. coalesce() combina partições adjacentes sem fazer um shuffle completo — mais barato. repartition() sempre faz um shuffle completo redistribuindo todos os dados, o que é desnecessário aqui. Ambos resultariam em 2 arquivos, mas coalesce() chegará lá gastando menos recursos.

Semana 10
Você analisa o plano de execução com df.explain("formatted") e encontra o nó BatchEvalPython e o campo PushedFilters: []. O que isso indica?
A
O Spark está usando broadcast join automaticamente — sinal de boa otimização
B
Uma UDF Python está sendo usada, bloqueando o predicate pushdown e lendo todas as colunas desnecessariamente
C
O Spark não conseguiu achar o arquivo Parquet e está usando um caminho alternativo
D
O DataFrame foi cacheado corretamente e a leitura virá da memória
✅ B. BatchEvalPython é o nó que aparece quando o Spark precisa chamar código Python externo (UDF). PushedFilters: [] significa que o filtro NÃO foi empurrado até o FileScan — o Catalyst não consegue "ver" o que a UDF faz internamente, então não pode otimizar. O resultado é que todos os dados são lidos do disco antes de qualquer filtragem. Solução: substituir a UDF por funções nativas do pyspark.sql.functions.

Semana 10
Um DataFrame resultado de um join pesado será usado em 4 análises diferentes (count, show, groupBy e write). Qual é a estratégia correta?
A
Não fazer nada — o Spark já cacheia automaticamente resultados usados mais de uma vez
B
Chamar .cache() após o join e disparar uma action para materializar, depois executar as 4 análises
C
Salvar em Parquet e reler 4 vezes — mais seguro que cache
D
Converter para RDD antes das análises — RDD é cacheado automaticamente
✅ B. O Spark NÃO cacheia automaticamente. Sem cache, cada uma das 4 actions recalcula todo o DAG desde o início — incluindo o join pesado — desperdiçando recursos. Com .cache(), o join é calculado uma única vez (na 1ª action após o cache) e os resultados ficam em memória para as próximas 3 análises. Lembre-se: cache() é lazy — só materializa quando uma action é chamada explicitamente após o persist.

Semana 11
Em um modelo dbt, você escreve FROM {{ ref('stg_orders') }}. O que acontece quando o dbt executa dbt run?
A
O dbt simplesmente substitui ref() pelo nome literal da tabela stg_orders
B
O dbt registra a dependência no DAG e garante que stg_orders seja materializado ANTES deste modelo
C
ref() executa testes de qualidade no modelo referenciado antes de continuar
D
ref() aponta para a tabela bruta na zona Raw do Data Lake
✅ B. ref() é muito mais que uma substituição de nome. Ele sinaliza ao dbt que este modelo DEPENDE de stg_orders, o que faz o dbt: (1) construir o DAG de dependências completo, (2) garantir que stg_orders seja executado antes deste modelo, (3) resolver o nome correto da tabela no schema/banco configurado. É essa função que elimina a necessidade de gerenciar manualmente a ordem de execução de transformações SQL.

Semana 12
Em uma DAG Airflow, você tem: ingest >> [dbt_staging, validate_schema] >> dbt_marts >> notify. Quando notify vai executar?
A
Assim que a primeira das duas tasks paralelas terminar (dbt_staging OU validate_schema)
B
Somente depois que dbt_staging E validate_schema terminarem com sucesso, e dbt_marts também terminar
C
Independente das outras tasks — notify não tem dependências
D
notify não será executado — o Airflow não aceita listas como dependência intermediária
✅ B. O operador >> com lista [dbt_staging, validate_schema] cria dependência para AMBAS as tasks em paralelo. dbt_marts espera que as DUAS terminem com sucesso (trigger_rule padrão = all_success). notify, por sua vez, espera que dbt_marts também termine com sucesso. A ordem completa é: ingest → {dbt_staging e validate_schema em paralelo} → dbt_marts → notify.

Semana 12
Um FileSensor configurado com mode="poke" está esperando um arquivo que demora 4 horas para aparecer. Qual é o problema e a solução?
A
Não há problema — poke é o modo mais eficiente para esperas longas
B
No modo poke, o sensor ocupa um worker inteiro por 4 horas — usar mode="reschedule" libera o worker entre verificações
C
FileSensor não suporta esperas de mais de 1 hora — usar um cron externo
D
Aumentar o timeout para 8 horas — o modo poke fica mais eficiente com timeout maior
✅ B. No modo poke, o sensor cria uma task que fica ativa continuamente verificando a condição — ocupando um slot de worker inteiro durante todo o período de espera. Com 4 horas de espera, isso bloqueia um worker que poderia estar processando outras tasks. O modo reschedule resolve isso: o sensor verifica a condição, se não estiver pronta, marca a task como "reschedule" e libera o worker, reativando a verificação após o poke_interval.

Uma empresa de saúde precisa sincronizar dados de prontuários (com CPF, CID e diagnósticos) de um sistema legado para um Data Warehouse moderno na nuvem, com frequência mensal para analytics. Qual abordagem é obrigatória por compliance (LGPD)?
A
ETL — anonimizar/pseudonimizar os dados sensíveis ANTES de chegarem ao Data Warehouse na nuvem
B
ELT — carregar os dados brutos e anonimizar com dbt dentro do warehouse
C

Questão Dissertativa 1 — Semanas 8, 9 e 10
Um job Spark que faz join entre uma tabela de 500M pedidos (50 GB) e uma tabela de 200 produtos demora 45 minutos. Você analisa com explain() e vê SortMergeJoin, PushedFilters: [] e o plano lê todas as 80 colunas da tabela de pedidos. Identifique os três problemas e proponha as correções com código PySpark.
Problema 1 — SortMergeJoin em vez de BroadcastHashJoin: a tabela de produtos tem 200 linhas (~poucos KB). O Catalyst não usou broadcast automaticamente (talvez por falta de estatísticas). Correção: df_pedidos.join(broadcast(df_produtos), "product_id").

Problema 2 — PushedFilters: []: um filtro no pipeline depende de uma UDF Python, que é opaca para o Catalyst. Resultado: lê os 50 GB completos antes de filtrar. Correção: substituir a UDF por funções nativas (when(), col() etc.) para o Catalyst poder aplicar predicate pushdown no FileScan.

Problema 3 — Column Pruning ausente (lê 80 colunas): o código não seleciona apenas as colunas necessárias antes de operações pesadas. Correção: adicionar .select("order_id", "product_id", "total", "order_date") antes do join, reduzindo o volume de dados transferidos no shuffle.

Código corrigido: 
resultado = (df_pedidos
    .filter(col("total") > 0)           # filtro nativo → PushedFilters preenchido
    .select("order_id","product_id","total")  # column pruning
    .join(broadcast(df_produtos), "product_id")  # BroadcastHashJoin
    .groupBy("product_name")
    .agg(sum("total").alias("receita"))
)
Questão Dissertativa 2 — Semanas 7 e 11
Uma startup de e-commerce está construindo seu primeiro Data Lake. Eles têm: (a) um banco PostgreSQL de pedidos com 200M de linhas e coluna updated_at, (b) uma integração com Stripe (pagamentos via API REST), (c) eventos de clique do app mobile. Para cada fonte, indique a estratégia de ingestão e, após o dado chegar ao Data Lake, qual seria a estrutura de modelos dbt (camadas e nomes de modelos).
(a) PostgreSQL — pedidos: ingestão incremental com cursor em updated_at via Airbyte (ou dbt_source + query incremental). Sem necessidade de CDC pois não há requisito de capturar DELETEs e a latência de horas é aceitável.

(c) Eventos de clique mobile: streaming direto via Kafka. O app mobile publica um evento no Kafka a cada clique, sem banco de dados de origem intermediário (não é CDC).

Questão Dissertativa 3 — Semana 12
Descreva como um pipeline de dados completo seria orquestrado com Airflow para o seguinte fluxo: (1) extrair dados do PostgreSQL, (2) esperar o arquivo aparecer no S3, (3) rodar os modelos dbt de staging e marts em paralelo após a espera, (4) rodar os testes dbt, (5) notificar o time via e-mail em caso de sucesso. Esboce a estrutura da DAG em pseudocódigo e justifique as escolhas de operador para cada task.
Estrutura da DAG: 
extract_postgres  →  wait_s3  →  [dbt_staging, dbt_marts]  →  dbt_test  →  notify_email

Task 1 — extract_postgres (BashOperator): executa o script de extração que usa psycopg2 ou Airbyte CLI. BashOperator é adequado pois é uma chamada a um script externo.

Task 2 — wait_s3 (S3KeySensor): sensor que verifica se o arquivo Parquet apareceu no bucket S3. Deve usar mode="reschedule" para não ocupar worker durante a espera. poke_interval=60, timeout=3600.

Tasks 3a/3b — dbt_staging e dbt_marts (BashOperator em paralelo): dois BashOperators rodando em paralelo. Nota: tecnicamente dbt_marts depende de dbt_staging — se estiverem no mesmo projeto dbt, basta um único dbt run. Para paralelismo real, seria necessário que staging e marts fossem projetos separados ou usar seletores distintos garantindo que staging já foi materializado.

Task 4 — dbt_test (BashOperator): executa dbt test. Depende de dbt_staging e dbt_marts (ambos devem ter terminado). Se falhar, dbt_test fica failed e notify_email recebe status upstream_failed.

Task 5 — notify_email (EmailOperator ou PythonOperator): dispara apenas se todas as tasks anteriores tiveram sucesso (trigger_rule=all_success, padrão). Para garantir notificação mesmo em falha, adicionar on_failure_callback no default_args.
Questão Dissertativa 4 — Semanas 8, 9 e 12
Compare MapReduce e Apache Spark em termos de: (a) onde os resultados intermediários são armazenados, (b) como cada um lida com algoritmos iterativos (ex: k-means que passa pelos dados 50 vezes), e (c) o que são "stages" no Spark e o que determina a fronteira entre dois stages.
(a) Armazenamento de resultados intermediários: no MapReduce, após cada fase Map e cada fase Reduce, os resultados são escritos obrigatoriamente no HDFS (disco distribuído). Na próxima etapa, os dados são lidos de volta do disco. Em um pipeline com N etapas, há 2N operações de I/O em disco. O Spark mantém os resultados intermediários em memória RAM entre as etapas — só lê do disco uma vez (a entrada) e escreve uma vez (a saída final). Se os dados não couberem na RAM, o Spark "derrama" (spill) para disco, mas o design favorece memória.

(b) Algoritmos iterativos: no MapReduce, cada iteração do k-means é um job separado que lê do HDFS e escreve de volta no HDFS. Com 50 iterações, são 100 ciclos de I/O completos. A diferença de performance pode ser de 100x ou mais para esse tipo de workload. No Spark, os dados podem ser cacheados em memória com .cache() ou .persist(), e cada iteração simplesmente lê da RAM — sem tocar o disco. O MLlib do Spark usa esse modelo para todos os algoritmos iterativos (k-means, regressão logística, SVD).

(c) Stages no Spark: um stage é um conjunto de tasks que podem ser executadas em paralelo sem precisar de comunicação entre partições (sem shuffle). A fronteira entre dois stages é sempre uma wide transformation — uma operação que exige redistribuição dos dados entre executors (shuffle), como groupBy(), join(), orderBy() ou distinct(). Dentro de um stage, todas as transformações são narrow (map, filter, select) e cada task processa sua partição independentemente. Minimizar o número de stages (e portanto o número de shuffles) é a principal alavanca de performance em Spark.
