Data Lake, Data Warehouse e Data Lakehouse — as três gerações
Semana 2

Nas semanas 1 e 2 entendemos por que Big Data existe e como arquitetar o fluxo de processamento (Lambda/Kappa). Agora a pergunta é: onde os dados ficam armazenados? A resposta evoluiu em três gerações ao longo de 30 anos.

Geração 1: Data Warehouse — o armazém organizado

O Data Warehouse surgiu nos anos 1990 como solução para consolidar dados de diferentes sistemas transacionais (ERP, CRM, POS) em um único lugar para análise. A lógica é schema-on-write: você define rigorosamente a estrutura dos dados antes de carregá-los.

  • SQL rápido e otimizado para analytics
  • Schema forte — dados sempre consistentes
  • ACID transactions garantidas
  • Fácil de usar para analistas de negócio
  • Só aceita dados estruturados (sem imagens, textos livres)
  • Custo muito alto em escala de PB
  • Schema rígido — mudanças são lentas e caras
  • Não suporta dados de ML diretamente
Exemplos: Amazon Redshift, Google BigQuery, Snowflake, Microsoft Azure Synapse, IBM Db2 Warehouse. Ainda amplamente usados — especialmente para BI e relatórios estruturados.
Geração 2: Data Lake — o lago de dados brutos

Em 2010, James Dixon (Pentaho) cunhou o termo Data Lake: em vez de um armazém onde tudo deve entrar organizado, um lago onde você despeja os dados brutos como chegam e decide o que fazer com eles depois. A lógica é schema-on-read: a estrutura é aplicada na hora da leitura, não da escrita.

Metáfora do próprio James Dixon: "Se você pensar em um Data Mart como um reservatório de água engarrafada — limpa, embalada e estruturada para consumo fácil — um Data Lake é um grande corpo de água natural. O conteúdo flui de uma fonte para o lago; vários usuários podem examinar, mergulhar e buscar amostras."
  • Aceita qualquer tipo de dado: estruturado, semiestruturado, não estruturado
  • Custo muito baixo — armazena em S3/HDFS (commodity storage)
  • Flexível — schema aplicado só quando necessário
  • Ideal para ML — dados de treino ficam aqui
  • Sem ACID — updates e deletes são complexos
  • Consultas lentas sem otimização (sem índices nativos)
  • Risco de virar "Data Swamp" sem governança
  • Difícil para analistas de negócio sem ferramentas
Geração 3: Data Lakehouse — o melhor dos dois mundos

Proposto pela Databricks em 2019 com o Delta Lake, o Lakehouse combina o baixo custo e a flexibilidade do Data Lake com a confiabilidade e performance do Data Warehouse. A tecnologia central é uma camada de metadados transacional sobre o armazenamento de objetos (S3/HDFS).

A metáfora da refinaria de petróleo

Um Data Lake sem organização é inútil — ninguém sabe o que há lá dentro, ninguém confia nos dados e ninguém consegue encontrar o que precisa. A solução é organizar o Data Lake em zonas de maturidade dos dados, exatamente como uma refinaria de petróleo processa o óleo bruto em produtos cada vez mais refinados.

Formatos de arquivo — por que CSV não serve para Big Data
O problema com CSV e JSON em escala

CSV e JSON são formatos universais, mas foram projetados para legibilidade humana e intercâmbio de dados, não para processamento analítico de bilhões de registros. Em escala, os problemas aparecem rapidamente.

📄 CSV — o que dá errado
  • Para ler uma coluna, você lê o arquivo inteiro (row-based)
  • Sem schema — "1" pode ser número ou string
  • Sem compressão nativa — 1 GB de dados → 1 GB no disco
  • Não suporta tipos aninhados (arrays, objetos)
📋 JSON — o que dá errado em escala
  • Chaves repetidas em cada registro — 30-40% de overhead
  • Sem compressão nativa eficiente
  • Parse lento para bilhões de linhas
  • Schema implícito — pode mudar a qualquer momento
Colunar vs. Row-based — a diferença que muda tudo

A inovação central dos formatos modernos é o armazenamento colunar. Em vez de armazenar os dados linha por linha (como o CSV), os dados são armazenados coluna por coluna. Isso parece detalhe, mas muda completamente a performance de queries analíticas.

Por que colunar é mais rápido para analytics? (1) Você lê só as colunas necessárias — em vez de ler 100 colunas, lê as 3 da sua query. (2) Dados do mesmo tipo comprimem muito melhor juntos — coluna de preços (float) comprime diferente de coluna de nomes (string). (3) Operações vetorizadas na CPU são mais eficientes em blocos de mesmo tipo.
Apache Parquet Colunar
Armazenamento colunar com compressão por coluna
Criado pelo Twitter e Cloudera em 2013
Suporta tipos complexos: arrays, maps, structs

✅ Use para:
Sistemas com Hive, ecossistemas Hadoop legados; em novos projetos prefira Parquet
Compressão — o impacto no tamanho e na velocidade

A escolha do algoritmo de compressão é um trade-off entre velocidade de leitura/escrita e taxa de compressão. Um arquivo menor economiza storage e acelera transferências de rede, mas compressão mais agressiva leva mais tempo de CPU.

Regra prática: use Snappy para dados que serão lidos frequentemente (mais rápido de descomprimir) e Zstd para dados arquivados ou lidos raramente (menor tamanho). Gzip é o mais compacto mas o mais lento — evite para analytics interativo.
Lendo e escrevendo Parquet com PySpark
PySpark — operações com Parquet
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, to_date

spark = SparkSession.builder.appName("DataLake").getOrCreate()

# ── 1. Ler JSON da zona Raw ──────────────────────────────────
df_raw = spark.read.json("s3://meu-ecommerce/raw/orders/year=2024/month=01/")

# ── 2. Transformações para zona Silver ───────────────────────
df_silver = (df_raw
  .dropDuplicates(["order_id"])             # deduplicar
  .dropna(subset=["order_id", "total"])      # remover nulos críticos
  .withColumn("order_date", to_date("created_at"))  # padronizar data
  .filter(col("total") > 0)                  # remover pedidos inválidos
)

# ── 3. Escrever Silver em Parquet com Snappy + particionamento
(df_silver
  .write
  .mode("overwrite")
  .partitionBy("year", "month")             # particionar por data
  .option("compression", "snappy")
  .parquet("s3://meu-ecommerce/silver/orders/")
)

# ── 4. Criar tabela Gold — receita diária por categoria ───────
df_gold = (df_silver
  .groupBy("order_date", "category")
  .agg({"total": "sum", "order_id": "count"})
  .withColumnRenamed("sum(total)",    "revenue")
  .withColumnRenamed("count(order_id)", "orders")
)

df_gold.write.mode("overwrite").parquet("s3://meu-ecommerce/gold/revenue_daily/")

Data Swamp — o Data Lake que deu errado
O que é um Data Swamp

Um Data Swamp (pântano de dados) é o que acontece com um Data Lake sem governança. Com o tempo, ninguém sabe o que há lá dentro, não há como confiar nos dados e o lago se torna inutilizável — um pântano.

Pesquisa Gartner (2022): mais de 60% dos projetos de Data Lake fracassam ou não entregam o valor esperado. A principal causa não é tecnologia — é falta de governança, catalogação e qualidade de dados.
🦟 Sinais de um Data Swamp
  • Arquivos sem nome ou com nomes genéricos (data_final_v3.csv)
  • Ninguém sabe a origem de um arquivo
  • Dados duplicados em vários diretórios sem documentação
  • Formatos misturados sem padronização (CSV, JSON, Excel juntos)
  • Dados sem data de criação ou fonte identificada
✅ Como evitar o Data Swamp
  • Zonas claras: Raw/Silver/Gold com regras explícitas
  • Catálogo: cada dataset documentado (OpenMetadata, Datahub)
  • Linhagem: rastrear de onde cada dado veio e quem o transformou
  • Testes de qualidade: regras automáticas no dbt ou Great Expectations
  • Data Owner: cada dataset tem um responsável nomeado

Parte 3 — Desafio: governança anti-swamp (15 min)

Seu e-commerce cresceu e agora tem 15 engenheiros de dados. Três deles começaram a jogar arquivos CSV diretamente na zona Silver. Defina as 3 regras técnicas e as 3 regras de processo que você implementaria para evitar que o Data Lake se torne um Data Swamp.

Governança
Solução: Regras anti-Data Swamp
🔧 Regras Técnicas (IAC/automação)
  • IAM Policy no S3: só o pipeline de ingestão tem permissão de escrita na Silver — humanos podem apenas ler
  • Schema validation no CI/CD: todo PR que adiciona uma nova tabela precisa ter schema documentado em JSON Schema ou Avro
  • Testes automáticos de qualidade: Great Expectations ou dbt tests rodam em cada ingestão — falhou, o dado não entra na Silver
📋 Regras de Processo (cultura e governança)
  • Data Owner obrigatório: nenhum dataset é adicionado sem um responsável nomeado que aprove e documente
  • Catálogo como requisito: toda nova tabela Silver/Gold deve ter entrada no OpenMetadata antes de ir para produção
  • Revisão quinzenal: reunião de "limpeza do lago" — datasets não acessados há 90 dias são arquivados ou deletados com aprovação do owner
