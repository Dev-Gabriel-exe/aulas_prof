O que é Kafka — o log imutável que mudou o streaming
Semana 2

Na Semana 2 vimos que a Arquitetura Kappa resolve o problema de Lambda usando streaming puro. A peça central da Kappa é um log imutável e replay-able. Esse log tem um nome: Apache Kafka. Hoje vamos entender como ele funciona por dentro.

O problema antes do Kafka

Em 2010, o LinkedIn tinha dezenas de sistemas diferentes se comunicando: o sistema de perfis precisava avisar o sistema de recomendações, que precisava avisar o sistema de notificações, que precisava avisar o sistema de analytics... O resultado era uma teia de integrações ponto-a-ponto impossível de manter.

❌ Antes do Kafka — integração ponto-a-ponto
  • N sistemas × M destinos = N×M integrações para manter
  • Se um sistema destino cai, o dado se perde
  • Adicionar um novo consumidor exige modificar o produtor
  • Impossível "rebobinar" para reprocessar dados antigos
✅ Com Kafka — hub centralizado de eventos
  • Todo sistema publica eventos no Kafka — N+M integrações
  • Kafka retém os dados — consumidores leem quando puderem
  • Novo consumidor? Basta se inscrever no tópico — ninguém muda
  • Replay: rebobina o tópico e reprocessa desde o início
O conceito central: log imutável e append-only

A grande sacada do Kafka é tratar dados como um log — exatamente como o log de um banco de dados (Write-Ahead Log). Um log tem duas propriedades fundamentais:

📝 Append-only
Mensagens só podem ser adicionadas ao final. Nunca editadas. Nunca deletadas (até expirar por retenção). Imutabilidade total.
🔢 Ordered
Cada mensagem tem um número sequencial (offset). Dentro de uma partição, a ordem é sempre garantida. Offset 5 sempre veio depois do offset 4.
A consequência mais poderosa: como o log é imutável e ordenado, qualquer consumidor pode ler de qualquer ponto no tempo. Um bug no código de processamento? Corrija o código, volte o offset para 0 e reprocesse tudo — os dados ainda estão lá. Isso é a essência da Arquitetura Kappa.

Tópicos, Partições e Brokers — a anatomia do Kafka
Os três conceitos fundamentais
Broker
Um servidor Kafka. Um cluster tem múltiplos brokers (tipicamente 3+). Cada broker armazena algumas partições. Responsável por aceitar escritas e servir leituras.
Topic
Canal nomeado de eventos. Como uma tabela de banco de dados, mas append-only. Produtores publicam em tópicos. Consumidores se inscrevem em tópicos.
Partition
Um tópico é dividido em N partições. Cada partição é um log ordenado e independente. Mais partições = mais paralelismo de produção e consumo.

Cada partição é armazenada em um Broker diferente (Leader). Com replication-factor=2, cada partição tem também um Follower em outro Broker. Cada mensagem tem um offset único por partição (começando em 0).
Particionamento por chave — a garantia de ordem

Quando um producer envia uma mensagem, ele pode informar uma chave. O Kafka garante que todas as mensagens com a mesma chave vão para a mesma partição — o que garante a ordem para aquela chave.

Python — Producer com chave de particionamento
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers='localhost:9092',
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    key_serializer=lambda k: k.encode('utf-8')
)

# Todos os pedidos do cliente "C001" vão para a mesma partição
# Isso garante que os eventos desse cliente chegam em ordem
producer.send(
    topic='orders',
    key='C001',      # ← chave = customer_id
    value={'order_id': 'ord_001', 'total': 299.90, 'status': 'created'}
)
producer.send(
    topic='orders',
    key='C001',      # ← mesmo cliente → mesma partição → ordem garantida
    value={'order_id': 'ord_001', 'total': 299.90, 'status': 'paid'}
)
producer.flush()  # garantir que as mensagens foram enviadas
Por que isso importa? Se o pedido "criado" chegar depois de "pago" no processador, você vai errar o estado do pedido. Ao usar customer_id como chave, todos os eventos daquele cliente ficam na mesma partição, na ordem certa.
Replicação no Kafka — Leader e Followers

Assim como o HDFS, o Kafka replica cada partição em múltiplos brokers. A diferença é que no Kafka existe uma hierarquia clara: Leader (recebe todas as leituras e escritas) e Followers (replicam o leader).

ComponenteResponsabilidadeO que acontece se cair
Leader da partiçãoRecebe escritas do producer. Serve leituras do consumer. Replicas seguem o leader.Um Follower do ISR é eleito novo Leader automaticamente. Sem downtime.
Follower / ISRReplica os dados do Leader continuamente. ISR = In-Sync Replica (suficientemente atualizado).Cluster continua funcionando. Leader redistribui réplicas para manter o fator de replicação.
Broker inteiroHospeda partições (Leaders e Followers) e serve clientes.Partições que eram Leader naquele broker elegem novos Leaders. Se replication-factor ≥ 2, sem perda de dados.

Consumers, Consumer Groups e Offset — lendo o log com controle
Consumer Groups — o segredo do paralelismo

Um Consumer Group é um conjunto de consumers que cooperam para ler um tópico. O Kafka garante que cada partição é lida por exatamente um consumer dentro do grupo ao mesmo tempo. Isso permite escalar horizontalmente o processamento.

Producers

Regra de ouro dos Consumer Groups: para máximo paralelismo, o número de consumers no grupo deve ser igual ao número de partições. Se tiver mais consumers que partições, os consumers extras ficam ociosos. Se tiver menos, alguns consumers leem mais de uma partição.
Múltiplos grupos — cada um tem sua cópia completa

Diferentes Consumer Groups são completamente independentes. Se você tem 3 grupos (fraud-detector, analytics, notifications), cada um lê o tópico inteiro do início — como se cada grupo tivesse sua cópia dos dados.

Isso é radicalmente diferente de filas tradicionais (RabbitMQ): em uma fila, cada mensagem é consumida UMA VEZ por qualquer consumer. No Kafka, cada grupo consome tudo independentemente.

Offset — a posição no log

O offset é um número inteiro que identifica a posição de uma mensagem dentro de uma partição (começa em 0). Cada Consumer Group mantém seu próprio offset por partição — registrado no tópico interno __consumer_offsets.

Na prática: at-least-once é o padrão mais usado. O processamento deve ser idempotente — processar a mesma mensagem duas vezes deve produzir o mesmo resultado. No contexto de pedidos: ao receber "pedido criado" duas vezes, verificar se já existe antes de inserir.

      - KAFKA_CFG_ADVERTISED_LISTENERS=PLAINTEXT://localhost:9092
      - KAFKA_CFG_CONTROLLER_LISTENER_NAMES=CONTROLLER
      - KAFKA_CFG_AUTO_CREATE_TOPICS_ENABLE=false
      - ALLOW_PLAINTEXT_LISTENER=yes
Comandos essenciais — tópicos, producers e consumers
Kafka CLI — comandos essenciais
# ── Criar e listar tópicos ─────────────────────────────────────
kafka-topics.sh --create --topic orders \
  --partitions 3 --replication-factor 1 \
  --bootstrap-server localhost:9092

kafka-topics.sh --list --bootstrap-server localhost:9092

kafka-topics.sh --describe --topic orders \
  --bootstrap-server localhost:9092

# ── Produzir mensagens ─────────────────────────────────────────
kafka-console-producer.sh --topic orders \
  --bootstrap-server localhost:9092
# > ord_001:{"customer":"C001","total":299}   ← digita e pressiona Enter
# > ord_002:{"customer":"C002","total":89}

# Produtor com chave explícita
kafka-console-producer.sh --topic orders \
  --property "parse.key=true" \
  --property "key.separator=:" \
  --bootstrap-server localhost:9092

# ── Consumir mensagens ─────────────────────────────────────────
kafka-console-consumer.sh --topic orders \
  --from-beginning \
  --bootstrap-server localhost:9092

# Consumir como parte de um grupo
kafka-console-consumer.sh --topic orders \
  --group fraud-detector \
  --from-beginning \
  --bootstrap-server localhost:9092

# ── Verificar offsets do grupo ────────────────────────────────
kafka-consumer-groups.sh --describe \
  --group fraud-detector \
  --bootstrap-server localhost:9092

Exercício 3 — Desafio: LAG e reprocessamento (10 min)

Publique mais 20 mensagens com o consumer do grupo-a parado. Em seguida: (a) verifique o LAG do grupo-a com kafka-consumer-groups.sh --describe — quantas mensagens estão pendentes? (b) reinicie o consumer do grupo-a sem o --from-beginning — ele lê as mensagens antigas ou só as novas? (c) use --from-beginning — ele reprocessa tudo? Por quê isso é útil na Arquitetura Kappa?

Conceitual + Prática
Resposta esperada
  • (a) O LAG será 20 — o grupo-a tem 20 mensagens pendentes de processamento. O Kafka reteve tudo enquanto o consumer estava parado.
  • (b) Sem --from-beginning, o consumer retoma do último offset commitado — lê as 20 mensagens pendentes. Não relê as antigas.
  • (c) Com --from-beginning, o consumer reinicia o offset para 0 — reprocessa tudo desde o início. Isso é exatamente o mecanismo de reprocessamento da Arquitetura Kappa: se descobrimos um bug, rebobinamos o offset e reprocessamos todos os eventos com o código corrigido.

O que você precisa ter gravado
Log Imutável

Kafka é um log append-only. Mensagens têm offset sequencial. Replay disponível por retenção configurável.

Tópicos e Partições

Tópico = canal. Partição = unidade de paralelismo. Mesma chave → mesma partição → ordem garantida.

Consumer Groups

Cada partição = 1 consumer por grupo. Grupos diferentes = cópias independentes. Offset por grupo por partição.

Offset e Semânticas

At-least-once é o padrão. LAG = mensagens pendentes. Replay: setar offset para 0 reprocessa tudo.
