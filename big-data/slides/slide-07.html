O que é ingestão de dados — o primeiro passo de todo pipeline
Semanas 3–5

Nas semanas anteriores vimos onde os dados ficam armazenados (Data Lake, HDFS, S3) e como fluem em tempo real (Kafka). Mas ainda não respondemos: como os dados chegam até lá? Qualquer pipeline começa com a ingestão — e a escolha errada aqui compromete tudo que vem depois.

Definição e o problema central

Ingestão de dados é o processo de mover dados de sistemas de origem (bancos de dados, APIs, arquivos, sensores) para o sistema de destino (Data Lake, Data Warehouse, Kafka). Parece simples, mas em larga escala envolve decisões críticas:

⏱ Frequência
Mover tudo de uma vez (full load), só o que mudou (incremental) ou dado a dado em tempo real (streaming)?
📦 Volume
Como mover 10 TB de um banco Oracle legado para o S3 sem derrubar o sistema de origem?
🔄 Confiabilidade
O que acontece se a ingestão falhar no meio? Como garantir que nenhum registro seja perdido ou duplicado?
O mapa completo das estratégias
Batch (lote)
Dados movidos em blocos periódicos (hora, dia, semana)
Full Load: copia tudo — simples mas pesado
Incremental: copia só o que mudou — eficiente
Ferramentas: Airbyte, Sqoop, AWS Glue, Fivetran
✅ Relatórios diários, migração de BD, cargas noturnas
Streaming (evento a evento)
Dados movidos continuamente conforme chegam
Latência de milissegundos a segundos
CDC: captura cada INSERT/UPDATE/DELETE do BD de origem
Ferramentas: Kafka, Kinesis, Debezium, Flink
✅ Detecção de fraude, dashboards ao vivo, IoT, audit trail
Regra prática: começa com batch. Se a latência de batch (horas) não resolver o problema de negócio, evolui para streaming. Streaming é mais complexo e caro — só use quando realmente necessário.

Ingestão Batch — Airbyte e Sqoop na prática
Os três modos de ingestão batch
Full Load
  • Copia a tabela inteira toda vez
  • Simples de implementar
  • Pesado: uma tabela com 500M linhas copiada toda noite?
  • Útil para tabelas pequenas ou que não têm coluna de timestamp
Incremental (Append)
  • Copia só os registros novos desde a última execução
  • Usa coluna updated_at ou id sequencial como cursor
  • Muito mais eficiente que full load
  • Não captura deleções ou updates antigos
Incremental (Dedup)
  • Igual ao anterior, mas faz upsert no destino
  • Detecta registros que mudaram e os atualiza
  • Usa chave primária para identificar duplicatas
  • Padrão do Airbyte para a maioria dos conectores
Airbyte Open-source
Plataforma de integração com +300 conectores prontos (APIs, BDs, SaaS)
UI visual para configurar pipelines sem código
Suporte a full load, incremental e incremental dedup
Pode ser self-hosted (Docker) ou cloud (Airbyte Cloud)
Escrito em Java/Python; conectores customizáveis
Alternativas comerciais: Fivetran, Stitch, AWS Glue
Melhor para: conectar SaaS (Salesforce, Stripe, HubSpot), APIs REST e bancos relacionais ao Data Lake com mínimo de código.
Apache Sqoop Legado
Ferramenta para importar/exportar dados entre RDBMS e HDFS
Usa MapReduce por baixo — paralelismo automático
Divide a tabela em chunks e lê em paralelo com múltiplos mappers
Muito eficiente para grandes volumes de dados tabulares
Legado: ativamente mantido mas superado por Airbyte/Spark JDBC
Alternativa moderna: Spark com JDBC ou Airbyte
Melhor para: migrações de bancos legados (Oracle, Teradata) para HDFS em ambientes Hadoop tradicionais.
Ingestão incremental com Airbyte — conceito e exemplo

A ideia central do incremental é usar um cursor — uma coluna que cresce monotonicamente, como updated_at ou um ID sequencial. O Airbyte salva o último valor lido e, na próxima execução, começa dali.

SQL — Lógica do incremental manual (o que o Airbyte faz por baixo)
-- Última execução: cursor_value = '2024-01-14 23:59:59'
-- Próxima execução lê somente o que mudou depois disso

SELECT *
FROM orders
WHERE updated_at > '2024-01-14 23:59:59'   -- cursor da última run
ORDER BY updated_at ASC;

-- Airbyte salva o novo cursor: MAX(updated_at) desta execução
-- Próxima run usará esse novo valor como ponto de partida

-- Cuidado com o problema do "late arriving data":
-- Registros com updated_at no passado (chegam atrasados) serão ignorados!
-- Para resolver: usar janela de segurança (overlap de 1-2 horas)

SELECT *
FROM orders
WHERE updated_at > ('2024-01-14 23:59:59' - INTERVAL '2 hours')
-- overlap de 2h para capturar dados com atraso
Late arriving data: um pedido criado às 23:58 mas processado pelo sistema de origem às 00:05 terá updated_at = 00:05 — fora da janela da última run. A solução é adicionar um overlap de segurança e usar deduplicação no destino.

CDC — Change Data Capture: a evolução da ingestão incremental

O incremental com cursor tem uma limitação grave: ele não captura deleções. Se um registro for deletado do banco de origem, o incremental jamais saberá — o registro simplesmente some. O CDC resolve isso capturando diretamente do transaction log do banco de dados.

Todo banco relacional mantém um log de transações (Write-Ahead Log no PostgreSQL, Binary Log no MySQL, Redo Log no Oracle). O CDC lê esse log em tempo real e emite um evento para cada INSERT, UPDATE e DELETE — exatamente como aconteceram, na ordem exata.

Debezium — o CDC open-source

Debezium é a ferramenta CDC mais usada do mercado. Roda como um conector do Kafka Connect e lê o transaction log do banco de origem sem impacto significativo de performance.

JSON — Configuração do conector Debezium para PostgreSQL
{
  "name": "orders-cdc-connector",
  "config": {
    "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
    "database.hostname": "postgres-prod",
    "database.port":     "5432",
    "database.user":     "debezium",
    "database.password": "${file:/opt/kafka/secrets:db.password}",
    "database.dbname":   "ecommerce",
    "database.server.name": "orders",        // prefixo do tópico Kafka
    "table.include.list": "public.orders,public.customers",
    "plugin.name":       "pgoutput",          // replication plugin do PG
    "slot.name":         "debezium_orders",   // replication slot
    "publication.name":  "dbz_publication",

    // Snapshot inicial: ler todos os dados existentes antes de iniciar o CDC
    "snapshot.mode":     "initial",

    // Converter para Avro (Schema Registry)
    "value.converter":   "io.confluent.kafka.serializers.KafkaAvroSerializer",
    "schema.registry.url": "http://schema-registry:8081"
  }
}
Snapshot mode: na primeira execução, o Debezium faz um snapshot completo da tabela (lê tudo como eventos "r" - read) antes de começar a capturar mudanças em tempo real. Isso garante que o destino começa com o estado atual do banco.

Guia de decisão — escolhendo a estratégia certa
Árvore de decisão para ingestão
❓ Latência aceitável > 1 hora?
→ Batch — mais simples, menor custo. Escolha entre Full Load ou Incremental.
❓ Tabela tem coluna updated_at ou ID sequencial?
→ Sim: Incremental — eficiente, não sobrecarrega a origem. Não: Full Load — sem jeito de saber o que mudou.
❓ Precisa capturar deleções ou latência < 1 minuto?
→ CDC com Debezium — captura INSERTs, UPDATEs e DELETEs diretamente do transaction log.
❓ A fonte são eventos de aplicação (cliques, sessões, logs) e não um BD?
→ Kafka direto — o app publica no Kafka, sem intermediário. CDC é para bancos, não para eventos de app.

Atividade · Análise em grupos
Escolha de estratégia de ingestão para 4 cenários reais
Atividade em trios. Para cada cenário, o grupo deve: (a) identificar a estratégia mais adequada, (b) indicar a ferramenta específica e (c) justificar com critérios técnicos — latência, volume, necessidade de capturar deleções, impacto na fonte.
Cenário 1 — Plataforma de Marketplace

Um marketplace com 2 milhões de produtos precisa sincronizar o catálogo do PostgreSQL de origem para o Data Lake no S3 diariamente. Os produtos raramente são deletados (sellers preferem desativar). A tabela tem coluna updated_at. Volume: ~50 mil updates por dia.

Análise
Solução: Incremental com Airbyte
  • Estratégia: Incremental Dedup (Airbyte) usando updated_at como cursor
  • Por que não Full Load? 2M produtos copiados diariamente é desnecessário quando só 50k mudam
  • Por que não CDC? Deleções são raras e latência de horas é aceitável para catálogo
  • Configuração: Airbyte com conector PostgreSQL → S3, sync_mode=incremental, cursor=updated_at, frequência=1h
Cenário 2 — Sistema de Pagamentos Financeiros

Uma fintech precisa manter um audit trail completo de todas as transações financeiras: criações, alterações de status (pending→paid→refunded) e cancelamentos (DELETE). Qualquer mudança no banco de produção deve aparecer no Data Lake em menos de 30 segundos. Conformidade regulatória exige rastreabilidade de todas as operações.

Análise
Solução: CDC com Debezium + Kafka
  • Estratégia: CDC via Debezium lendo o WAL do PostgreSQL → Kafka → S3 via Kafka Connect Sink
  • Por que CDC? Dois requisitos obrigatórios: captura de DELETEs + latência <30s. Apenas CDC atende ambos.
  • Formato: Avro com Schema Registry — cada evento tem before/after state para audit trail completo
  • Impacto na origem: mínimo — Debezium lê o WAL, não executa queries na tabela
Cenário 3 — App de E-commerce (Events)

Um e-commerce quer capturar todos os eventos de comportamento do usuário no app: visualizações de produto, adição ao carrinho, início de checkout, abandono. São ~50 milhões de eventos por dia, gerados pelo frontend e backend do app em tempo real. Não há banco de dados de origem — os eventos são gerados diretamente pelo código.

Análise
Solução: Kafka direto (sem CDC)
  • Estratégia: App publica diretamente no Kafka — producer no backend (Node.js/Python) envia evento a cada ação do usuário
  • Por que não CDC? CDC é para bancos de dados. Aqui os eventos nascem no app, não existem num BD de origem
  • Por que não Batch? 50M eventos/dia — armazenar no app e enviar em batch aumenta risco de perda
  • Destino: Kafka → Kafka Connect Sink → S3 (Raw zone em Avro) → Spark para processar → Silver/Gold em Parquet
Cenário 4 — Migração de ERP Legado

Uma indústria precisa migrar 10 anos de dados do SAP (Oracle) para um Data Lake moderno. São 500 GB de dados históricos, tabelas sem coluna updated_at (sistema legado dos anos 2000). Após a migração inicial, as sincronizações serão mensais e apenas para relatórios analíticos — sem necessidade de tempo real.

Análise + Estratégia de migração
Solução: Full Load com Sqoop / Spark JDBC + Full Load mensal
  • Migração inicial: Sqoop ou Spark com JDBC para Oracle — paralelismo automático divide a tabela em chunks e lê em paralelo, minimizando o tempo de migração
  • Por que Full Load? Sem coluna updated_at, é impossível fazer incremental — não há como saber o que mudou
  • Sincronizações mensais: Full Load (tabelas pequenas) ou comparação via hash de linha (tabelas maiores)
  • Horário: rodar fora do horário de pico (madrugada) para não impactar o SAP
  • Consideração: após migrar, avaliar se vale adicionar updated_at nas tabelas do Oracle para futuras sincronizações incrementais

Batch Incremental

Cursor em updated_at. Eficiente. Não captura DELETEs. Airbyte para SaaS, Sqoop para HDFS legado.

Streaming

Kafka para eventos de app, Kinesis na AWS. Latência de ms. Producer publica, consumer lê com offset.

Decisão

Latência ok? → Batch. DELETE necessário? → CDC. Eventos de app? → Kafka direto. Sem updated_at? → Full Load.
