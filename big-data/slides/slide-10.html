Catalyst Optimizer — o cérebro por trás de toda query Spark
Semana 9

Você já sabe que broadcast join evita shuffle e que filtrar antes de juntar é mais rápido. Mas quem decide automaticamente essas otimizações quando você não as especifica manualmente? A resposta é o Catalyst Optimizer — o "cérebro" do Spark SQL.

O que é o Catalyst — e por que ele existe

Toda vez que você escreve uma operação DataFrame ou uma query Spark SQL, o código não é executado diretamente. Primeiro, o Spark constrói uma árvore de operações (Abstract Syntax Tree) e passa por um pipeline de otimização chamado Catalyst. Ele analisa, reescreve e escolhe o plano de execução mais eficiente — tudo isso automaticamente, sem que você precise pensar nisso.

Por que isso é poderoso? O mesmo código DataFrame pode gerar planos de execução completamente diferentes dependendo do tamanho dos dados, estatísticas das tabelas e configurações do cluster. Você escreve a intenção; o Catalyst decide o como.
As 4 fases do Catalyst Optimizer
1
Análise (Analysis)
Resolve referências de colunas e tabelas, valida tipos. Transforma o código em um Unresolved Logical Plan e depois em um Resolved Logical Plan, consultando o catálogo de metadados (schema).
2
Otimização Lógica (Logical Optimization)
Aplica regras baseadas em álgebra relacional: predicate pushdown (empurra filtros o mais perto possível da fonte), column pruning (remove colunas não usadas), constant folding (resolve expressões constantes em tempo de compilação).
3
Planejamento Físico (Physical Planning)
Gera múltiplos Physical Plans candidatos (ex: diferentes estratégias de join) e usa um modelo de custo (Cost-Based Optimizer) para escolher o mais eficiente — decide aqui se usa Broadcast Hash Join ou Shuffle Hash Join.
4
Geração de Código (Code Generation)
Usa o Tungsten para gerar bytecode Java otimizado diretamente — eliminando overhead de interpretação e aproveitando recursos de CPU como vetorização e gerenciamento de memória off-heap.
Predicate Pushdown — a otimização mais comum

Quando você escreve um filtro depois de várias operações, o Catalyst frequentemente "empurra" esse filtro para o mais próximo possível da leitura dos dados — às vezes até para dentro do próprio arquivo Parquet, evitando ler dados que seriam descartados de qualquer forma.

PySpark — código que você escreve vs o que o Catalyst executa
# ── O QUE VOCÊ ESCREVE (lógica de negócio, ordem natural) ─────
resultado = (df
    .join(df_clientes, "customer_id")
    .select("order_id", "total", "customer_name", "state")
    .filter(col("state") == "SP")   # filtro escrito DEPOIS do join
)

# ── O QUE O CATALYST REALMENTE EXECUTA (otimizado) ────────────
# O Catalyst detecta que o filtro de "state" só depende da tabela
# de clientes, então ele MOVE o filtro para ANTES do join:

# 1. Lê apenas as colunas necessárias (column pruning)
# 2. Filtra clientes de SP ANTES de juntar (predicate pushdown)
# 3. Junta um volume muito menor de dados
# 4. Resultado final é IDÊNTICO, mas muito mais rápido
Você não precisa fazer isso manualmente! Diferente do que vimos na Semana 9 sobre "filtrar antes de juntar" — esse era um exemplo manual didático. Na prática, o Catalyst já reordena filtros automaticamente na maioria dos casos. Mas ele não é mágico: para joins complexos ou UDFs, vale a pena confirmar com explain() que a otimização realmente ocorreu.

explain() — lendo o plano de execução como um mapa
df.explain() — a ferramenta mais importante desta semana

O método .explain() mostra o plano de execução de qualquer DataFrame ou query SQL — sem executar nada de fato. É a forma de "ver" o que o Catalyst decidiu fazer antes de gastar tempo de cluster.

PySpark — modos do explain()
df.explain()                  # plano físico simples (padrão)
df.explain("extended")       # mostra logical E physical plan
df.explain("formatted")      # formato legível, com numeração
df.explain("cost")           # inclui estimativas de custo (estatísticas)
df.explain("codegen")        # mostra o código Java gerado pelo Tungsten
Lendo um plano de execução real

Vamos analisar o plano de uma query: df_pedidos.filter(col("total") > 100).join(broadcast(df_categorias), "category_id")

== Physical Plan ==
*(2) BroadcastHashJoin [category_id], [category_id], LeftOuter :- *(2) Project [order_id#1, customer_id#2, total#3, category_id#4] : +- *(2) Filter (isnotnull(total#3) AND (total#3 > 100.0)) : +- *(2) ColumnarToRow : +- FileScan parquet orders PushedFilters: [IsNotNull(total), GreaterThan(total,100.0)] : ReadSchema: struct<order_id:int,customer_id:int,total:double,category_id:int> +- BroadcastExchange HashedRelationBroadcastMode +- *(1) FileScan parquet categories
✅ Sinais bons neste plano
  • PushedFilters: o filtro foi empurrado até o FileScan — só lê linhas com total > 100
  • BroadcastHashJoin: o Catalyst escolheu broadcast — sem shuffle pesado
  • ReadSchema reduzido: só as colunas necessárias foram lidas (column pruning)
🔍 O que observar
  • Exchange / ShuffleExchange: indica shuffle — procure por esses termos no plano
  • SortMergeJoin: join sem broadcast — geralmente mais caro
  • BatchScan sem PushedFilters: sinal de que o filtro não foi otimizado
Quando o Predicate Pushdown FALHA — o problema das UDFs

Lembra das UDFs (Semana 9)? Elas são uma "caixa preta" para o Catalyst. Ele não consegue analisar o que está dentro de uma função Python customizada — então não pode otimizar filtros que dependem dela.

== Physical Plan com UDF (SEM otimização) ==
*(1) Filter pythonUDF0#15 +- BatchEvalPython [classificar_cliente(total#3)], [pythonUDF0#15] +- *(1) FileScan parquet orders PushedFilters: [] ReadSchema: struct<order_id:int,...,total:double,...todas_colunas>
Repare: PushedFilters: [] está vazio e todas as colunas foram lidas — o Catalyst não conseguiu otimizar nada porque o filtro depende de uma UDF opaca. Isso confirma a lição da semana passada: evite UDFs quando uma função nativa do pyspark.sql.functions resolver o mesmo problema.
🙋 Quiz — você analisa o explain() de uma query e vê "PushedFilters: []" mesmo tendo escrito um filter(). O que provavelmente está acontecendo?
A
É normal — Spark nunca empurra filtros para o FileScan
B
O filtro provavelmente depende de uma UDF, que é opaca para o Catalyst Optimizer
C
O arquivo de origem não é Parquet, então não há como aplicar filtros
D
O cluster não tem memória suficiente para aplicar o filtro

Cache e Persist — evitando recomputar o que já foi calculado
O problema: lazy evaluation recalcula tudo

Lembre-se da Semana 8: transformações são lazy. Isso significa que se você usar o mesmo DataFrame em múltiplas actions, o Spark recalcula todo o DAG desde o início cada vez — incluindo leitura do disco e todas as transformações anteriores.

❌ Sem cache — recalcula tudo
df_caro = df.join(...).groupBy(...).agg(...)

df_caro.count()      # executa TUDO (1ª vez)
df_caro.show()       # executa TUDO de novo!
df_caro.write.parquet("out/")  # executa TUDO 3ª vez!
O join pesado e o groupBy são recalculados 3 vezes — desperdício de tempo de cluster.
✅ Com cache — calcula uma vez
df_caro = df.join(...).groupBy(...).agg(...)
df_caro.cache()      # marca para cachear

df_caro.count()      # executa E guarda em memória
df_caro.show()       # usa o cache — rápido!
df_caro.write.parquet("out/")  # usa o cache!
O resultado é calculado uma única vez e reutilizado nas próximas actions.
cache() vs persist() — níveis de armazenamento

cache() é um atalho para persist(StorageLevel.MEMORY_AND_DISK). Mas o persist() permite escolher exatamente onde guardar os dados.

StorageLevelOnde guardaQuando usar
MEMORY_ONLYSó RAM. Se não couber, recalculaDataset pequeno que cabe na memória, acesso rápido
MEMORY_AND_DISKRAM, e se não couber, derrama pro discoPadrão recomendado — equilíbrio entre velocidade e segurança
DISK_ONLYSó discoDataset muito grande, memória limitada
MEMORY_ONLY_SERRAM, mas serializado (mais compacto)Economizar memória ao custo de CPU para des/serializar
PySpark — persist com nível customizado e unpersist
from pyspark import StorageLevel

df_caro = df.join(df2, "id").groupBy("categoria").agg(sum("total"))

# Persist com nível específico
df_caro.persist(StorageLevel.MEMORY_AND_DISK)

# IMPORTANTE: cache/persist é lazy também! Só materializa na 1ª action
df_caro.count()    # aqui sim o cache é preenchido

# ... usar df_caro várias vezes ...

# SEMPRE libere a memória quando não precisar mais!
df_caro.unpersist()
Erro comum: cachear DataFrames pequenos ou que são usados só uma vez. Cache tem overhead — só use quando o DataFrame for reutilizado múltiplas vezes e for caro de recalcular (após joins, agregações pesadas). Cachear tudo "por garantia" desperdiça memória do cluster.
Tungsten — a camada de execução nativa

Tungsten é o projeto que revolucionou a performance do Spark a partir da versão 1.5, focando em três pilares: gerenciamento de memória eficiente (off-heap), geração de código (whole-stage code generation) e algoritmos cache-friendly (vetorização de CPU).

Memória off-heap
Spark gerencia memória diretamente em formato binário, evitando overhead do garbage collector da JVM.
Whole-Stage CodeGen
Combina várias operações em uma única função Java compilada, eliminando overhead de chamadas virtuais.
Cache-aware
Algoritmos desenhados para aproveitar o cache da CPU (L1/L2/L3), reduzindo acessos à RAM.
Você não precisa configurar nada manualmente — o Tungsten roda automaticamente em todo DataFrame/Spark SQL. Mencionar isso é importante para entender por que DataFrame é tão mais rápido que RDD: RDD não passa pelo Tungsten, só pelo Catalyst (parcial); DataFrame passa pelos dois.
🙋 Quiz — você tem um DataFrame resultado de um join pesado que será usado em 5 actions diferentes. O que você deve fazer?
A

B
Usar .cache() ou .persist() após o join, e chamar uma action para materializar antes das próximas 5 actions
C

Caso de estudo — query lenta para diagnosticar

O professor vai apresentar a seguinte query lenta. Sua missão: usar explain() para identificar os problemas antes de otimizar.

PySpark — query original (lenta)
from pyspark.sql.functions import udf
from pyspark.sql.types import StringType

@udf(returnType=StringType())
def categoria_valor(total):
    if total > 1000: return "alto"
    elif total > 100: return "medio"
    else: return "baixo"

# Lê TODAS as colunas de duas tabelas grandes
df_pedidos = spark.read.parquet("data/pedidos/")    # 5M linhas
df_produtos = spark.read.parquet("data/produtos/")   # 200 linhas (!)

# Join SEM broadcast explícito + UDF + filtro depois do join
resultado = (df_pedidos
    .join(df_produtos, "product_id")
    .withColumn("faixa", categoria_valor(col("total")))
    .filter(col("faixa") == "alto")
    .groupBy("product_name")
    .agg(sum("total").alias("receita"))
)
resultado.show()
Sua tarefa: execute resultado.explain("formatted") e identifique pelo menos 3 problemas de performance neste código antes de ver a solução.
Solução otimizada — comparando lado a lado
PySpark — query otimizada
from pyspark.sql.functions import col, when, sum, broadcast

df_pedidos = spark.read.parquet("data/pedidos/")
df_produtos = spark.read.parquet("data/produtos/")

resultado = (df_pedidos
    .filter(col("total") > 1000)              # filtro ANTES, sem UDF
    .select("product_id", "total")         # só colunas necessárias
    .join(broadcast(df_produtos), "product_id")  # broadcast explícito
    .groupBy("product_name")
    .agg(sum("total").alias("receita"))
)
resultado.show()

Filtro de "faixa" aplicado DEPOIS do joinFiltro de total aplicado ANTES do joinJunta muito menos linhas
