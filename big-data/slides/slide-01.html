Onde você está pisando

Você está no terceiro período de um curso de Inteligência Artificial. Até agora você aprendeu a programar, modelar bancos de dados e entender algoritmos. Este semestre dá um salto de escala: em vez de milhares de registros, vamos lidar com bilhões.

Big Data e Engenharia de Dados é a disciplina que responde à pergunta que todo projeto de IA real enfrenta: "nosso modelo funcionou no notebook com 10 mil linhas — e agora, com 10 bilhões?". A resposta envolve arquiteturas distribuídas, frameworks específicos e uma mentalidade completamente diferente de como processar dados.

Por que Big Data é inseparável de IA? Modelos de machine learning são tão bons quanto os dados que os treinaram. GPT-4 foi treinado com ~1 trilhão de tokens. O sistema de recomendação do Netflix usa dados de 238 milhões de usuários. Sem Big Data, IA de escala real simplesmente não existe.

V1 — Volume: o primeiro problema

Volume é o V mais intuitivo: a quantidade de dados gerada cresce em ritmo que os bancos de dados tradicionais não conseguem acompanhar. Não é só uma questão de armazenamento — é uma questão de processamento, custo e arquitetura.

  • Câmeras, radar, acelerômetro, GPS
  • Alimenta modelos de piloto automático
Por que o PostgreSQL não aguenta?

Um banco de dados relacional como PostgreSQL ou MySQL é incrível para dezenas de GB ou poucos TB — e você vai continuar usando eles para isso. O problema começa quando os dados crescem além da capacidade de uma única máquina.

🐘 PostgreSQL — o que acontece com 10 TB
Query que rodava em 2 segundos passa a demorar 20 minutos
Backup demora horas — janela de manutenção impossível
INSERT/UPDATE durante carga pesada causa lock e timeout
Adicionar RAM e disco tem limite físico e fica caro demais
Solução: escala vertical (máquina maior) — mas tem teto
VS
⚡ Spark + HDFS — o que muda
Os 10 TB são divididos em blocos de 128 MB espalhados em 100 máquinas
Query processada em paralelo por todas as máquinas ao mesmo tempo
Falha de uma máquina? Os dados existem em 3 cópias (replicação)
Precisou de mais capacidade? Adiciona máquinas ao cluster
Solução: escala horizontal — sem teto prático
Escala vertical vs. horizontal: trocar o servidor por um maior (vertical) tem limite físico e custo exponencial. Adicionar mais servidores comuns ao cluster (horizontal) é como contratar mais funcionários — você divide o trabalho e o custo é linear.
V2 — Velocidade: a dimensão do tempo

Velocidade não é só "os dados chegam rápido". É sobre qual é o tempo máximo aceitável entre a geração do dado e a tomada de decisão. Esse tempo muda tudo na arquitetura.

ModeloLatência aceitávelTecnologia típicaExemplo
Batch Horas a dias Hadoop MapReduce, Spark batch Relatório de vendas do mês
Micro-batch Segundos a minutos Spark Structured Streaming Dashboard de pedidos por hora
Streaming Milissegundos Kafka + Flink / Storm Detecção de fraude no cartão
Real-time < 1ms Hardware dedicado, FPGA Trading de alta frequência
Exemplo concreto — detecção de fraude no Nubank: quando você passa o cartão, o sistema tem menos de 200ms para decidir aprovar ou recusar. Isso significa que o modelo de ML precisa rodar, o resultado precisa chegar ao terminal e a resposta precisa aparecer — tudo em menos de um piscar de olhos. Isso só é possível com processamento em streaming.
🙋 Quiz — uma empresa quer analisar os logs do servidor para detectar ataques em andamento. Qual modelo de processamento é adequado?
A

B

V3, V4 e V5 — Variedade, Veracidade e Valor
V3 — Variedade

Os dados do mundo real não chegam arrumados em tabelas. Eles chegam em dezenas de formatos diferentes — e um sistema de Big Data precisa lidar com todos eles juntos.

ESTRUTURADO
Tabelas SQL, planilhas Excel, CSVs com schema fixo. Fácil de consultar.
SEMIESTRUTURADO
JSON, XML, Parquet, logs de sistema. Tem estrutura, mas flexível.
NÃO ESTRUTURADO
Imagens, vídeos, áudios, textos livres. Representa ~80% dos dados existentes.
~80% dos dados do mundo são não estruturados
V4 — Veracidade

Veracidade é sobre a confiabilidade dos dados. A questão não é só se o dado está lá — é se ele é correto, consistente e confiável o suficiente para tomar decisões.

  • Sensor com defeito transmitindo temperatura de -999°C
  • Usuário digitando "aaaa" no campo de nome
  • Data de nascimento futura: 15/03/2089
  • Duplicatas: mesma transação registrada três vezes
  • Solução: validação na ingestão + regras de qualidade no dbt
Garbage in, garbage out: se o dado de entrada for ruim, o modelo de IA treinado nele vai ser ruim — não importa quão sofisticado seja o algoritmo.
V5 — Valor: o único V que realmente importa no final

Volume, velocidade, variedade e veracidade são desafios técnicos. Valor é o objetivo de negócio. Ter petabytes de dados não tem nenhum valor por si só — o valor está no que você consegue extrair deles.

📊
Insight
"Nossa taxa de churn é maior às quintas-feiras para usuários do plano básico"
🤖
Predição
"Este cliente tem 87% de probabilidade de cancelar nos próximos 30 dias"
⚡
Ação
"Disparar oferta personalizada de upgrade antes que ele cancele"
BD Tradicional vs. Big Data — o mapa completo
CritérioBD Tradicional (PostgreSQL, MySQL)Big Data (Spark, HDFS, Kafka)
VolumeMB a alguns TB (1 máquina)TB a ZB (centenas de máquinas)
VelocidadeBatch, queries pontuaisBatch + streaming em tempo real
VariedadeDados estruturados, schema fixoEstruturado, semi e não estruturado
EscalabilidadeVertical (máquina maior)Horizontal (mais máquinas)
ConsistênciaACID forte (transações garantidas)Eventual consistency (BASE)
SchemaSchema-on-write (defina antes de inserir)Schema-on-read (define ao ler)
LinguagemSQL padrãoSQL + Python/Scala + APIs específicas
Custo inicialBaixo (1 servidor)Alto (cluster de máquinas)
Maturidade50+ anos de mercado15–20 anos, ecossistema em evolução
Não caia na armadilha: Big Data não substitui banco de dados relacional. Uma startup com 50 mil clientes que usa Kafka + Spark está gastando dinheiro à toa e adicionando complexidade desnecessária. Use Big Data quando o problema realmente exige — e não antes.

Big Data como combustível da IA — o pipeline real

Modelos de IA não surgem do nada — eles são treinados, validados e monitorados em dados. A engenharia de dados é a infraestrutura invisível que torna tudo isso possível.

Pipeline de dados para IA — visão geral
# 1. INGESTÃO — dados brutos chegam de várias fontes
kafka: eventos de clique do app → tópico "user_events"
airbyte: sincroniza banco de dados transacional → Data Lake (raw)

# 2. TRANSFORMAÇÃO — limpeza e enriquecimento (dbt ou Spark)
SELECT
  user_id,
  COUNT(event) AS clicks_today,
  MAX(timestamp) AS last_active,
  AVG(session_duration) AS avg_session_sec
FROM raw.user_events
WHERE DATE(timestamp) = CURRENT_DATE
GROUP BY user_id

# 3. FEATURE STORE — features prontas para treino e inferência
feast.materialize(feature_views=["user_engagement"], start=yesterday)

# 4. TREINO — modelo consome features históricas
df = feast.get_historical_features(entity_df, feature_refs)
model.fit(df[features], df["target_churn"])

# 5. INFERÊNCIA — modelo usa features em tempo real
features_live = feast.get_online_features(user_id=12345)
prediction = model.predict(features_live)  # → 0.87 (87% churn risk)
Feature Store: é o "supermercado" de features para IA. Você calcula as features uma vez usando Big Data e as reutiliza tanto no treino (dados históricos) quanto na inferência em produção (dados em tempo real). Vamos construir isso na Semana 17.

Organização
Parte 2 — Mapeamento dos 5 Vs (20 min)

Para o caso real escolhido pelo grupo, identifique e justifique cada V:

  • Volume: qual é a escala dos dados? (KB, MB, GB, TB, PB)
  • Velocidade: batch, micro-batch ou streaming? Qual latência é aceitável?
  • Variedade: quais tipos de dados estão envolvidos? (estruturado, semiestruturado, não estruturado)
  • Veracidade: quais problemas de qualidade de dados existem nesse contexto?
  • Valor: qual é o insight, predição ou ação gerada pelos dados?
Análise crítica
Parte 3 — Desafio: qual seria o maior risco? (10 min)

Para o caso do seu grupo: se a veracidade dos dados fosse comprometida (dados corrompidos, sensor com defeito, dados maliciosamente manipulados), qual seria o impacto real no negócio e nas pessoas? Quem seria prejudicado? Como isso poderia ser mitigado?

Ética e impacto
Exemplo: Agricultura de precisão — Big Data no campo
  • Volume: 1 sensor por 2 hectares em uma fazenda de 10.000 ha = 5.000 sensores × 1 leitura/minuto = ~7M leituras/dia (~500 MB/dia)
  • Velocidade: micro-batch de 15 minutos — dados de umidade do solo chegam e ativam irrigação. Streaming para alertas de chuva ou geada.
  • Variedade: sensores (estruturado), imagens de drone (não estruturado), previsão do tempo via API (semiestruturado/JSON), histórico de safras (planilhas)
  • Veracidade: sensor com defeito transmitindo umidade 0% — sistema liga irrigação desnecessariamente → desperdício de água e custo elevado
  • Valor: redução de 30% no consumo de água, aumento de 15% na produtividade, alerta precoce de pragas com modelo de visão computacional
Risco de veracidade: se sensores de umidade falharem silenciosamente (leituras incorretas mas plausíveis), o modelo de irrigação vai tomar decisões erradas. A mitigação é validação cruzada entre sensores próximos + alertas de anomalia estatística. Isso é V4 — Veracidade — na prática.

O que você precisa ter gravado
Volume

Escala que uma máquina não aguenta. BD tradicional → vertical. Big Data → horizontal.

Velocidade

Batch (horas), micro-batch (minutos) ou streaming (ms). Latência define a arquitetura.

Variedade

Estruturado, semiestruturado e não estruturado. ~80% dos dados do mundo são não estruturados.

Veracidade

Dados ruins → decisões ruins. Qualidade na entrada é tão importante quanto o algoritmo.

Valor

O único objetivo final. Dado sem valor extraído é custo, não ativo.
