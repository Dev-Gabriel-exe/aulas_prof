Por que governança de dados importa
Semana 15

Você já sabe ingerir, processar em batch e streaming, transformar com dbt, orquestrar com Airflow e armazenar com garantias ACID via Delta Lake. Ao longo de 15 semanas, seu pipeline já gerou dezenas de tabelas — mas quem sabe o que cada uma significa, de onde vem e quem pode acessá-la? É disso que trata a governança.

O que é governança de dados

Governança de dados é o conjunto de processos, papéis, políticas e ferramentas que garantem que os dados de uma organização sejam confiáveis, descobríveis, seguros e usados de forma responsável. Não é uma ferramenta única — é uma combinação de catalogação, linhagem, qualidade (Semana 11) e conformidade regulatória.

🔍 Descobribilidade
Qualquer pessoa consegue encontrar e entender uma tabela sem perguntar a quem a criou.
🔗 Rastreabilidade
É possível saber de onde um dado veio e para onde ele vai (linhagem).
🔒 Conformidade
Dados sensíveis são identificados e tratados conforme LGPD e políticas internas.
O que acontece sem governança
❌ "Tribal knowledge"
  • Só uma pessoa sabe o que a tabela fct_v2_final_use_this_one significa
  • Essa pessoa sai da empresa — o conhecimento vai junto
  • Dados órfãos: tabelas duplicadas, sem dono, sem documentação
✅ Com governança
  • Cada tabela tem descrição, dono (owner) e tags de negócio
  • Novos membros do time encontram e entendem os dados sozinhos
  • Mudanças em uma tabela mostram imediatamente quem é impactado (linhagem)
Caso real: imagine descobrir, meses depois, que uma coluna de CPF estava sendo replicada sem anonimização em 8 dashboards diferentes de BI — porque ninguém sabia que aquela tabela alimentava tantos consumidores. Isso é exatamente o tipo de problema que linhagem e catalogação resolvem antes que vire incidente de compliance.
🙋 Quiz — qual das opções melhor descreve o principal risco de uma empresa NÃO ter governança de dados?
A
As queries SQL ficam mais lentas de executar
B
Conhecimento sobre os dados fica concentrado em poucas pessoas, e mudanças podem impactar consumidores desconhecidos
C
O Spark não consegue processar os dados corretamente
D
O custo de armazenamento em nuvem aumenta automaticamente

Data Catalog — o "Google" dos dados da empresa
O que é um Data Catalog

Um Data Catalog é um inventário centralizado e pesquisável de todos os ativos de dados de uma organização — tabelas, dashboards, pipelines, modelos de ML — com metadados que descrevem o que cada um é, quem o mantém e como ele se conecta a outros ativos.

Metadados técnicos
Schema (colunas, tipos)
Volume, frequência de atualização
Localização física (path, banco)
Metadados de negócio
Descrição em linguagem natural
Dono (owner) e time responsável
Tags de domínio (finance, marketing)
Metadados operacionais
Última execução, status do job
SLA e freshness (atualidade)
Histórico de qualidade (testes dbt)
Metadados de governança
Classificação de sensibilidade (PII)
Políticas de acesso e retenção
Conformidade regulatória (LGPD)
OpenMetadata — arquitetura e conceitos principais

OpenMetadata é uma plataforma open-source de catalogação que se conecta a dezenas de fontes (Postgres, Snowflake, dbt, Airflow, Kafka, Delta Lake) via conectores automáticos, extraindo metadados sem exigir documentação manual completa.

ConceitoO que representa
ServiceUma conexão configurada com uma fonte (ex: "postgres-producao", "dbt-cloud")
Database / SchemaEstrutura hierárquica espelhando a organização real do banco de dados
Table / Topic / DashboardOs ativos de dados propriamente ditos, catalogados com metadados completos
GlossaryDicionário de termos de negócio compartilhado (ex: "o que é 'cliente ativo'?")
Tag / ClassificationRótulos aplicados a colunas ou tabelas (ex: PII.Sensitive, Tier.Gold)
Ingestão automática: o OpenMetadata roda "connectors" agendados (parecido com um Airflow interno) que se conectam à fonte, leem o schema e populam o catálogo automaticamente — reduzindo drasticamente o esforço manual de manter a documentação atualizada.
🙋 Quiz — qual é a principal vantagem de um Data Catalog como o OpenMetadata usar conectores automáticos em vez de documentação manual?
A
Os conectores tornam as queries SQL mais rápidas
B
Reduzem o esforço manual e mantêm o catálogo atualizado automaticamente conforme os dados mudam
C
Eliminam a necessidade de ter donos (owners) para as tabelas
D
Garantem automaticamente a conformidade com a LGPD

Linhagem — rastreando de onde vêm e para onde vão os dados
O que é linhagem de dados (data lineage)

Linhagem é o mapeamento visual e rastreável de como um dado se transforma ao longo do pipeline: de qual sistema de origem ele veio, quais transformações sofreu, e em quais tabelas, dashboards ou modelos ele termina sendo usado.

postgres.orders
stg_orders
fct_revenue_daily
Dashboard: Receita Executiva

Essa cadeia mostra que qualquer mudança em postgres.orders (uma nova coluna, um tipo alterado) tem impacto rastreável até o dashboard executivo — sem linhagem, essa conexão só existiria na cabeça de quem construiu o pipeline.

Linhagem upstream vs downstream
⬆️ Upstream (de onde vem)
Pergunta: "quais fontes alimentam esta tabela?"
Útil para debugar: "por que este número está errado?"
Rastreia até a fonte bruta original
⬇️ Downstream (para onde vai)
Pergunta: "o que depende desta tabela?"
Útil para avaliar impacto: "posso deletar esta coluna com segurança?"
Rastreia até dashboards e modelos finais
Conexão com o que você já sabe: o ref() do dbt (Semana 11) já constrói uma linhagem implícita entre modelos dbt. Ferramentas como OpenMetadata estendem essa ideia para TODO o ecossistema — incluindo fontes fora do dbt, dashboards de BI e até modelos de Machine Learning.
Como a linhagem é capturada na prática
  • Análise estática de SQL: a ferramenta lê as queries (dos modelos dbt, views, stored procedures) e identifica automaticamente quais tabelas alimentam quais.
  • Metadados de execução: Airflow e Spark expõem logs de execução que revelam quais inputs geraram quais outputs.
  • Anotação manual: quando a transformação acontece fora do alcance automático (ex: script Python customizado), a linhagem é declarada manualmente na ferramenta.
🙋 Quiz — um engenheiro precisa saber se é seguro remover uma coluna de uma tabela staging sem quebrar nenhum dashboard. Que tipo de linhagem ele deve consultar?
A
Upstream — para ver de onde a coluna se origina
B
Downstream — para ver tudo que depende dessa tabela/coluna
C
Nenhuma — linhagem não ajuda nesse tipo de decisão
D
Apenas a documentação técnica da tabela, sem linhagem

Classificação de dados sensíveis e LGPD aplicada
Classificação de dados sensíveis

Antes de aplicar qualquer política de proteção, é preciso saber onde os dados sensíveis estão. A classificação identifica e rotula colunas conforme seu nível de sensibilidade:

PII (Personally Identifiable Information)
CPF, e-mail, telefone, endereço — qualquer dado que identifica uma pessoa diretamente.
Dados sensíveis (LGPD Art. 5º)
Origem racial, saúde, orientação sexual, dados biométricos — exigem tratamento ainda mais restrito.
Confidencial (negócio)
Dados financeiros internos, contratos, estratégia — sensíveis para a empresa, não necessariamente pessoais.
Público
Dados que podem ser compartilhados livremente, sem restrição de acesso.
Na prática: ferramentas de catalogação como o OpenMetadata podem aplicar classificação automática via reconhecimento de padrões (regex para CPF, e-mail) combinada com revisão humana — reduzindo o risco de uma coluna sensível passar despercebida.
LGPD aplicada à engenharia de dados
  • Princípio da minimização: só colete e mantenha os dados pessoais estritamente necessários para a finalidade declarada.
  • Anonimização/pseudonimização: dados sensíveis devem ser tratados antes de chegarem a ambientes de analytics amplamente acessíveis (conexão direta com a Semana 11 — ETL vs ELT em contextos regulados).
  • Direito ao esquecimento: um titular pode solicitar a exclusão de seus dados — isso exige que a empresa saiba EXATAMENTE onde esses dados estão replicados (linhagem é essencial aqui).
  • Registro de tratamento: é preciso documentar por que cada dado pessoal é coletado e processado — o catálogo de dados é a ferramenta natural para esse registro.
Como tudo se conecta: classificação identifica ONDE estão os dados sensíveis; linhagem mostra PARA ONDE eles se propagam; e só com essas duas peças é possível atender de forma confiável a uma solicitação de exclusão sob a LGPD.

Abordagem esperada
  • Consultar a linhagem DOWNSTREAM a partir de customers.email, não upstream — a pergunta é "o que quebra se eu remover", não "de onde vem".
