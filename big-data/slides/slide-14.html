Por que um Data Lake "puro" não é suficiente
Pós-P2

Você já sabe ingerir dados (Semana 7), processá-los com Spark (Semanas 8-10) e transformá-los com dbt orquestrado por Airflow (Semanas 11-12). Falta uma peça: onde e como esses dados ficam armazenados de forma confiável. É disso que trata o Delta Lake.

O problema do Data Lake tradicional

Um Data Lake tradicional é, na prática, uma pasta cheia de arquivos Parquet em um object storage (S3, HDFS, ADLS). Isso funciona bem para leitura, mas quebra em cenários comuns de engenharia de dados:

❌ Sem Delta Lake
  • Sem atomicidade — job falha no meio da escrita e a pasta fica com dados parciais
  • Sem isolamento — leitor pode ver arquivos novos e antigos misturados
  • Parquet é imutável — corrigir uma linha exige reescrever o arquivo inteiro
  • Sem histórico — dado sobrescrito desaparece, impossível auditar
✅ Com Delta Lake
  • Escritas atômicas — tudo ou nada, nunca dados parciais
  • Isolamento — leitores sempre veem a última versão consistente
  • UPDATE/DELETE/MERGE eficientes, sem reescrever a tabela inteira
  • Histórico completo — time travel para qualquer versão anterior
Origem: o Delta Lake foi criado pela Databricks em 2019 como formato aberto exatamente para resolver esses problemas sem abandonar o ecossistema de object storage barato que já conhecemos.
A arquitetura Lakehouse

O termo Lakehouse descreve a combinação do custo baixo e flexibilidade de um Data Lake (armazenamento em object storage, formatos abertos) com as garantias transacionais e de performance que antes só existiam em Data Warehouses.

Delta Lake não substitui o Parquet — ele é uma camada de metadados sobre arquivos Parquet que adiciona um log de transações. É por isso que se diz que o Delta Lake "traz ACID ao Data Lake".

Conexão com o que você já sabe: as zonas raw/silver/gold do Data Lake (Semana 3) continuam existindo — o Delta Lake apenas torna cada uma dessas zonas transacional e versionada, em vez de ser só uma pasta de arquivos Parquet soltos.

ACID, o _delta_log e o MERGE INTO
As 4 garantias ACID aplicadas ao Data Lake
  • Atomicity: uma escrita só é considerada concluída se TODOS os arquivos Parquet envolvidos foram gravados com sucesso. Se algo falhar no meio, nada é commitado.
  • Consistency: o schema da tabela é validado a cada escrita — não é possível deixar a tabela em um estado inválido.
  • Isolation: leitores sempre veem uma versão consistente da tabela, nunca um estado intermediário de uma escrita em andamento.
  • Durability: uma vez commitado, o estado fica registrado permanentemente no transaction log, no próprio object storage.
Como o _delta_log funciona na prática

Toda tabela Delta tem uma subpasta _delta_log/ ao lado dos arquivos de dados. Cada escrita gera um novo arquivo JSON numerado sequencialmente, descrevendo quais arquivos foram adicionados e quais foram removidos naquela transação:

v000000000000000000000.json — criação da tabela (INSERT inicial)
v100000000000000000001.json — primeiro INSERT (append de novos pedidos)
v200000000000000000002.json — um UPDATE via MERGE INTO
v300000000000000000003.json — um DELETE de registros de teste

Os arquivos Parquet antigos não são apagados imediatamente — eles ficam disponíveis para leitura de versões anteriores, o que possibilita o time travel (Aula 3).

Como a leitura funciona: quando um engine lê uma tabela Delta, ele primeiro lê o transaction log para saber quais arquivos Parquet compõem a versão atual — e só então lê os dados. É essa indireção que garante isolamento e consistência.
Criando uma tabela e usando MERGE INTO
Python — PySpark + Delta Lake
from pyspark.sql import SparkSession
from delta.tables import DeltaTable

spark = (SparkSession.builder
    .appName("delta-lake-demo")
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension")
    .config("spark.sql.catalog.spark_catalog", "org.apache.spark.sql.delta.catalog.DeltaCatalog")
    .getOrCreate())

df = spark.read.parquet("s3://meu-bucket/raw/pedidos/")

# Escreve como tabela Delta
df.write.format("delta").mode("overwrite").save("s3://meu-bucket/delta/pedidos/")

# MERGE INTO — upsert atômico (insere novos, atualiza existentes)
delta_tabela = DeltaTable.forPath(spark, "s3://meu-bucket/delta/pedidos/")

(delta_tabela.alias("destino")
    .merge(df_novos_pedidos.alias("origem"), "destino.id = origem.id")
    .whenMatchedUpdateAll()
    .whenNotMatchedInsertAll()
    .execute())
MERGE INTO é a operação mais importante do Delta Lake para engenharia de dados: resolve, em uma única transação atômica, o clássico problema de "como faço upsert em um Data Lake" — algo impossível de fazer com segurança em Parquet puro.

Time Travel, Schema Evolution e Performance
Time Travel — consultando o passado dos dados

Como o transaction log preserva o histórico de versões, é possível consultar a tabela exatamente como ela estava em qualquer ponto do tempo — por número de versão ou por timestamp:

Python / SQL — Time Travel
# Por número de versão
df_v5 = spark.read.format("delta").option("versionAsOf", 5).load("s3://.../delta/pedidos/")

# Por timestamp
df_ontem = (spark.read.format("delta")
    .option("timestampAsOf", "2026-07-01")
    .load("s3://.../delta/pedidos/"))

# Via SQL
spark.sql("SELECT * FROM pedidos VERSION AS OF 5")
spark.sql("SELECT * FROM pedidos TIMESTAMP AS OF '2026-07-01'")

# Ver o histórico completo de transações
delta_tabela.history().show()
Casos de uso: auditoria regulatória, reproduzir um relatório antigo, debugar "por que esse número mudou", e desfazer um DELETE ou UPDATE acidental com um RESTORE.
Limitação: depende dos arquivos Parquet antigos ainda existirem. Rodar VACUUM com retenção curta apaga fisicamente esses arquivos e destrói o histórico.
Schema Enforcement vs Schema Evolution
🔵 Schema Enforcement (padrão)
O Delta Lake REJEITA qualquer escrita cujo schema não bata com o atual
Coluna faltando, tipo incompatível ou coluna extra geram erro
Protege contra corrupção silenciosa de schema
🟢 Schema Evolution (opt-in)
Habilitada explicitamente com mergeSchema=true
Usada quando a mudança é intencional (nova coluna na origem)
NÃO é o padrão — proteção, não bug
Python — habilitando schema evolution
df_novo.write.format("delta") \
    .mode("append") \
    .option("mergeSchema", "true") \
    .save("s3://.../delta/pedidos/")
OPTIMIZE, Z-Ordering e VACUUM

Jobs incrementais frequentes (streaming, micro-batches) geram muitos arquivos Parquet pequenos — o "small files problem", que degrada a leitura. OPTIMIZE compacta esses arquivos:

SQL — OPTIMIZE, ZORDER e VACUUM
OPTIMIZE pedidos;

-- Compacta e já ordena fisicamente pelas colunas mais filtradas
OPTIMIZE pedidos ZORDER BY (cliente_id, data_pedido);

-- Remove arquivos Parquet não referenciados por nenhuma versão
-- nos últimos N dias (default: 7 dias / 168 horas)
VACUUM pedidos RETAIN 168 HOURS;
TécnicaQuando usarLimitação
ParticionamentoColuna de baixa cardinalidade, sempre no filtro (ex: data)Muitas partições pequenas geram overhead; só otimiza 1 coluna bem
Z-OrderMúltiplas colunas de filtro, alta cardinalidade (ex: cliente_id)Precisa rodar OPTIMIZE periodicamente — não é automático
VACUUM com retenção curta é perigoso: se houver uma query de time travel ou um job longo lendo uma versão antiga, ele pode quebrar por os arquivos terem sido apagados fisicamente.

Delta Lake vs Iceberg e construção prática
Delta Lake vs Apache Iceberg

Ambos resolvem o mesmo problema (ACID em Data Lake) com abordagens de metadados diferentes. O essencial é reconhecer que são concorrentes na mesma categoria — não confundir com Parquet/Avro, que são formatos de arquivo, um nível abaixo.

AspectoDelta LakeApache Iceberg
OrigemDatabricksNetflix, depois Apache
Engine principalFortemente acoplado ao SparkAgnóstico — Spark, Flink, Trino, Snowflake
Formato de logJSON + checkpoints ParquetManifestos Avro + metadata JSON
AdoçãoForte no ecossistema DatabricksForte em ambientes multi-engine

Apache Hudi é o terceiro nome relevante nesse espaço, com foco historicamente maior em upserts de baixa latência (streaming).

Lakehouse na prática: essas três tecnologias (Delta, Iceberg, Hudi) são as implementações que tornam viável a promessa do Lakehouse — custo de Data Lake com confiabilidade de Data Warehouse.

Time Travel + OPTIMIZE
Comportamento esperado
  • Time travel: a versão 0 deve ter menos linhas que a versão atual (antes do MERGE do Exercício 2), confirmando que o dado antigo continua acessível mesmo após alterações posteriores.
  • OPTIMIZE: o número de arquivos Parquet deve diminuir sensivelmente após a compactação — vários arquivos pequenos viram poucos arquivos maiores, sem alterar o conteúdo lógico da tabela.
  • ZORDER: não reduz necessariamente o número de arquivos, mas reorganiza os dados internamente para acelerar filtros pelas colunas escolhidas.

O que você precisa ter gravado
ACID + _delta_log

Camada de transaction log sobre arquivos Parquet. Garante atomicidade, isolamento e histórico de versões.

MERGE INTO

Upsert atômico — insere e atualiza em uma única transação, sem reescrever a tabela inteira.

Time Travel

versionAsOf / timestampAsOf consultam estados anteriores da tabela. Depende dos arquivos antigos não terem sido removidos por VACUUM.

OPTIMIZE / Z-Order

Compacta arquivos pequenos e reorganiza dados para acelerar filtros em múltiplas colunas de alta cardinalidade.
